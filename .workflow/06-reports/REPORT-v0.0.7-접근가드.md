# 최종 보고서 (Final Report)

## 요약
- 버전: v0.0.7
- 작업명: 접근가드
- 최종 상태: 완료

## 작업 개요
- 커뮤니티 전용 웹앱의 모든 페이지 진입 시, 요청 헤더 기반으로 접근 가능 여부를 판정하고 비허용 사용자를 `block` 페이지로 보내는 구조를 정리했다.
- 초기에는 프론트 라우트 미들웨어와 클라이언트 상태 보관 방식으로 접근했지만, 마스터 설명에 맞춰 "요청 헤더" 기준의 서버 미들웨어 구조로 재설계했다.

## 결과 요약
### 주요 수정 파일
- `src/server/utils/access.ts`: `X-Discord-ID`, 개발 환경 여부, 유저 존재 여부를 통합 판정하는 공통 접근 체크 유틸 추가
- `src/server/middleware/access.ts`: 페이지 요청을 전역으로 검사하고 실패 시 `/block`으로 리다이렉트하는 Nitro 서버 미들웨어 추가
- `src/server/api/users/check.get.ts`: 공통 접근 체크 유틸 재사용으로 정리
- `src/app/pages/block.vue`: 비허용 접근자용 차단 페이지 추가
- `src/app/components/layout/common-layout/CommonHeader.vue`: 디스코드 ID 표시 제거
- `src/app/layouts/main.vue`: 디스코드 ID 표시 제거
- `src/app/utils/api-client.ts`: 클라이언트 측 `discordId` 헤더 주입 제거
- `src/app/stores/user.store.ts`: 접근용 `discordId` 상태 제거

### 주요 기능 변경
- 모든 페이지 직접 진입은 서버 미들웨어가 먼저 검사한다.
- `/api`와 정적 자산 경로는 예외 처리하여 앱 로딩이 깨지지 않도록 했다.
- `users/check`와 실제 페이지 차단 로직이 한 유틸을 공유하도록 정리해 기준이 갈라지지 않게 만들었다.
- 디스코드 ID를 프론트에서 보여 주거나 보관하지 않도록 수정했다.

## 특이 사항
- 이 구조는 상위 프록시 또는 진입 경로가 페이지 요청에 `X-Discord-ID`를 실어 준다는 운영 전제가 필요하다.
- 브라우저 단독 환경만으로는 완전한 인증이 아니므로, 향후 운영 확장 시 로그인 또는 서명된 접근 토큰 구조가 필요하다.
- 작업 중 `pnpm.cmd exec nuxi typecheck`는 `vue-tsc` 조회 과정에서 네트워크/권한 문제로 끝까지 수행하지 못했다.

## 후속 작업
- 실제 배포 경로에서 페이지 요청에 `X-Discord-ID`가 안정적으로 전달되는지 확인
- 필요 시 `/api` 요청도 동일한 상위 프록시 정책 아래에서 일관되게 헤더 주입되는지 점검
- 운영 환경까지 확장할 계획이 생기면 로그인 또는 서명된 게이트웨이 토큰 구조 설계
