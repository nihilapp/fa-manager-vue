# 분석 (Analysis)

## 요약
- 요청 내용: `src/app/pages/players/index.vue` 진입 시 `DataTable` 렌더 함수에서 발생하는 예외를 해결한다.
- 범위: 플레이어 목록 렌더링 경로, 상태 표시 슬롯, 플레이어 상태 맵 사용처
- 구현 가능성: 가능

## 요구사항 요약
- 플레이어 목록 페이지가 예외 없이 렌더링되어야 한다.
- 플레이어 상태 값이 비정상적이거나 누락되어도 화면 전체가 깨지지 않아야 한다.
- 기존 플레이어 도메인 스토어 정비 작업과 충돌 없이 최소 범위에서 안정성을 확보해야 한다.

## 현재 상태
### 관련 파일
- `src/app/pages/players/index.vue`: 플레이어 목록 페이지 진입점이며 `PlayerList`만 렌더링한다.
- `src/app/components/players/PlayerList.vue`: 목록 조회 컴포저블을 호출하고 스토어의 `playerList`, `playerPageData`, `isAdmin`을 `PlayerListTable`로 전달한다.
- `src/app/components/players/PlayerListTable.vue`: `DataTable`의 `status` 슬롯에서 `playerStatusMap.get(value as PlayerStatus)!`를 사용해 상태 뱃지를 그린다.
- `src/app/components/table/DataTable.vue`: 전달받은 슬롯을 각 셀 렌더링 시 호출한다. 슬롯 내부 예외는 전체 렌더 함수 예외로 전파된다.
- `src/app/stores/player.store.ts`: `playerStatusMap`을 보유하지만, 상태 조회 실패 시 사용할 공통 fallback 액션/헬퍼는 없다.
- `src/app/components/players/PlayerDetail.vue`: 상세 화면에서도 동일한 `playerStatusMap.get(... )!` 단정 접근을 사용한다.

### 현재 로직 분석
- 목록 페이지 마운트 시 `useGetPlayerList`가 `/players` API를 호출한다.
- 성공 시 응답 `list`는 `playerStore.playerList`에 저장되고 `pageData`는 `playerPageData`에 저장된다.
- 테이블 렌더링 시 `DataTable`이 각 행의 `status` 값을 슬롯으로 넘긴다.
- `PlayerListTable`는 상태 값이 맵에 반드시 존재한다고 가정하고 non-null assertion(`!`)으로 `name`, `color`를 참조한다.
- 런타임에서 `value`가 `undefined`, `null`, 혹은 예상 외 문자열이면 `Map.get()`이 `undefined`를 반환하고 `.name` 접근에서 렌더 예외가 발생한다.

## 영향도 평가
### 직접 영향
- `src/app/components/players/PlayerListTable.vue`의 상태 뱃지 렌더링
- `src/app/stores/player.store.ts`의 상태 표시 유틸리티 책임

### 간접 영향
- `src/app/components/players/PlayerDetail.vue`도 동일 패턴을 사용하므로 재발 가능성이 있다.
- 플레이어 상태 값이 서버/DB 정합성 문제로 일시적으로 비정상이어도 UI가 복구 불가능한 상태가 되지 않도록 방어 계층이 필요하다.
- 스토어 정비 작업의 Task 4(기존 화면 참조 호환성 확인)와 결과가 맞물린다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 근본 원인이 서버 데이터 불일치인지, 프런트 타입-런타임 간극인지 현재 로그만으로는 단정할 수 없다.
- 그러나 UI 방어 로직은 데이터 원인과 무관하게 즉시 적용 가능하다.

### 가정 / 불확실성
- 현재 예외 스택만 보면 `PlayerListTable` 상태 슬롯이 1차 원인으로 보이지만, 실제 입력값이 무엇이었는지는 별도 런타임 로그가 없다.
- `PlayerDetail.vue`는 현재 제보 대상은 아니지만 같은 패턴이므로 함께 정리하는 편이 안전하다.
