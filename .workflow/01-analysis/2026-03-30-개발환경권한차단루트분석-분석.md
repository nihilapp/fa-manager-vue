## 요약

- 요청 내용: 개발 환경에서 계정 유무나 관리자 권한과 무관하게 테스트를 진행하려면 `src/server/api`의 모든 라우트에서 개별 처리가 필요한지, 아니면 공통 차단 루트 한 곳을 조정하면 되는지 파악
- 범위: `src/server/utils/auth.ts`, `src/server/utils/access.ts`, `src/server/middleware/access.ts`, `src/server/api` 전역의 `authHelper()` 소비 패턴, 프런트 `/block` 리다이렉트 흐름
- 구현 가능성: 가능

## 요구사항 요약

- 개발 환경에서 권한 때문에 테스트가 막히는 실제 차단 루트를 먼저 명확히 알아야 한다.
- 차단이 페이지 전역 미들웨어인지, API 공통 헬퍼인지, 혹은 각 라우트의 개별 권한 체크인지 구분해야 한다.
- 이후 수정 단계에서 "한 곳 수정으로 해결 가능한 범위"와 "라우트별로 따로 손봐야 하는 범위"를 분리해 바로 설계할 수 있어야 한다.

## 현재 상태

### 관련 파일

- `src/server/utils/auth.ts`: 디스코드 식별자 우선순위(헤더 -> 쿠키 -> 개발용 `ADMIN_DISCORD_ID`)를 해석하고, `user`, `error`, `isDevelopmentBypass`, `hasPermission`, `isAdmin`을 포함한 인증 컨텍스트를 만든다.
- `src/server/utils/access.ts`: 페이지 접근 허용 여부를 판정하는 공통 유틸이다. 현재는 `authHelper()`가 돌려준 `isDevelopmentBypass`를 먼저 보면 페이지 접근은 우회된다.
- `src/server/middleware/access.ts`: 비 API 페이지 요청마다 `checkCommunityAccess()`를 호출하고 실패 시 `/block`으로 리다이렉트한다.
- `src/server/api/players/me.get.ts`: 앱 부트스트랩 시 가장 먼저 호출되는 사용자 정보 API다. 현재는 `isDevelopmentBypass`일 때만 예외적으로 `error`를 무시한다.
- `src/app.vue`: `/block`이 아니면 앱 시작 시 항상 `/players/me`를 호출한다.
- `src/app/composables/players/useGetMyInfo.ts`: `/players/me` 실패 시 클라이언트에서 `/block`으로 이동시킨다.
- `src/server/api/log-histories/*.ts`: `isAdmin`을 직접 검사하는 대표 라우트군이다. 공통 헬퍼만 바꿔도 여전히 관리자 전용 제한이 남는다.
- `src/server/api/sessions/index.post.ts`, `src/server/api/currency-transactions/*.ts`: `hasPermission()`을 사용하는 대표 라우트군이다. 개발 우회가 `hasPermission()`에 반영되면 추가 수정 없이 통과 가능하다.
- `src/server/api/campaigns/mine.get.ts` 등 다수 라우트: `authHelper()` 호출 뒤 `if (error) return error;` 다음에 `user!.id`를 사용한다. 개발 우회만으로는 `user === null` 상태를 처리하지 못한다.

### 현재 로직 분석

- `src/server/utils/auth.ts:22`~`45`에서 식별자는 헤더, 쿠키, 개발환경 `ADMIN_DISCORD_ID` 순으로 선택된다. 즉 잘못된 쿠키가 있으면 개발 우회보다 쿠키가 우선된다.
- `src/server/utils/auth.ts:126`~`154`에서 개발환경이면서 식별자가 없거나, 개발용 식별자로 조회한 사용자가 없어도 `isDevelopmentBypass`를 `true`로 반환할 수 있다. 하지만 동시에 `error`도 함께 반환한다.
- `src/server/utils/access.ts:10`~`15`는 `isDevelopmentBypass`를 `error`보다 먼저 확인하므로, 페이지 요청 자체는 공통 지점 한 곳에서 우회된다.
- `src/server/middleware/access.ts:27`~`37`는 `/api`가 아닌 요청에만 접근 가드를 적용한다. 따라서 `src/server/api` 전체를 여기서 한 번에 열 수는 없다.
- `src/server/api/players/me.get.ts:2`~`9`는 예외적으로 `isDevelopmentBypass`를 먼저 보고 성공 응답을 준다. 이 한 라우트 덕분에 앱 초기 진입의 `/block` 일부는 막아낸다.
- `src/app.vue:7`~`9`와 `src/app/composables/players/useGetMyInfo.ts:20`~`29` 때문에 `/players/me`만 실패해도 사용자는 곧바로 `/block`으로 이동한다. 체감상 전역 차단처럼 보이는 1차 원인이다.
- 반면 대부분의 API는 `if (error) return error;` 패턴을 유지한다. 따라서 개발 우회 플래그가 있어도 `error`가 남아 있는 한 각 라우트는 계속 막힌다.
- `src/server/api/log-histories/index.get.ts:5`~`10`처럼 `isAdmin`을 직접 확인하는 라우트는 공통 페이지 가드와 무관한 2차 차단 지점이다.
- `src/server/api/sessions/index.post.ts:4`~`22`처럼 `hasPermission()`을 쓰는 라우트는 `authHelper()`의 개발 우회가 이미 함수 내부에 반영되어 있어, `error`만 제거되면 대체로 공통 처리로 해결 가능하다.
- 그러나 `src/server/api/campaigns/mine.get.ts:5`~`27`처럼 `user!.id`를 직접 참조하는 라우트는 `error`만 우회해도 `user`가 없으면 런타임 오류 또는 잘못된 쿼리가 발생한다. 이 부류는 라우트별 또는 공통 `requireUser` 계약 정리가 필요하다.

## 영향도 평가

### 직접 영향

- 페이지 진입 차단 자체는 `src/server/utils/access.ts`와 `src/server/middleware/access.ts`의 공통 루트로 설명할 수 있다.
- API 차단은 단일 루트만으로 끝나지 않는다. `authHelper()` 결과 소비 방식에 따라
  - `error`만 보고 바로 반환하는 라우트
  - `isAdmin`을 개별 확인하는 라우트
  - `hasPermission()`을 사용하는 라우트
  - `user!.id`를 직접 전제하는 라우트
  로 나뉜다.
- 즉 "개발 환경에서는 전 라우트를 전부 손봐야 하나"에 대한 답은 "아니지만, 모든 라우트가 한 지점 수정만으로 자동 해결되지는 않는다"이다.

### 간접 영향

- `discord_id` 쿠키가 남아 있으면 개발환경에서도 `ADMIN_DISCORD_ID` 우회가 선택되지 않아 분석과 실제 체감이 달라질 수 있다.
- `players/me`가 성공하더라도 이후 CUD API가 `user!.id`에 의존하면 개발용 무계정 상태에서 다른 런타임 오류가 이어질 수 있다.
- 개발 환경 전체 우회를 무제한 허용하면 로그 이력, 세션 생성, 플레이어 수정 등 작성자 ID가 필요한 API에서 데이터 무결성 문제가 생길 수 있다.

## 실행 판단

### 구현 가능성

- 가능

### 예상 기술 난제

- `authHelper()`가 "에러가 있지만 개발 우회 가능"이라는 모순된 상태를 반환하고 있어 소비자별 해석이 갈린다. 반환 계약을 먼저 정리해야 한다.
- `user`가 없는 개발 우회를 허용할지, 아니면 개발 전용 가상 사용자 컨텍스트를 만들지 결정해야 한다.
- `isAdmin` 직접 검사 라우트는 공통 우회 정책을 별도로 전달하지 않으면 계속 막힌다.

### 가정 / 불확실성

- 현재 세션에는 context7 MCP 가 노출되지 않아 로컬 코드만 기준으로 분석했다.
- 환경 변수 실제 값은 확인하지 않았으므로, `ADMIN_DISCORD_ID` 존재 여부와 쿠키 상태는 코드 흐름 기준으로만 판단했다.
- `src/server/api` 전체 라우트 수동 전수 테스트는 수행하지 않았고, 패턴 검색과 대표 라우트 샘플을 기준으로 분류했다.
