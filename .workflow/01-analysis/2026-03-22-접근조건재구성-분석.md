# 분석 (Analysis)

## 요약
- 요청 내용: `src/server/utils/auth.ts`, `src/server/utils/access.ts`, `src/server/api/users/check.get.ts`에서 접근 조건을 4단계 우선순위로 재구성한다. 1차는 개발 환경 여부, 2차는 디스코드 ID 헤더 존재 여부, 3차는 관리자 여부, 4차는 본인 여부이며, 어느 조건에도 부합하지 않으면 `/block` 흐름으로 보낸다.
- 범위: 공통 인증 컨텍스트(`authHelper`), 페이지/엔드포인트 접근 체크(`checkCommunityAccess`), 접근 상태 확인 API(`users/check`)
- 구현 가능성: 가능

## 요구사항 요약
- 운영 환경이 아닌 개발 환경이면 후속 인증/인가 조건을 평가하지 않고 통과해야 한다.
- 운영 환경에서는 요청 헤더에 디스코드 ID가 있어야 한다.
- 운영 환경에서 디스코드 ID가 있을 때 관리자(`ROLE_ADMIN`, `ROLE_SUPER_ADMIN`)면 통과해야 한다.
- 운영 환경에서 디스코드 ID가 있을 때 요청 대상이 본인이면 통과해야 한다.
- 위 조건 어느 것에도 맞지 않으면 접근 차단 응답 또는 `/block` 리다이렉트로 이어져야 한다.
- `auth.ts`, `access.ts`, `check.get.ts`는 같은 기준을 공유해야 한다.

## 현재 상태
### 관련 파일
- `src/server/utils/auth.ts`: 공통 인증 컨텍스트를 만든다. 현재는 개발 환경일 때 디스코드 ID가 없어도 DB에서 관리자 또는 첫 사용자를 찾아 `isDevelopmentBypass=true`로 무조건 통과시키며, 운영 환경에서는 디스코드 ID로 유저를 조회해 `hasPermission(resourceUserId)`와 `checkAdmin()`을 제공한다.
- `src/server/utils/access.ts`: 페이지 접근 가능 여부를 판단한다. 현재는 `authHelper(event)`의 `error` 유무만 보고 접근 허용 또는 차단을 반환한다.
- `src/server/api/users/check.get.ts`: 접근 체크 API다. 현재는 `checkCommunityAccess(event)` 결과를 그대로 반환하는 얇은 래퍼다.
- `src/server/utils/request-header.ts`: `x-discord-id` 헤더를 읽는 단일 유틸이다. 운영 환경 2차 조건의 기준점이다.
- `src/server/middleware/access.ts`: 페이지 라우트 진입 시 `checkCommunityAccess(event)`가 실패하면 `/block`으로 리다이렉트한다.

### 현재 로직 분석
- `authHelper`는 이미 개발 환경 우회, 관리자 판정, 본인 판정(`hasPermission`)을 모두 가지고 있다.
- 다만 현재 개발 환경 우회는 “무조건 통과 + 대체 사용자 주입” 방식이라, 요청하신 1차 조건은 충족하지만 어떤 사용자로 간주할지 명시적이지 않다.
- 운영 환경에서는 디스코드 ID 헤더가 없으면 즉시 `REQUIRE_DISCORD_ID` 에러를 반환하므로 2차 조건 일부는 이미 구현되어 있다.
- 관리자 여부와 본인 여부는 `hasPermission(resourceUserId)` 호출 시에만 평가된다. 즉 현재 `checkCommunityAccess`와 `users/check`는 헤더만 있으면 일반 사용자도 통과할 수 있어, 요청하신 3차/4차 조건을 실제로는 적용하지 않는다.
- `users/check.get.ts`는 별도 판단이 없고 `checkCommunityAccess` 위임만 하므로, 접근 정책 변화는 `access.ts`와 `auth.ts`에 집중된다.

## 영향도 평가
### 직접 영향
- `src/server/utils/auth.ts`
  - 개발 환경 우회 방식과 운영 환경 인증 실패 응답을 유지 또는 조정해야 한다.
  - 접근 체크에서 재사용 가능한 관리자/본인 판정 인터페이스를 더 명확히 노출해야 할 수 있다.
- `src/server/utils/access.ts`
  - 현재는 `error`만 보고 통과시키므로, 요청하신 3차/4차 조건을 반영할 실제 판정 로직이 필요하다.
  - “본인” 판정을 위해 요청 대상 사용자 식별자를 어디서 받을지 계약 정의가 필요하다.
- `src/server/api/users/check.get.ts`
  - 공통 접근 판단 결과를 그대로 반환하되, 필요 시 대상 사용자 정보를 전달하도록 구조가 바뀔 수 있다.

### 간접 영향
- `src/server/middleware/access.ts`는 `checkCommunityAccess` 반환 결과 형식이 바뀌면 연쇄 수정이 필요하다.
- `authHelper`를 사용하는 다수의 CUD API는 기존 `hasPermission(resourceUserId)` 계약에 의존하므로, 공통 인증 컨텍스트 구조 변경 시 회귀 영향이 생길 수 있다.
- “본인 여부”를 페이지 접근에도 적용하려면 현재 URL만으로는 대상 사용자 ID를 알 수 없는 페이지가 존재할 수 있어, 프런트 또는 라우팅 정책과의 추가 정합성 검토가 필요하다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 요청 문구의 2차 조건을 문자 그대로 적용하면 “헤더에 디스코드 ID가 있으면 즉시 패스”가 되어 3차 관리자, 4차 본인 조건은 도달 불가능한 죽은 조건이 된다.
- “본인 여부”를 `checkCommunityAccess(event)`만으로 판단하려면 대상 사용자 ID가 필요하지만, 현재 함수 시그니처에는 그런 입력이 없다.
- 개발 환경에서 통과시킬 때 현재처럼 대체 사용자를 주입할지, 사용자 없이 통과만 허용할지 설계 기준이 필요하다.

### 가정 / 불확실성
- 실질 의도는 다음 둘 중 하나일 가능성이 높다.
  - 해석 A: 개발 환경이면 통과, 운영 환경이면 디스코드 ID 헤더만 있으면 통과
  - 해석 B: 개발 환경이면 통과, 운영 환경이면 디스코드 ID 헤더가 있어야 하며 그 뒤 관리자 또는 본인일 때만 통과
- 현재 요청의 “2차 조건 … 있으면 패스” 표현은 해석 A에 가깝지만, 이어지는 3차/4차 조건 존재를 고려하면 해석 B가 더 자연스럽다.
- 본 분석 단계에서는 해석 충돌을 명시하고, 설계 단계에서 어떤 계약으로 정리할지 확정해야 한다.
