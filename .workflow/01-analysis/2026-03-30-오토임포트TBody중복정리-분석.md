# 분석 (Phase 1 - Analysis)

## 요약
- 작업명: 오토임포트TBody중복정리
- 참조 초기화 문서: `.workflow/00-init/2026-03-30-오토임포트TBody중복정리-초기화.md`
- 요구사항 요약:
  - 개발 서버 경고에 출력된 `Duplicated imports "TBody"` 원인을 식별한다.
  - Nuxt 오토임포트가 중복 전역 선언을 만들지 않도록 구조를 정리한다.
  - 프로젝트 규칙상 직접 import 추가 없이 오토임포트 규칙을 유지하는 방향을 우선 검토한다.

## 요구사항 재해석
- 이번 작업의 본질은 런타임 기능 수정이 아니라 개발 환경 품질 저하 요소 제거다.
- 경고는 단순 로그 노이즈에 그치지 않고, 오토임포트 스캔 결과가 예기치 않은 이름을 전역에 노출하고 있음을 의미한다.
- 따라서 `TBody`라는 내부 제네릭 이름이 왜 외부 export처럼 간주되는지 확인하고, 공용 API 표면을 의도한 이름만 노출하도록 제한해야 한다.

## 현황 파악
### 관련 파일과 역할
- `src/app/composables/query/useDelete.ts`
  - DELETE 요청용 composable과 `UseDeleteReturn`, `UseDeleteOptions` 타입을 export한다.
  - 최상위 export 타입의 제네릭 파라미터 이름으로 `TBody`를 사용한다.
- `src/app/composables/query/usePatch.ts`
  - PATCH 요청용 composable과 대응 타입을 export한다.
  - 동일하게 제네릭 파라미터 이름으로 `TBody`를 사용한다.
- `src/app/composables/query/usePost.ts`
  - POST 요청용 composable과 대응 타입을 export한다.
  - 동일하게 제네릭 파라미터 이름으로 `TBody`를 사용한다.
- `src/app/composables/query/usePut.ts`
  - PUT 요청용 composable과 대응 타입을 export한다.
  - 동일하게 제네릭 파라미터 이름으로 `TBody`를 사용하며, Nuxt 생성 결과상 최종 surviving import source가 된다.
- `src/app/composables/query/useApiRequest.ts`
  - `ApiRequestBody` 등 공통 요청 타입과 fetch 옵션 헬퍼를 제공한다.
  - 자체 제네릭에도 `TBody`를 사용하지만, 현재 경고는 이 파일이 아니라 CRUD composable 4개 중복에서 발생한다.
- `nuxt.config.ts`
  - `imports.dirs`에 `app/composables/**`가 포함되어 있어 위 파일들이 모두 Nuxt 오토임포트 스캔 대상이다.
- `.nuxt/imports.d.ts`
  - 생성 결과에서 `export { usePut, TBody, UsePutReturn, UsePutOptions }`가 실제 포함되어 있다.
  - `useDelete`, `usePatch`, `usePost`는 `TBody` export가 중복 경고로 탈락하고, `usePut` 쪽만 남은 상태다.
- `.nuxt/types/imports.d.ts`
  - 생성 결과에서 `const TBody: typeof import('../../src/app/composables/query/usePut').TBody`가 전역 선언되어 있다.
  - 내부 제네릭 식별자가 전역 오토임포트 심볼로 잘못 승격된 사실을 확인할 수 있다.
- `src/app/types/manual-app-imports.d.ts`
  - 현재 수동 전역 선언에는 `TBody`가 없다.
  - 즉 이번 경고는 수동 선언 충돌이 아니라 Nuxt 자동 스캔 결과에서 유발된 문제다.

### 현재 동작
- 개발 서버 시작 시 Nuxt 오토임포트 스캐너가 `useDelete.ts`, `usePatch.ts`, `usePost.ts`, `usePut.ts`를 순서대로 읽는다.
- 스캐너는 각 파일에서 `TBody`를 export 심볼로 인식하고 동일 이름 중복 경고를 출력한다.
- 마지막 파일인 `usePut.ts`의 `TBody`만 전역 오토임포트 심볼로 남고, 나머지는 무시된다.
- 애플리케이션 동작 자체는 계속되지만, 전역 공간에 의도하지 않은 타입/심볼이 노출된 상태다.

## 영향도 평가
### 직접 영향
- 개발 서버 경고가 반복 출력되어 실제 문제 탐지가 어려워진다.
- 오토임포트 공용 표면에 의도하지 않은 `TBody`가 포함되어 타입 공간이 오염된다.
- 향후 동일 이름의 실사용 타입 또는 변수 선언이 추가될 경우 충돌 가능성이 높아진다.

### 간접 영향
- composable 파일을 계속 같은 패턴으로 복제할 경우 `TQuery`, `TData` 등 다른 제네릭 이름도 유사하게 잘못 노출될 위험이 있다.
- Nuxt 버전, unimport 파서 버전 변화에 따라 경고가 에러 수준으로 승격되거나 생성 결과가 달라질 수 있다.
- 팀 규칙상 오토임포트를 신뢰해야 하므로, 잘못된 export 표면은 다른 파일의 타입 작성 방식에도 악영향을 줄 수 있다.

## 원인 분석
- 핵심 원인은 `imports.dirs`에 포함된 composable 파일의 top-level export 타입 선언을 Nuxt 오토임포트 스캐너가 처리하는 과정에서, 제네릭 파라미터 식별자 `TBody`를 독립 export로 오인하는 생성 결과가 발생한 점이다.
- 확인 근거:
  - 개발 서버 경고 메시지가 네 파일의 `TBody` 중복을 직접 지목한다.
  - `.nuxt/imports.d.ts`에 `TBody` export가 실제 생성되어 있다.
  - `.nuxt/types/imports.d.ts`에 전역 `const TBody` 선언이 생성되어 있다.
- 현재 코드 검색 결과, 네 CRUD composable 외부에서 `UseDeleteOptions`, `UsePatchOptions`, `UsePostOptions`, `UsePutOptions` 같은 타입을 직접 소비하는 사용처는 발견되지 않았다.
- 따라서 export 표면 축소 또는 타입 선언 위치 조정이 기능 회귀 위험이 낮은 방향으로 보인다.

## 실행 판단
- 2단계 설계로 진행 가능
- 권장 설계 방향:
  - Nuxt 오토임포트 대상 composable에서 내부 전용 타입 export를 최소화하거나 제거한다.
  - 필요 시 공용 타입은 별도 `app/types` 또는 query 전용 타입 파일로 이동해 의도적으로 이름을 노출한다.
  - 내부 제네릭 파라미터 이름이 스캐너에 노출되지 않도록, composable 파일의 export 표면을 함수 중심으로 단순화한다.

## 불확실성 및 확인 필요 사항
- Nuxt/unimport가 왜 제네릭 파라미터 `TBody`를 export 심볼로 잘못 인식하는지는 생성물로 증상 확인은 가능하지만 내부 파서 구현까지는 이번 세션에서 확인하지 못했다.
- 현재 사용처 검색에서는 옵션/리턴 타입 직접 사용이 없었지만, 문자열 검색으로 잡히지 않는 타입 참조나 IDE 전용 사용은 추가 검증이 필요할 수 있다.
