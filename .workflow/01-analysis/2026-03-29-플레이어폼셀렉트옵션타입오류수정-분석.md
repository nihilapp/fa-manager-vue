# 분석 (Analysis)

## 요약
- 요청 내용: `src/app/components/players/NewPlayerForm.vue`에서 `FormSelect`에 전달하는 `options` 프롭의 TS4104 readonly 배열 타입 오류 해결
- 범위: `NewPlayerForm.vue`, `FormSelect.vue`, 관련 셀렉트 옵션 타입 계약
- 구현 가능성: 가능

## 요구사항 요약
- `NewPlayerForm.vue`에서 `as const`로 선언된 상태/권한 셀렉트 옵션 배열을 `FormSelect`에 타입 오류 없이 전달할 수 있어야 한다.
- 기존 옵션 표시, 선택, vee-validate 연동 동작은 유지되어야 한다.
- 호출부에서 불필요한 복사나 강제 타입 단언 없이 공용 컴포넌트 계약이 실제 사용 패턴을 수용해야 한다.

## 현재 상태
### 관련 파일
- `PRD/PRD.md`: 플레이어 관리 기능이 포함된 제품 요구사항 기준 문서다. 이번 수정은 플레이어 등록 폼 안정성과 직접 연결된다.
- `PRD/Coding-Rules.md`: 타입 안정성과 기존 구조 보존 기준 문서다. 공용 컴포넌트 계약을 수정하되 화면 흐름은 유지해야 한다.
- `src/app/components/players/NewPlayerForm.vue`: 플레이어 등록 폼이다. 상태/권한 옵션 배열을 `as const`로 선언해 zod `z.enum`과 UI 옵션을 함께 사용한다.
- `src/app/components/common/FormSelect.vue`: 공용 셀렉트 UI 컴포넌트다. `options`를 순회하며 `valueKey`, `labelKey`를 이용해 렌더링하고 선택 값을 갱신한다.

### 현재 로직 분석
- `NewPlayerForm.vue`는 `playerStatusOptions`, `playerRoleOptions`를 `z.enum` 입력으로 쓰기 위해 readonly tuple로 유지하고 있다.
- 같은 파일의 `playerStatusSelectOptions`, `playerRoleSelectOptions`도 `as const`로 선언되어 각 원소와 배열 전체가 readonly가 된다.
- `FormSelect.vue`는 `options?: SelectOption[]`로 선언되어 mutable 배열만 허용하므로 readonly 배열 전달 시 TS4104가 발생한다.
- `FormSelect.vue` 내부 로직은 `find`, `v-for` 순회만 사용하고 `options`를 변경하지 않는다. 따라서 실제 런타임 계약은 readonly 배열을 받아도 문제없다.

## 영향도 평가
### 직접 영향
- `src/app/components/common/FormSelect.vue`의 `options` 프롭 타입 선언
- `src/app/components/players/NewPlayerForm.vue`의 타입 오류 해소 여부

### 간접 영향
- 다른 화면에서 `FormSelect`에 `readonly` 배열을 전달하는 경우도 추가 이점이 생긴다.
- `FormSelect`가 배열을 변경하지 않는다는 계약이 타입 수준에서 더 정확하게 표현된다.
- 옵션 기본값 팩토리 `options: () => []`가 readonly 배열 타입과 호환되는지 확인이 필요하다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `defineProps`와 `withDefaults` 조합에서 `readonly` 배열 기본값이 문제 없이 추론되는지 확인이 필요하다.
- 프로젝트 전체 타입체크 명령이 별도 스크립트로 정의되어 있지 않아 검증 명령을 직접 선택해야 한다.

### 가정 / 불확실성
- context7 MCP 선검색 지침이 있으나 현재 세션에서 사용 가능한 context7 리소스를 확인하지 못했다. 이번 분석은 로컬 코드베이스 기준이다.
- `FormSelect`의 `SelectOption`이 외부 공용 타입으로 분리되어 있지 않아 수정 범위는 현재 파일 내부에 한정한다.
