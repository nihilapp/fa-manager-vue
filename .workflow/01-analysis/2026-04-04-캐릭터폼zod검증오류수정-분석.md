# 분석 (Analysis)

## 요약
- 요청 내용: 캐릭터 편집 폼에서 발생하는 `Cannot read properties of undefined (reading 'flatMap')` 오류 원인 파악 및 수정
- 범위: 캐릭터 폼 검증 스키마, `vee-validate`와 `zod` 연동부
- 구현 가능성: 가능

## 요구사항 요약
- 폼 뒤로 가기 등 검증이 트리거되는 시점에 발생하는 런타임 예외를 제거해야 한다.
- 패키지 버전 전체를 흔들지 않고 현재 코드 레벨에서 안전하게 우회하는 것이 우선이다.

## 현재 상태
### 관련 파일
- `src/app/components/characters/CharacterEditorForm.vue`: 편집 폼에서 `useForm`과 캐릭터 스키마를 사용한다.
- `src/app/components/characters/CharacterCreateForm.vue`: 동일 스키마를 생성 폼에서도 사용한다.
- `src/app/stores/character.store.ts`: `toTypedSchema(z.object(...))` 기반 캐릭터 폼 검증 규칙을 정의한다.
- `node_modules/@vee-validate/zod/dist/vee-validate-zod.mjs`: `invalid_union` 처리 시 `issue.unionErrors.flatMap(...)`를 호출한다.
- `node_modules/zod/src/v4/core/errors.ts`: Zod v4의 `invalid_union` 이슈 구조가 `errors` 필드를 사용함을 보여준다.

### 현재 로직 분석
- 캐릭터 폼은 `campaignId`, `startLevel`, 시작 자금 필드들에 `z.union([z.string(), z.number()])` 또는 `z.union([z.number(), z.null()])`를 사용한다.
- 입력값이 기대 타입과 어긋나면 Zod가 `invalid_union` 이슈를 만든다.
- 현재 설치 조합에서는 `@vee-validate/zod`가 Zod v4 이슈 구조를 잘못 읽어 런타임 예외가 발생한다.

## 영향도 평가
### 직접 영향
- 캐릭터 생성/수정 폼의 스키마 정의
- 폼 검증 중 발생하는 예외 처리 흐름

### 간접 영향
- 동일 스토어 스키마를 공유하는 생성 폼도 같이 안정화된다.
- 패키지 버전 정합성 문제는 남아 있으므로 다른 화면에 `union` 기반 스키마가 추가되면 재발 가능성이 있다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 기존 입력 허용 범위는 유지하면서 `union` 없이 동일한 검증 의미를 보존해야 한다.

### 가정 / 불확실성
- 이번 오류 재현 경로는 캐릭터 폼 기준으로 확인했다.
- 다른 도메인 폼에 같은 패턴이 있으면 별도 점검이 필요하다.
