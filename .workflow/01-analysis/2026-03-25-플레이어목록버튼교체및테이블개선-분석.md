# 분석 (Analysis)

## 요약
- 요청 내용: `players/index.vue`의 기존 버튼을 공통 `Button.vue`로 대체하고, 수정 버튼은 관리자만 보이게 처리하며, 플레이어 목록 테이블을 더 에어테이블 같은 방향으로 개선
- 범위: `src/app/pages/players/index.vue`, `src/app/components/table/DataTable.vue`, 필요 시 `src/app/components/table/Paginator.vue`와 사용자 권한 참조부
- 구현 가능성: 가능

## 요구사항 요약
- 플레이어 목록 액션 셀의 기존 `<button>`을 공통 버튼 컴포넌트로 교체해야 한다.
- 두 번째 버튼은 수정 버튼으로 정의하고, `src/app/stores/user.store.ts`의 `isAdmin` 기준으로만 노출해야 한다.
- 단순 색상 변경 수준이 아니라 테이블 자체를 에어테이블처럼 더 가볍고 정돈된 카드형 데이터 그리드 느낌으로 개선하고 싶다.
- 테이블 디자인 개선은 한 페이지 전용 땜질보다 `DataTable.vue`를 손봐서 전체 재사용 품질을 올리는 방향이 적절하다.

## 현재 상태
### 관련 파일
- `src/app/pages/players/index.vue`: 플레이어 목록 페이지다. `DataTable` 슬롯을 사용하고 있으며 액션 영역에 원시 `<button>` 하나와 빈 버튼 하나가 있다.
- `src/app/components/common/Button.vue`: 직전 작업으로 공통 버튼 기능이 보강된 상태다. `fill`, `ghost`, `outline` 모드와 커스텀 클래스 병합이 가능하다.
- `src/app/stores/user.store.ts`: `myInfo.role` 및 개발환경 여부를 기반으로 `isAdmin` computed를 제공한다.
- `src/app/components/table/DataTable.vue`: 현재는 회색 배경, dashed border 중심의 다소 무거운 박스형 테이블이다. 헤더/행/셀 스타일이 전반적으로 고정돼 있어 에어테이블 같은 밀도와 여백을 표현하기 어렵다.
- `src/app/pages/players/[id].vue`: 플레이어 상세 페이지가 이미 존재하므로 상세보기 버튼은 최소한 상세 라우팅과 연결할 수 있다.

### 현재 로직 분석
- `players/index.vue`는 `storeToRefs(userStore)`에서 `userList`, `userPageData`만 꺼내고 있어 관리자 여부 분기를 아직 사용하지 않는다.
- 액션 셀에는 상세보기 버튼만 실제 UI가 있고 두 번째 버튼은 비어 있다.
- `DataTable.vue`는 `role="table"` 기반의 div 테이블 구조를 사용하며, 헤더와 행 스타일이 컴포넌트 내부에 강하게 박혀 있다.
- 행 배경은 `odd/even` 줄무늬, 헤더는 짙은 배경, 전체 컨테이너는 굵은 테두리라 에어테이블 특유의 밝고 얇은 그리드, 넓은 여백, hover 중심 강조와는 방향이 다르다.

## 영향도 평가
### 직접 영향
- `src/app/pages/players/index.vue`의 액션 버튼 렌더링, 관리자 분기, 상세/수정 UI 구조 변경
- `src/app/components/table/DataTable.vue`의 레이아웃, 헤더, 행, 셀 스타일 변경

### 간접 영향
- `DataTable.vue`를 공통 컴포넌트로 사용 중인 다른 페이지도 동일한 테이블 스타일 변화를 받게 된다.
- `isAdmin`은 개발환경에서 항상 true가 될 수 있으므로 로컬 개발 시 수정 버튼이 항상 보일 수 있다.
- 수정 버튼의 실제 라우트가 아직 별도로 없으므로, 이번 작업에서 버튼 노출만 할지 상세 페이지 재사용 링크로 둘지 설계에서 결정해야 한다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 에어테이블 느낌은 단순 클래스 교체가 아니라 헤더 대비, 행 hover, 셀 여백, 액션 버튼 밀도, 페이지네이터 톤까지 같이 맞춰야 어색하지 않다.
- 공통 `DataTable.vue`를 바꾸면 다른 화면에도 영향이 갈 수 있으므로 지나치게 특화된 스타일은 피해야 한다.
- 수정 버튼의 실제 목적지 경로가 명확하지 않아, 라우팅 전략을 설계에서 먼저 고정해야 한다.

### 가정 / 불확실성
- `context7 MCP`는 지침상 요구되지만 현재 세션에서 제공되지 않아 로컬 코드 기준으로 분석을 수행했다.
- `players/[id].vue`는 존재하지만 수정 전용 페이지는 확인되지 않았다. 따라서 수정 버튼의 최종 동작은 설계에서 명시해야 한다.
- 에어테이블 스타일은 기능 추가보다 시각 언어 조정의 비중이 크므로, 이번 작업에서는 구조를 유지한 채 색상/간격/상호작용을 개선하는 선이 현실적이다.
