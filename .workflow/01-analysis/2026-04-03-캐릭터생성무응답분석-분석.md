# 분석 (Analysis)

## 요약
- 요청 내용: 캐릭터 생성 버튼을 눌렀을 때 아무 반응이 없는 현상의 원인을 확인한다
- 범위: 캐릭터 생성 페이지, 생성 폼 submit 흐름, 생성 API, 에러 처리 경로
- 구현 가능성: 가능

## 요구사항 요약
- 생성 버튼 클릭 시 실제로 어떤 요청이 실행되는지 확인해야 한다
- 요청이 실패한다면 실패 원인이 프런트 입력값 문제인지 서버 검증 문제인지 분리해야 한다
- 사용자는 현재 "신호가 가는지조차 모르겠다"고 느끼므로, 요청 실패 시 사용자 피드백 부재 여부도 함께 확인해야 한다

## 현재 상태
### 관련 파일
- `src/app/pages/characters/add-character.vue`: 캐릭터 생성 페이지 래퍼로, 실제 렌더링은 `CharacterCreateForm`에 위임한다
- `src/app/components/characters/CharacterCreateForm.vue`: 생성 폼과 submit 로직을 담당한다. 현재 `useCreateNewCharacter().execute(...)`를 호출한 뒤 클래스 생성 요청과 상세 페이지 이동을 이어서 수행한다
- `src/app/composables/characters/useCreateNewCharacter.ts`: `/characters` POST 요청을 보내는 생성 뮤테이션 컴포저블이다. 에러 콜백은 선택 사항이며 현재 생성 폼에서는 전달하지 않는다
- `src/server/api/characters/index.post.ts`: 캐릭터 생성 API다. `body.name`과 `body.race`가 모두 있어야 하며, 둘 중 하나라도 비어 있으면 `BAD_REQUEST`를 반환한다
- `src/app/composables/query/usePost.ts`: POST 요청 공통 래퍼다. 응답이 에러여도 네트워크 예외가 아니면 반환값으로만 전달되고, 별도 UI 알림은 없다
- `src/app/utils/api-error-handler.ts`: 에러 응답을 처리하지만, 현재 코드 기준으로 `BAD_REQUEST`는 콘솔 로그도 없고 콜백이 없으면 화면 피드백도 없다

### 현재 로직 분석
- 생성 폼 submit 핸들러는 `buildCharacterSubmitPayload(formValues)` 결과에 `race: ''`를 덮어쓴 뒤 `/characters` POST를 호출한다
- 서버 API는 `if (!body || !body.name || !body.race)` 조건으로 필수값을 검사하므로, 현재 프런트 요청은 이름이 있어도 `race: ''` 때문에 항상 `BAD_REQUEST` 대상이 된다
- 생성 폼은 `useCreateNewCharacter()`를 에러 콜백 없이 사용한다
- `usePost`와 `handleApiResponse` 구조상 400 응답은 throw 없이 반환될 수 있으며, 이 경우 폼 submit 함수는 `response?.data?.id`가 없어서 조용히 `return`된다
- 결과적으로 사용자는 버튼 클릭 후 로딩이 짧게 끝나고 화면 변화, 에러 메시지, 이동이 모두 없어서 "무응답"으로 인식하게 된다
- 세션 내 context7 MCP 서버는 확인되지 않아 외부 참조 검색은 수행하지 못했고, 로컬 코드 기준으로 분석했다

## 영향도 평가
### 직접 영향
- `src/app/components/characters/CharacterCreateForm.vue`
- `src/server/api/characters/index.post.ts`
- `src/app/composables/characters/useCreateNewCharacter.ts`

### 간접 영향
- 캐릭터 수정 화면은 다른 컴포넌트를 사용하므로 현재 증상과 직접 연결되지는 않지만, 생성/수정 폼 간 필드 계약 불일치가 누적될 가능성은 있다
- 에러 피드백 체계가 부족해 다른 폼에서도 유사하게 "무응답"처럼 보이는 문제가 재발할 수 있다

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 서버가 실제로 `race`를 여전히 필수로 요구하는지, 아니면 최근 UI 단순화 후 계약이 남아 있는지 판단해야 한다
- 수정 방향은 두 갈래다: 프런트에서 `race` 입력을 복원하거나, 서버 계약을 현재 폼 구조에 맞게 완화해야 한다
- 단순히 필수값만 맞춰도 생성은 복구되지만, 사용자 피드백 부재는 별도 보완이 필요하다

### 가정 / 불확실성
- 현재 생성 폼에는 `race` 입력 필드가 없으므로, 최근 UI 변경 과정에서 서버 계약과 프런트 payload가 어긋난 것으로 보인다
- API 실패 시 사용자 메시지를 보여줄 공통 UI가 별도로 존재하는지는 아직 확인하지 못했다
