# 분석 (Analysis)

## 요약
- 요청 내용: 개발서버 실행 후 첫 화면 진입 시 `Failed to resolve component: Icon`, `Component <Anonymous> is missing template or render function`, 화면 500 오류 원인 파악
- 범위: `GetIcon.vue`, 푸터/헤더 등 아이콘 사용 컴포넌트, `nuxt.config.ts`의 오토임포트 설정, 생성된 `.nuxt` 산출물
- 구현 가능성: 가능

## 요구사항 요약
- 개발서버 접속 직후 발생하는 500 오류의 직접 원인을 특정해야 한다.
- 단순 로그 확인이 아니라 어떤 등록 경로가 끊겼는지 코드 구조 기준으로 설명할 수 있어야 한다.
- 이후 수정 단계로 넘어갈 경우 최소 변경 지점을 바로 확정할 수 있어야 한다.

## 현재 상태
### 관련 파일
- `src/app/components/GetIcon.vue`: 공용 아이콘 래퍼 컴포넌트. 템플릿에서 `<Icon :icon="iconName" />`를 직접 렌더링한다.
- `src/app/components/layout/common-layout/CommonFooter.vue`: 첫 진입 레이아웃에서 `GetIcon`을 호출하는 지점. 초기 렌더링 중 즉시 문제를 노출한다.
- `src/app/layouts/default.vue`, `src/app/layouts/main.vue`: 공통 푸터를 포함해 오류가 전체 레이아웃 500으로 전파되는 경로다.
- `nuxt.config.ts`: `@iconify/vue`의 `Icon`을 `imports.presets`에 등록해 스크립트 오토임포트만 설정한다.
- `.nuxt/imports.d.ts`, `.nuxt/types/imports.d.ts`: `Icon`이 자동 임포트 심볼로 생성된 상태를 보여주지만, 컴포넌트 자동 등록 목록에는 포함되지 않는다.
- `.nuxt/components.d.ts`: `GetIcon` 등 프로젝트 컴포넌트는 등록되지만 `Icon`은 컴포넌트 목록에 없다.

### 현재 로직 분석
- `GetIcon.vue`는 `<script setup>`에서 `Icon`을 직접 참조하지 않고, 템플릿에서만 `<Icon />`를 사용한다.
- 현재 `nuxt.config.ts`의 `imports.presets`는 함수/심볼 자동 임포트 체계이며, Vue 템플릿에서 사용할 전역 컴포넌트 등록과는 별개다.
- 따라서 SFC 컴파일 시 `Icon`에 대한 스크립트 임포트가 주입되지 않고, 런타임은 템플릿 태그를 전역 컴포넌트로 해석하려다 실패한다.
- 그 결과 `GetIcon` 내부에서 `<Icon>`이 해석되지 않아 `<Anonymous>` 렌더링 경고가 연쇄로 발생하고, 레이아웃 렌더링 전체가 500으로 실패한다.

## 영향도 평가
### 직접 영향
- `GetIcon.vue`를 사용하는 모든 뷰에서 동일한 오류가 발생할 수 있다.
- 현재는 `CommonFooter.vue`가 레이아웃 초기 렌더링에 포함되어 있어 접속 즉시 장애가 노출된다.
- `DataTable.vue`, `CommonHeader.vue` 등 다른 `GetIcon` 사용 지점도 동일한 결함을 공유한다.

### 간접 영향
- Nuxt 오토임포트에 대한 팀 규칙 해석이 혼동될 수 있다. 특히 "수동 import 금지"와 "템플릿 컴포넌트 사용 가능"을 동일시하면 유사 장애가 반복될 수 있다.
- `manual-app-imports.d.ts`와 `.nuxt/types/imports.d.ts`가 타입 수준에서는 `Icon`을 전역 심볼처럼 보여 주므로, 런타임 등록 문제를 타입이 가려 버리는 상태다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 프로젝트 규칙상 직접 import를 피해야 하므로, 수정 시에도 수동 import 대신 Nuxt가 인식할 수 있는 컴포넌트 등록 방식 또는 스크립트 참조 방식 중 규칙에 맞는 방식을 선택해야 한다.
- 기존 `nuxt.config.ts`, `GetIcon.vue`가 이미 수정 중인 상태일 수 있어 사용자 변경과 충돌 없이 최소 수정으로 정리해야 한다.

### 가정 / 불확실성
- 현재 분석은 로컬 생성 산출물과 소스 코드 기준이며, 별도 브라우저 런타임 재현 절차 없이도 경고 메시지와 코드 구조가 일치한다는 점을 근거로 한다.
- `context7 MCP` 사용 지침은 있으나 현재 세션 도구 목록에 해당 도구가 제공되지 않아 로컬 코드와 생성 산출물 분석으로 대체했다.
