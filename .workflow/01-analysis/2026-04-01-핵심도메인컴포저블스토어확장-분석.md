# 분석 (Analysis)

## 요약
- 요청 내용: `src/server/api/campaigns`, `src/server/api/characters`, `src/server/api/sessions` 전부에 맞춰 프런트 컴포저블을 만들고, 상태 관리는 캠페인/캐릭터/세션/화폐 현황 4개 스토어로 정리할 수 있도록 분석한다.
- 범위: 세 도메인 서버 API, 화폐 현황 도메인 기준, `src/app/stores`, `src/app/composables`, `src/app/utils/query-keys.ts`
- 구현 가능성: 가능

## 요구사항 요약
- players 도메인에서 사용한 패턴처럼 세 도메인도 API 단위로 대응하는 프런트 컴포저블을 만들어야 한다.
- 각 컴포저블은 조회 성공 시 도메인 전담 스토어 축에 데이터를 저장하고, 실패 시 해당 축을 초기화해야 한다.
- 기본 CRUD뿐 아니라 서버 API에 이미 존재하는 하위 리소스 경로도 프런트에서 호출 가능한 컴포저블 표면으로 노출해야 한다.
- 하위 리소스는 관련 도메인 스토어가 직접 관리해야 한다. 캠페인 관련 상태는 전부 캠페인 스토어, 세션 관련 상태는 전부 세션 스토어에 둔다.
- 최종 스토어 기준은 `campaign`, `character`, `session`, `currency` 4개다.
- 구현은 한 번에 몰아서 변경하더라도 설계와 태스크는 도메인별, API별로 분해 가능한 수준이어야 한다.

## 현재 상태
### 관련 파일
- `src/app/composables/players/useGetPlayerCampaignList.ts`: 플레이어별 캠페인 목록 응답을 스토어에 저장하는 기준 패턴이다.
- `src/app/composables/players/useGetPlayerSessionList.ts`: 플레이어별 세션 목록 응답을 세션 스토어 파생 축에 저장하는 기준 패턴이다.
- `src/app/composables/players/useCreateNewPlayer.ts`: 생성 후 상세 저장과 목록 재조회, 라우팅 후처리까지 포함한 변경형 컴포저블 패턴이다.
- `src/app/stores/campaign.store.ts`: 기본 캠페인 목록/페이지/상세와 플레이어별 캠페인 파생 축만 가진 최소 스토어다.
- `src/app/stores/session.store.ts`: 기본 세션 목록/페이지/상세와 플레이어별 세션 파생 축만 가진 최소 스토어다.
- `src/server/api/currency-transactions/*`: 화폐 현황 목록/상세/생성/수정/삭제 API가 구현돼 있으며 전담 스토어 후보 범위를 제공한다.
- `src/app/stores/player.store.ts`: 기본 축, 내 정보 축, 액션 분리 방식을 확인할 수 있는 기준 스토어다.
- `src/app/utils/query-keys.ts`: `campaigns`, `characters`, `sessions`의 기본 목록/상세 키와 `sessions.logs*`, `characters.classes`, `players.campaigns`, `players.sessions` 키가 정의돼 있다.
- `src/server/api/campaigns/*`: 목록, 내 목록, 상세, 생성, 수정, 삭제, 상태 변경, 멤버 추가/삭제, 캐릭터 추가/삭제 API가 구현돼 있다.
- `src/server/api/characters/*`: 목록, 내 목록, 상세, 생성, 수정, 삭제, 상태 변경, 클래스 추가/수정/삭제 API가 구현돼 있다.
- `src/server/api/sessions/*`: 목록, 내 목록, 상세, 생성, 수정, 삭제, 로그 CRUD, 참여자 추가/삭제 API가 구현돼 있다.

### 현재 로직 분석
- 현재 앱 레이어에는 `campaigns`, `characters`, `sessions` 전용 컴포저블 폴더가 없다.
- `campaign.store.ts`는 플레이어별 캠페인 목록을 위한 축만 있고 `mine` 축이나 생성/수정 후 반영 규칙은 아직 없다.
- 캠페인 멤버, 캠페인 캐릭터 연결 상태를 캠페인 스토어 안에서 어떻게 보관할지 아직 정해지지 않았다.
- `session.store.ts`는 플레이어별 세션 목록 축만 있고 `mine` 축, 로그 축, 참여자 변경 후 동기화 규칙은 아직 없다.
- characters 도메인은 전용 스토어가 없어 목록/상세/내 목록/클래스 상태를 저장할 위치 자체가 없다.
- currency 도메인은 현재 전용 스토어와 전용 컴포저블이 없어 화폐 현황을 어디에 저장할지 먼저 고정해야 한다.
- `query-keys.ts`는 기본 조회 키는 갖췄지만 캠페인 멤버/캠페인 캐릭터/세션 참여자 변경 후 어떤 키를 무효화 또는 재조회할지 정책이 아직 문서화되지 않았다.
- players 기준 컴포저블은 `useGet`, `usePost`, `usePut`, `useDelete`를 감싸며, 성공 시 스토어 액션 실행, 실패 시 대응 clear 액션 실행, 필요 시 관련 목록을 `refetch`하는 구조를 사용한다.

## 도메인별 분석
### Campaigns
- 서버 API 범위:
  - `GET /campaigns`
  - `GET /campaigns/mine`
  - `GET /campaigns/:id`
  - `POST /campaigns`
  - `PUT /campaigns/:id`
  - `DELETE /campaigns/:id`
  - `PUT /campaigns/:id/status`
  - `POST /campaigns/:id/members/:userId`
  - `DELETE /campaigns/:id/members/:userId`
  - `POST /campaigns/:campaignId/characters/:characterId`
  - `DELETE /campaigns/:campaignId/characters/:characterId`
- 프런트 간극:
  - 기본 목록/내 목록/상세/생성/수정/삭제/상태 변경을 호출하는 전용 컴포저블이 없다.
  - 멤버 및 캐릭터 연결 API에 대응하는 액션형 컴포저블도 없다.
  - 스토어에는 `myCampaignList`, `myCampaignPageData`가 없어 `/campaigns/mine` 결과를 독립 저장할 수 없다.
  - 캠페인 멤버 및 캠페인 캐릭터 관련 상태를 캠페인 스토어 안에서 도메인 소유로 유지할 축이 필요하다.

### Characters
- 서버 API 범위:
  - `GET /characters`
  - `GET /characters/mine`
  - `GET /characters/:id`
  - `POST /characters`
  - `PUT /characters/:id`
  - `DELETE /characters/:id`
  - `PUT /characters/:id/status`
  - `POST /characters/:id/classes`
  - `PUT /characters/:id/classes/:className`
  - `DELETE /characters/:id/classes/:className`
- 프런트 간극:
  - 전용 스토어와 전용 컴포저블이 모두 없다.
  - `queryKeys.characters.classes`는 정의돼 있지만 클래스 목록/상태를 저장하는 축이 없다.
  - 플레이어별 캐릭터 API는 현재 직접 존재하지 않으므로 기본 목록과 내 목록 중심으로 먼저 맞추는 편이 안전하다.

### Sessions
- 서버 API 범위:
  - `GET /sessions`
  - `GET /sessions/mine`
  - `GET /sessions/:id`
  - `POST /sessions`
  - `PUT /sessions/:id`
  - `DELETE /sessions/:id`
  - `GET /sessions/:id/logs`
  - `POST /sessions/:id/logs`
  - `GET /sessions/:id/logs/:logId`
  - `PUT /sessions/:id/logs/:logId`
  - `DELETE /sessions/:id/logs/:logId`
  - `POST /sessions/:id/players`
  - `DELETE /sessions/:id/players/:userId`
- 프런트 간극:
  - 플레이어별 세션 목록 외에는 세션 도메인 전용 컴포저블이 없다.
  - `/sessions/mine` 결과를 저장할 `mySessionList`, `mySessionPageData`가 없다.
  - 세션 로그 조회/상세/변경 결과를 저장할 `sessionLogList`, `sessionLogPageData`, `sessionLogInfo` 축이 없다.
  - 참여자 추가/삭제 이후 어떤 축을 재동기화할지 기준이 없다.

### Currency
- 서버 API 범위:
  - `GET /currency-transactions`
  - `GET /currency-transactions/:id`
  - `POST /currency-transactions`
  - `PUT /currency-transactions/:id`
  - `DELETE /currency-transactions/:id`
- 프런트 간극:
  - 화폐 현황 전용 스토어와 전용 컴포저블이 없다.
  - 캐릭터/플레이어별 파생 목록을 두더라도 상태 소유권은 화폐 현황 스토어가 가져야 한다.
  - 현재 범위의 직접 구현 API는 아니더라도, 네 번째 스토어 기준선으로 함께 설계해야 이후 도메인 분리가 흔들리지 않는다.

- 추후 페이지/컴포넌트가 어떤 스토어 축을 읽을지에 대한 네이밍 기준

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 세 도메인을 한 번에 다루면 스토어 축이 빠르게 늘어나므로 `기본 축`, `mine 축`, `부모 기준 파생 축`, `하위 리소스 축`을 명확히 분리해야 한다.
- 도메인 경계를 유지하면서도 최종 스토어 수는 4개로 제한해야 하므로, 하위 리소스와 파생 목록을 별도 스토어가 아니라 도메인 내부 축으로 설계해야 한다.
- 하위 리소스 변경 API는 응답 본문만으로 전체 상태를 갱신하기 어렵기 때문에 관련 상세/목록을 재조회할지, 국소 업데이트할지 기준이 필요하다.
- 캐릭터 클래스와 세션 로그는 조회 DTO와 상세 DTO 사용 방식이 다를 수 있어 저장 타입 경계가 필요하다.

### 가정 / 불확실성
- 현재 직접 API 구현 범위는 `campaigns`, `characters`, `sessions`를 우선으로 하되, 스토어 기준은 `currency`까지 포함한 4개로 확정한다.
- 프런트 호출부 연결보다 먼저 컴포저블/스토어 기반을 정비하는 것이 우선이다.
- 일부 하위 리소스 API는 즉시 화면 연결 대상이 없을 수 있으므로, 설계 단계에서 우선 구현 순서를 조정할 여지가 있다.
