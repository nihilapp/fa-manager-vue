# 분석 (Analysis)

## 요약
- 요청 내용: 운영 환경에서는 `X-Discord-ID`가 있어야 접근 가능하고 일반 사용자는 본인 리소스만 수정 가능해야 하며, `ROLE_ADMIN`/`ROLE_SUPER_ADMIN`은 우회 가능해야 한다. 개발 환경에서는 위 제한을 완화해야 한다.
- 범위: `authHelper`, 페이지 접근 가드, `users/check`, CUD API 전반의 인증/인가 진입 로직
- 구현 가능성: 가능

## 요구사항 요약
- 페이지와 인증 필요 API는 공통으로 같은 규칙을 따라야 한다.
- 운영 환경에서는 디스코드 ID 헤더가 없는 사용자는 접근 불가다.
- 운영 환경에서는 일반 사용자가 본인 리소스만 수정할 수 있어야 한다.
- 운영 환경에서는 `ROLE_ADMIN`, `ROLE_SUPER_ADMIN`만 소유권 제한을 우회할 수 있어야 한다.
- 개발 환경에서는 위 제한을 풀어 로컬 개발과 점검이 가능해야 한다.
- 현재 일부 구현은 이 규칙 일부만 만족하고 있으므로, 단일 공통 진입점으로 정리되어야 한다.

## 현재 상태
### 관련 파일
- `src/server/utils/auth.ts`: 대부분의 인증 필요 API가 사용하는 공통 헬퍼다. 현재는 항상 `X-Discord-ID`를 요구하고, 유저 조회 후 `isAdmin`, `hasPermission`, `checkAdmin`을 제공한다.
- `src/server/utils/request-header.ts`: `X-Discord-ID`를 읽는 공통 유틸이다. 서버 전체 헤더 식별 기준점이다.
- `src/server/utils/access.ts`: 최근 추가된 페이지 접근 체크 유틸이다. 현재는 `DEV가 아니면 무조건 차단`으로 구현되어 있어 마스터 요구와 반대다.
- `src/server/middleware/access.ts`: 페이지 요청 시 `/block` 리다이렉트를 수행하는 Nitro 서버 미들웨어다. 내부적으로 `checkCommunityAccess`를 사용한다.
- `src/server/api/users/check.get.ts`: 접근 가능 여부를 응답하는 엔드포인트다. 페이지 가드와 같은 기준을 따라야 한다.
- `src/server/api/users/[id].put.ts`: `authHelper + hasPermission`으로 본인 또는 관리자 수정 제한을 구현하고 있다.
- `src/server/api/characters/[id].put.ts`: 캐릭터 소유자 또는 관리자만 수정 가능하게 구현되어 있다.
- `src/server/api/campaigns/[id].put.ts`: 캠페인 소유자 또는 관리자만 수정 가능하게 구현되어 있다.
- `src/server/api/sessions/[id].put.ts`: 세션이 속한 캠페인 소유자 또는 관리자만 수정 가능하게 구현되어 있다.
- `src/server/api/sessions/[id].delete.ts`: `authHelper`를 쓰면서도 `getDiscordId`를 중복 호출해 별도 체크를 수행하는 예외 케이스다.
- `src/server/api/campaigns/[id]/members/[userId].post.ts`: 현재는 `hasPermission(userId)`만으로 타깃 사용자 본인 또는 관리자만 캠페인 참여를 처리한다. 운영 규칙과는 맞지만, 개발 환경 우회 규칙이 반영되지 않는다.

### 현재 로직 분석
- 현재 CUD API 다수는 이미 `authHelper`를 사용하므로 “운영에서는 디스코드 ID 필수, 운영에서 어드민 우회” 구조의 절반은 갖춰져 있다.
- 하지만 `authHelper`는 `import.meta.env.DEV`를 고려하지 않기 때문에 개발 서버 예외가 없다.
- 반대로 새로 만든 `checkCommunityAccess`는 개발/운영 판정을 뒤집어 구현해 두어서 전체 정책과 충돌한다.
- 페이지 접근 가드와 API 인증 규칙이 서로 다른 유틸에 나뉘어 있어 기준이 분기되고 있다.
- 일부 API는 `authHelper` 외에 직접 `getDiscordId`를 다시 검사하고 있어, 나중에 규칙이 바뀌면 누락·불일치가 생기기 쉽다.
- “본인 것만 수정 가능” 규칙은 현재 `hasPermission(resourceUserId)`가 담당하지만, 개발 환경 완화 정책을 어디에 넣을지 정해져 있지 않다.
- 조회(GET) API 중 공개 허용 API와 보호 대상 API가 섞여 있으므로, 이번 작업은 CUD 중심 인증/인가 통합과 페이지 접근 규칙 통합을 우선 범위로 보는 것이 적절하다.

## 영향도 평가
### 직접 영향
- `src/server/utils/auth.ts`: 운영/개발 분기와 관리자 우회 정책을 담는 핵심 수정 대상
- `src/server/utils/access.ts`: 페이지 접근 가드가 `authHelper`와 같은 기준을 따르도록 재작성 필요
- `src/server/middleware/access.ts`: 공통 접근 판단 결과를 `/block` 리다이렉트에 연결하는 얇은 어댑터로 유지 가능
- `src/server/api/users/check.get.ts`: 공통 접근 판단 결과를 그대로 반환하도록 조정 필요
- `src/server/api/sessions/[id].delete.ts` 등 예외 엔드포인트: 중복 헤더 체크 제거 필요

### 간접 영향
- `authHelper`를 사용하는 모든 POST/PUT/DELETE API의 동작이 변경된다
- 개발 환경에서는 지금까지 실패하던 호출이 통과할 수 있어 테스트 시나리오가 바뀐다
- 운영 환경에서는 페이지 진입과 API 호출이 동일 기준으로 막히므로, 상위 프록시나 요청 주체가 `X-Discord-ID`를 실제로 넣는지 확인이 더 중요해진다
- 관리자 우회가 `ROLE_ADMIN`, `ROLE_SUPER_ADMIN`에만 허용되므로 유저 role 데이터 정합성이 중요해진다

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 개발 환경 완화의 범위를 정확히 어디까지 볼지 정해야 한다. 마스터 표현대로면 인증·소유권 제한 모두 완화로 해석되지만, 생성/수정 시 작성자 메타(`creatorId`, `updaterId`)를 어떤 사용자로 기록할지 별도 기준이 필요하다.
- 개발 환경에 헤더가 없을 때 `authHelper`가 어떤 가상 사용자 컨텍스트를 반환할지 결정해야 한다. `null user + always allow`로 갈지, 첫 관리자 계정/더미 계정을 찾을지 정책이 필요하다.
- 페이지 접근 가드와 API 인증을 동일 유틸로 묶을 때, 페이지는 “접근 가능 여부”만 있으면 되지만 CUD API는 “현재 사용자 ID”가 필요하다. 따라서 단순 boolean이 아니라 인증 컨텍스트 객체를 표준화해야 한다.
- 일부 API는 “본인 또는 캠페인 마스터 또는 관리자”처럼 소유권 기준이 복합적이므로 `hasPermission`만으로 충분한지, 도메인별 보조 헬퍼가 필요한지 검토가 필요하다.

### 가정 / 불확실성
- 본 작업은 CUD API와 페이지 접근 규칙을 우선 대상으로 하며, 공개 GET API 전체를 비공개로 바꾸는 작업까지는 당장 포함하지 않는다.
- “어드민은 나 하나”라는 운영 사실은 role 데이터 기준으로 이미 표현된다고 가정한다. 즉 코드상 특정 디스코드 ID 하드코딩보다는 `ROLE_ADMIN`, `ROLE_SUPER_ADMIN` 판정이 기준이 된다.
- 개발 환경 완화 시 실제 DB 유저가 없어도 일부 API가 통과할 수 있도록 할지 여부는 설계에서 명시해야 한다.
