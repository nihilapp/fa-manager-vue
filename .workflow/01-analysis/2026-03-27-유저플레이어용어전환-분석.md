# 분석 (Analysis)

## 요약
- 요청 내용: 프로젝트 전반에서 `users` 용어를 `players`로 전환한다
- 범위: 프런트 컴포넌트/스토어/컴포저블, 서버 API/DTO/DB 테이블, PRD 및 워크플로우 문서 전반
- 구현 가능성: 가능. 다만 단순 문자열 치환이 아니라 경로, 자동임포트, API 계약, DB 스키마 명칭을 단계적으로 재설계해야 한다

## 요구사항 요약
- 마스터 요청은 `users`라는 단어 자체를 `players`로 바꾸는 대규모 리네이밍이다.
- 구현 관점에서는 세 가지 층위가 동시에 맞아야 한다.
  - 사용자에게 보이는 UI/문서 용어를 `플레이어` 기준으로 통일한다.
  - 프런트 내부 식별자와 파일 경로에서 `user`/`users` 명칭을 `player`/`players`로 재정렬한다.
  - 서버 API, DTO, DB 테이블, 인증/권한 로직에서 `users`를 어떤 수준까지 `players`로 바꿀지 계약 단위로 정해야 한다.
- 특히 현재 프로젝트는 이미 페이지 경로 `/players`와 내부 구현 `users`가 혼재되어 있어, 이번 작업은 혼재 상태를 제거하는 정합성 회복 작업에 가깝다.

## 현재 상태
### 관련 파일
- `PRD/PRD.md`: 핵심 도메인을 "플레이어(User)"로 설명하고 있어 상위 요구사항은 이미 플레이어 중심이지만, 하위 구현 명칭과 완전히 일치하지 않는다.
- `PRD/Task-List.md`: 관리자/감사로그/API 태스크에 `users` 테이블, `/api/users/:id` 등 기존 명칭이 남아 있어 용어 기준선 충돌이 발생한다.
- `PRD/Api-Specification.md`: `POST /api/users`, `GET /api/users`, `PATCH /api/users/{id}`와 `users` 테이블 명세를 직접 정의하고 있어 외부 계약 문서 영향이 크다.
- `src/app/pages/players/index.vue`: 실제 라우트는 이미 `/players`를 사용하지만 내부 렌더링은 `UserListTable`에 의존한다.
- `src/app/components/users/UserListTable.vue`: 플레이어 목록 화면의 실질 구현체지만 컴포넌트 폴더명과 내부 저장소/컴포저블 사용은 전부 `users` 기준이다.
- `src/app/stores/user.store.ts`: `useUserStore`, `userList`, `userPageData`, `myInfo` 등 전역 상태의 중심 축이다. 프런트 리네이밍 시 연쇄 영향의 핵심이다.
- `src/app/composables/users/useGetUserList.ts`, `src/app/composables/users/useGetMyInfo.ts`: `/users`, `/users/me` API를 직접 호출하며 쿼리 키도 `queryKeys.users`에 묶여 있다.
- `src/app/utils/query-keys.ts`: `users`를 쿼리 키 네임스페이스로 고정하고 있어 캐시 계약 변경 대상이다.
- `src/server/api/users/*.ts`: 서버 라우트 경로가 모두 `users`에 종속되어 있다. 프런트 경로 변경과 별개로 실제 API 계약을 유지할지 이전할지 결정이 필요하다.
- `src/server/types/dto/user.dto.ts`: `UserQueryDto`, `UserCreateDto`, `UserUpdateDto`, `UserOutDto`를 정의한다. 서버 전역 자동임포트와 다른 DTO의 중첩 타입이 여기에 기대고 있다.
- `src/server/db/table/users.table.ts`: 실제 DB 테이블명이 `users`이며, Drizzle 스키마와 인덱스 명도 여기에 종속된다.
- `src/server/db/table/relations.ts`: 캠페인, 세션, 캐릭터, 문서, 로그, 화폐 트랜잭션이 모두 `usersTable`과 관계를 맺고 있다.
- `src/server/utils/auth.ts`: 인증 결과 타입 `AuthResolvedUser`, `UserOutDto`, `USER_NOT_FOUND` 메시지, `resourceUserId` 권한 검증 규칙이 사용자 모델 명칭에 의존한다.
- `src/server/types/manual-server-imports.d.ts`: `TypeUser`, `UserQueryDto`, `UserOutDto`, `UserStatus` 등 전역 타입 선언을 유지하고 있어 서버 전역 리네이밍의 허브 역할을 한다.

### 현재 로직 분석
- 프런트 라우트와 메뉴는 이미 `플레이어`와 `/players`를 사용한다. 반면 목록 컴포넌트, 저장소, 컴포저블, 쿼리 키는 `users` 명칭으로 남아 있다.
- 서버는 인증과 CRUD의 기준 엔티티를 `users`로 유지한다. `/users/me` 같은 공통 인증 API도 이 계약을 따른다.
- DTO와 DB는 `User*`, `usersTable`, `users` 테이블 기준으로 정렬되어 있고, 다른 도메인들이 이를 참조한다.
- 따라서 현재 구조는 "표시 명칭은 플레이어, 내부 계약은 사용자"가 혼합된 상태이며, 이번 작업은 이 둘을 어느 레벨까지 통합할지 결정해야 한다.

## 영향도 평가
### 직접 영향
- 프런트 파일/식별자 리네이밍
  - `components/users`, `composables/users`, `user.store.ts`, `useUserStore`, `useGetUserList` 등 명칭과 경로 변경이 필요하다.
- 프런트 상태/쿼리 계약 변경
  - `userList`, `userPageData`, `userInfo`, `UserListTable`, `queryKeys.users`를 `player*` 구조로 바꾸면 호출부 전수 수정이 필요하다.
- 서버 API 경로 변경 가능성
  - `/api/users`, `/api/users/me`, `/api/users/[id]`를 `/api/players`로 바꾸려면 프런트 호출과 PRD API 명세를 동시에 바꿔야 한다.
- DTO/타입 리네이밍 가능성
  - `UserQueryDto`, `UserOutDto`, `UserStatus`, `UserRole`의 명칭 변경은 자동임포트 선언과 다수 DTO 참조 파일을 연쇄 수정하게 만든다.
- DB 스키마 변경 가능성
  - `usersTable`과 실제 `users` 테이블명까지 변경할 경우 Drizzle 스키마, relation, DDL, 마이그레이션 전략이 필요하다.

### 간접 영향
- 인증/인가 규칙
  - `auth.ts`의 `hasPermission(resourceUserId)`와 `/users/me` 기반 현재 인증 흐름은 명칭 변경 시 오해를 낳기 쉬워, 함수명과 메시지까지 정비 대상이 될 수 있다.
- 감사 로그 및 관리자 기능
  - `PRD/Task-List.md`에 남아 있는 `users` 관련 관리자 태스크, 감사 로그 설명, 역할 변경 방어 규칙이 새 용어 기준선과 충돌한다.
- 다른 도메인 DTO의 참조 타입
  - `CampaignOutDto`, `CharacterOutDto`, `SessionLogOutDto`, `LogHistoryOutDto` 등이 `user: UserOutDto | null`을 포함하고 있어 단일 파일 변경으로 끝나지 않는다.
- 문서 및 이전 워크플로우 참조
  - 기존 `.workflow` 산출물에는 `users/me.get.ts`, `userStore`, `useGetUserList()`가 다수 기록되어 있어, 이번 작업 이후에도 과거 문서와 현재 구조 사이의 용어 차이가 남을 수 있다.

## 실행 판단
### 구현 가능성
- 가능
- 다만 한 번에 치환하면 위험하다. 최소한 아래 선택지를 설계 단계에서 명확히 결정해야 한다.
  - 선택지 A: UI/프런트 내부 명칭만 `player`로 바꾸고 서버 API 및 DB 계약은 `users`로 유지
  - 선택지 B: 프런트와 서버 API 계약까지 `players`로 바꾸고 DB 테이블은 `users`로 유지
  - 선택지 C: DB 테이블까지 포함해 전면적으로 `players`로 전환

### 예상 기술 난제
- 자동임포트 체계 때문에 타입명과 파일명만 바꾸면 되는 작업이 아니다. `manual-app-imports.d.ts`, `manual-server-imports.d.ts`와 Nuxt 오토임포트 경로를 함께 맞춰야 한다.
- API 경로를 변경하면 기존 `/users/me` 인증 의존 지점, 쿼리 키, 공통 에러 메시지, 문서가 모두 함께 바뀌어야 한다.
- DB 테이블명까지 변경하면 코드 리네이밍을 넘어 데이터 마이그레이션과 운영 배포 순서까지 설계해야 한다.
- `UserRole`, `UserStatus`는 실제로는 플레이어뿐 아니라 관리자 권한 의미도 담고 있어, 단순 `PlayerRole`로 바꿀 때 의미 왜곡 가능성이 있다.

### 가정 / 불확실성
- 마스터 요청의 "단어 자체를 바꾼다"가 DB 테이블명과 실제 API 엔드포인트까지 포함하는지, 아니면 코드/문서/프런트 계약 중심인지는 아직 확정되지 않았다.
- `users` 테이블에 대한 실제 마이그레이션 관리 파일 존재 여부는 이번 분석에서 확인하지 않았다. DB 물리 변경 범위는 설계 단계에서 추가 확인이 필요하다.
- 기존 외부 클라이언트나 문서 소비자가 `/api/users`를 사용 중인지 확인되지 않았다. 외부 호환성 요구가 있으면 이중 라우팅이나 호환 기간 설계가 필요할 수 있다.
