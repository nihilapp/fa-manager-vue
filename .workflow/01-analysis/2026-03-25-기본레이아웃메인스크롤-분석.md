# 분석 (Analysis)

## 요약
- 요청 내용: `src/app/layouts/default.vue`에서 헤더와 푸터는 고정하고, 그 사이 `main` 영역에만 스크롤이 발생하도록 조정
- 범위: `src/app/layouts/default.vue`, `src/app/components/layout/common-layout/CommonContent.vue`
- 구현 가능성: 가능

## 요구사항 요약
- 브라우저 전체 문서가 아닌 기본 레이아웃 내부의 콘텐츠 영역만 세로 스크롤을 가져야 한다.
- `CommonHeader`와 `CommonFooter`는 같은 위치를 유지하고, 콘텐츠가 길어질 때 `main`만 독립적으로 스크롤되어야 한다.
- 기존 Nuxt 앱 루트에서 `overflow: hidden`을 사용하고 있으므로, 내부 flex 레이아웃에서 남은 높이 점유와 overflow 책임을 명확히 나눠야 한다.

## 현재 상태
### 관련 파일
- `src/app/layouts/default.vue`: 기본 레이아웃의 최상위 래퍼에서 `CommonHeader`, `CommonContent`, `CommonFooter`를 세로로 배치한다. 현재 `h-full`만 사용해 내부 콘텐츠의 남은 높이 계산이 불명확하다.
- `src/app/components/layout/common-layout/CommonContent.vue`: `<main class="flex-1 ...">`만 선언되어 있으며, 실제 스크롤 처리(`overflow-y-auto`)와 자식 축소 허용(`min-h-0`)이 없다.
- `src/app/assets/styles/alt-body.css`: `#__nuxt`를 `height: 100dvh; overflow: hidden;`으로 고정해 전역 문서 스크롤을 막고 있다.
- `src/app/assets/styles/tailwind.css`: 커스텀 Tailwind 토큰/유틸이 정의되어 있어 기본 클래스 사용 시 프로젝트 파일 기준으로 안전성 확인이 필요하다.

### 현재 로직 분석
- 현재 전역 컨테이너는 이미 `100dvh`와 `overflow: hidden`을 가지므로, 원하는 동작은 레이아웃 내부에서 해결하는 편이 맞다.
- 하지만 `default.vue`의 최상위 래퍼가 `h-full`만 사용할 경우 자식 flex 아이템이 남은 높이를 안정적으로 분배받지 못할 수 있고, `main` 쪽도 `overflow-y-auto`가 없어 실제 스크롤 컨테이너가 명시되지 않는다.
- 따라서 상위 래퍼에는 `min-h-0`을 포함한 세로 flex 컨테이너 제약을, `CommonContent`에는 `flex-1 min-h-0 overflow-y-auto`를 부여하는 방식이 가장 직접적이다.

## 영향도 평가
### 직접 영향
- `src/app/layouts/default.vue`의 최상위 레이아웃 클래스 조정
- `src/app/components/layout/common-layout/CommonContent.vue`의 `main` 스크롤 책임 명시

### 간접 영향
- `default` 레이아웃을 사용하는 모든 페이지가 새 스크롤 컨테이너를 공유한다.
- 페이지 내부에서 `h-full` 또는 `min-h-full`에 의존하는 일부 화면은 부모의 `min-h-0` 제약을 전제로 동작이 더 안정화될 가능성이 있다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 일부 페이지가 문서 스크롤을 전제로 스타일을 작성했다면 내부 스크롤 컨테이너 전환 후 높이 계산이 달라질 수 있다.
- Tailwind 기본 클래스의 가용성이 프로젝트 설정에 따라 달라질 수 있으므로 현재 사용 중인 유틸/클래스 체계와 충돌이 없는 조합으로 적용해야 한다.

### 가정 / 불확실성
- `context7 MCP`는 현재 세션에 제공되지 않아 로컬 코드 검색과 프로젝트 문서 기준으로 분석을 수행했다.
- `CommonHeader`와 `CommonFooter` 자체 높이는 콘텐츠에 따라 결정되며, 별도 고정 높이 계산은 필요 없다고 가정했다.
