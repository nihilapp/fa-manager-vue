# 분석 (Analysis)

## 요약
- 요청 내용: 캐릭터 생성 페이지와 캐릭터 수정 페이지의 UI를 먼저 구현하고, 실제 저장 로직은 후속 작업으로 남긴다.
- 범위: 생성 페이지 래퍼, 수정 페이지 래퍼, 생성/수정 공용 캐릭터 입력 UI 컴포넌트
- 구현 가능성: 가능

## 요구사항 요약
- 캐릭터 생성 페이지는 더미 안내 박스를 제거하고 실제 입력 가능한 폼 UI로 바꾼다.
- 캐릭터 수정 페이지는 신규 라우트로 추가하고, 기존 캐릭터 상세 조회 결과를 폼 기본값에 반영한다.
- 저장/수정 API 실행 로직은 마스터가 후속 구현할 예정이므로 UI와 기본 상태 표현에 집중한다.
- 페이지 파일은 렌더링 컴포넌트만 감싸고, 실제 화면 로직은 컴포넌트로 분리한다.

## 현재 상태
### 관련 파일
- `src/app/pages/characters/add-character.vue`
  - 현재는 `SectionPage` 안에 안내 문구만 출력하는 임시 페이지다.
- `src/app/pages/players/add-player.vue`
  - 페이지 파일이 `NewPlayerForm`만 렌더링하는 래퍼 패턴의 기준이다.
- `src/app/components/players/NewPlayerForm.vue`
  - `SectionPage` 기반 입력 폼 구조, 버튼 배치, `useForm` 사용 패턴의 기준 컴포넌트다.
- `src/app/composables/characters/useGetCharacterInfo.ts`
  - 수정 페이지에서 기존 캐릭터 정보를 불러와 스토어에 적재하는 기준 컴포저블이다.
- `src/app/stores/character.store.ts`
  - `characterInfo`, `characterClassList`를 보유하고 있어 수정 화면 기본값 주입의 상태 기준선이다.
- `src/server/types/dto/character.dto.ts`
  - 생성/수정 UI에 노출할 수 있는 캐릭터 필드 목록과 타입을 제공한다.
- `src/app/components/common/FormInput.vue`
  - 텍스트, 숫자, 텍스트에어리어 입력을 모두 지원한다.
- `src/app/components/common/FormSelect.vue`
  - 셀렉트 기반 상태 선택 UI를 제공한다.

### 현재 로직 분석
- 캐릭터 생성 화면은 아직 실사용 가능한 폼이 없다.
- 캐릭터 수정 전용 라우트와 폼 컴포넌트도 없다.
- 캐릭터 상세 조회는 이미 구현돼 있으므로 수정 화면은 새 API 없이 기존 조회 결과를 이용할 수 있다.
- 공용 입력 UI를 하나로 만들면 생성/수정 페이지가 같은 레이아웃과 필드 배치를 재사용할 수 있다.

## 영향도 평가
### 직접 영향
- `src/app/pages/characters/add-character.vue`
- `src/app/pages/characters/edit/[id].vue` 신규 추가
- `src/app/components/characters/CharacterEditorForm.vue` 신규 추가

### 간접 영향
- `useGetCharacterInfo`를 호출하는 수정 페이지는 진입 시 캐릭터 스토어를 사용하므로 기존 캐릭터 상세 흐름과 상태 공유가 발생한다.
- 후속 저장 로직 구현 시 공용 폼의 제출 이벤트 계약을 그대로 이어받을 가능성이 높다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 필드 수가 많아 화면 가독성을 해치지 않도록 섹션 분리와 2열 그리드 구성이 필요하다.
- 수정 페이지에서 비동기 로딩 이전 상태와 로딩 이후 기본값 갱신을 모두 안전하게 처리해야 한다.

### 가정 / 불확실성
- 상태 선택지는 현재 코드베이스에서 사용 중인 `ACTIVE`, `INACTIVE`, `REST`를 재사용한다.
- 캐릭터 클래스 편집 로직은 별도 API와 화면이 필요하므로 이번 작업에서는 읽기 전용 요약 카드로만 노출한다.
- 저장 버튼은 동작하지 않고, `submit` 이벤트만 방출하는 UI 계약만 제공한다.
