## 요약
- 요청 내용: `players/me`는 본인 데이터 전용, `players/[id]`는 어드민의 대상 플레이어 관리 라우트로 정상화
- 범위: `src/server/api/players/[id].get.ts`, `src/server/api/players/[id].put.ts`, `src/server/api/players/[id].delete.ts`
- 구현 가능성: 가능

## 요구사항 요약
- `me`와 `[id]` 라우트 책임을 분리해야 한다.
- `[id]`는 실제 URL 파라미터의 플레이어 ID를 사용해야 한다.
- `[id]`는 어드민 전용 대상 플레이어 조회/수정/삭제로 동작해야 한다.

## 현재 상태
### 관련 파일
- `src/server/api/players/me.get.ts`: 본인 플레이어 정보 조회
- `src/server/api/players/me.put.ts`: 본인 플레이어 정보 수정
- `src/server/api/players/me.delete.ts`: 본인 플레이어 soft delete
- `src/server/api/players/[id].get.ts`: 파일명과 다르게 본인 수정 로직이 들어 있었음
- `src/server/api/players/[id].put.ts`: 파일명과 다르게 본인 수정 로직이 들어 있었음
- `src/server/api/players/[id].delete.ts`: 파일명과 다르게 본인 수정 로직이 들어 있었음

### 현재 로직 분석
- `[id]` 계열 세 파일 모두 `getRouterParam(event, 'id')`를 사용하지 않았다.
- 세 파일 모두 `user!.id`만 사용해 사실상 `me`와 동일 대상을 조작했다.
- `GET` 파일조차 update body를 읽고 수정 응답을 반환하는 잘못된 상태였다.

## 영향도 평가
### 직접 영향
- 어드민 대상 플레이어 관리 API가 정상 동작하지 않았다.
- `[id]` 호출이 기대와 다르게 자기 자신을 수정하는 오동작을 일으킬 수 있었다.

### 간접 영향
- API 경로 의미와 실제 동작이 어긋나 유지보수 혼란이 있었다.
- 권한 정책 문서와 실제 구현 불일치가 생겼다.
