# 분석 (Analysis)

## 요약
- 요청 내용: `DataTableColumn`이 제네릭 타입으로 바뀐 뒤 오토임포트 선언과 관련 사용 유틸을 그 변경에 맞춰 수정한다.
- 범위: `src/app/types/manual-app-imports.d.ts`, `src/app/composables/useColumns.ts`, `src/app/utils/column-builder.ts`, `src/app/components/table/DataTable.vue`
- 구현 가능성: 가능

## 요구사항 요약
- 전역 타입 선언에서 `DataTableColumn`을 제네릭으로 사용할 수 있어야 한다.
- `DataTableColumn<T>`를 소비하는 유틸 함수와 빌더 클래스도 제네릭을 잃지 않도록 타입 시그니처를 정리해야 한다.
- 기존 Nuxt 오토임포트/전역 선언 규칙을 깨지 않도록 직접 import 추가 없이 선언 파일에서 해결해야 한다.

## 현재 상태
### 관련 파일
- `src/app/types/common.types.ts`: `DataTableColumn<T = any>` 정의가 들어 있다.
- `src/app/types/manual-app-imports.d.ts`: 전역 타입 별칭으로 `DataTableColumn`을 노출하지만 현재 제네릭 타입 파라미터를 전달하지 않는다.
- `src/app/composables/useColumns.ts`: `DataTableColumn[]`를 그대로 받는 단순 래퍼 함수라 제네릭 정보가 소실된다.
- `src/app/utils/column-builder.ts`: `DataTableColumn`을 내부 상태로 들고 있어 `keyof T` 기반 키 타입 정보를 유지하지 못한다.
- `src/app/components/table/DataTable.vue`: props는 `DataTableColumn<T>[]`를 쓰지만 내부 헬퍼 함수는 비제네릭 `DataTableColumn`을 사용한다.

### 현재 로직 분석
- 현재 전역 타입 별칭은 `type DataTableColumn = import(...).DataTableColumn;` 형태라 타입 인자를 받을 수 없다.
- `useColumns`와 `ColumnBuilder`는 열 정의를 보존/생성하는 역할인데, 제네릭이 빠져 있으면 컬럼 key에 대한 타입 안정성이 약해진다.
- `DataTable.vue`는 실제 렌더링은 동작하더라도 내부 함수 시그니처가 넓어져 제네릭 효과를 충분히 살리지 못한다.

## 영향도 평가
### 직접 영향
- 전역 타입 별칭 시그니처 변경
- 열 배열 유틸 함수 시그니처 변경
- 컬럼 빌더 클래스의 제네릭화
- `DataTable.vue` 헬퍼 함수의 타입 파라미터 반영

### 간접 영향
- `ColumnBuilder` 사용처가 있으면 새 제네릭 기본값 덕분에 대부분 호출부 수정 없이 유지될 가능성이 높다.
- 전역 `DataTableColumn<T>` 사용 패턴이 타입 검사에서 더 엄격해질 수 있다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `ColumnBuilder`는 key 타입을 `keyof T | (string & {})`와 맞춰야 하므로 생성자/정적 팩토리의 타입 인자를 함께 정리해야 한다.
- `DataTable.vue`는 `<script setup generic>` 환경이라 내부 헬퍼 함수도 `DataTableColumn<T>`를 명시해 동일한 제네릭 컨텍스트를 유지해야 한다.

### 가정 / 불확실성
- 현재 세션에서 context7 MCP는 확인되지 않아 로컬 코드 기준으로만 판단했다.
- `ColumnBuilder` 실제 사용처 전부를 수정할 필요는 없을 것으로 보이나, 타입 검증에서 추가 오류가 나오면 연쇄 수정이 필요할 수 있다.
