# 분석 (Phase 1 - Analysis)

## 요약
- 요청 내용: 플레이어 GET 응답에도 관계 테이블 정보를 포함하도록 개선
- 범위: `players/index.get.ts`, `players/[id].get.ts`, `players/me.get.ts`, `PlayerOutDto`
- 구현 가능성: 가능

## 요구사항 요약
- 플레이어 조회 API가 다른 도메인 API처럼 관계 테이블 정보를 포함해 응답해야 한다.
- 특히 단건 조회와 내 정보 조회는 현재 플레이어 기본 컬럼만 반환하고 있어 관계 데이터가 누락된다.
- 관계 정의 기준으로 빠진 항목 없이 응답 계약을 맞춰야 한다.

## 현재 상태
### 관련 파일
- `src/server/api/players/index.get.ts`: 플레이어 목록 조회. 이미 여러 관계를 `with`로 포함하지만 `currencyTransactions`는 빠져 있다.
- `src/server/api/players/[id].get.ts`: 관리자용 플레이어 단건 조회. 현재 `findFirst`만 수행하고 관계 `with`가 전혀 없다.
- `src/server/api/players/me.get.ts`: 로그인된 내 플레이어 정보 조회. `authHelper`가 반환한 기본 유저 객체를 그대로 응답해 관계 정보가 없다.
- `src/server/types/dto/player.dto.ts`: 플레이어 응답 DTO. `campaigns`, `campaignMembers`, `characters`, `sessionPlayers`, `sessionLogs`, `docs`, `logHistories`까지만 정의되어 있고 `currencyTransactions`가 없다.
- `src/server/db/table/relations.ts`: `playersRelations`에 총 8개 관계(`campaigns`, `campaignMembers`, `characters`, `sessionPlayers`, `sessionLogs`, `docs`, `logHistories`, `currencyTransactions`)가 정의되어 있다.

### 현재 로직 분석
- 목록 조회는 Drizzle relation query를 사용해 대부분의 관계를 포함한다.
- 단건 조회와 내 정보 조회는 relation query가 아니라 기본 player row만 반환한다.
- DTO가 `currencyTransactions`를 누락하고 있어 목록 조회에서도 전체 관계를 표현하지 못한다.

## 영향도 평가
### 직접 영향
- 플레이어 GET 응답 payload 구조가 확장된다.
- `PlayerOutDto`를 참조하는 서버/클라이언트 코드에서 추가 필드 사용이 가능해진다.
- 단건 조회 및 내 정보 조회의 DB 접근이 relation query로 변경된다.

### 간접 영향
- 응답 크기가 증가해 플레이어 상세/내 정보 호출의 네트워크 비용이 소폭 늘 수 있다.
- `currencyTransactions`에 중첩 관계(`character`, `user`)를 어디까지 포함할지에 따라 순환 참조나 과도한 payload 위험이 있을 수 있다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `currencyTransactions` 관계를 포함할 때 중첩 관계를 최소화해야 한다.
- `me.get.ts`는 `authHelper` 반환 객체를 그대로 쓰지 말고, 권한 검증 후 player id 기준으로 다시 조회해야 한다.

### 가정 / 불확실성
- 이번 요구의 "관계 테이블 정보도 다"는 `playersRelations`에 정의된 8개 관계를 기준으로 해석한다.
- `currencyTransactions` 내부의 `user`, `character`를 모두 중첩해 내려야 하는 요구는 명시되지 않았으므로 우선 기본 row 배열만 포함하는 방향이 안전하다.
