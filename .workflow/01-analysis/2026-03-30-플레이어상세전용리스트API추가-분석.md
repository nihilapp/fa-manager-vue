# 분석 (Phase 1 - Analysis)

## 요약
- 요청 내용: 플레이어 상세 화면용 전용 리스트 API 존재 여부를 점검하고, 부족한 엔드포인트를 추가한다.
- 범위: 플레이어 참여 캠페인 목록 API, 플레이어 참여 세션 목록 API, 관련 워크플로우 문서
- 구현 가능성: 가능

## 요구사항 요약
- 플레이어 상세 화면에서 캐릭터/참여 캠페인/참여 세션을 각각 `pageSize=10` 기반으로 독립 페이징해야 한다.
- 캐릭터 목록은 기존 공용 API를 재사용해도 되지만, 참여 캠페인과 참여 세션은 전용 엔드포인트가 필요하다.
- 모든 목록은 최소한 페이지네이션과 필터링이 가능해야 한다.

## 현재 상태
### 관련 파일
- `src/app/components/players/PlayerDetail.vue`
  - 현재 `playerInfo.campaignMembers`, `playerInfo.characters`, `playerInfo.sessionPlayers`를 그대로 출력한다.
  - 독립적인 page/size/filter 제어 구조가 없다.
- `src/server/api/characters/index.get.ts`
  - `userId`, `campaignId`, `name`, `status` 등 필터와 `page`, `size`, `sort`를 지원한다.
  - 특정 플레이어 캐릭터 목록 API로 재사용 가능하다.
- `src/server/api/campaigns/index.get.ts`
  - `CampaignQueryDto.userId`는 캠페인 소유자 필터다.
  - 참여 캠페인 목록에는 사용할 수 없다.
- `src/server/api/sessions/index.get.ts`
  - 세션 자체 필터와 페이징은 있으나, 특정 플레이어 참여 여부 기준 필터가 없다.
- `src/server/db/table/relations.ts`
  - 참여 캠페인은 `campaignMembersTable`, 참여 세션은 `sessionPlayersTable`를 기준으로 추적 가능하다.

### 현재 로직 분석
- 캐릭터 목록은 `/api/characters?userId={playerId}` 호출만으로 페이징/필터링이 가능하다.
- 참여 캠페인 목록은 `campaignMembersTable.userId = playerId`를 기준으로 캠페인 목록을 가져오는 전용 GET 엔드포인트가 없다.
- 참여 세션 목록은 `sessionPlayersTable.userId = playerId`를 기준으로 세션 목록을 가져오는 전용 GET 엔드포인트가 없다.

## 영향도 평가
### 직접 영향
- `players/[id]/campaigns.get.ts`, `players/[id]/sessions.get.ts` 신규 엔드포인트가 추가된다.
- 플레이어 상세 화면은 이후 대용량 관계 배열 대신 독립 리스트 API를 사용할 수 있게 된다.

### 간접 영향
- 프런트는 `PlayerDetail.vue`에서 더 이상 `playerInfo` 내부 배열 전체를 신뢰하지 않고 목록 API를 병행 사용하게 될 가능성이 높다.
- 참여 리스트 기준이 소유(owner) 기준이 아니라 membership 기준으로 분리되므로, 유사 기능을 다른 화면에서도 재사용할 수 있다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 조인 테이블을 기준으로 대상 id 집합을 만들고, 그 위에 기존 `CampaignQueryDto`, `SessionQueryDto` 필터를 합성해야 한다.
- 세션 참여 목록은 조인 테이블 필터(`role`, `characterId`)와 세션 본체 필터(`campaignId`, `name`, `status`)를 함께 처리해야 한다.

### 가정 / 불확실성
- 캐릭터 목록은 기존 `/api/characters` 재사용으로 충분하다고 판단한다.
- 참여 캠페인 목록은 `campaignMembers.role`, 참여 세션 목록은 `sessionPlayers.role`, `sessionPlayers.characterId` 필터가 있으면 상세 화면 확장성에 유리하다고 본다.
