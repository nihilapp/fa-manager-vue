# 분석 (Analysis)

## 요약
- 요청 내용: `ListPageData`를 서버 응답 타입 파일로 이동하고, `ListDataType`의 nullable 페이지 필드를 falsy 기본값으로 정리
- 범위: 서버 응답 타입, 리스트 직렬화 유틸, 프런트 테이블 소비부
- 구현 가능성: 가능

## 요구사항 요약
- `ListPageData`는 프런트 공통 타입이 아니라 서버 응답 구조의 파생 타입으로 두는 편이 자연스럽다.
- `ListDataType`에서 `list`, `totalElements`, `filteredElements` 외 필드도 `null`이 아니라 `0/false`를 사용해야 한다.
- 프런트 소비부에서 `null` 방어 로직을 줄일 수 있어야 한다.

## 현재 상태
### 관련 파일
- `src/server/types/response.types.ts`: 공통 응답 타입 정의
- `src/server/utils/list-data.ts`: 목록 응답 직렬화 클래스
- `src/app/types/common.types.ts`: 프런트 공통 타입에 `ListPageData`가 위치
- `src/app/types/manual-app-imports.d.ts`: 전역 타입 선언
- `src/app/components/table/DataTable.vue`: 페이지 정보 정규화
- `src/app/components/table/Paginator.vue`: 페이지 이동 버튼 로직

### 현재 로직 분석
- 서버 응답 타입은 페이지 필드를 `number | null`, `boolean | null`로 두고 있었다.
- `ListData`는 비페이징일 때 `null`을 반환했고, 프런트는 이를 다시 정규화해야 했다.
- `ListPageData`는 실제로 서버 응답 `ListDataType`에서 `list`만 뺀 타입인데 프런트 공통 타입 파일에 있었다.

## 영향도 평가
### 직접 영향
- 공통 응답 타입과 리스트 직렬화 값
- 프런트 테이블/페이지네이터의 타입과 분기

### 간접 영향
- 목록 응답을 쓰는 다른 화면에서도 `null` 방어 코드가 줄어들 수 있다.
