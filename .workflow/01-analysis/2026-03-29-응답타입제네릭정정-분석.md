# 분석 (Analysis)

## 요약
- 요청 내용: `BaseApiResponse<T>`의 `data`를 `T | null`이 아니라 `T`로 유지하고, 에러 응답만 `BaseApiResponse<null>`로 표현
- 범위: 서버 응답 타입 정의, 앱 공용 응답 타입 사용부, 직전 워크플로 문서 보정
- 구현 가능성: 가능

## 요구사항 요약
- 성공 응답 타입이 실패 응답 때문에 강제로 `null` 가능성을 가지면 안 된다.
- 성공인데 `data`가 비어 있는 응답은 `BaseApiResponse<null>`처럼 제네릭 자체로 표현해야 한다.
- 따라서 `BaseApiResponse<T>`는 항상 `data: T` 계약을 가져야 한다.

## 현재 상태
### 관련 파일
- `src/server/types/response.types.ts`: 현재 `BaseApiResponse<T>`의 `data`를 `T | null`로 선언하고 있다.
- `src/server/utils/base-response.ts`: `data`, `page`, `error` 메서드 반환 제네릭을 `BaseApiResponse<T>` 기준으로 제공한다.
- `src/app/composables/query/useApiRequest.ts`: 앱이 서버 `BaseApiResponse<T>`를 공용 응답 타입으로 사용한다.
- `src/app/utils/api-error-handler.ts`: 에러 응답을 `ApiErrorResponse = BaseApiResponse<null>`로 해석한다.

### 현재 로직 분석
- 현재 선언은 에러 응답을 한 타입으로 수용하려고 `data: T | null`로 넓혀 두었다.
- 하지만 이 방식은 `BaseApiResponse<UserDto>` 같은 성공 타입에도 불필요하게 `null`이 섞인다.
- 마스터 요구대로라면 성공 시 `null` 응답도 `BaseApiResponse<null>` 또는 `BaseApiResponse<MyType | null>`처럼 제네릭으로 표현해야 하므로, 인터페이스가 별도로 `null`을 더할 이유가 없다.

## 영향도 평가
### 직접 영향
- `src/server/types/response.types.ts`의 `data` 선언
- 관련 검증 문서와 보고 문구

### 간접 영향
- 앱 `onSuccess` 콜백과 응답 처리부에서 성공 데이터 타입이 더 정확해진다.
- 에러 응답은 기존대로 `BaseApiResponse<null>` 별칭으로 유지되므로 런타임 동작 변화는 없다.
