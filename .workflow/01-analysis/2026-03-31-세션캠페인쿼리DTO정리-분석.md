# 분석 (Analysis)

## 요약
- 요청 내용: `SessionQueryDto`, `CampaignQueryDto` 자체에 필터 필드를 포함시키고, 플레이어 하위 API 및 쿼리 키 타입을 DTO 기준으로 정리
- 범위: 서버 DTO, `players/[id]/sessions.get.ts`, `players/[id]/campaigns.get.ts`, `src/app/utils/query-keys.ts`
- 구현 가능성: 가능

## 요구사항 요약
- `SessionQueryDto & { role?: SessionRole; characterId?: number }` 같은 임시 확장 대신 `SessionQueryDto`가 해당 필드를 직접 소유해야 한다.
- `CampaignQueryDto & { role?: CampaignRole }` 같은 임시 확장 대신 `CampaignQueryDto`가 `role` 필드를 직접 소유해야 한다.
- 플레이어 하위 API와 쿼리 키 타입은 새 DTO 정의를 기준으로 맞춰야 한다.

## 현재 상태
### 관련 파일
- `src/server/types/dto/session.dto.ts`: 세션 목록 조회용 쿼리 DTO 정의, 현재 `role`, `characterId` 없음
- `src/server/types/dto/campaign.dto.ts`: 캠페인 목록 조회용 쿼리 DTO 정의, 현재 `role` 없음
- `src/server/api/players/[id]/sessions.get.ts`: `SessionQueryDto`에 `role`, `characterId`를 교차 타입으로 덧붙여 사용
- `src/server/api/players/[id]/campaigns.get.ts`: `CampaignQueryDto`에 `role`을 교차 타입으로 덧붙여 사용
- `src/app/utils/query-keys.ts`: 플레이어 세션/캠페인 하위 키에서 DTO와 경로 파라미터 조합을 타입으로 정의

### 현재 로직 분석
- 서버 API는 `getQuery<...>`에서 확장 타입을 직접 선언해 필터 필드를 우회적으로 받는다.
- DTO 정의와 실제 API 수용 필드가 달라 타입 기준점이 분산돼 있다.
- 쿼리 키는 플레이어 하위 라우트의 `playerId`를 별도 타입으로 포함해야 하므로 DTO와 경로 파라미터를 함께 표현해야 한다.

## 영향도 평가
### 직접 영향
- 세션/캠페인 쿼리 DTO 타입 정의
- 플레이어 하위 세션/캠페인 목록 API의 `getQuery` 타입
- 플레이어 하위 세션/캠페인 쿼리 키 타입 정의

### 간접 영향
- 이후 프론트/서버 양쪽에서 세션·캠페인 목록 필터를 사용할 때 DTO만 보면 수용 가능한 파라미터를 바로 알 수 있다.
- `query-keys.ts`는 `playerId` 경로 파라미터를 여전히 별도로 가져야 하므로 DTO 완전 동일화가 아니라 `DTO + path param` 구조가 필요하다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `query-keys.ts`는 경로 파라미터 때문에 DTO만 단독으로 쓸 수 없으므로, DTO 확장 제거와 경로 파라미터 보존을 동시에 만족하는 형태가 필요하다.

### 가정 / 불확실성
- `context7` MCP는 현재 세션에 노출되지 않아 저장소 코드 분석으로 대체했다.
