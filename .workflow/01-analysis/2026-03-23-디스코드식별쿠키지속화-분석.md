# 분석 (Analysis)

## 요약
- 요청 내용: 최초 진입 시 `x-discord-id` 헤더로 식별한 값을 쿠키에 저장하고, 이후 새로고침/후속 API 요청에서는 쿠키를 fallback으로 사용하도록 인증 흐름을 개선
- 범위: 서버 인증 유틸, 요청 헤더 읽기, 쿠키 저장/조회 규칙, `/users/me`를 포함한 공통 API 인증 흐름
- 구현 가능성: 가능

## 요구사항 요약
- 디스코드 버튼으로 열린 최초 요청에서 전달된 `x-discord-id`를 서버가 인식해야 한다.
- 해당 식별값은 최초 요청 이후에도 유지되어야 하므로 서버가 쿠키에 저장해야 한다.
- 후속 요청에서는 헤더가 없더라도 쿠키 값으로 동일 사용자를 식별해야 한다.
- 모든 API가 동일 규칙을 사용하도록 공통 인증 유틸에 반영해야 한다.

## 현재 상태
### 관련 파일
- `src/server/utils/auth.ts`: 모든 보호 API가 사용하는 공통 인증 헬퍼다. 현재는 헤더의 `x-discord-id`만 읽고, 개발 환경에서는 `ADMIN_DISCORD_ID`를 fallback으로 사용한다.
- `src/server/utils/request-header.ts`: 요청 헤더를 읽는 유틸이며 `getDiscordId(event)`를 제공한다.
- `src/server/utils/access.ts`: 페이지 접근 허용 여부를 `authHelper(event)` 결과로 판단한다.
- `src/server/api/users/me.get.ts`: `authHelper(event)` 결과의 `user`를 그대로 반환하므로, 식별 지속화가 적용되면 헤더 없이도 `myInfo` 조회가 가능해진다.
- `src/app/utils/api-client.ts`: 클라이언트 API 호출은 쿠키를 자동 첨부할 수 있지만, 현재 `x-discord-id` 헤더는 직접 붙이지 않는다.

### 현재 로직 분석
- 현재 인증 흐름은 요청마다 `x-discord-id` 헤더가 존재해야 사용자를 찾는다.
- 최초 브라우저 진입 시 헤더가 있더라도, 이후 브라우저 내부의 Axios 요청이나 새로고침 재진입에서 같은 헤더가 자동 보장되지 않는다.
- 따라서 `/users/me` 같은 앱 공통 API는 헤더가 없는 시점에 `USER_NOT_FOUND` 또는 `REQUIRE_DISCORD_ID`로 떨어질 수 있다.
- 개발 환경에서는 `ADMIN_DISCORD_ID`가 fallback이지만, 이것은 실제 사용자의 식별 지속화 문제를 해결하지 못한다.

## 영향도 평가
### 직접 영향
- `authHelper(event)`의 식별 우선순위가 바뀐다.
- 서버가 헤더 값으로 쿠키를 갱신하는 부수 효과가 추가된다.
- `/users/me`와 모든 `authHelper` 사용 API가 쿠키 fallback 혜택을 받는다.

### 간접 영향
- 접근 미들웨어와 API 인증 로직이 같은 쿠키를 참조하게 되므로 브라우저 재진입 경험이 일관된다.
- 잘못된 쿠키가 남아 있을 때 사용자 조회 실패 처리와 쿠키 정리 정책이 필요할 수 있다.
- 운영 환경에서는 `secure`, `httpOnly`, `sameSite` 같은 쿠키 정책을 명확히 해야 한다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 헤더와 쿠키 값이 다를 때 어떤 값을 신뢰할지 정해야 한다.
- 사용자 조회 실패 시 stale 쿠키를 지울지 유지할지 정책이 필요하다.
- 개발 환경 fallback(`ADMIN_DISCORD_ID`)과 쿠키 fallback의 우선순위를 충돌 없이 정리해야 한다.

### 가정 / 불확실성
- 프런트의 Axios 요청은 same-origin 기준으로 쿠키를 자동 전송한다고 가정한다.
- 최초 디스코드 진입 요청에 실제로 `x-discord-id` 헤더가 포함된다는 전제는 마스터 설명을 기준으로 한다.
