# 분석 (Analysis)

## 요약
- 요청 내용: `md` 폴더 문서를 현재 사양 기준으로 갱신
- 범위: `md/api.md`, `md/db-table-dto-reference.md`, `md/dto-migration-guide.md`, 권한 계획 문서
- 구현 가능성: 가능

## 요구사항 요약
- UI와 API 연동 전에 참고 문서가 현재 코드와 같은 명칭/경로/타입을 설명해야 한다.
- `users` 기준 옛 문서, 과거 마이그레이션 상태 설명, 현재와 다른 권한 정책을 정리해야 한다.
- API 문서는 프런트가 바로 붙일 수 있는 수준의 공통 응답 형식과 현재 경로를 포함해야 한다.

## 현재 상태
### 관련 파일
- `md/api.md`: API 전체 스펙 문서이나 `users` 기준 경로와 오래된 설명이 섞여 있었음
- `md/db-table-dto-reference.md`: 테이블/DTO 참조 문서이나 `User*Dto` 기준 설명이 남아 있었음
- `md/dto-migration-guide.md`: 과거 작업 로그 중심이라 현재 기준 문서로 보기 어려웠음
- `md/2026-03-18-admin-permission-expansion-plan.md`: 계획 문서가 현재 권한 정책과 일부 어긋나 있었음
- `src/server/api/*`: 현재 실제 엔드포인트 기준
- `src/server/types/dto/*`: 현재 DTO 분리 기준
- `src/app/composables/players/*`: 프런트에서 실제 연동 중인 API 경로 기준

### 현재 로직 분석
- 현재 서버 API는 `/api/players` 기준으로 이미 전환되어 있음
- 인증은 `X-Discord-ID` 헤더뿐 아니라 `discord_id` 쿠키도 사용함
- 목록 응답은 `BaseResponse.page()` + `ListDataType` 구조를 사용함
- `docs` DTO만 아직 배럴 본문에 남아 있고, 나머지는 도메인 파일 분리가 끝난 상태임

## 영향도 평가
### 직접 영향
- `md` 폴더 문서 참조 경로와 명칭
- API/UI 연동 시 참고할 인증, 응답, DTO 구조 설명

### 간접 영향
- 이후 작업자가 과거 `users` 명칭을 다시 사용하는 실수를 줄일 수 있음
- 권한 정책 문서가 최신 코드와 맞아야 QA/수동 검증 기준도 정리됨

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 저장소에 이미 대규모 리네이밍 변경이 있어 문서가 어떤 시점 기준인지 혼동되기 쉬움
- 모든 API 세부 필드를 완전 복제하기보다 연동에 필요한 현재 기준만 남기는 재구성이 더 적절함

### 가정 / 불확실성
- `context7 MCP`는 현재 세션 도구에 노출되지 않아 로컬 코드 기준으로 문서를 갱신함
