# 분석 (Analysis)

## 요약
- 요청 내용: 캐릭터 생성/수정 폼에 `useForm`을 적용한다.
- 범위: `CharacterCreateForm.vue`, `CharacterEditorForm.vue`, `character.store.ts`
- 구현 가능성: 가능

## 요구사항 요약
- 캐릭터 생성 폼과 수정 폼의 기본 입력 상태를 스토어 직접 바인딩 대신 `vee-validate`의 `useForm`으로 관리한다.
- 기존 생성/수정 요청 payload 구조와 캠페인 목록 로드, 클래스 입력 추가/삭제 흐름은 유지한다.
- 페이지 래핑 구조는 유지하고 렌더링 로직은 기존 컴포넌트 내부에서만 수정한다.

## 현재 상태
### 관련 파일
- `src/app/components/characters/CharacterCreateForm.vue`
  - 기본 입력 필드를 `characterStore.characterForm`에 직접 바인딩하고 제출 상태도 스토어에서 관리한다.
- `src/app/components/characters/CharacterEditorForm.vue`
  - 수정 폼도 동일하게 스토어 폼 상태를 직접 사용하며 캐릭터 상세 로드 후 스토어에 값을 동기화한다.
- `src/app/stores/character.store.ts`
  - 빈 폼 생성, 상세 응답 기반 폼 동기화, 제출 payload 생성 로직을 모두 스토어 내부 상태 기준으로 제공한다.
- `src/app/components/players/NewPlayerForm.vue`
  - 프로젝트 내 `useForm` 기준 예시로, `name` 기반 필드 연결과 `handleSubmit`, `resetForm`, `isSubmitting` 사용 패턴을 보여 준다.
- `src/app/types/manual-app-imports.d.ts`
  - `useForm`, `toTypedSchema`, `Field`는 오토임포트/전역 선언 대상이라 직접 import 하면 안 된다.

### 현재 로직 분석
- 캐릭터 폼은 스토어 ref를 통해 입력 상태를 공유하지만 실제 사용처는 생성/수정 컴포넌트 두 곳에 국한된다.
- 공용 `FormInput`, `FormSelect`는 `name` 속성이 있으면 `Field`와 연결되므로 `useForm` 전환이 가능하다.
- 생성 화면의 클래스 반복 입력은 `useFieldArray` 패턴이 아직 도입되지 않았고 현재 스토어 유틸리티가 안정적으로 동작한다.
- 현재 세션에서 `context7` MCP 서버는 등록되어 있지 않아 검색을 수행할 수 없었다.

## 영향도 평가
### 직접 영향
- `src/app/components/characters/CharacterCreateForm.vue`
- `src/app/components/characters/CharacterEditorForm.vue`
- `src/app/stores/character.store.ts`

### 간접 영향
- 제출 버튼 로딩 상태가 스토어 플래그에서 `useForm.isSubmitting`으로 이동한다.
- 수정 화면의 초기화 동작은 스토어 동기화 대신 `resetForm` 기반으로 바뀐다.
- 클래스 입력 배열은 기존 스토어 상태를 유지하므로 메인 폼과 클래스 입력의 상태 출처가 분리된다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 수정 화면에서 상세 응답 도착 시 `useForm` 값으로 재동기화하는 시점을 안정적으로 맞춰야 한다.
- 스토어 기반 payload 빌더를 함수형 입력값 기반으로 바꿔도 기존 CUD 컴포저블과 타입 정합성이 유지되어야 한다.

### 가정 / 불확실성
- 이번 범위의 `useForm` 적용은 기본 캐릭터 입력 필드에 한정하고, 동적 클래스 배열은 기존 스토어 로직을 유지한다.
- `context7` MCP 미등록 상태는 이번 세션 한정일 수 있으므로 문서에 한계로 남긴다.
