# 분석 (Analysis)

## 요약
- 요청 내용: 컴포저블과 `src/server/api` 응답 타입을 1:1로 대조하고, 불필요한 유니온 응답 타입을 실제 API 계약 그대로 단순화하며 `players` 패턴처럼 파일당 하나의 컴포저블 구조로 정리
- 범위: 기존 묶음 파일 기준 `campaigns`, `characters`, `sessions`, `currency-transactions` 도메인 컴포저블 및 대응 서버 API
- 구현 가능성: 가능

## 요구사항 요약
- 컴포저블이 사용하는 제네릭 응답 타입이 실제 서버 응답과 일치해야 한다.
- 동일 도메인 안에서 생성 API와 상세/수정 API의 응답 래퍼가 다르더라도, 공용 유니온으로 뭉개지 말고 엔드포인트별 정확 타입을 사용해야 한다.
- 각 도메인 컴포저블은 `players` 디렉터리처럼 파일당 하나의 컴포저블만 가져야 한다.
- 목록/단건 추출 유틸은 남길 수 있으나, "배열일 수도 있고 단건일 수도 있고 페이지일 수도 있다"는 식의 과도한 유니온 선언은 제거해야 한다.
- `sessions/logs`처럼 API 이름과 실제 응답 데이터가 어긋나는 구간은 현재 서버 계약을 우선 기준으로 삼아 프론트 타입을 맞춰야 한다.

## 현재 상태
### 관련 파일
- `src/app/composables/campaigns/useCampaigns.ts`: 캠페인 목록/상세/생성/수정/멤버/캐릭터 연동 컴포저블. `CampaignListResponse`, `CampaignItemResponse` 유니온으로 GET/POST/PUT/DELETE 전부를 포괄
- `src/app/composables/characters/useCharacters.ts`: 캐릭터 목록/상세/생성/수정/클래스 관리 컴포저블. 캠페인 도메인과 동일한 방식의 유니온 응답 타입 사용
- `src/app/composables/sessions/useSessions.ts`: 세션 목록/상세/세션 로그/참가자 관리 컴포저블. `SessionLogResponse`가 `SessionLogOutDto`, `SessionOutDto`, `ListData<...>`, 배열을 모두 포함하는 가장 넓은 유니온
- `src/app/composables/currency-transactions/useCurrencyTransactions.ts`: 거래 목록/상세/생성/수정/삭제 컴포저블. 실제 API는 명확하지만 유니온으로 선언돼 있음
- `src/app/composables/query/useDomainResource.ts`: `extractItemPayload`, `extractItemById`, `extractListPayload` 등 응답 정규화 유틸 제공. 유니온 제거 후에도 보조적으로 활용 가능
- `src/server/api/campaigns/index.get.ts`: `BaseApiResponse.page(ListData<CampaignOutDto>)`
- `src/server/api/campaigns/index.post.ts`: `BaseApiResponse.data(CampaignOutDto)`
- `src/server/api/campaigns/[id].get.ts`, `[id].put.ts`, `[id].delete.ts`, `[id]/status.put.ts`, `[id]/members/[userId].post.ts`, `[id]/members/[userId].delete.ts`, `[campaignId]/characters/[characterId].post.ts`, `[campaignId]/characters/[characterId].delete.ts`: 모두 `BaseApiResponse.page(ListData<CampaignOutDto>)`
- `src/server/api/characters/index.get.ts`, `mine.get.ts`: `BaseApiResponse.page(ListData<CharacterOutDto>)`
- `src/server/api/characters/index.post.ts`: `BaseApiResponse.data(CharacterOutDto)`
- `src/server/api/characters/[id].get.ts`, `[id].put.ts`, `[id].delete.ts`, `[id]/status.put.ts`, `[id]/classes/index.post.ts`, `[id]/classes/[className].put.ts`, `[id]/classes/[className].delete.ts`: 모두 `BaseApiResponse.page(ListData<CharacterOutDto>)`
- `src/server/api/sessions/index.get.ts`, `mine.get.ts`: `BaseApiResponse.page(ListData<SessionOutDto>)`
- `src/server/api/sessions/index.post.ts`: `BaseApiResponse.data(SessionOutDto)`
- `src/server/api/sessions/[id].get.ts`, `[id].put.ts`, `[id].delete.ts`, `[id]/players/index.post.ts`, `[id]/players/[userId].delete.ts`: 모두 `BaseApiResponse.page(ListData<SessionOutDto>)`
- `src/server/api/sessions/[id]/logs/index.get.ts`, `[id]/logs/index.post.ts`, `[id]/logs/[logId].get.ts`, `[id]/logs/[logId].put.ts`, `[id]/logs/[logId].delete.ts`: 이름은 로그 API지만 실제로는 모두 `BaseApiResponse.page(ListData<SessionOutDto>)`
- `src/server/api/currency-transactions/index.get.ts`: `BaseApiResponse.page(ListData<CurrencyTransactionOutDto>)`
- `src/server/api/currency-transactions/index.post.ts`, `[id].get.ts`, `[id].put.ts`, `[id].delete.ts`: 모두 `BaseApiResponse.data(CurrencyTransactionOutDto)`

### 현재 로직 분석
- 캠페인/캐릭터/거래 목록 컴포저블은 실제로는 항상 `ListData<T>`를 받지만 `ListData<T> | T[] | T` 유니온을 사용하고 있다.
- 캠페인/캐릭터/세션 단건 계열은 생성 API만 `T`, 나머지는 `ListData<T>`인데, 현재는 공용 `ItemResponse = T | ListData<T>` 유니온으로 선언되어 있다.
- `sessions/logs`는 실제 서버가 로그 단건/목록을 직접 내려주지 않고, 세션을 `ListData<SessionOutDto>`로 내려주며 그 안의 `logs`를 읽도록 되어 있다. 현재 컴포저블은 이 계약을 반영하지 못하고 `SessionLogOutDto` 관련 유니온까지 포함해 타입을 넓혀 두었다.
- `useDomainResource.ts`는 다양한 응답 형태를 흡수하기 위해 설계됐지만, 현재 도메인 컴포저블의 타입 선언이 과도하게 느슨해지는 원인이기도 하다.
- 묶음 파일 구조는 기본 규칙인 "파일당 하나의 컴포저블"을 위반하므로 최종 결과에서는 제거 대상이다.

## 영향도 평가
### 직접 영향
- 도메인 컴포저블의 제네릭 타입 파라미터와 옵션 인터페이스
- 성공 콜백 시 스토어 동기화 로직
- `sessions/logs` 응답을 해석하는 타입 가드 및 추출 흐름

### 간접 영향
- 해당 컴포저블을 사용하는 페이지/컴포넌트의 콜백 시그니처가 더 정확해짐
- 잘못된 응답 타입 가정이 컴파일 단계에서 더 빨리 드러날 수 있음
- 서버 API 자체 계약은 유지하므로 런타임 동작 변화는 최소화 가능

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 동일 도메인 안에서도 엔드포인트별 응답 래퍼가 달라 공용 `ActionOptions`를 그대로 재사용하기 어렵다.
- `sessions/logs`는 명칭상 로그 응답처럼 보이지만 실제 타입은 세션 목록 페이지 응답이라, 컴포저블 내부 로직도 현재 계약에 맞게 단순화해야 한다.
- 서버 `currency-transactions/[id].get.ts`, `[id].put.ts`, `[id].delete.ts` 구현 내용이 동일하게 보이므로 타입 정비 범위와 별개로 API 구현 오류 가능성이 있으나, 이번 작업에서는 응답 타입 정합성만 우선 처리한다.

### 가정 / 불확실성
- 이번 작업에서는 서버 API의 응답 구조 자체를 바꾸지 않고, 프론트 컴포저블 타입을 현재 서버 계약에 맞추는 것으로 범위를 제한한다.
- `sessions/logs` API가 향후 진짜 `SessionLogOutDto`를 직접 반환하도록 바뀔 가능성은 있으나, 현재 저장소 기준 계약은 `ListData<SessionOutDto>`이다.
