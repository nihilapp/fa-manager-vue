# 분석 (Analysis)

## 요약
- 요청 내용: `src/app/components/players`와 연계된 커스텀 훅에서 콜백이 `BaseApiResponse.data`만 받지 않고 응답 전체를 받도록 변경
- 범위: 플레이어 전용 컴포저블과 이를 호출하는 공통 쿼리 래퍼의 성공 콜백 계약
- 구현 가능성: 가능

## 요구사항 요약
- 플레이어 관련 커스텀 훅의 `callback`은 `data` 조각이 아니라 `BaseApiResponse<T>` 전체를 받아야 한다.
- 사용자는 콜백에서 `response.data`뿐 아니라 `code`, `message`, `error` 같은 메타데이터도 사용할 수 있어야 한다.
- 플레이어 목록, 내 정보, 상세 조회, 생성, 수정 훅 모두 같은 방향으로 정리해야 한다.

## 현재 상태
### 관련 파일
- `src/app/composables/query/useApiRequest.ts`: 공통 응답 처리 유틸리티이며 현재 `handleApiResponse`가 `response.data`만 `onSuccess`에 전달한다.
- `src/app/composables/query/useGet.ts`: GET 요청 래퍼이며 `onSuccess` 시그니처가 데이터 payload 기준이다.
- `src/app/composables/query/usePost.ts`: POST 요청 래퍼이며 `onSuccess` 시그니처가 데이터 payload 기준이다.
- `src/app/composables/query/usePut.ts`: PUT 요청 래퍼이며 `onSuccess` 시그니처가 데이터 payload 기준이다.
- `src/app/composables/query/usePatch.ts`: PATCH 요청 래퍼이며 공통 계약을 공유한다.
- `src/app/composables/query/useDelete.ts`: DELETE 요청 래퍼이며 공통 계약을 공유한다.
- `src/app/composables/players/useGetPlayerList.ts`: 플레이어 목록 조회 후 스토어를 갱신하고 외부 콜백에는 `list`, `totalCnt`만 넘긴다.
- `src/app/composables/players/useGetPlayerInfo.ts`: 플레이어 상세 조회 성공 시 `PlayerOutDto`만 외부 콜백에 넘긴다.
- `src/app/composables/players/useGetMyInfo.ts`: 내 플레이어 정보 조회 성공 시 `PlayerOutDto`만 외부 콜백에 넘긴다.
- `src/app/composables/players/useCreateNewPlayer.ts`: 생성 성공 시 `PlayerOutDto`만 외부 콜백에 넘긴다.
- `src/app/composables/players/useUpdatePlayer.ts`: 수정 성공 시 `PlayerOutDto`만 외부 콜백에 넘긴다.

### 현재 로직 분석
- 공통 `handleApiResponse`는 성공 응답에서 `checkAndHandleApiError` 이후 `response.data as TData`를 `onSuccess`에 전달한다.
- 플레이어 전용 훅은 이 데이터만 전달받아 스토어를 갱신한 뒤 외부 `callback`에도 데이터만 넘긴다.
- 따라서 외부 호출자는 성공 응답의 `code`, `message`를 사용할 수 없다.

## 영향도 평가
### 직접 영향
- 공통 쿼리 래퍼 `useGet`, `usePost`, `usePut`, `usePatch`, `useDelete`의 `onSuccess` 타입
- 플레이어 전용 훅 5종의 `callback` 타입 및 내부 데이터 분해 방식

### 간접 영향
- 플레이어 외 다른 도메인이 향후 동일 공통 훅을 사용할 때도 성공 콜백에서 응답 메타데이터 접근이 가능해진다.
- 현재 저장소 검색 기준으로 `src/app` 내 명시적 `onSuccess` 사용처는 플레이어 GET 훅 3곳뿐이라 즉시 영향은 제한적이다.
- 타입 계약이 바뀌므로 공통 훅을 신규 작성하는 코드에서는 `response.data` 접근을 명시해야 한다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 공통 계약을 바꾸지 않고 플레이어 훅만 수정하면 타입 계약이 어긋나므로 공통 유틸리티부터 변경해야 한다.
- 목록 조회 훅은 기존 `list`, `totalCnt` 이원 인자를 제거하고 응답 전체를 단일 인자로 통일해야 한다.

### 가정 / 불확실성
- `context7` MCP는 현재 세션에서 사용할 수 없어 저장소 코드 기준으로 분석했다.
- 워크트리에 기존 변경사항이 많아 이번 작업은 관련 파일만 선택적으로 수정해야 한다.
