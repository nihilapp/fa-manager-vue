# 분석 (Analysis)

## 요약

- 요청 내용: 남아 있던 플레이어 상세 조회 쿼리 키 불일치를 정리
- 범위: `useGetPlayerInfo`, 연관된 `useGetPlayerSessionList`
- 구현 가능성: 가능

## 요구사항 요약

- `/players/:id` 상세 조회는 `players.me`가 아니라 `players.detail({ id })` 키를 사용해야 한다.
- 연관된 플레이어 하위 세션 조회 훅이 미완성 상태면 함께 정리해 타입 검증이 통과해야 한다.

## 현재 상태

### 관련 파일

- `src/app/composables/players/useGetPlayerInfo.ts`: `/players/${id}` 요청인데 `queryKeys.players.me({})`를 사용
- `src/app/composables/players/useGetMyInfo.ts`: `/players/me` 요청이며 `queryKeys.players.me({})` 사용, 정상 기준
- `src/app/composables/players/useGetPlayerSessionList.ts`: `queryKeys.players.sessions` 호출이 끊긴 미완성 상태
- `src/app/utils/query-keys.ts`: `players.detail`, `players.sessions` 키가 이미 정의되어 있음

### 현재 로직 분석

- 상세 조회 키가 내 정보 키와 동일하면 캐시 의미가 섞인다.
- `useGetPlayerSessionList`는 문법상 미완성이라 타입 체크 전체를 깨뜨린다.

## 영향도 평가

### 직접 영향

- 플레이어 상세 조회 캐시 키 정합성
- 플레이어 하위 세션 조회 훅의 타입 안정성

### 간접 영향

- 플레이어 상세/내정보 캐시 무효화 기준이 분리된다.
- 전체 `tsc` 검증이 다시 통과 가능해진다.

## 실행 판단

### 구현 가능성

- 가능

### 예상 기술 난제

- 별도 구조 변경 없이 키 한 줄 교체와 미완성 훅 보완으로 해결 가능

### 가정 / 불확실성

- `context7` MCP는 현재 세션에 노출되지 않아 저장소 코드 분석으로 대체했다.
