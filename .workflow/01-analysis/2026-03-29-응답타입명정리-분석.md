# 분석 (Analysis)

## 요약
- 요청 내용: `src/app`과 `src/server` 전반의 `BaseResponse` 계열 명칭을 `BaseApiResponse`로 통일하고 앱의 `ApiErrorResponse`도 `BaseApiResponse<null>`로 정리
- 범위: 서버 응답 타입 정의, 서버 응답 헬퍼 사용처, 앱 공용 요청 훅, 앱/서버 수동 오토임포트 선언
- 구현 가능성: 가능

## 요구사항 요약
- 서버에서 사용하는 응답 타입과 응답 생성 헬퍼의 기준 명칭을 `BaseApiResponse`로 바꿔야 한다.
- 앱에서 사용하는 공용 응답 타입도 동일한 이름과 동일한 구조를 사용해야 한다.
- 앱의 `ApiErrorResponse`는 독립 구조가 아니라 `BaseApiResponse<null>` 별칭으로 정리해야 한다.
- 수동 오토임포트 선언도 새 명칭 기준으로 갱신해 호출부에서 추가 import 없이 같은 이름을 사용할 수 있어야 한다.

## 현재 상태
### 관련 파일
- `PRD/PRD.md`: 프론트와 서버가 같은 도메인 계약을 공유해야 하는 프로젝트 기준 문서다.
- `PRD/Coding-Rules.md`: 통합 풀스택 구조와 타입 안정성 유지 기준 문서다.
- `src/server/types/response.types.ts`: 서버 응답 인터페이스 `BaseResponseType`, 리스트 응답 보조 타입을 정의한다.
- `src/server/utils/base-response.ts`: 서버 API가 공통으로 쓰는 응답 생성 헬퍼 `BaseResponse`를 제공한다.
- `src/server/types/manual-server-imports.d.ts`: 서버 전역 타입 오토임포트 별칭을 수동 선언한다.
- `src/server/utils/auth.ts`, `src/server/utils/access.ts`: 서버 유틸리티에서 응답 타입을 직접 타입 주석에 사용한다.
- `src/server/api/**`: 대부분의 엔드포인트가 `BaseResponse.data/page/error` 호출에 의존한다.
- `src/app/composables/query/useApiRequest.ts`: 앱 공용 API 응답 타입 `BaseApiResponse`와 `ApiErrorResponse`를 정의한다.
- `src/app/types/manual-app-imports.d.ts`: 앱 전역 타입 오토임포트 선언을 제공한다.
- `src/app/composables/query/useGet.ts`, `usePost.ts`, `usePut.ts`, `usePatch.ts`, `useDelete.ts`: 앱 요청 훅이 공용 응답 타입과 에러 타입을 광범위하게 사용한다.

### 현재 로직 분석
- 서버는 실제 응답 객체 구조를 `BaseResponseType<T>`로 정의하고, 런타임 생성은 `BaseResponse` 클래스가 담당한다.
- 앱은 이미 `BaseApiResponse`라는 이름을 쓰고 있지만 내부적으로는 전역 `BaseResponse<T>` 별칭과 `ApiErrorResponse` 유니온으로 조합하고 있다.
- 따라서 현재는 이름이 다를 뿐 아니라 앱 응답 타입이 서버 정의와 직접 연결되지 않는다.
- 수동 오토임포트 선언에도 `BaseResponseType`, `BaseResponse`가 남아 있어 서버와 앱의 전역 타입 명칭이 불일치한다.

## 영향도 평가
### 직접 영향
- `src/server/types/response.types.ts`의 인터페이스 명칭과 데이터 필드 제네릭 계약
- `src/server/utils/base-response.ts`의 클래스명과 반환 타입
- `src/server/api/**`, `src/server/utils/**`의 `BaseResponse` 호출부와 타입 주석
- `src/app/composables/query/useApiRequest.ts`의 응답/에러 타입 정의
- `src/app/types/manual-app-imports.d.ts`, `src/server/types/manual-server-imports.d.ts`의 전역 타입 별칭

### 간접 영향
- 앱의 `useGet`/`usePost` 계열 훅에서 에러 응답과 성공 응답을 같은 공통 타입으로 다루는 방식이 단순화된다.
- 서버 자동 오토임포트로 `BaseResponse`를 참조하던 파일이 전부 `BaseApiResponse`로 바뀌므로 검색성과 일관성이 좋아진다.
- 응답 타입의 `data` 필드가 성공/실패를 모두 수용하도록 넓어지면 일부 처리부에서 추가 타입 단언이 유지될 수 있다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 앱과 서버가 같은 인터페이스를 바라보게 만들면서도 기존 호출부를 최대한 깨지 않도록 `data` 제네릭 계약을 조정해야 한다.
- 서버 전역 타입 별칭에서 기존 `responseTime` 확장 계약을 유지할지 결정해야 한다.
- 사용처가 많아 일괄 치환 후 타입체크로 누락 여부를 확인해야 한다.

### 가정 / 불확실성
- context7 MCP 선검색 지침이 있으나 현재 세션에는 `context7` 서버가 연결되어 있지 않아 로컬 코드베이스 기준으로 분석했다.
- `BaseResponse` 문자열은 `src/app`, `src/server` 외 문서(`md/`, `PRD/`)에도 남아 있으나 이번 요청 범위는 코드와 오토임포트 정리에 한정한다.
