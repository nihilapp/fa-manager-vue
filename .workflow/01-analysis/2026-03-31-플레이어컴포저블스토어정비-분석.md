# 분석 (Analysis)

## 요약
- 요청 내용: players API 대응 컴포저블 잔여분을 분석하고, 스토어 액션 중심 저장 구조로 정리
- 범위: `src/app/composables/players`, `src/app/stores/player.store.ts`, `src/app/stores/session.store.ts`, players 관련 페이지/컴포넌트
- 구현 가능성: 가능

## 요구사항 요약
- players 도메인 컴포저블은 API 응답 성공 시 스토어 액션으로 데이터를 저장해야 한다.
- 모든 스토어는 기본적으로 목록 변수, 페이지 정보 변수, 개별 정보 변수를 가져야 한다.
- 모든 스토어는 목록 세팅, 개별 정보 세팅, 목록 초기화, 개별 정보 초기화, 페이지 정보 세팅 액션을 가져야 한다.
- campaign, session, character처럼 플레이어별 파생 목록이 필요한 도메인은 기본 축 외에 `player별 목록/페이지 정보` 축을 둘 수 있다.
- 현재 작업은 players/session 스토어와 players 컴포저블을 우선 정리하고, 이후 다른 도메인으로 확장 가능한 패턴을 만드는 데 초점을 둔다.

## 현재 상태
### 관련 파일
- `src/app/stores/player.store.ts`: 플레이어 목록, 상세, 내 정보를 보관하지만 액션 명명과 책임이 일관되지 않음
- `src/app/stores/session.store.ts`: 기본 세션 목록과 플레이어별 세션 목록을 함께 관리하나 초기화 범위가 뭉쳐 있음
- `src/app/composables/players/useGetPlayerList.ts`: 목록 조회 후 스토어 저장 수행
- `src/app/composables/players/useGetPlayerInfo.ts`: 상세 조회 후 스토어 저장 수행
- `src/app/composables/players/useGetMyInfo.ts`: 내 정보 조회 후 스토어 저장 수행
- `src/app/composables/players/useGetPlayerSessionList.ts`: 플레이어별 세션 목록 조회 후 세션 스토어 저장 수행
- `src/app/composables/players/useCreateNewPlayer.ts`: 생성 후 목록 재조회만 수행하고 스토어 직접 저장 규칙은 약함
- `src/app/composables/players/useUpdatePlayer.ts`: 수정 후 목록 재조회만 수행하고 상세/개별 정보 갱신은 없음
- `src/app/utils/query-keys.ts`: players의 `check`, `campaigns`, `sessions` 키는 있으나 일부는 실제 컴포저블이 없음
- `src/server/api/players/*`: 프런트에서 아직 대응하지 않은 players API 목록 존재

### 현재 로직 분석
- 현재 players 컴포저블은 조회형 일부만 존재하며, `GET /players/check`, `GET /players/:id/campaigns`, `PUT /players/me`, `DELETE /players/me`, `DELETE /players/:id` 대응이 비어 있다.
- `player.store.ts`는 `playerListCount`, `myInfo`처럼 기본 축 밖 상태가 섞여 있어 패턴화가 어렵다.
- `session.store.ts`는 플레이어별 세션 목록까지 관리하지만 `clearSessionList`가 기본 목록과 파생 목록을 동시에 비워 세밀한 제어가 어렵다.
- 페이지/컴포넌트는 스토어 값을 읽지만, 어떤 액션이 어떤 상태를 책임지는지 추론해야 하는 구조다.

- `src/app/stores/session.store.ts`
- `src/app/composables/players/*.ts`
- players 상세/목록 관련 페이지와 컴포넌트

### 간접 영향
- 향후 campaign/session/character 스토어 설계 패턴
- `query-keys` 사용 규칙
- 내 정보 기반 네비게이션(`CommonHeader.vue`, `common.store.ts`)의 참조 상태명

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `myInfo`를 별도 축으로 둘지 `playerInfo` 재사용 축으로 합칠지 명확한 기준이 필요하다.
- 플레이어별 파생 목록은 기본 축과 동일한 이름 패턴을 유지하면서도 충돌 없이 분리해야 한다.
- 생성/수정/삭제 뮤테이션에서 직접 스토어 갱신과 재조회 중 어느 쪽을 표준으로 둘지 정해야 한다.

### 가정 / 불확실성
- 현재 범위는 players 중심 정비이며 campaign/session/character 전체 컴포저블 작성은 포함하지 않는다.
- `GET /players/check`의 반환 타입은 접근 체크용 공통 응답으로 가정하되, 실제 사용처가 없어 최소 래핑만 우선 구현한다.
