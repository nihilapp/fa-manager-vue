# 분석 (Analysis)

## 요약
- 요청 내용: `players/detail/[id]` 페이지 진입 시 발생하는 `Cannot read properties of null (reading 'name')` 오류의 원인을 파악한다
- 범위: 플레이어 상세 페이지 setup 흐름, players 조회 컴포저블, player/session 스토어, 메타 설정 타이밍
- 구현 가능성: 가능

## 요구사항 요약
- 플레이어 상세 페이지 진입 직후 발생하는 런타임 예외의 직접 원인을 식별해야 한다
- 단순 증상 설명이 아니라, 어떤 시점에 어떤 값이 null이며 왜 그 값이 아직 준비되지 않았는지 코드 흐름으로 설명할 수 있어야 한다
- 현재 활성 작업인 플레이어 컴포저블/스토어 정비와 연관된 회귀 가능성도 함께 확인해야 한다

## 현재 상태
### 관련 파일
- `src/app/pages/players/detail/[id].vue`: 플레이어 상세 페이지 엔트리이며 `useGetPlayerInfo`, `useGetPlayerSessionList`, `useSetMeta` 호출을 수행한다
- `src/app/composables/players/useGetPlayerInfo.ts`: `/players/:id` GET 성공 시 `playerStore.playerInfo`를 채우고 실패 시 비운다
- `src/app/composables/players/useGetPlayerSessionList.ts`: `/players/:id/sessions` GET 성공 시 세션 목록과 페이지 정보를 저장한다
- `src/app/composables/query/useGet.ts`: 요청은 자동 실행되지 않으며 `execute()`가 호출될 때만 실제 GET 요청이 발생한다
- `src/app/stores/player.store.ts`: `playerInfo`의 초기값은 `null`이다
- `src/app/components/players/PlayerDetail.vue`: 템플릿에서 `v-if="playerInfo"`로 null 상태를 방어하지만, 페이지 setup 이전의 메타 계산은 방어하지 못한다

### 현재 로직 분석
- `src/app/pages/players/detail/[id].vue`는 setup 시점에 `useGetPlayerInfo(+route.params.id!)`와 `useGetPlayerSessionList(+route.params.id!)`를 준비한다
- 같은 setup 내부에서 `useSetMeta({ title: \`플레이어 [${playerInfo.value!.name}] 정보\` })`를 즉시 호출한다
- 실제 데이터 요청은 `onMounted` 내부의 `playerInfoReq.execute()`와 `playerSessionListReq.execute()`에서 시작된다
- `useGet.ts` 구현상 `execute()` 이전에는 네트워크 요청이 발생하지 않으므로, setup 시점의 `playerInfo.value`는 기본값 `null`이다
- 따라서 메타 타이틀 생성 시 `playerInfo.value!.name` 평가가 먼저 일어나며, 여기서 null dereference가 발생한다

## 영향도 평가
### 직접 영향
- `src/app/pages/players/detail/[id].vue`의 13번째 줄 강제 단언(`playerInfo.value!.name`)이 직접적인 예외 발생 지점이다
- 상세 페이지는 mount 전에 setup 단계에서 중단되므로, 뒤이어 실행될 `playerInfoReq.execute()`와 `playerSessionListReq.execute()`까지 도달하지 못할 수 있다

### 간접 영향
- 현재 players 상세 페이지는 템플릿 레벨에서는 null 방어가 있으나, 메타 설정만 예외적으로 선행 평가되어 회귀가 발생한다
- 동일 패턴으로 "스토어 값 초기값은 null인데 setup에서 non-null assertion으로 메타/계산값 생성"을 사용하는 다른 상세 페이지도 잠재 리스크가 있다
- `useGet`가 자동 실행형이 아닌 수동 실행형으로 통일된 현재 구조에서는, mount 이전 데이터를 전제하는 모든 코드가 동일한 문제를 일으킬 수 있다

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 수정 자체는 단순하지만, 메타 타이틀을 빈 값/기본값으로 둘지, `watch` 또는 `computed` 기반으로 데이터 도착 후 갱신할지 설계 기준이 필요하다
- 플레이어 상세 페이지 외에 유사 패턴이 있는지 점검 범위를 정해야 한다

### 가정 / 불확실성
- 현재 수집한 로그와 코드 기준으로 직접 원인은 확정 가능하다
- 실제 브라우저에서 예외가 발생한 뒤 `onMounted`가 일부 상황에서 전혀 실행되지 않는지 여부는 별도 재현 검증이 필요하다
- context7 MCP 사용이 지침상 요구되지만, 이번 세션에서는 관련 MCP 리소스가 노출되지 않아 로컬 코드 기준 분석으로 대체했다
