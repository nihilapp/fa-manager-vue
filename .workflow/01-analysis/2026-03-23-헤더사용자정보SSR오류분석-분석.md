# 분석 (Analysis)

## 요약
- 요청 내용: SSR 중 발생한 `Cannot read properties of null (reading 'name')` 오류 원인 파악
- 범위: 헤더 사용자 정보 렌더링, 사용자 스토어 초기 상태, 앱 초기 데이터 패칭 시점
- 구현 가능성: 가능

## 요구사항 요약
- 헤더에서 사용자 이름을 렌더링할 때 `null` 접근이 발생하는 이유를 코드 흐름 기준으로 식별해야 한다.
- 단순 예외 위치 확인이 아니라, SSR 시점과 사용자 정보 조회 시점 간 불일치를 분리해 설명해야 한다.
- 이후 수정 단계에서 바로 설계로 넘어갈 수 있도록 직접 원인과 간접 영향 범위를 명시해야 한다.

## 현재 상태
### 관련 파일
- `src/app/components/layout/common-layout/CommonHeader.vue`: 전역 헤더 UI를 렌더링하며 사용자 아이콘과 `myInfo!.name`를 직접 출력한다. `myInfo`가 `null`일 수 있다는 방어가 없다.
- `src/app/stores/user.store.ts`: `myInfo`를 `ref<UserOutDto | null>(null)`로 선언하고 `getMyInfo()` 성공 시에만 값을 채운다. 실패 시 다시 `null`로 유지한다.
- `src/app/app.vue`: `userStore.getMyInfo()`를 `onMounted` 안에서만 호출한다. 따라서 서버 렌더링 중에는 `myInfo`가 초기값 `null` 상태로 남는다.
- `src/app/layouts/default.vue`: 기본 레이아웃에서 `CommonHeader`를 항상 렌더링한다. 사용자 정보 준비 여부와 무관하게 SSR 대상이 된다.
- `src/app/stores/common.store.ts`: `isAdmin.value`를 스냅샷처럼 사용해 `navItems`를 초기화한다. 이번 예외의 직접 원인은 아니지만, 사용자 정보가 늦게 채워질 때 네비게이션 공개 여부가 반응형으로 갱신되지 않을 가능성이 있다.

### 현재 로직 분석
- SSR 시작 시 `default.vue`가 `CommonHeader`를 포함해 렌더링된다.
- `CommonHeader.vue`는 Pinia의 `myInfo` ref를 읽고 템플릿에서 `myInfo!.name`를 바로 평가한다.
- 하지만 `myInfo`는 `user.store.ts`에서 기본값이 `null`이며, 이를 채우는 `getMyInfo()` 호출은 `app.vue`의 `onMounted`에서만 실행된다.
- `onMounted`는 클라이언트 마운트 이후 훅이므로 서버 렌더 단계에서는 실행되지 않는다.
- 결과적으로 SSR 중 `myInfo`는 `null`이고, non-null assertion `!`는 타입스크립트 타입만 억제할 뿐 런타임 안전성을 제공하지 못해 `null.name` 접근 예외가 발생한다.

## 영향도 평가
### 직접 영향
- `CommonHeader.vue` SSR 렌더링 실패로 인해 기본 레이아웃 전체가 깨질 수 있다.
- 인증된 사용자 정보를 기대하는 다른 공용 레이아웃 컴포넌트도 동일 패턴이면 같은 종류의 SSR 예외를 유발할 수 있다.

### 간접 영향
- `user.store.ts`의 사용자 정보 조회 시점이 클라이언트 전용이어서, SSR 기반 초기 렌더 품질과 인증 상태 일관성이 낮다.
- `common.store.ts`의 `navItems`는 `isAdmin.value` 초기값을 정적으로 캡처하므로, 사용자 정보 조회 후에도 관리자 메뉴 노출 상태가 기대와 다를 수 있다.
- `/users/me` 호출 실패 시 클라이언트에서만 `/block`으로 이동하므로, 서버 단계에서는 예외 대신 빈 상태 렌더링 전략이 필요하다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 수정 방향에 따라 두 갈래가 있다.
- 단기 방어: 헤더 템플릿에서 `myInfo` 존재 여부를 조건부 렌더링해 SSR 예외를 차단한다.
- 구조 개선: 사용자 정보 조회를 SSR 가능 시점(`callOnce`, `useAsyncData`, 레이아웃/플러그인 초기화 등)으로 이동하고, 관련 스토어가 늦은 초기화에도 반응하도록 정리해야 한다.
- 관리자 메뉴 공개 여부도 사용자 정보 반응성 기준으로 다시 설계할 필요가 있다.

### 가정 / 불확실성
- `useGet()`가 SSR에서도 동작 가능한 구현인지는 아직 확인하지 않았다. 다만 현재 호출 위치가 `onMounted`이므로 이번 예외 원인 판단에는 영향이 없다.
- `/block` 경로의 헤더 노출 정책은 아직 확인하지 않았다. 수정 시 레이아웃 분기 또는 헤더 내부 방어 중 어느 방식이 더 적절한지는 설계 단계에서 확정해야 한다.
