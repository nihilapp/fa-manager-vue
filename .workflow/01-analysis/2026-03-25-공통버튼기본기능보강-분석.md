# 분석 (Analysis)

## 요약
- 요청 내용: `src/app/components/common/Button.vue`를 공통 버튼으로 사용하기 위한 기본 기능만 보강하고, 디자인은 호출부에서 커스터마이즈 가능하게 유지
- 범위: `src/app/components/common/Button.vue` 중심, 관련 스타일 유틸과 아이콘 사용 계약 확인 포함
- 구현 가능성: 가능

## 요구사항 요약
- 버튼 컴포넌트는 기본적인 접근성, 타입 지정, 비활성화 처리, 클릭 이벤트 전달, 슬롯/클래스 기반 커스터마이즈를 지원해야 한다.
- 디자인 시스템을 강제하기보다 공통 베이스만 제공하고 실제 시각 스타일은 `class` 또는 명시적 커스텀 클래스 전달로 확장 가능해야 한다.
- 아이콘 버튼과 텍스트+아이콘 버튼을 모두 수용하되, 현재처럼 템플릿을 이원화하기보다 하나의 계약으로 정리하는 것이 유지보수에 유리하다.

## 현재 상태
### 관련 파일
- `src/app/components/common/Button.vue`: 현재 공통 버튼 컴포넌트다. `iconName`, `label`, `run` emit만 제공하며 클릭 이벤트 원본, 슬롯, 로딩 상태, `aria` 보조 속성 전달, 사용자 정의 클래스 분리 등이 부족하다.
- `src/app/assets/styles/tailwind.css`: 프로젝트에서 실제 사용 가능한 유틸리티와 버튼 기본 동작(`button`의 cursor, disabled 상태)을 정의한다.
- `src/app/assets/styles/utility/button.css`: `button-base`, `button-disabled`, `button-outline-*`, `button-normal-*`, `button-ghost-*`, `button-focus-*` 등 버튼 관련 커스텀 유틸리티를 제공한다.

### 현재 로직 분석
- 현재 컴포넌트는 `label` 존재 여부로 버튼 구조를 분기하고 있다.
- 아이콘 클래스에 `buttonClass`가 잘못 재사용되는 구간이 있어 의도한 아이콘 스타일 분리가 되지 않는다.
- `button`의 기본 `type`은 옵셔널이지만 기본값이 없어 폼 내부 사용 시 의도치 않게 submit 동작을 만들 수 있다.
- 클릭 핸들러는 무조건 `run`만 emit하고 브라우저의 `MouseEvent`를 노출하지 않아 호출부에서 세부 제어가 어렵다.
- `disabled` 외 상태가 없어서 비동기 처리 중 중복 클릭 방지 같은 공통 기능을 컴포넌트 차원에서 제공하지 못한다.
- 기본 스타일 클래스가 비어 있어 공통 버튼으로서 레이아웃 최소 계약도 정의되지 않았다.

## 영향도 평가
### 직접 영향
- `src/app/components/common/Button.vue`의 props, emits, 템플릿 구조, 클래스 조합 로직 수정
- 버튼 기본 속성 계약 재정의로 기존 또는 미래 사용처의 적용 방식 정리

### 간접 영향
- 향후 버튼 사용처가 `run` 이벤트 또는 `label`/`ariaLabel` 계약에 의존할 경우 호환성 고려가 필요하다.
- 프로젝트는 Nuxt 오토임포트 규칙을 사용하므로 `cn`, `GetIcon` 등은 직접 import 없이 유지해야 한다.
- 커스텀 스타일은 `tailwind.css`와 `button.css`에 정의된 유틸리티 범위 안에서만 안전하게 제공해야 한다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 공통 버튼답게 기능을 늘리되 디자인 책임까지 과도하게 끌어오지 않도록 props 범위를 절제해야 한다.
- 현재 사용처가 많지 않더라도 향후 확장성을 위해 기본값과 호환 emit 계약을 함께 고려해야 한다.

### 가정 / 불확실성
- `context7 MCP` 사용이 지침상 요구되지만 현재 세션에서 해당 도구가 제공되지 않아 로컬 코드와 문서 기준으로 분석을 수행했다.
- 현재 `Button.vue`의 실제 사용처는 검색 결과 명확히 확인되지 않아, 이번 작업은 미래 공통 사용성을 기준으로 최소 계약을 정의하는 방향으로 설계해야 한다.
