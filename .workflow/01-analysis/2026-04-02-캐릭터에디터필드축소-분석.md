# 분석 (Analysis)

## 요약
- 요청 내용: 캐릭터 생성/수정 UI에서 이름, 상태, 시작 레벨, 시작 자금, 클래스 정보만 남기고 나머지 필드를 제거한다.
- 범위: `CharacterEditorForm` UI, 폼 상태 타입, 제출 payload
- 구현 가능성: 가능

## 요구사항 요약
- 생성/수정 화면을 더 간략하게 만든다.
- 유지 대상은 이름, 상태, 시작 레벨, 시작 자금, 클래스 정보다.
- 클래스 정보는 별도 테이블 기반이므로 현재처럼 요약 정보만 남겨도 된다.
- 나머지 식별/능력치/장비/메모 필드는 모두 제거한다.

## 현재 상태
### 관련 파일
- `src/app/components/characters/CharacterEditorForm.vue`
  - 현재는 기본 정보, 클래스, 능력치, 특성, 자금, 장비, 요구치 메모까지 포함한 대형 폼이다.
- `src/app/pages/characters/add-character.vue`
  - 공용 폼 생성 모드 래퍼다.
- `src/app/pages/characters/edit/[id].vue`
  - 공용 폼 수정 모드 래퍼다.
- `src/server/types/dto/character.dto.ts`
  - 시작 레벨, 상태, 자금, 클래스 정보를 수정 화면 기본값에 공급한다.

### 현재 로직 분석
- 공용 폼이 많은 필드를 직접 관리하고 있어 UI가 과도하게 크다.
- 수정 기본값 주입은 `characterInfo` 전체를 폼 구조로 매핑하는 방식이다.
- 자금은 현재 CP/SP/EP/GP/PP 입력으로 제공되고 있다.

## 영향도 평가
### 직접 영향
- `src/app/components/characters/CharacterEditorForm.vue`

### 간접 영향
- 생성/수정 페이지 래퍼는 동일 컴포넌트를 사용하므로 별도 구조 수정은 거의 없다.
- 후속 저장 로직은 축소된 submit payload 기준으로 붙여야 한다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 수정 화면 기본값 매핑에서 제거된 필드를 안전하게 정리해야 한다.
- 시작 자금을 현재처럼 화폐 단위별로 유지할지, 단일 입력으로 바꿀지 명시적 요청이 없으므로 기존 단위 분해 UI를 유지하면서 나머지 필드만 제거하는 편이 안전하다.

### 가정 / 불확실성
- "시작 자금"은 기존 DTO 구조와 맞춰 CP/SP/EP/GP/PP 묶음 입력으로 유지한다.
- 클래스 정보는 수정 모드에서 요약 칩, 생성 모드에서 안내 박스로 유지한다.
