## 요구사항 요약
- 서버 응답의 `ListDataType<T>`에서 `list`는 각 도메인 스토어의 배열 ref로 분리 저장하고, 나머지 페이징 메타는 별도 ref로 분리 저장하는 패턴을 도입해야 함.
- 유저 도메인에서는 `getUserList` 호출 시 실제 목록은 `userList` 성격의 배열 ref에 저장하고, 페이지 메타는 `userPageData`라는 별도 ref에 저장하는 형태로 개선해야 함.
- 페이지 메타 타입은 서버의 `ListDataType<T>`에서 `list`만 제외한 `Omit<ListDataType<T>, 'list'>` 기반 타입으로 정의해야 함.
- `DataTable.vue`는 분리된 `items`와 `pagination` 메타를 입력으로 받는 구조를 전제로 정리되어야 하며, 이후 다른 모든 리스트 도메인에도 같은 패턴을 확장할 계획이 문서에 포함되어야 함.

## 현황 파악
### 관련 파일
- `src/app/components/DataTable.vue`: 테이블 렌더링과 페이지네이터 UI를 함께 담당. `items`와 `pagination` 메타를 받아 표시 구간과 페이지 이동 버튼을 계산함.
- `src/server/types/response.types.ts`: 프런트가 소비하는 `ListDataType` 계약 정의. `currentPage`, `totalPages`, `hasPrev`, `nextPage` 등 다수의 파생 필드를 함께 노출함.
- `src/server/utils/list-data.ts`: 서버에서 `ListDataType` JSON을 생성하는 원본 클래스. `filteredElements`, `pageSize`, `currentPage`를 바탕으로 각 파생 값을 계산함.
- `src/app/stores/user.store.ts`: 현재 `getUserList`가 `res.data.list`와 `res.data.totalElements`만 개별 저장하고 있으며, 페이지 메타를 별도 상태로 보존하지 않음.
- `src/app/pages/players/index.vue`: 현재 `DataTable`의 시각 테스트 페이지. 아직 페이징 메타를 넘기지 않는 단순 사용 예시만 존재함.

### 현재 로직 분석
- `DataTable.vue`는 `startItem`, `endItem`, `pageNumbers`, `goToPage` 등 페이지네이터 계산 로직을 내부에 가지고 있음.
- 다만 해당 계산은 `currentPage`, `pageSize`, `firstPage`, `lastPage`, `hasPrev`, `hasNext`, `prevPage`, `nextPage`, `filteredElements`, `totalPages` 같은 값이 모두 신뢰 가능하다는 전제 위에 작성되어 있음.
- `items`는 `ListDataType<any>[]`로 선언되어 있으나 실제 의미는 "현재 페이지의 row 배열"이므로 타입 의미가 잘못 연결되어 있음.
- `pagination` prop은 사실상 `ListDataType`의 메타 부분만 필요하지만 현재 타입 선언만으로는 `list` 포함 여부가 불명확함.
- `src/app/stores/user.store.ts`의 `getUserList`는 현재 `res.data.list`와 `res.data.totalElements`만 저장하므로, 테이블이 필요로 하는 `currentPage`, `pageSize`, `hasNext` 같은 메타를 재사용할 수 없음.
- 결과적으로 실제 화면에서 유저 목록에 페이지네이터를 연결하려면 응답 원본을 다시 들고 있거나, 각 페이지에서 메타를 임의로 재조합해야 하는 구조임.

## 영향도 평가
### 직접 영향
- `src/app/stores/user.store.ts`의 목록 상태 구조와 `getUserList` 응답 저장 방식
- `src/app/components/DataTable.vue`의 props 타입 정의와 페이징 계산 로직
- `pageChange` 이벤트 계약과 버튼 활성화 조건
- 현재 페이지 범위 표시(`startItem`, `endItem`, `filteredElements`) 방식

### 간접 영향
- `ListDataType`을 반환하는 API 응답과 프런트 소비 코드의 계약 해석 방식
- 다른 목록 스토어(`campaign`, `character`, `session`, `log` 등)에도 동일 패턴을 적용하기 위한 공통 규칙 수립
- 향후 `DataTable`을 실제 목록 페이지에 연결할 때 필요한 데이터 매핑 방식
- 테스트/스토리 성격의 `src/app/pages/players/index.vue` 예시 코드 확장 가능성

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `Omit<ListDataType<T>, 'list'>`를 어디에 둘지 결정해야 함. 서버 타입을 그대로 프런트에서 참조할지, 앱 레이어에서 별칭 타입으로 감쌀지 설계 판단이 필요함.
- 유저 스토어만 우선 수정하더라도 이후 다른 리스트 스토어가 같은 패턴으로 자연스럽게 따라올 수 있도록 네이밍과 상태 구조를 일반화해야 함.
- `DataTable`이 메타를 신뢰하되 일부 필드 누락 상황을 방어할지, 스토어 단계에서 완전한 메타만 넘기도록 보장할지 책임 경계를 정해야 함.

### 가정 / 불확실성
- 마스터 요청은 우선 "개선 방향 정리" 수준으로 해석했으며, 아직 구체적인 UX 변경 범위는 확정되지 않음.
- `context7 MCP` 사용이 프로젝트 규칙에 명시되어 있으나 현재 세션에서는 해당 도구가 제공되지 않아 로컬 코드 기준으로 분석을 수행함.
- 참조 규칙에 있는 `project-init-guideline.md`가 로컬에 없어 그 문서의 추가 요구사항은 반영하지 못했음.

## 제안하는 설계 진입 포인트
- 앱 레이어에 `type ListPageData<T> = Omit<ListDataType<T>, 'list'>` 형태의 별칭을 두고, 테이블과 스토어가 같은 메타 계약을 공유
- `user.store.ts`에서 `getUserList` 성공 시 `list`는 목록 ref, 나머지는 `userPageData` ref에 저장하는 패턴으로 재구성
- 같은 패턴을 다른 리스트 스토어로 확장하기 위한 네이밍 규칙과 적용 대상 목록을 설계 문서에 포함
- `DataTable`은 `items`와 `pagination`을 명시적으로 분리받고, 페이지 메타가 비어 있으면 페이지네이터를 숨기거나 안전한 기본값을 사용하도록 책임을 명확화
