# 분석 (Analysis)

## 요약

- 요청 내용: `useCreateNewPlayer` 같은 래퍼 컴포저블의 반환값이 `Promise`가 아니라 내부 `usePost`/`useGet` 반환 객체 그 자체가 되도록 정리
- 범위: `src/app/composables/query/useGet.ts`, 플레이어 래퍼 컴포저블들
- 구현 가능성: 가능

## 요구사항 요약

- `useGet`~`useDelete` 계열은 내부 실행 함수(`execute`, `mutateAsync`)만 비동기면 충분하다.
- 이를 감싸는 래퍼 컴포저블은 `await` 없이 즉시 사용할 수 있어야 한다.
- 특히 `src/app/composables/players/useCreateNewPlayer.ts`는 `usePost` 반환 객체를 그대로 반환해야 한다.

## 현재 상태

### 관련 파일

- `src/app/composables/query/useGet.ts`: 현재 top-level 함수 자체가 `async`라 `Promise<UseGetReturn<T>>`를 반환한다.
- `src/app/composables/query/usePost.ts`: 이미 동기 함수이며 내부 `execute`만 비동기다.
- `src/app/composables/players/useGetMyInfo.ts`: `useGet`를 래핑한다.
- `src/app/composables/players/useGetPlayerList.ts`: `useGet`를 래핑한다.
- `src/app/composables/players/useCreateNewPlayer.ts`: `usePost`를 래핑한다.

### 현재 로직 분석

- `useGet` 본문에는 top-level `await`가 없고, 비동기는 내부 `execute`, `refetch` 안에만 있다.
- 따라서 `useGet`에 `async`가 붙어 있을 이유가 없다.
- 이 때문에 `useGetMyInfo`, `useGetPlayerList`는 겉으로는 동기처럼 보여도 실제 반환은 `Promise`가 된다.
- `useCreateNewPlayer`는 `usePost`를 직접 반환하므로 이미 구조상 동기 반환이 가능하지만, 성공 콜백은 현재 응답 객체/데이터 계약 혼용 가능성이 있다.

## 영향도 평가

### 직접 영향

- `src/app/composables/query/useGet.ts` 시그니처
- `src/app/composables/players/useGetMyInfo.ts`
- `src/app/composables/players/useGetPlayerList.ts`
- `src/app/composables/players/useCreateNewPlayer.ts`

### 간접 영향

- `useGet` 래퍼를 `await`해서 쓰던 호출부가 있으면 제거 가능성이 생긴다.
- 타입 기준이 단순해져 composable 사용 ergonomics가 개선된다.
