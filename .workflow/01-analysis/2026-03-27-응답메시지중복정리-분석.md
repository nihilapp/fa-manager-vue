# 분석 (Analysis)

## 요약
- 요청 내용: `src/server/constant/response-message.ts`의 중복 메시지를 점검하고 적절한 문맥으로 정리
- 범위: 응답 메시지 상수 파일과 직접 사용 중인 서버 API 핸들러
- 구현 가능성: 가능

## 요구사항 요약
- 중복 문자열보다 먼저 중복 상수 선언 여부를 확인한다.
- 각 메시지가 실제 어느 권한 검증, 어느 등록 중복, 어느 도메인 리소스를 설명하는지 문맥을 분리한다.
- 과도하게 범용적인 메시지는 구체적 도메인 메시지로 대체한다.

## 현재 상태
### 관련 파일
- `src/server/constant/response-message.ts`: 서버 응답 메시지 상수를 도메인별로 정의하는 단일 상수 파일
- `src/server/api/players/[id].put.ts`: 플레이어 수정 권한 실패 시 메시지 반환
- `src/server/api/players/[id].delete.ts`: 플레이어 삭제 권한 실패 시 메시지 반환
- `src/server/api/campaigns/[id].put.ts`, `delete.ts`, `status.put.ts`: 캠페인 소유권 검증 실패 시 메시지 반환
- `src/server/api/campaigns/[id]/members/[userId].post.ts`, `delete.ts`: 대상 플레이어 권한 검증 실패 시 메시지 반환
- `src/server/api/campaigns/[campaignId]/characters/[characterId].post.ts`, `delete.ts`: 캠페인 소유권 검증 실패 시 메시지 반환
- `src/server/api/sessions/[id]/players/index.post.ts`: 세션 플레이어 등록 중 권한 및 캠페인 소속 불일치 메시지 반환

### 현재 로직 분석
- `PLAYER_FORBIDDEN`이 두 번 선언되어 뒤쪽 선언이 앞쪽 의미를 덮어쓰는 상태였다.
- 캠페인 소유권 검증에서도 플레이어 권한 메시지를 재사용하고 있어 응답 문맥이 부정확했다.
- 세션 플레이어 등록에서 캐릭터의 캠페인 불일치를 권한 실패 메시지로 처리하고 있었다.
- 세션 등록 중복 메시지 중 하나는 네이밍만 남고 실제 사용되지 않았다.

## 영향도 평가
### 직접 영향
- 응답 메시지 상수 키와 문자열
- 권한 실패 응답을 반환하는 플레이어/캠페인/세션 API 핸들러

### 간접 영향
- 프런트 에러 핸들러는 코드 중심으로 분기하므로 메시지 문구 변경 영향은 낮음
- 테스트 스냅샷이 있다면 문자열 기대값이 변할 수 있으나 현재는 로컬 검색 기준 별도 테스트 자산이 확인되지 않음

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 이미 진행 중인 대규모 `user -> player` 변경과 파일 수정 범위가 겹치므로 기존 변경을 덮어쓰지 않도록 최소 범위 패치가 필요함

### 가정 / 불확실성
- `context7 MCP`는 현재 세션 도구에 노출되지 않아 로컬 코드 검색 기준으로 판단함
- 문자열 자체를 소비하는 외부 클라이언트가 있다면 문구 변경 검증이 추가로 필요할 수 있음
