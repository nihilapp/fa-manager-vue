# 분석 (Analysis)

## 요약
- 요청 내용: 삭제된 아이콘 타입 생성 스크립트를 복구하고, 단독 실행만으로 `src/app/types/icons` 아래 유니온 타입 파일을 다시 생성할 수 있게 한다.
- 범위: `package.json`, 신규 `scripts/get-icon-names.ts`, 생성 결과물인 `src/app/types/icons/*.types.ts`, `src/app/types/icons/index.ts`
- 구현 가능성: 가능

## 요구사항 요약
- `package.json`에 선언된 `@iconify-json/*` 패키지를 읽어 대상 아이콘 세트를 동적으로 수집해야 한다.
- 스크립트는 루트 기준 단독 실행 가능해야 하며, `scripts/get-icon-names.ts` 경로에 위치해야 한다.
- 실행 결과는 각 세트별 `set:name` 문자열 리터럴 유니온 타입 파일과 통합 `IconName` 타입 파일이어야 한다.
- 마스터 요청에 따라 마지막에 `package.json` 실행 명령을 함께 제공해야 한다.

## 현재 상태
### 관련 파일
- `package.json`: 프로젝트 스크립트와 `@iconify-json/*` devDependencies 선언 위치다.
- `src/server/api/icons/index.get.ts`: 기존 타입 생성 로직이 남아 있는 서버 API 파일이다. 패키지 목록, 출력 경로, 타입명 생성 규칙의 기준 구현이다.
- `src/app/types/icons/*.types.ts`: 현재 생성 결과가 남아 있는 유니온 타입 파일들이다. 출력 형식 호환 기준이다.
- `src/app/types/icons/index.ts`: 각 세트 타입을 합쳐 `IconName`으로 노출하는 통합 타입 파일이다.
- `src/app/types/manual-app-imports.d.ts`: 앱 전역에서 `IconName`을 참조하는 선언 파일이다. 생성 결과 호환성이 중요하다.

### 현재 로직 분석
- 기존 생성 책임은 `src/server/api/icons/index.get.ts`에 들어 있다.
- API 호출 시 출력 폴더를 비우고, 각 Iconify JSON 패키지의 `icons.json`을 읽어 `prefix.types.ts`와 `index.ts`를 생성한다.
- 현재 방식은 서버 라우트를 통해서만 실행 가능하므로, 삭제된 별도 스크립트를 대체하지 못한다.
- 현재 타입 파일은 이미 남아 있어 원하는 출력 포맷을 직접 비교 기준으로 삼을 수 있다.

## 영향도 평가
### 직접 영향
- `scripts/get-icon-names.ts` 추가
- `package.json` 실행 스크립트 추가
- 실행 시 `src/app/types/icons/*.types.ts`, `src/app/types/icons/index.ts` 재생성

### 간접 영향
- `src/app/types/manual-app-imports.d.ts` 및 아이콘 이름을 소비하는 전역 타입 사용처가 생성 결과 형식에 의존한다.
- 기존 서버 API 생성 로직과 출력 형식이 달라지면 추후 수동/자동 생성 경로가 이원화될 수 있다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `package.json`에서 Iconify 패키지 목록을 동적으로 읽되, 현재 선언 순서를 가능한 유지해야 한다.
- ESM TypeScript 스크립트에서 `icons.json`을 안정적으로 불러오기 위해 `createRequire` 기반 로딩이 필요하다.
- 기존 타입 사용처와 호환되도록 `index.ts`의 `IconName` 구조와 빈 문자열 허용 여부를 유지해야 한다.

### 가정 / 불확실성
- context7 MCP 사용 지침이 있으나 현재 세션에서는 관련 리소스를 확인하지 못했다.
- 기존 삭제된 스크립트의 정확한 로그 메시지나 부가 기능은 알 수 없으므로, 남아 있는 서버 로직과 현재 생성물 형식을 기준으로 복구한다.
