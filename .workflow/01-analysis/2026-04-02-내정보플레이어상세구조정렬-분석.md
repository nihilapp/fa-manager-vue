# 분석 (Analysis)

## 요약
- 요청 내용: `src/app/pages/me/index.vue`를 플레이어 상세와 동일한 구조로 맞추고, 같은 렌더링 컴포넌트를 사용하도록 정리한다.
- 범위: `me` 페이지, 플레이어 상세 페이지, 플레이어 상세 렌더링 컴포넌트, 관련 `mine`/플레이어별 조회 컴포저블과 스토어 축
- 구현 가능성: 가능

## 요구사항 요약
- `me` 페이지가 `myInfo`만 읽는 임시 상태에서 벗어나 실제 내 캐릭터/캠페인/세션 목록까지 함께 렌더링해야 한다.
- `players/detail/[id].vue`와 `me/index.vue`는 구조와 렌더링 컴포넌트를 동일하게 사용해야 한다.
- `me`는 `mine` 전용 API와 스토어 축을 그대로 유지하고, 일반 플레이어 상세는 기존 `playerId` 기준 조회 흐름을 유지해야 한다.

## 현재 상태
### 관련 파일
- `src/app/pages/me/index.vue`
  - 메타 설정과 `myInfo` 참조만 있고 실제 조회 실행과 렌더링 구조가 없다.
- `src/app/pages/players/detail/[id].vue`
  - `playerInfo`, `playerSessionList`, `playerCampaignList`를 상세 페이지에 연결하지만 캠페인 호출 실행과 캐릭터 목록 연결이 빠져 있다.
- `src/app/components/players/PlayerDetail.vue`
  - 플레이어 기본 정보와 3개 섹션 틀은 있지만 실제 리스트 렌더링이 비어 있고, 플레이어 상세 전용 이름으로 묶여 있다.
- `src/app/composables/players/useGetMyInfo.ts`
  - `/players/me` 단건 정보를 `myInfo` 축에 저장한다.
- `src/app/composables/campaigns/useGetMyCampaignList.ts`
  - `/campaigns/mine` 목록과 페이지 데이터를 `myCampaignList`, `myCampaignPageData`에 저장한다.
- `src/app/composables/characters/useGetMyCharacterList.ts`
  - `/characters/mine` 목록과 페이지 데이터를 `myCharacterList`, `myCharacterPageData`에 저장한다.
- `src/app/composables/sessions/useGetMySessionList.ts`
  - `/sessions/mine` 목록과 페이지 데이터를 `mySessionList`, `mySessionPageData`에 저장한다.
- `src/app/composables/players/useGetPlayerInfo.ts`, `useGetPlayerCampaignList.ts`, `useGetPlayerSessionList.ts`
  - 일반 플레이어 상세용 단건/하위 목록 축을 저장한다.
- `src/app/composables/characters/useGetCharacterList.ts`
  - `/characters?userId=` 형태로 특정 플레이어의 캐릭터 목록을 기존 index API로 조회할 수 있다.

### 현재 로직 분석
- `me` 페이지는 이미 필요한 `mine` 계열 컴포저블이 준비돼 있지만 페이지 연결이 없다.
- 플레이어 상세 페이지는 공통 레이아웃 후보가 있으나 실제 리스트 데이터 주입이 불완전하다.
- 캐릭터 목록은 전용 `/players/:id/characters` API 없이도 `useGetCharacterList({ query: { userId: id } })`로 가져올 수 있다.
- 현재 상세 렌더링 컴포넌트는 페이지 책임과 표현 책임이 섞여 있어 `me`와의 재사용성이 낮다.

## 영향도 평가
### 직접 영향
- `src/app/pages/me/index.vue`
- `src/app/pages/players/detail/[id].vue`
- `src/app/components/players/*` 공통 상세 렌더러
- `src/app/stores/character.store.ts` 또는 캐릭터 목록 참조 축 사용 방식

### 간접 영향
- `mine` 축과 일반 상세 축의 역할 분리가 화면 레벨에서도 명확해진다.
- 플레이어 상세 화면이 실제로 캐릭터/캠페인/세션 데이터까지 모두 표시하게 된다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `me`와 일반 플레이어 상세가 같은 뷰를 쓰더라도 데이터 소스 축은 다르므로 props 계약을 일반화해야 한다.
- 캐릭터 목록은 `mine` 축과 일반 `index` 축을 각각 써야 하므로 공통 컴포넌트가 스토어가 아닌 props만 보도록 정리해야 한다.

### 가정 / 불확실성
- 이번 범위에서는 플레이어 상세 캐릭터 조회를 기존 `/characters?userId=` 재사용으로 처리한다.
- 내 정보 화면의 상단 액션은 플레이어 상세와 완전히 같을 필요는 없고, 동일한 레이아웃 안에서 버튼 노출 여부만 달라질 수 있다.
