# 20260320-뷰쿼리전환-분석

## 1. 개요
현재 Nuxt 3의 기본 데이터 패칭 수단인 `useFetch`와 `$fetch`를 `@tanstack/vue-query`로 전환할 때의 영향도를 분석합니다.

## 2. 현황 분석
### 2.1. 데이터 패칭 구조
- **GET 요청**: `src/app/composables/query/useGet.ts`에서 `useFetch`를 래핑하여 사용 중.
- **그 외 요청 (POST, PUT, PATCH, DELETE)**: `src/app/composables/query/useApiRequest.ts`의 `createApiMutation` 함수에서 `$fetch`를 사용 중.
- **상태 관리**: API 응답 결과를 Pinia 스토어(`user.store.ts`)의 `ref` 변수에 직접 저장하는 방식.

### 2.2. 사용 빈도 (src/app 기준)
- **커스텀 컴포저블 사용**: `user.store.ts`에서 `useGet` 1회 사용 확인.
- **페이지/컴포넌트 직접 사용**: `pages/index.vue` 등 주요 지점에서 아직 직접적인 사용 없음 (초기 단계).

## 3. 영향도 평가
### 3.1. 코드 수정 범위 (낮음)
- `src/app/composables/query/` 내의 기본 래퍼 함수들만 수정하면 대부분의 로직을 흡수 가능합니다.
- 현재 API 호출이 집중된 곳이 적어 리팩토링 비용이 매우 낮습니다.

### 3.2. 아키텍처 영향 (중간~높음)
- **SSR 전략**: Nuxt의 네이티브 SSR 통합 방식에서 `vue-query`의 프리페칭(Prefetching) 및 하이드레이션(Hydration) 방식으로 전환 설정이 필요합니다.
- **캐시 중심 개발**: 기존의 "API 호출 -> 스토어 저장" 방식에서 "Query Key를 통한 캐시 참조" 방식으로의 패러다임 변화가 필요합니다. 이는 Pinia 스토어의 역할을 축소시키고 데이터 일관성을 높이는 효과가 있습니다.
- **쿼리 키 관리**: 프로젝트 전반에 걸친 `Query Key` 네이밍 규칙 및 관리 체계 수립이 필요합니다.

### 3.3. 기술적 요구사항
- 패키지 설치: `@tanstack/vue-query`
- Nuxt 플러그인 등록: SSR 지원을 위한 클라이언트/서버 설정

## 4. 결론 및 제언
현재 프로젝트가 초기 단계이므로, 더 복잡한 로직이 쌓이기 전에 **지금 전환하는 것이 가장 비용 효율적**입니다. 전환 시 `useQuery`와 `useMutation`을 기반으로 하는 표준화된 API 레이어를 구축할 것을 권장합니다.
