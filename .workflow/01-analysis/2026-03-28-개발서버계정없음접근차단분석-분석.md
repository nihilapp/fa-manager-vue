## 요구사항 요약

- 개발서버에서는 사용자가 없더라도 접근 가능하도록 설계된 것처럼 보이는데, 실제로 어느 분기에서 차단되는지 확인해야 한다.
- DB가 완전히 비어 있는 상태를 전제로 서버 미들웨어와 클라이언트 부트스트랩 로직을 함께 추적해야 한다.
- 이후 수정 단계에서 바로 설계할 수 있도록 직접 원인과 간접 영향 범위를 분리해 문서화해야 한다.

## 현재 상태

### 관련 파일

- `src/server/utils/auth.ts`: 디스코드 식별자를 해석하고 플레이어 조회 후 `isDevelopmentBypass`와 `error`를 함께 담은 인증 컨텍스트를 생성한다.
- `src/server/utils/access.ts`: `authHelper()` 결과를 기반으로 커뮤니티 접근 허용 여부를 판정한다. 현재는 `error`를 가장 먼저 반환한다.
- `src/server/middleware/access.ts`: 페이지 요청마다 `checkCommunityAccess()`를 호출하고, 오류가 있으면 `/block`으로 리다이렉트한다.
- `src/server/api/players/me.get.ts`: 현재 사용자 조회 API로 `authHelper()`에서 `error`가 오면 그대로 실패를 반환한다.
- `src/app/composables/players/useGetMyInfo.ts`: `/players/me` 실패 시 플레이어 스토어를 비우고 클라이언트에서 `/block`으로 이동한다.
- `src/app.vue`: `/block`이 아닐 때 앱 시작 시 무조건 `useGetMyInfo().execute()`를 호출한다.

### 현재 로직 분석

- `src/server/utils/auth.ts`에서 개발환경이고 `ADMIN_DISCORD_ID`가 있으면 식별자 소스를 `development`로 결정한다.
- 같은 파일에서 DB 조회 결과 사용자가 없으면 `error = PLAYER_NOT_FOUND`, `isDevelopmentBypass = source === 'development'`인 컨텍스트를 반환한다.
- 하지만 `src/server/utils/access.ts`는 `isDevelopmentBypass`를 보기 전에 `if (error) return error;`를 먼저 실행한다.
- 그 결과 개발 우회 플래그가 `true`여도 `PLAYER_NOT_FOUND`가 우선 반환되고, `src/server/middleware/access.ts`가 이를 받아 `/block`으로 리다이렉트한다.
- 서버 미들웨어를 우회하더라도 `src/app.vue`가 앱 시작 시 `/players/me`를 호출하고, `src/server/api/players/me.get.ts`가 같은 `error`를 그대로 반환하므로
  `useGetMyInfo.ts`의 `onError`에서 다시 `/block`으로 이동한다.

## 영향도 평가

### 직접 영향

- 개발환경에서 `ADMIN_DISCORD_ID`는 존재하지만 대응하는 플레이어 레코드가 없을 때 접근 우회가 동작하지 않는다.
- DB 초기화 직후나 로컬 개발 초기 세팅 중 화면 진입 자체가 막혀 초기 관리자 계정 생성 플로우 테스트가 어렵다.

### 간접 영향

- `authHelper()`의 반환 계약이 모호하다. 현재는 "개발 우회 허용"과 "에러 발생"이 동시에 존재할 수 있는데, 소비자마다 우선순위를 다르게 해석하면 버그가 반복될 수 있다.
- `/players/me`를 앱 부트스트랩에서 강제 호출하는 구조 때문에, 페이지 접근 가드만 수정해도 클라이언트에서 동일 문제가 재발할 수 있다.
- 개발 우회가 허용된 상태에서 `user === null`일 수 있으므로, 이후 API나 UI가 `user!.id` 같은 가정을 하고 있으면 별도 보호 규칙이 필요하다.

## 실행 판단

### 구현 가능성

- 가능

### 예상 기술 난제

- 서버 가드와 `/players/me` API가 공유하는 인증 계약을 하나로 정리해야 한다.
- "개발 우회 허용"과 "플레이어 미존재"를 동시에 표현할지, 개발환경에서는 별도의 초기 관리자 생성 흐름으로 분리할지 설계 결정이 필요하다.
- 무계정 개발환경에서 허용 가능한 기능 범위를 정하지 않으면, 화면 진입은 되더라도 CUD API에서 `user!.id` 접근으로 런타임 오류가 날 수 있다.

### 가정 / 불확실성

- 현재 세션에서는 `context7` MCP 서버가 등록되어 있지 않아 로컬 코드 기준으로만 분석했다.
- `ADMIN_DISCORD_ID`가 실제로 설정되어 있는지는 환경파일을 열지 않았으므로, 분석은 코드 경로와 마스터 설명을 기준으로 했다.
- 의도된 동작이 "완전 무계정도 전체 접근 허용"인지, "초기 관리자 생성 화면까지만 허용"인지는 설계 단계에서 확정이 필요하다.
