# 분석 (Analysis)

## 요약
- 요청 내용: players 이후 남은 세션, 캠페인, 캐릭터, 화폐 도메인의 프런트 컴포저블/스토어 확장 범위를 분석해 이후 작업 재개 기준을 마련
- 범위: `src/server/api/{sessions,campaigns,characters,currency-transactions}`, `src/app/composables`, `src/app/stores`, 관련 쿼리 키 및 현재 사용처
- 구현 가능성: 가능

## 요구사항 요약
- players 도메인에 적용한 패턴을 나머지 도메인에도 확장해야 한다.
- 각 도메인은 기본적으로 목록 변수, 페이지 정보 변수, 개별 정보 변수를 갖는 스토어가 필요하다.
- 플레이어별 또는 부모 리소스별 파생 목록이 필요한 경우, 도메인 스토어 내부에 별도 파생 목록/페이지 정보 축을 둔다.
- 모든 도메인 컴포저블은 API 성공 시 스토어 액션으로 상태를 저장하고, 실패 시 대응 초기화 액션을 호출해야 한다.
- 이후 작업은 한 번에 전체 구현하더라도 도메인별 순서를 가져야 하며, 설계와 태스크 분해가 가능한 수준으로 사전 범위를 명확히 해야 한다.

## 현재 상태
### 관련 파일
- `src/app/composables/players/*`: players만 구현되어 있으며 현재 프로젝트의 기준 패턴으로 사용할 수 있다.
- `src/app/stores/player.store.ts`: 기본 축과 내 정보 축이 공존하는 players 기준 스토어
- `src/app/stores/session.store.ts`: 기본 세션 목록과 플레이어별 세션 목록을 보관하는 파생 축 사례
- `src/app/stores/campaign.store.ts`: 플레이어별 캠페인 목록 축까지 포함한 최소 구조가 방금 추가됨
- `src/app/stores`: character, currency 전용 스토어는 아직 없음
- `src/app/utils/query-keys.ts`: campaigns, characters, sessions, currency-transactions에 대한 조회 키는 이미 일부 정의돼 있음
- `src/server/api/campaigns/*`: 목록/내 목록/상세/생성/수정/삭제/상태변경/멤버 관리/캠페인 소속 캐릭터 추가·삭제 API 존재
- `src/server/api/sessions/*`: 목록/내 목록/상세/생성/수정/삭제/세션 로그 CRUD/세션 참여자 추가·삭제 API 존재
- `src/server/api/characters/*`: 목록/내 목록/상세/생성/수정/삭제/상태변경/클래스 추가·수정·삭제 API 존재
- `src/server/api/currency-transactions/*`: 목록/상세/생성/수정/삭제 API 존재

### 현재 로직 분석
- 현재 앱 레이어에서 players 외 도메인용 컴포저블은 없다.
- 세션은 스토어만 일부 존재하고, 캠페인은 스토어 골격만 있으며 실제 조회/변경 컴포저블은 없다.
- 캐릭터와 화폐는 스토어와 컴포저블 모두 비어 있다.
- `query-keys.ts`는 기본 목록/상세 조회 기준은 갖추고 있으나, 세션 로그나 캠페인 멤버 조작처럼 하위 리소스용 키 확장이 필요할 수 있다.
- 현재 UI 사용처는 players 중심이어서 다른 도메인 컴포저블은 즉시 연결 대상이 많지 않다. 따라서 우선 "저장 계약이 맞는 컴포저블/스토어 기반"을 만드는 것이 먼저다.

## 도메인별 미구현 범위
### 1. Sessions
- 기본 조회/변경
  - `GET /sessions`
  - `GET /sessions/mine`
  - `GET /sessions/:id`
  - `POST /sessions`
  - `PUT /sessions/:id`
  - `DELETE /sessions/:id`
- 파생/하위 리소스
  - `GET /sessions/:id/logs`
  - `POST /sessions/:id/logs`
  - `GET /sessions/:id/logs/:logId`
  - `PUT /sessions/:id/logs/:logId`
  - `DELETE /sessions/:id/logs/:logId`
  - `POST /sessions/:id/players`
  - `DELETE /sessions/:id/players/:userId`
- 스토어 필요
  - 기본 축: `sessionList`, `sessionPageData`, `sessionInfo`
  - 파생 축: `mySessionList`, `mySessionPageData`, `playerSessionList`, `playerSessionPageData`, `sessionLogList`, `sessionLogPageData`, `sessionLogInfo`

### 2. Campaigns
- 기본 조회/변경
  - `GET /campaigns`
  - `GET /campaigns/mine`
  - `GET /campaigns/:id`
  - `POST /campaigns`
  - `PUT /campaigns/:id`
  - `DELETE /campaigns/:id`
  - `PUT /campaigns/:id/status`
- 파생/하위 리소스
  - `POST /campaigns/:id/members/:userId`
  - `DELETE /campaigns/:id/members/:userId`
  - `POST /campaigns/:campaignId/characters/:characterId`
  - `DELETE /campaigns/:campaignId/characters/:characterId`
- 스토어 필요
  - 기본 축: `campaignList`, `campaignPageData`, `campaignInfo`
  - 파생 축: `myCampaignList`, `myCampaignPageData`, `playerCampaignList`, `playerCampaignPageData`

### 3. Characters
- 기본 조회/변경
  - `GET /characters`
  - `GET /characters/mine`
  - `GET /characters/:id`
  - `POST /characters`
  - `PUT /characters/:id`
  - `DELETE /characters/:id`
  - `PUT /characters/:id/status`
- 파생/하위 리소스
  - `POST /characters/:id/classes`
  - `PUT /characters/:id/classes/:className`
  - `DELETE /characters/:id/classes/:className`
- 스토어 필요
  - 기본 축: `characterList`, `characterPageData`, `characterInfo`
  - 파생 축: `myCharacterList`, `myCharacterPageData`, `playerCharacterList`, `playerCharacterPageData`, `characterClassList`
- 추가 고려
  - 플레이어별 캐릭터 목록 API는 현재 서버에 직접 노출돼 있지 않으므로, 우선 `mine`과 일반 `index` 중심으로 설계하고 플레이어별 목록은 추후 서버 API 필요 여부를 판단해야 한다.

### 4. Currency Transactions
- 기본 조회/변경
  - `GET /currency-transactions`
  - `GET /currency-transactions/:id`
  - `POST /currency-transactions`
  - `PUT /currency-transactions/:id`
  - `DELETE /currency-transactions/:id`
- 스토어 필요
  - 기본 축: `currencyTransactionList`, `currencyTransactionPageData`, `currencyTransactionInfo`
  - 파생 축 후보: `characterCurrencyTransactionList`, `characterCurrencyTransactionPageData`, `playerCurrencyTransactionList`, `playerCurrencyTransactionPageData`
- 추가 고려
  - 현재 쿼리 DTO에 어떤 필터가 있는지에 따라 캐릭터별/플레이어별 파생 목록은 같은 `index` API를 필터링하여 저장할 가능성이 높다.

## 영향도 평가
### 직접 영향
- `src/app/stores/*`
- `src/app/composables/{campaigns,sessions,characters,currency-transactions}` 신규 폴더 또는 파일 구조
- `src/app/utils/query-keys.ts`
- 각 도메인 페이지/컴포넌트가 추가되면 해당 호출부

### 간접 영향
- 공통 데이터 패칭 규칙의 일관성
- 이후 UI 페이지 구현 시 상태 참조 네이밍
- 하위 리소스(세션 로그, 캐릭터 클래스, 캠페인 멤버)의 스토어 분리 기준

## 우선순위 제안
1. Sessions
   - 이미 `session.store.ts`가 있고 players 상세와 간접 연결되어 있어 확장 기준으로 가장 적합
2. Campaigns
   - `campaign.store.ts` 골격이 생겼고 플레이어별 캠페인 목록 컴포저블도 존재
3. Characters
   - 캠페인/세션과 연결이 많아 상위 도메인 정리 후 구현하는 편이 안정적
4. Currency Transactions
   - 독립 CRUD 성격이 강해 마지막에 패턴 복제 적용이 가장 수월

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 세션 로그와 캐릭터 클래스처럼 하위 리소스를 메인 스토어에 포함할지 별도 스토어로 분리할지 경계 설정이 필요하다.
- `mine`, `player별`, `character별` 파생 목록이 동시에 존재하는 도메인은 상태 축이 빠르게 늘어날 수 있다.
- currency-transactions는 현재 프런트 사용처가 보이지 않아, 필수 파생 목록을 어디까지 선제 구현할지 선을 정해야 한다.

### 가정 / 불확실성
- 현재 단계는 분석 문서 작성만 수행하며 설계/구현은 이후 세션에서 이어서 진행한다.
- 캐릭터의 플레이어별 목록은 서버 API가 직접 제공되지 않으므로 현재는 설계 후보로만 본다.
- currency-transactions의 파생 목록은 `index` API의 필터 가능 범위를 확인한 뒤 최종 설계에서 축소될 수 있다.
