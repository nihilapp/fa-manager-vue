## 2. 요구사항 재해석
- `api-error-handler.ts`에서 실제 사용되지 않는 토스트 관련 상태와 미사용 helper를 제거해야 합니다.
- `usePost`, `usePut`, `usePatch`, `useDelete`는 `createApiMutation` 경유가 아니라 각 파일에서 직접 `useMutation`을 사용해야 합니다.
- `useGet`과 mutation 훅은 제네릭과 body 전달 차이를 제외하면 같은 형태의 옵션과 반환 형태를 가져야 합니다.

## 3. 현황 분석
- `useGet`은 위치 기반 인자(`url`, `callback`, `errorCallback`, `options`)를 받고 있습니다.
- mutation 훅은 동일한 위치 기반 인자를 받으며 내부적으로 `useApiRequest.ts`의 `createApiMutation`에 의존합니다.
- `api-error-handler.ts`는 `shouldShowToast`, `toastMessage`, `throwIfError`를 제공하지만 현재 코드베이스에서 소비되지 않습니다.
- `useGet`의 확인된 사용처는 `src/app/stores/user.store.ts` 1곳이며, mutation 훅의 확인된 직접 사용처는 아직 없습니다.

## 4. 영향도 평가
- 직접 영향: `src/app/composables/query` 계층과 `user.store.ts`
- 간접 영향: 이후 mutation 훅을 사용하는 화면이 추가될 때 공통 인터페이스를 그대로 따르게 됩니다.
- 불확실성: 아직 mutation 훅 소비처가 거의 없어 런타임 사용성은 제한적으로만 확인 가능합니다. 따라서 타입 검증이 중요합니다.

## 5. 실행 판단
- 현재 사용처가 적어 인터페이스 정리 비용이 낮습니다.
- 공통 로직을 `handleApiResponse`, `handleApiRequestError` 수준으로 축소하면 중복은 줄이고 결합도는 과도하게 높이지 않을 수 있습니다.
