# 분석 (Analysis)

## 요약
- 요청 내용: 캐릭터 생성/수정 폼을 `NewPlayerForm.vue`처럼 `zod` 기반으로 구성한다.
- 범위: `CharacterCreateForm.vue`, `CharacterEditorForm.vue`
- 구현 가능성: 가능

## 요구사항 요약
- 캐릭터 기본 입력 필드에 `zod` 스키마와 `toTypedSchema`를 적용해 `useForm` 수준에서 입력 검증을 수행한다.
- 플레이어 폼과 동일하게 `validationSchema`를 `useForm`에 연결하는 방향이 요구사항의 핵심이다.
- 기존 생성/수정 요청 흐름과 클래스 배열 입력 구조는 유지하되, 최소한 기본 필드의 필수값/형식 제약은 명시해야 한다.

## 현재 상태
### 관련 파일
- `src/app/components/players/NewPlayerForm.vue`
  - `z.object(...)`와 `toTypedSchema(...)`를 정의하고 `useForm`에 `validationSchema`를 주입하는 기준 예시다.
- `src/app/components/characters/CharacterCreateForm.vue`
  - `useForm`은 적용돼 있지만 `validationSchema`가 없어 필수 입력 누락 시에도 submit 콜백이 실행될 수 있다.
- `src/app/components/characters/CharacterEditorForm.vue`
  - 수정 폼 역시 동일하게 스키마 없이 `useForm` 초기값과 제출만 관리한다.
- `src/app/stores/character.store.ts`
  - 캐릭터 폼 값 타입과 payload 변환 로직을 제공한다. 현재 검증 책임은 없다.

### 현재 로직 분석
- 캐릭터 폼의 검증 가능한 필드는 `name`, `status`, `startLevel`, 시작 자금 5종, `campaignId`다.
- `FormInput`과 `FormSelect`는 이미 `name` 기반 `Field` 연결을 사용하므로 스키마만 추가하면 에러 메시지가 자동 노출된다.
- `campaignId`는 `number | null`, 숫자 입력 필드는 문자열 기반이라 `zod`에서 현재 문자열 입력 포맷을 허용하는 설계가 필요하다.
- 클래스 배열 입력은 `useForm` 필드가 아니므로 이번 요청 범위에서 포함하면 추가 설계가 필요하다.
- 현재 세션에서는 `context7` MCP 서버가 등록되어 있지 않아 외부 검색은 수행하지 못했다.

## 영향도 평가
### 직접 영향
- `src/app/components/characters/CharacterCreateForm.vue`
- `src/app/components/characters/CharacterEditorForm.vue`

### 간접 영향
- 생성/수정 시 빈 이름, 잘못된 상태 값, 숫자 형식 오류를 폼 레벨에서 차단하게 된다.
- 스키마가 추가되면 수정 화면의 초기값 재설정 시 타입과 일치하는 값 형태를 유지해야 한다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 숫자 입력 필드가 문자열로 보관되므로 빈 문자열 허용과 숫자 문자열 허용 규칙을 함께 정의해야 한다.
- `campaignId`는 `null` 허용이 필요한데, 선택 필수 여부를 어디까지 강제할지 설계가 필요하다.

### 가정 / 불확실성
- 이번 범위에서는 `name` 필수, `status` 허용값 제한, 숫자 필드의 숫자 문자열 검증을 우선 반영하는 것이 적절하다.
- 클래스 배열 입력의 `zod` 검증은 현재 요청 범위 밖으로 두는 쪽이 안전하다.
