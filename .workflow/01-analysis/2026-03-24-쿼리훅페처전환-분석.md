# 분석 (Analysis)

## 요약
- 요청 내용: `useGet ~ useDelete`의 외부 이름과 기본 사용성은 유지하되, 내부 구현을 Vue Query 중심에서 Nuxt 4 기본 fetcher(`useFetch`, `$fetch`) 기반으로 교체한다.
- 범위: `src/app/composables/query/*.ts`, 관련 공통 유틸, 실제 호출부(`user.store.ts`), Vue Query 설정 잔존물
- 구현 가능성: 가능

## 요구사항 요약
- `useGet`, `usePost`, `usePut`, `usePatch`, `useDelete`라는 훅 이름과 기본 호출 진입점은 유지한다.
- 내부는 Nuxt 4 권장 fetcher 방향으로 교체한다.
- `GET` 요청은 기본 자동 실행을 끄고, 원하는 시점에 수동으로 실행할 수 있어야 한다.
- 기존 `onSuccess`, `onError`, `enabled` 같은 옵션 의미는 최대한 유지한다.

## 현재 상태
### 관련 파일
- `src/app/composables/query/useGet.ts`: `useQuery` 기반이며 `await useGet(...)` 직후 `suspense()`를 호출해 자동 요청까지 수행한다.
- `src/app/composables/query/usePost.ts`, `usePut.ts`, `usePatch.ts`, `useDelete.ts`: `useMutation` 기반이며 `execute`를 `mutateAsync`에 매핑한다.
- `src/app/composables/query/useApiRequest.ts`: 공통 타입과 응답/에러 처리 유틸을 제공하지만, 에러 처리가 Axios 에러 형식에 묶여 있다.
- `src/app/utils/api-client.ts`: Axios 인스턴스와 토큰 인터셉터를 제공한다.
- `src/app/plugins/vue-query.ts`: Vue Query 플러그인과 SSR hydrate/dehydrate를 등록한다.
- `src/app/stores/user.store.ts`: 실제 `useGet` 호출부 2곳이 존재하며, 둘 다 현재는 `await useGet(...)`만으로 요청이 실행되는 전제를 가진다.
- `nuxt.config.ts`: Vue Query 자동 임포트와 Vite optimizeDeps가 남아 있다.

### 현재 로직 분석
- `useGet`은 기본 `enabled: true`이며 `await useGet(...)`만 호출해도 즉시 요청을 발생시킨다.
- `usePost` 계열은 `execute`를 직접 호출하는 구조라 수동 실행 모델과 잘 맞는다.
- 현재 호출부 기준 실제 사용 중인 훅은 `useGet`뿐이다.
- Vue Query 플러그인과 auto-import 설정은 지금 훅 구현이 의존하는 기반이지만, fetcher 전환 후에는 불필요해질 가능성이 높다.

## 영향도 평가
### 직접 영향
- `src/app/composables/query` 내부 구현 전면 수정
- `src/app/stores/user.store.ts`의 `useGet` 호출 방식 수정 필요
- `src/app/composables/query/useApiRequest.ts`의 에러 처리 일반화 필요
- `nuxt.config.ts`, `src/app/plugins/vue-query.ts` 정리 여부 결정 필요

### 간접 영향
- 기존에 Vue Query 반환형에 맞춰 기대하던 `data`, `error`, `status`, `execute` 형태를 완전히 깨면 향후 호출부 확장 시 연쇄 수정이 커질 수 있다.
- GET 기본 자동 실행을 끄면 기존 호출부를 명시 실행 방식으로 바꾸지 않을 경우 데이터가 비어 있게 된다.
- Axios 제거 수준까지 한 번에 밀면 범위가 커질 수 있으므로, 우선 훅 내부의 기본 fetcher만 `useFetch/$fetch`로 바꾸고 나머지 잔존 의존은 최소 범위로 정리하는 편이 안전하다.

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `useGet`의 "수동 실행 + 기존 반환 인터페이스 유지"를 동시에 만족시키려면 `useFetch` 기반 호환 래퍼가 필요하다.
- `useMutation` 반환형을 더 이상 쓸 수 없으므로 POST/PUT/PATCH/DELETE의 상태 표현(`pending`, `status`, `response`, `execute`)을 자체적으로 맞춰야 한다.
- Axios 에러와 Fetch 에러를 모두 처리할 수 있게 공통 에러 처리 유틸을 일반화해야 한다.

### 가정 / 불확실성
- 현재 프로젝트 내 실제 `useGet` 호출부는 `user.store.ts` 2곳뿐이며, 다른 숨은 소비자는 없는 것으로 본다.
- `context7 MCP`는 현재 세션에서 사용할 수 없어 로컬 코드 기준으로 분석했다.
