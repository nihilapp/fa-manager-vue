# 분석 (Analysis)

## 요약
- 요청 내용: 모든 프론트 페이지 진입 전에 `GET /api/users/check`를 공통 호출하고, 실패 시 `block` 라우트로 이동시키는 구조를 설계
- 범위: Nuxt 프론트 라우팅 진입점, 공통 레이아웃, 헤더 표시 구조, `users/check` API 응답 처리
- 구현 가능성: 가능

## 요구사항 요약
- 모든 페이지는 헤더에 디스코드 ID가 표시되어야 한다.
- 접근 가능 조건은 개발 버전(`import.meta.env.DEV`)이면서 `discordId` 헤더가 존재하는 경우다.
- 외부 검색 등으로 직접 유입한 사용자는 정상 페이지 대신 차단 페이지(`block` 라우트)로 보내야 한다.
- 단일 페이지별 중복 구현이 아니라 전역 진입 가드 형태가 필요하다.

## 현재 상태
### 관련 파일
- `src/server/api/users/check.get.ts`: `discordId` 헤더와 개발 환경 여부를 검사하고 실패 시 `TEAM_ONLY` 에러 응답을 반환한다.
- `src/app/app.vue`: 모든 페이지 렌더링의 최상위 진입점이다. 여기서는 레이아웃과 페이지 슬롯만 감싼다.
- `src/app/layouts/default.vue`: 대부분의 내부 페이지 공통 레이아웃으로 보이며 `CommonHeader`를 항상 렌더링한다.
- `src/app/layouts/main.vue`: 홈(`/`)에서 사용하는 별도 레이아웃이다. 이 레이아웃도 자체 헤더를 가진다.
- `src/app/components/layout/common-layout/CommonHeader.vue`: 현재 공통 헤더 컴포넌트다. 디스코드 ID 표시는 아직 없다.
- `src/app/utils/api-client.ts`: Axios 인스턴스가 정의되어 있으며 현재는 `token` 쿠키만 요청 헤더에 주입한다.
- `src/app/composables/query/useGet.ts`: Vue Query 기반 GET 공통 훅이다. 페이지/스토어 수준 데이터 조회에 쓰이지만 라우트 진입 제어용으로는 너무 늦다.
- `src/app/utils/api-error-handler.ts`: API 응답 코드에 따라 일부 리다이렉트는 가능하나 `TEAM_ONLY`에 대한 전역 차단 분기는 아직 없다.
- `nuxt.config.ts`: `dir.middleware`가 `app/middleware`로 지정되어 있으나 실제 `src/app/middleware` 폴더는 아직 없다.

### 현재 로직 분석
- 현재 프론트 진입 시 `users/check`를 자동 호출하는 지점이 없다.
- `CommonHeader`와 `main` 레이아웃 헤더는 각각 별개여서, 디스코드 ID를 표시하려면 공통 상태 저장소나 전역 상태가 필요하다.
- Axios 인터셉터에 `discordId` 헤더 주입이 아직 없어, `users/check` 호출을 넣더라도 식별자 전달 전략을 먼저 정해야 한다.
- 페이지 내부의 `useGet` 또는 개별 `onMounted` 호출로 접근 검사를 넣으면 페이지 수만큼 중복되고, SSR/초기 라우팅 시점 차단이 늦어진다.
- Nuxt 전역 라우트 미들웨어는 페이지 전환 전 실행되므로 요구사항인 "모든 페이지를 걸친 전역 차단"에 가장 직접적으로 맞는다.

## 영향도 평가
### 직접 영향
- `src/app/middleware` 신규 생성 및 전역 라우트 미들웨어 추가 가능성
- `src/app/utils/api-client.ts`의 공통 헤더 주입 로직 수정 가능성
- `src/app/stores/user.store.ts` 또는 신규 접근 상태 store 추가 가능성
- `src/app/components/layout/common-layout/CommonHeader.vue`와 `src/app/layouts/main.vue`의 헤더 표시 로직 수정 가능성
- `src/app/pages/block.vue` 신규 생성 필요

### 간접 영향
- SSR/CSR 혼합 환경에서 초기 네비게이션 성능에 영향 가능
- `TEAM_ONLY` 응답을 다른 API에서도 재사용할 경우 전역 에러 처리 정책과 충돌 가능
- 개발 환경 외 배포에서 전부 차단되는 의도라면 운영 환경 접근 정책과 SEO/robots 처리까지 검토가 필요
- `discordId` 확보 수단이 쿠키인지 쿼리인지 별도 주입인지 불명확하여, 실제 헤더 전달 방법 설계가 선행되어야 한다

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- `discordId`를 어떤 저장소에서 읽어 Axios 헤더로 넣을지 아직 확정되지 않음
- SSR에서도 동일하게 차단할지, 클라이언트 진입만 차단할지 정책 결정을 해야 함
- `block` 라우트 자체는 가드 대상에서 제외하지 않으면 리다이렉트 루프가 생길 수 있음
- `main` 레이아웃과 `default` 레이아웃의 헤더가 분리되어 있어 디스코드 ID 노출 방식을 통일해야 함

### 가정 / 불확실성
- `discordId`는 최종적으로 쿠키, 로컬 상태, 혹은 초기 쿼리 파라미터 중 하나에서 확보될 것으로 가정함
- 현재 요청은 "어디에 넣는 게 좋은가"에 대한 구조 설계 판단이며, 아직 실제 구현 승인 전 단계로 해석함
- `users/check` 응답은 HTTP 에러가 아니라 `BaseResponse.error` 형태이므로, 미들웨어에서는 응답 바디의 `error/code`를 직접 해석해야 함
