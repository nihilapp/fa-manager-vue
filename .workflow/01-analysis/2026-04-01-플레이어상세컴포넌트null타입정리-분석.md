# 분석 (Analysis)

## 요약
- 요청 내용: `PlayerDetail.vue`에서 `playerInfo` null 가능성 타입 경고가 반복되는 구조를 개선한다
- 범위: `src/app/components/players/PlayerDetail.vue`, `src/app/pages/players/detail/[id].vue`
- 구현 가능성: 가능

## 요구사항 요약
- 플레이어 상세 화면은 런타임에서 데이터 존재 여부를 분기하지만, 정적 타입 기준으로도 nullable 경고가 남지 않도록 구조를 정리해야 한다
- 최소 수정으로 화면 동작을 유지하면서 타입 추론을 더 명확하게 만들어야 한다

## 현재 상태
### 관련 파일
- `src/app/components/players/PlayerDetail.vue`: 현재 `playerStore.playerInfo`와 `playerStatusMap`을 직접 읽으며, `v-if="playerInfo"` 아래에서도 nullable 추론이 남는다
- `src/app/pages/players/detail/[id].vue`: 상세 데이터를 조회하고 `PlayerDetail`을 렌더하는 엔트리 페이지다
- `src/app/stores/player.store.ts`: `playerInfo`의 기본값은 `null`이며 타입은 `PlayerOutDto | null`이다

### 현재 로직 분석
- `PlayerDetail.vue`는 스토어의 nullable ref를 직접 사용한다
- 템플릿의 `v-if="playerInfo"`는 런타임 분기에는 충분하지만, Volar/TS는 스토어 ref와 템플릿 자동 unref가 섞인 경우 null 제거를 완전히 보장하지 않는다
- 현재 사용처는 `players/detail/[id].vue` 한 곳뿐이므로, 부모가 데이터 존재 시에만 컴포넌트를 렌더하고 자식에 필수 props를 전달하는 구조로 바꾸기 쉽다

## 영향도 평가
### 직접 영향
- `PlayerDetail.vue`는 더 이상 player store의 nullable 상태를 직접 읽지 않게 된다
- 상세 페이지는 `playerInfo` 존재 시 자식 컴포넌트를 렌더하고, 없으면 오류 화면을 직접 보여줘야 한다

### 간접 영향
- props 기반 확정 타입 구조로 바뀌면 향후 `PlayerDetail` 재사용 시 필요한 입력 계약이 명확해진다
- 스토어 상태와 표현 컴포넌트의 결합도가 낮아진다

## 실행 판단
### 구현 가능성
- 가능

### 예상 기술 난제
- 템플릿 조건 분기와 props 타입을 함께 맞춰야 하므로 부모/자식을 같이 수정해야 한다

### 가정 / 불확실성
- 현재 `PlayerDetail` 사용처는 한 곳만 확인됐다
- `playerStatusMap` 직접 접근 대신 스토어의 `getPlayerStatusInfo`를 사용하는 편이 non-null 단언 제거에 유리하다
