## 설계 준수 여부
- 항목 1:
  - 설계 기대값: `authHelper`가 운영/개발 공통 진입점이 되고, `user`, `isAdmin`, `isDevelopmentBypass`, `hasPermission`, `requireUser`, `error`를 포함한 단일 인증 컨텍스트를 제공해야 한다.
  - 실제 구현: `src/server/utils/auth.ts`에서 공통 컨텍스트 생성 함수를 두고 운영/개발 모두 동일 필드 구조를 반환하도록 정리했다. 개발 환경에서는 fallback user를 우선 탐색하고, 없으면 에러를 반환하도록 처리했다.
  - 일치 여부: 부분 일치

- 항목 2:
  - 설계 기대값: 페이지 접근 규칙은 `authHelper` 또는 그 하위 공통 함수만 재사용해야 하며 독립 규칙을 가지면 안 된다.
  - 실제 구현: `src/server/utils/access.ts`의 `checkCommunityAccess(event)`가 `authHelper(event)`의 `error`만 소비하는 경량 래퍼로 변경되었고, `src/server/middleware/access.ts`와 `src/server/api/users/check.get.ts`가 이를 그대로 사용한다.
  - 일치 여부: 일치

- 항목 3:
  - 설계 기대값: `sessions/[id].delete.ts` 같은 예외 엔드포인트의 직접 헤더 검사와 중복 인증 분기를 제거하고 `authHelper` 결과만으로 판단해야 한다.
  - 실제 구현: `src/server/api/sessions/[id].delete.ts`에서 `getDiscordId(event)` 기반 직접 검사 코드를 제거하고 `authHelper` 결과만 사용하도록 정리했다.
  - 일치 여부: 일치

- 항목 4:
  - 설계 기대값: 개발 환경 우회 시 `user!.id`, `creatorId`, `updaterId`를 사용하는 CUD API가 null 참조 없이 동작할 수 있는 사용자 식별 전략이 필요하다.
  - 실제 구현: `src/server/utils/auth.ts`에서 개발 환경일 때 요청 헤더 사용자 조회 실패 시에도 fallback user를 재시도하고, 최종적으로 사용자를 찾지 못하면 에러를 반환해 쓰기 API가 null 참조 전에 중단되도록 했다.
  - 일치 여부: 부분 일치

## 테스트 결과
### 단위 테스트
- 별도 단위 테스트는 작성하지 않았다.

### 수동 테스트
- 코드 경로 기준 수동 점검:
  - 운영 환경에서는 `X-Discord-ID`가 없을 때 `authHelper`가 `UNAUTHORIZED`를 반환한다.
  - 운영 환경에서는 헤더는 있으나 유저가 없을 때 `NOT_FOUND`를 반환한다.
  - 개발 환경에서는 요청 헤더 유저가 없더라도 fallback user가 있으면 우회 컨텍스트를 반환한다.
  - 개발 환경에서 fallback user도 없으면 `USER_NOT_FOUND` 에러를 반환해 CUD API의 `user!.id` 접근 전에 차단된다.
  - 페이지 접근 유틸은 `authHelper` 결과를 재사용하므로 API와 페이지의 차단 기준이 동일하다.

## 미결 사항
- 설계 문서에서 제안된 `requireUser()` 보조 함수를 실제 API 호출부에서 적극 사용하도록 전면 정리하지는 않았다.
- 개발 환경에 DB 유저가 전혀 없는 경우까지 완전 우회시키지는 않았다. 현재는 null 참조 방지를 우선해 `USER_NOT_FOUND`로 차단한다.
- `authHelper`를 사용하는 모든 GET/마이페이지 계열 API까지 호출부 안정성을 전수 점검한 것은 아니며, 이번 작업은 문서 범위대로 페이지 접근과 CUD 중심 통합에 우선 집중했다.
- `pnpm.cmd exec nuxi typecheck`는 이번 작업에서 재시도하지 않았다.
