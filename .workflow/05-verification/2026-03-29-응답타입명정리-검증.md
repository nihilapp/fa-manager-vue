## 설계 준수 여부
- 항목 1: 서버 응답 인터페이스와 헬퍼 명칭 통일
  - 설계 기대값: `response.types.ts`와 `base-response.ts`가 모두 `BaseApiResponse` 기준으로 동작해야 한다.
  - 실제 구현: `src/server/types/response.types.ts`에 `BaseApiResponse<T>`를 정의하고 `src/server/utils/base-response.ts` 클래스명 및 반환 타입을 전환했다.
  - 일치 여부: 일치
- 항목 2: 서버 사용처 전면 치환
  - 설계 기대값: `src/server/api/**`, `src/server/utils/**`에서 구 명칭 사용이 제거되어야 한다.
  - 실제 구현: 서버 API 템플릿, 유틸리티, 엔드포인트 전반의 `BaseResponse`/`BaseResponseType`를 `BaseApiResponse`로 치환했다.
  - 일치 여부: 일치
- 항목 3: 앱 응답 타입 통합
  - 설계 기대값: 앱 `BaseApiResponse`가 서버 정의를 재사용하고 `ApiErrorResponse`는 `BaseApiResponse<null>` 별칭이어야 한다.
  - 실제 구현: `src/app/composables/query/useApiRequest.ts`는 서버 타입을 로컬 별칭으로 사용하고, `src/app/types/manual-app-imports.d.ts`는 서버 타입 직접 참조로 통일했다.
  - 일치 여부: 일치
- 항목 4: 오토임포트 충돌 해소
  - 설계 기대값: 앱/서버 오토임포트가 같은 명칭을 쓰더라도 서버 값 심볼 사용이 깨지지 않아야 한다.
  - 실제 구현: `useApiRequest.ts`에서 `BaseApiResponse`, `ApiErrorResponse` export를 제거하고 수동 타입 선언으로만 제공해 서버 Nitro 오토임포트 값과 충돌하지 않게 했다.
  - 일치 여부: 일치

### 수동 테스트

### 통합 테스트
- `pnpm.cmd run postinstall` 실행 성공
- `pnpm.cmd exec tsc --noEmit --pretty false` 실행 성공
- `pnpm.cmd exec nuxi typecheck`는 `vue-tsc` 로컬 부재로 npm registry 접근을 시도하다 네트워크 제한(`EACCES`)으로 실패

## 미결 사항
- `nuxi typecheck`는 로컬 `vue-tsc`가 설치되지 않은 상태라 완전 검증으로 마무리하지 못했다.
- 이번 작업 범위는 `src/app`, `src/server`, 오토임포트 선언에 한정되어 `md/`, `PRD/` 문서 문자열은 갱신하지 않았다.
- 이후 타입 계약 보정에서 `BaseApiResponse<T>`의 `data`는 `T`로 유지하고, nullable 성공 응답은 제네릭으로 표현하는 방향으로 추가 정정했다.
