# 검증 (Verification)

## 요약
- 검증 대상: 개발환경 우선 권한 통과 구조와 owner 스코프 기반 CUD 라우트 우회 적용
- 전체 결과: 부분 수동 검증 + 타입 검증 통과
- 배포 가능 여부: 가능

## 설계 준수 여부
- 항목 1:
  - 설계 기대값: `authHelper()`가 개발환경에서는 우선 통과를 반환해야 한다.
  - 실제 구현: `src/server/utils/auth.ts`에서 개발환경이면 디스코드 식별 실패 여부와 무관하게 실제 사용자, 첫 사용자, 개발용 대체 사용자 중 하나를 반환하고 `error`를 없앴다.
  - 일치 여부: 일치

- 항목 2:
  - 설계 기대값: 어드민은 개발환경과 동일하게 owner 스코프를 우회해야 한다.
  - 실제 구현: 캠페인/캐릭터/세션 계열 CUD 라우트에서 `isAdmin`이면 `userId` 또는 소유 캠페인 필터를 적용하지 않도록 수정했다.
  - 일치 여부: 일치

- 항목 3:
  - 설계 기대값: 기존 API 계약은 유지하고 내부 권한 판정만 변경해야 한다.
  - 실제 구현: URL, DTO, 응답 포맷은 유지했고 내부 인증/조회 조건만 바꿨다.
  - 일치 여부: 일치

## 미결 사항
- 개발환경에서 실제 플레이어가 전혀 없을 경우 일부 CUD는 DB FK 제약 때문에 비즈니스 실패가 날 수 있다. 이는 권한 차단이 아니라 데이터 선행조건 부족 문제다.
- `players/[id].put.ts`, `players/[id].delete.ts`, `players/[id].get.ts`는 파일명과 구현 내용이 일치하지 않는 기존 이상 상태가 남아 있다.
- GET 라우트 전체를 어드민 전권 조회로 넓히지는 않았다. 이번 작업 범위는 마스터 요구대로 권한 차단 해소와 CUD owner 우선순위 정리에 집중했다.
