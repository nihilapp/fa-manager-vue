# 설계 (Design)

## 요약
- 변경 대상: `md` 폴더 문서 4종
- 설계 방향: 과거 작업 로그형 문서를 현재 참조 문서형 구조로 재정리
- 주요 결정 사항: API 문서는 공통 규칙 + 도메인별 엔드포인트 중심으로 재작성

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `md/api.md`: 공통 인증/응답 형식, 현재 엔드포인트, UI 연동 우선순위 중심으로 재작성
- `md/db-table-dto-reference.md`: `players` 기준 명칭과 현재 DTO 파일 구조 중심으로 재작성
- `md/dto-migration-guide.md`: 과거 진행 로그 대신 현재 DTO 전략과 남은 예외 범위 중심으로 재작성
- `md/2026-03-18-admin-permission-expansion-plan.md`: 계획 문서에서 현행 권한 정책 요약 문서로 전환

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 없음
- 문서만 갱신

## 로직 흐름
1. `md` 문서에서 오래된 `users` 기준 표현과 과거 완료 보고 성격의 내용을 제거한다.
2. 현재 실제 서버 API, DTO, 프런트 사용처를 기준으로 핵심 스펙을 다시 작성한다.
3. 검색으로 구형 명칭이 문서 내에서 어떤 의미로 남았는지 확인하고, 설명 목적의 잔존만 허용한다.
4. workflow 기록과 완료 이력을 남긴다.

## 대안 검토
### 옵션 A
- 설명: 기존 문서에 부분 패치만 적용
- 장점: 수정량이 적음
- 단점: 과거 진행 로그와 현재 기준이 계속 섞여 읽기 어려움
- 미채택 이유: API/UI 연동 기준 문서로 쓰기엔 재구성이 필요함

### 최종 선택
- 선택 이유: 실제 연동에 필요한 정보만 빠르게 찾을 수 있게 문서 목적을 다시 맞추는 편이 효율적임
- 기대 효과: 마스터와 이후 작업자가 `md` 폴더만 보고도 현재 구조를 빠르게 이해 가능

## 제약 및 참고 사항
- 문서는 현재 구현 기준만 반영하고, 미구현 기능은 계획 또는 예외로만 짧게 남긴다.
