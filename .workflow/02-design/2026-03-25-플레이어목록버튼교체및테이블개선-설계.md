# 설계 (Design)

## 요약
- 변경 대상: `src/app/pages/players/index.vue`, `src/app/components/table/DataTable.vue`, `src/app/components/table/Paginator.vue`
- 설계 방향: 플레이어 목록 액션은 공통 `Button.vue` 기반으로 정리하고, 테이블은 구조는 유지하되 밝고 얇은 경계, 넉넉한 여백, hover 중심 강조를 적용해 에어테이블에 가까운 데이터 그리드 톤으로 개선
- 주요 결정 사항:
  - 액션 버튼은 상세보기와 수정 두 개로 구성
  - 수정 버튼은 `useUserStore()`의 `isAdmin`이 true일 때만 렌더링
  - 상세보기는 기존 `players/[id].vue`로 라우팅
  - 수정 버튼은 수정 전용 페이지가 없으므로 동일 상세 페이지로 진입하되, 관리자 전용 액션으로 시각적 의미만 먼저 부여
  - 테이블 스타일은 특정 페이지 전용이 아니라 공통 `DataTable.vue`와 `Paginator.vue`에서 재사용 가능한 수준으로 정리

## 로직 흐름
1. 플레이어 목록 페이지가 마운트되면 기존과 동일하게 `useGetUserList().execute()`를 호출해 목록을 채운다.
2. 페이지는 `userStore`에서 `userList`, `userPageData`, `isAdmin`을 읽는다.
3. 각 행의 액션 셀은 공통 `Button.vue` 두 개로 구성한다.
4. 상세보기 버튼은 항상 렌더링되고, 해당 플레이어의 `/players/[id]` 경로로 이동한다.
5. 수정 버튼은 `isAdmin`이 true일 때만 렌더링된다.
6. 수정 버튼의 목적지는 당장 별도 수정 페이지가 없으므로 동일한 `/players/[id]` 경로를 사용하되, 아이콘/라벨로 관리자 액션임을 구분한다.
7. `DataTable.vue`는 기존 role 구조는 유지하면서 헤더, 행, 셀 클래스를 더 밝고 가벼운 디자인으로 재조합한다.
8. 행은 zebra 대비보다 hover와 active-like focus 대비를 더 강조하고, 셀 구분선은 dashed 중심에서 옅은 solid 중심으로 전환한다.
9. 상태 슬롯은 badge 스타일로 감싸 에어테이블식 필드 강조를 일부 반영한다.
10. `Paginator.vue`는 페이지 정보 라벨과 페이지 이동 버튼을 pill 그룹 형태로 재구성해 테이블과 시각 언어를 맞춘다.

## 대안 검토
### 옵션 A
- 설명: `players/index.vue`만 수정해 버튼 교체와 테이블 주변 래퍼 클래스만 덧씌움
- 장점: 변경 범위가 작음
- 단점: 공통 테이블 컴포넌트의 무거운 기본 스타일이 그대로 남아 에어테이블 느낌을 만들기 어려움
- 미채택 이유: 페이지별 땜질이 되어 재사용성과 일관성이 떨어짐

### 옵션 B
- 설명: `DataTable.vue`를 기능적으로 크게 개편해 정렬, 컬럼 고정, 선택 상태, 툴바까지 한 번에 추가
- 장점: 에어테이블과 더 가까운 사용성을 만들 수 있음
- 단점: 이번 요청 범위를 크게 넘고, 설계/검증 비용이 급격히 증가함
- 미채택 이유: 현재는 시각 언어와 액션 사용성 개선이 우선이며, 기능 확장은 별도 작업으로 분리하는 편이 안전함

### 최종 선택
- 선택 이유: 공통 테이블의 스타일 톤만 정돈해도 전체 인상이 크게 개선되고, 플레이어 목록의 버튼 정비와 자연스럽게 맞물린다
- 기대 효과: 버튼 사용 규칙이 통일되고, 플레이어 목록은 더 가볍고 읽기 쉬운 데이터 그리드로 보이게 된다

## 제약 및 참고 사항
- `isAdmin`은 개발환경에서 true가 될 수 있으므로 로컬에서는 수정 버튼이 항상 보일 수 있다.
- 수정 전용 페이지가 아직 없으므로 버튼 라벨은 "수정"이지만 실제 이동 경로는 상세 페이지 재사용으로 제한된다.
- 프로젝트 Tailwind 클래스 사용 범위가 제한될 수 있어, `tailwind.css`에서 확인된 클래스 중심으로 조합해야 한다.
