# 설계 (Design)

## 요약
- 변경 대상: `PlayerOutDto`, `players/index.get.ts`, `players/[id].get.ts`, `players/me.get.ts`
- 설계 방향: `playersRelations` 정의 기준 8개 관계를 플레이어 GET 응답 전반에 일관되게 포함한다.
- 주요 결정 사항:
  - 목록/단건/내 정보 조회 모두 동일한 `with` 구조를 사용한다.
  - `currencyTransactions`는 우선 1단계 관계 row만 포함하고 중첩 `user`, `character`는 추가하지 않는다.
  - `me.get.ts`는 `authHelper` 검증 뒤 player id 기준으로 DB를 다시 조회한다.

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/server/types/dto/player.dto.ts`
  - `currencyTransactions?: CurrencyTransactionOutDto[]`를 추가해 플레이어 응답 DTO를 관계 정의와 맞춘다.
- `src/server/api/players/index.get.ts`
  - 누락된 `currencyTransactions: true`를 `with`에 추가한다.
- `src/server/api/players/[id].get.ts`
  - 기본 `findFirst` 조회를 relation query로 전환하고 8개 관계를 모두 포함한다.
- `src/server/api/players/me.get.ts`
  - 인증 컨텍스트의 `user.id`를 기준으로 relation query를 수행해 내 정보 응답에도 8개 관계를 포함한다.

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 플레이어 GET 응답에 `currencyTransactions`가 추가된다.
- `[id].get`, `me.get` 응답에도 목록과 동일한 관계 필드가 포함된다.

## 로직 흐름
1. 플레이어 조회 요청에서 권한 또는 사용자 식별을 먼저 검증한다.
2. 조회 시 `playersTable` relation query에 8개 관계(`campaigns`, `campaignMembers`, `characters`, `sessionPlayers`, `sessionLogs`, `docs`, `logHistories`, `currencyTransactions`)를 공통으로 포함한다.
3. 조회 결과가 없으면 기존과 동일한 NOT_FOUND 응답을 반환한다.
4. 조회 결과가 있으면 `PlayerOutDto` 계약에 맞춰 관계 포함 payload를 응답한다.

## 대안 검토
### 옵션 A
- 설명: `[id].get.ts`만 수정한다.
- 장점: 변경 범위가 가장 작다.
- 단점: `me.get.ts`와 목록 조회의 관계 일관성이 여전히 깨진다.
- 미채택 이유: 마스터 요청의 "플레이어 GET" 범위를 충분히 충족하지 못한다.

### 옵션 B
- 설명: `authHelper` 자체를 relation query로 변경한다.
- 장점: `me.get.ts` 외에도 인증 컨텍스트가 richer payload를 갖게 된다.
- 단점: 인증에 의존하는 모든 서버 로직의 payload와 성능에 연쇄 영향이 생긴다.
- 미채택 이유: 이번 작업 범위 대비 영향 범위가 과도하다.

### 최종 선택
- 선택 이유: 플레이어 GET 엔드포인트만 명시적으로 보강하면 요구를 충족하면서도 영향 범위를 제한할 수 있다.
- 기대 효과: 플레이어 목록/상세/내 정보 조회 결과가 동일한 관계 기준을 따르게 된다.

## 제약 및 참고 사항
- `currencyTransactions` 중첩 관계는 이번 범위에서 추가하지 않는다.
- 기존 API 주석은 제거하지 않는다.
