# 설계 (Design)

## 요약
- 변경 대상: `src/server/utils/auth.ts`, `src/server/utils/request-header.ts`, 필요 시 공용 쿠키 상수 또는 유틸
- 설계 방향: 서버 인증 공통 경로에서 `x-discord-id` 헤더를 최우선 식별값으로 사용하고, 이를 쿠키에 저장하여 후속 요청에서는 쿠키로 동일 사용자를 복원한다.
- 주요 결정 사항:
  - 식별 우선순위는 `헤더 > 쿠키 > 개발 환경 ADMIN_DISCORD_ID`로 둔다.
  - 헤더가 있으면 쿠키를 즉시 갱신한다.
  - 쿠키 값으로 사용자 조회가 실패하면 stale 쿠키를 제거한다.

## 아키텍처 변화
### 신규 구성 요소
- 공용 쿠키 키 상수 또는 로컬 상수
  - 예: `DISCORD_ID_COOKIE_KEY = 'discord_id'`
- 필요 시 쿠키 옵션 계산 함수
  - 개발/운영 `secure` 분기와 `sameSite`, `httpOnly`, `path`, `maxAge`를 공통으로 유지

### 수정 구성 요소
- `src/server/utils/request-header.ts`
  - 현재 헤더 읽기 유틸은 유지
  - 필요 시 쿠키 읽기용 보조 함수 추가 가능
- `src/server/utils/auth.ts`
  - 현재 요청에서 `x-discord-id` 헤더를 먼저 읽는다.
  - 헤더가 있으면 해당 값을 쿠키에 저장한다.
  - 헤더가 없으면 쿠키의 `discord_id`를 읽어 식별값으로 사용한다.
  - 둘 다 없고 개발 환경이면 `ADMIN_DISCORD_ID`를 마지막 fallback으로 사용한다.
  - 사용자 조회 실패 시 식별 근원이 쿠키였다면 stale 쿠키를 삭제한다.

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 외부 API 응답 계약 변경 없음
- 내부 인증 계약 변경:
  - `authHelper(event)`는 헤더만이 아니라 쿠키 기반 식별도 지원한다.
  - 후속 API 요청은 별도 헤더 없이도 쿠키가 있으면 같은 사용자를 복원할 수 있다.

## 로직 흐름
1. 요청 수신 시 `x-discord-id` 헤더를 읽는다.
2. 헤더 값이 있으면 이를 현재 요청의 식별값으로 채택하고 쿠키 `discord_id`를 갱신한다.
3. 헤더 값이 없으면 쿠키 `discord_id`를 읽어 식별값으로 사용한다.
4. 운영 환경에서는 식별값이 없으면 비인증 에러를 반환한다.
5. 개발 환경에서는 헤더와 쿠키가 모두 없을 때만 `ADMIN_DISCORD_ID`를 fallback으로 사용한다.
6. 식별값으로 사용자 조회를 수행한다.
7. 조회 성공 시 인증 컨텍스트를 반환한다.
8. 조회 실패 시:
  - 식별 근원이 쿠키였다면 stale 쿠키를 제거한다.
  - 기존 규칙대로 `USER_NOT_FOUND` 또는 `REQUIRE_DISCORD_ID`를 반환한다.

## 대안 검토
### 옵션 A
- 설명: 프런트 Axios 인터셉터에서 `x-discord-id`를 직접 유지하도록 구현
- 장점:
  - 서버 쿠키 로직이 단순하다.
- 단점:
  - 최초 헤더 값을 클라이언트가 읽을 수 있어야 하는데, 현재 구조상 보장되지 않는다.
  - 보안/일관성 측면에서 서버 공통 처리보다 취약하다.
- 미채택 이유:
  - 최초 요청 헤더 기반 식별을 후속 요청까지 보존하는 목적에는 서버 쿠키 동기화가 더 적합하다.

### 옵션 B
- 설명: `authHelper`에서 헤더 우선, 쿠키 fallback, 개발 fallback을 공통 처리
- 장점:
  - 모든 API가 동일 정책을 공유한다.
  - `/users/me` 같은 앱 공통 호출이 자연스럽게 복구된다.
  - stale 쿠키 정리까지 한 곳에서 관리할 수 있다.
- 단점:
  - 서버 인증 유틸 책임이 조금 늘어난다.

### 최종 선택
- 선택 이유:
  - 인증 규칙의 단일 기준점을 유지하려면 `authHelper` 한 곳에서 식별 소스 우선순위를 관리하는 것이 가장 일관적이다.
- 기대 효과:
  - 디스코드 버튼 최초 진입 후 새로고침 및 후속 API 요청에서 사용자 식별 유지
  - 헤더 부재로 인한 `/users/me` 실패 감소
  - 잘못된 쿠키 자동 정리

## 제약 및 참고 사항
- 디스코드 ID는 민감한 인증 토큰은 아니지만 사용자 식별자이므로 `httpOnly` 쿠키로 관리하는 편이 안전하다.
- 운영 환경에서는 `secure: true`, 개발 환경에서는 `secure: false` 분기가 필요하다.
- 쿠키 만료 시간은 너무 짧지 않게 두되, 장기 세션 정책은 추후 별도 보안 요구사항과 맞춰 조정 가능하다.
