# 설계 (Design)

## 요약
- 변경 대상: `player.store.ts`, `PlayerListTable.vue`, `PlayerDetail.vue`
- 설계 방향: 플레이어 상태 조회를 공통 헬퍼로 추상화하고, 미정의/예상 외 상태에도 안전한 기본 표시를 제공한다.
- 주요 결정 사항: UI 컴포넌트에서 `Map.get(... )!` 직접 접근을 제거하고 스토어에서 fallback 정보를 반환한다.

- 없음

### 수정 구성 요소
- `src/app/stores/player.store.ts`
  - `playerStatusMap`을 그대로 유지한다.
  - 상태 문자열을 받아 안전한 `StatusInfo`를 반환하는 헬퍼를 추가한다.
  - 맵에 없는 상태는 기본 레이블과 회색 배지로 치환한다.
- `src/app/components/players/PlayerListTable.vue`
  - 상태 슬롯에서 직접 맵을 조회하지 않고 스토어 헬퍼를 통해 뱃지 정보를 계산한다.
- `src/app/components/players/PlayerDetail.vue`
  - 상세 화면 상태 표시도 동일 헬퍼를 사용해 목록/상세의 표시 규칙을 통일한다.

## 로직 흐름
1. 플레이어 목록 또는 상세 화면이 상태 값을 표시하려고 한다.
2. 컴포넌트는 전달받은 상태 문자열을 스토어 헬퍼에 넘긴다.
3. 헬퍼는 `playerStatusMap`에서 일치 항목을 찾는다.
4. 존재하면 해당 `name`, `color`를 반환한다.
5. 존재하지 않으면 기본값(`알 수 없음`, `gray`)을 반환한다.
6. 컴포넌트는 반환된 정보를 그대로 `StatusBadge`에 전달한다.

## 대안 검토
### 옵션 A
- 설명: `PlayerListTable.vue`와 `PlayerDetail.vue` 각각에서 `??` fallback 로직을 직접 구현
- 장점: 수정량이 적다.
- 단점: 상태 표시 규칙이 중복되고 향후 다른 화면에서 재사용되지 않는다.
- 미채택 이유: 동일 패턴 재발 가능성이 높아 공통 헬퍼가 더 유지보수에 유리하다.

### 옵션 B
- 설명: 서버 응답 직후 모든 플레이어 상태를 정규화한다.
- 장점: UI는 항상 정규화된 값만 받는다.
- 단점: 서버 계약과 다른 값을 프런트에서 묵시적으로 교정하게 되며, 데이터 이상 탐지가 어려워진다.
- 미채택 이유: 현재 문제는 렌더 안정성 확보가 우선이며, 표시 계층 방어가 더 단순하고 명확하다.

### 최종 선택
- 선택 이유: 스토어를 상태 표시 기준선으로 삼으면 목록/상세가 같은 규칙을 공유하면서 예외를 제거할 수 있다.
- 기대 효과: 플레이어 상태 값이 비정상이어도 목록 페이지와 상세 페이지가 모두 안전하게 렌더링된다.

## 제약 및 참고 사항
- Nuxt 오토임포트 규칙을 유지하고 불필요한 직접 import를 추가하지 않는다.
- 기존 `playerStatusMap` 소비 패턴과 충돌하지 않도록 헬퍼는 추가만 하고 기존 맵은 유지한다.
- 추후 서버 데이터 정합성 점검이 필요할 수 있으나 이번 작업 범위에는 포함하지 않는다.
