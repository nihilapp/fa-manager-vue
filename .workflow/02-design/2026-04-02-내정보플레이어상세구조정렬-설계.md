# 설계 (Design)

## 요약
- 변경 대상: `src/app/pages/me/index.vue`, `src/app/pages/players/detail/[id].vue`, `src/app/components/players/PlayerProfileDetail.vue`, `src/app/components/players/PlayerDetail.vue`
- 설계 방향: 페이지는 각자 데이터 로딩만 담당하고, 실제 렌더링은 공통 상세 컴포넌트 하나로 통합한다.
- 주요 결정 사항:
  - `me`는 `mine` 전용 API와 스토어 축을 유지한다.
  - 일반 플레이어 상세는 기존 `playerInfo`, `playerCampaignList`, `playerSessionList` 축을 유지하고 캐릭터는 `characters index + userId` 조회로 채운다.
  - 공통 렌더러는 스토어를 직접 읽지 않고 props만 받아 렌더링한다.

## 아키텍처 변화
### 신규 구성 요소
- `src/app/components/players/PlayerProfileDetail.vue`
  - 플레이어 기본 정보, 캐릭터/캠페인/세션 섹션, 선택적 뒤로가기 버튼을 렌더링하는 공통 상세 뷰

### 수정 구성 요소
- `src/app/pages/me/index.vue`
  - `myInfo`, `myCharacterList`, `myCampaignList`, `mySessionList` 및 각 페이지 데이터 축을 읽고 공통 뷰에 전달한다.
- `src/app/pages/players/detail/[id].vue`
  - 누락된 캠페인/캐릭터 조회를 추가하고 공통 뷰에 전달한다.
- `src/app/components/players/PlayerDetail.vue`
  - 기존 호출부 호환을 위해 신규 공통 뷰를 그대로 감싸는 래퍼로 유지하거나 동일 렌더링 컴포넌트로 대체한다.

## 로직 흐름
1. `me/index.vue`는 마운트 시 `useGetMyInfo`, `useGetMyCharacterList`, `useGetMyCampaignList`, `useGetMySessionList`를 실행한다.
2. `players/detail/[id].vue`는 마운트 시 `useGetPlayerInfo`, `useGetPlayerCampaignList`, `useGetPlayerSessionList`, `useGetCharacterList({ userId: id })`를 실행한다.
3. 두 페이지는 각각 제목과 버튼 옵션만 다르게 주고, 나머지 정보는 같은 props 계약으로 공통 뷰에 전달한다.
4. 공통 뷰는 플레이어 상태, 각 섹션 총 건수, 실제 리스트 항목, 빈 상태 메시지를 렌더링한다.

## 대안 검토
### 옵션 A
- 설명: `me` 전용 컴포넌트를 새로 만들고 `PlayerDetail`을 복제한다.
- 장점: 빠르게 구현 가능하다.
- 단점: 같은 화면 구조가 두 파일로 갈라져 이후 수정 시 중복이 생긴다.
- 미채택 이유: 마스터 요청이 동일 구조와 동일 렌더링 컴포넌트 사용에 가깝다.

### 옵션 B
- 설명: 기존 `PlayerDetail.vue`를 범용 props 구조로 확장해 `me`와 플레이어 상세가 함께 사용한다.
- 장점: 파일 수가 적다.
- 단점: 이름이 플레이어 상세 전용 의미를 유지해 `me` 사용처에서 의미가 흐려진다.
- 미채택 이유: 공통 역할이 드러나는 별도 이름이 유지보수에 더 명확하다.

### 최종 선택
- 선택 이유: 공통 렌더링을 별도 컴포넌트로 분리하면 페이지와 표현 책임이 명확해지고 `mine`/일반 상세 분리도 유지된다.
- 기대 효과: `me`와 플레이어 상세가 같은 UI 계약을 공유하면서도 서로 다른 데이터 축을 안정적으로 사용할 수 있다.

## 제약 및 참고 사항
- 캐릭터 상세 전용 플레이어 하위 API는 추가하지 않는다.
- Tailwind 사용은 현재 프로젝트 스타일 규칙을 따르며 기존 사용 중인 유틸 클래스 위주로 구성한다.
