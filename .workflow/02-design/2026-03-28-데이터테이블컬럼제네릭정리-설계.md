# 설계 (Design)

## 요약
- 변경 대상: `manual-app-imports.d.ts`, `useColumns.ts`, `column-builder.ts`, `DataTable.vue`
- 설계 방향: `DataTableColumn<T>`의 타입 인자를 전역 별칭, 유틸 함수, 빌더 클래스, 컴포넌트 내부 헬퍼까지 일관되게 전달한다.
- 주요 결정 사항: 호출부 부담을 줄이기 위해 기본 제네릭은 유지하고, 빌더 클래스는 `T = any` 기본값을 둔다.

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/app/types/manual-app-imports.d.ts`
  - 전역 타입 별칭을 `type DataTableColumn<T = any> = import(...).DataTableColumn<T>;` 형태로 수정
- `src/app/composables/useColumns.ts`
  - 입력 배열과 반환값에 동일한 `T`를 유지하는 제네릭 함수로 변경
- `src/app/utils/column-builder.ts`
  - 클래스와 정적 팩토리를 제네릭화하여 `key`와 `build()` 결과에 `T`가 반영되도록 변경
- `src/app/components/table/DataTable.vue`
  - 내부 헬퍼 시그니처를 `DataTableColumn<T>` 기반으로 변경

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 런타임 API 변경 없음
- 타입 레벨 인터페이스만 강화됨

## 로직 흐름
1. 전역 타입 별칭에서 `DataTableColumn<T>`를 그대로 노출한다.
2. `useColumns<T>`가 컬럼 배열을 받으면 타입 인자를 보존한 채 반환한다.
3. `ColumnBuilder<T>`가 `keyof T | (string & {})` 범위의 key를 받아 `DataTableColumn<T>`를 누적 생성한다.
4. `DataTable.vue` 내부 유틸 함수도 같은 `T` 컨텍스트를 사용해 컬럼 속성 접근 타입을 유지한다.

## 대안 검토
### 옵션 A
- 설명: 선언 파일만 수정하고 나머지 유틸은 그대로 둔다.
- 장점: 수정 범위가 가장 작다.
- 단점: 실제 사용 유틸에서 제네릭 정보가 계속 소실된다.
- 미채택 이유: 마스터 요청이 오토임포트 파일 중심이지만, 현재 연관 유틸도 같은 문제를 직접 갖고 있다.

### 옵션 B
- 설명: `DataTableColumn`의 기본 타입 매개변수를 `unknown`으로 바꿔 더 엄격하게 만든다.
- 장점: 타입 안정성이 높아진다.
- 단점: 기존 사용처에 불필요한 수정이 크게 번질 수 있다.
- 미채택 이유: 이번 작업 범위는 선언/시그니처 정합성 복구이며, 타입 정책 변경까지 확대하지 않는다.

### 최종 선택
- 선택 이유: 기본 제네릭을 유지한 채 연결 지점들만 정리하면 호출부 충격 없이 제네릭 이점을 복구할 수 있다.
- 기대 효과: 오토임포트 전역 타입과 컬럼 관련 유틸에서 `keyof T` 기반 타입 추론이 다시 동작한다.
