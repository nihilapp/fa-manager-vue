# 설계 (Design)

## 요약
- 변경 대상: 세션/캠페인 쿼리 DTO, 플레이어 하위 세션/캠페인 API, 플레이어 하위 쿼리 키 타입
- 설계 방향: 필터 필드는 DTO 본체로 흡수하고, 쿼리 키는 `DTO + playerId`를 명시적 인터페이스로 표현
- 주요 결정 사항: API의 교차 타입 선언을 제거하고 DTO를 단일 기준점으로 사용

## 아키텍처 변화
### 신규 구성 요소
- `PlayerSessionsQueryKeyParams`: `SessionQueryDto`에 `playerId`를 결합한 쿼리 키 전용 인터페이스
- `PlayerCampaignsQueryKeyParams`: `CampaignQueryDto`에 `playerId`를 결합한 쿼리 키 전용 인터페이스

### 수정 구성 요소
- `src/server/types/dto/session.dto.ts`
  - `role?: SessionRole`
  - `characterId?: number`
- `src/server/types/dto/campaign.dto.ts`
  - `role?: CampaignRole`
- `src/server/api/players/[id]/sessions.get.ts`
  - `getQuery<SessionQueryDto>`로 단순화
- `src/server/api/players/[id]/campaigns.get.ts`
  - `getQuery<CampaignQueryDto>`로 단순화
- `src/app/utils/query-keys.ts`
  - 플레이어 하위 키 파라미터를 명시적 인터페이스로 정의

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 변경 전: DTO 외부에서 교차 타입으로 필터 필드 추가
- 변경 후: DTO 본체에 필터 필드 포함, API는 DTO만 사용

## 로직 흐름
1. 클라이언트와 서버가 세션/캠페인 목록 필터를 각각 DTO 기준으로 해석한다.
2. 플레이어 하위 API는 `getQuery<DTO>`로 필터를 수신한다.
3. `role`, `characterId` 필터는 DTO에 정의된 필드로 직접 처리된다.
4. 플레이어 하위 쿼리 키는 동일 DTO 필드에 `playerId` 경로 파라미터를 더해 캐시 키를 구성한다.

## 대안 검토
### 옵션 A
- 설명: API 파일마다 기존처럼 교차 타입을 유지
- 장점: DTO 변경 없이 빠르게 동작
- 단점: 수용 가능한 쿼리 파라미터 정의가 파일별로 흩어진다
- 미채택 이유: 마스터 요청과 반대이며 타입 기준점이 불명확하다

### 최종 선택
- 선택 이유: DTO가 단일 진실 공급원 역할을 하게 되어 서버 API와 쿼리 키 정의가 함께 단순해진다
- 기대 효과: 필터 파라미터 추적성과 재사용성이 좋아진다

## 제약 및 참고 사항
- `playerId`는 라우트 경로 파라미터라 DTO에 넣지 않고 쿼리 키 전용 인터페이스에만 둔다.
