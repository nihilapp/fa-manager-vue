# 설계 (Design)

## 요약
- 변경 대상: `src/app/composables/query/*.ts`, `src/app/composables/query/useApiRequest.ts`, `src/app/stores/user.store.ts`, `nuxt.config.ts`, `src/app/plugins/vue-query.ts`
- 설계 방향: `useGet`은 `useFetch` 기반 수동 실행형으로, 나머지 mutation 계열은 `$fetch` 기반 수동 실행형으로 재구성한다.
- 주요 결정 사항:
  - `useGet`의 기본 자동 요청은 비활성화한다.
  - `useGet ~ useDelete`라는 훅 이름과 옵션 인터페이스는 유지한다.
  - 실제 GET 호출부는 `execute()`를 명시 호출하도록 수정한다.
  - Vue Query 전용 플러그인/자동 임포트는 제거한다.

### API / 인터페이스 변경
- 외부 컴포저블 이름 유지: `useGet`, `usePost`, `usePut`, `usePatch`, `useDelete`
- `UseGetOptions`, `UsePostOptions` 등 옵션 타입 유지

  - 기존: 생성 직후 자동 실행
  - 변경: 기본 수동 실행, 필요 시 `execute()` 호출

## 로직 흐름
1. `useGet` 호출 시 `useFetch` 인스턴스를 `immediate: false`로 생성한다.
2. 호출자는 원하는 시점에 `execute()`를 호출하고, 그때 현재 `query` 값을 기준으로 요청이 나간다.
3. 응답은 공통 `handleApiResponse`로 성공/에러를 정리하고 `response/data/status/error` 상태에 반영한다.
4. POST/PUT/PATCH/DELETE는 `$fetch`로 실행하고, 동일한 공통 응답/에러 처리 유틸을 사용한다.
5. 실제 사용 중인 `user.store.ts`는 GET 요청마다 `execute()`를 명시 호출하도록 맞춘다.
6. Vue Query 설정 파일과 Nuxt 설정 잔존물을 제거해 내부 구현과 설정 방향을 일치시킨다.

## 대안 검토
### 옵션 A
- 설명: Vue Query를 유지하되 내부 fetcher만 Axios에서 `$fetch`로 교체한다.
- 장점:
  - 호출부 수정이 거의 없다.
- 단점:
  - 마스터가 요청한 "Nuxt 4 기본 fetcher 방향"과 반쯤만 맞는다.
  - GET 기본 수동 실행 요구를 자연스럽게 반영하기 어렵다.
- 미채택 이유:
  - 방향 전환 효과가 약하고 Vue Query 구조 의존이 그대로 남는다.

### 옵션 B
- 설명: 훅 이름과 반환 형식까지 모두 버리고 `useFetch/$fetch`를 호출부에 직접 퍼뜨린다.
- 장점:
  - 구현이 가장 단순하다.
- 단점:
  - 기존 호출부를 대량 수정해야 한다.
  - 마스터가 요청한 "내부만 갈아끼우는 방식"과 어긋난다.
- 미채택 이유:
  - 변경 폭이 불필요하게 커진다.

### 최종 선택
- 선택 이유:
  - 호출 진입점 이름은 유지하면서 내부 구현을 Nuxt 기본 fetcher 방향으로 전환할 수 있다.
  - 실제 호출부 변경도 `useGet` 실행 시점 명시 정도로 제한할 수 있다.
- 기대 효과:
  - 데이터 요청 기본 전략이 Nuxt 4 권장 fetcher로 통일된다.
  - GET 요청은 원하는 시점에만 실행되며, 기존 공통 훅 레이어는 유지된다.

## 제약 및 참고 사항
- `useGet`의 자동 실행을 끄는 결정 때문에 실제 호출부 2곳 수정은 필수다.
- 패키지 의존성 제거(`package.json`, lockfile)까지 이번 턴에 강제하지는 않는다. 우선 동작 경로와 설정 정리를 우선한다.
- `context7 MCP`는 현재 세션에서 사용할 수 없어 로컬 코드 기준으로 설계했다.
