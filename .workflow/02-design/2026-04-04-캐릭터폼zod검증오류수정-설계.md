# 설계 (Design)

## 요약
- 변경 대상: `src/app/stores/character.store.ts`
- 설계 방향: `z.union(...)` 기반 필드를 `z.preprocess(...)` 기반 단일 스키마로 치환
- 주요 결정 사항: 패키지 변경 없이 런타임 예외를 차단하고 기존 폼 입력 허용 범위를 유지

## 아키텍처 변화
### 신규 구성 요소
- `normalizeNullableIntegerValue`: `campaignId`를 `number | null` 기준으로 정규화한다.
- `normalizeIntegerTextValue`: 숫자/문자/`null` 입력을 문자열 기반 검증으로 정규화한다.

### 수정 구성 요소
- `createOptionalIntegerStringSchema`: `union` 제거 후 문자열 검증으로 대체
- `createCharacterFormValidationSchema`: `campaignId`, `startLevel` 검증을 전처리 기반으로 변경

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 없음

## 로직 흐름
1. 폼 값이 Zod 스키마에 들어오기 전에 전처리 함수로 타입을 정규화한다.
2. `campaignId`는 빈 값이면 `null`, 정수 문자열이면 숫자로 변환한다.
3. `startLevel`과 자금 필드는 모두 문자열로 정규화한 뒤 비어 있음/정수 형식 여부를 검사한다.
4. `invalid_union` 자체를 만들지 않게 하여 `@vee-validate/zod` 내부 예외 경로를 우회한다.

## 대안 검토
### 옵션 A
- 설명: `zod` 또는 `@vee-validate/zod` 버전을 호환 조합으로 변경
- 장점: 근본 원인에 더 직접적이다.
- 단점: 현재 작업트리의 다른 변경과 충돌 가능성이 있고, 광범위한 회귀 확인이 필요하다.
- 미채택 이유: 긴급 폼 오류 수정 범위로는 영향이 크다.

### 최종 선택
- 선택 이유: 캐릭터 폼 범위에서 재현되는 오류를 가장 작고 안전한 변경으로 차단할 수 있다.
- 기대 효과: 생성/수정 폼 검증 시 런타임 예외 제거
