# 설계 (Design)

## 요약

- 변경 대상: 프런트 식별자/경로, 서버 API 엔드포인트, DTO/자동임포트, Drizzle 스키마/관계, PRD 및 워크플로우 문서
- 설계 방향: `users` 기반 내부 계약을 `players` 기준으로 전면 재정렬하고, 구현은 프런트-서버-DB-문서 순으로 분리된 태스크로 수행한다
- 주요 결정 사항:
  - API 경로도 `/api/players` 기준으로 전환한다
  - DTO, 스토어, 컴포저블, 쿼리 키 명칭도 `Player*`, `usePlayerStore`, `queryKeys.players`로 통일한다
  - DB 스키마도 `usersTable`와 실제 `users` 테이블명을 `playersTable`, `players`로 전환하는 것을 목표로 한다
  - 구현 중간 호환 계층은 최소화하고, 필요한 경우 검증 단계 전까지 일시적으로만 유지한다

## 아키텍처 변화

### 신규 구성 요소

- 신규 영구 구성 요소는 추가하지 않는다.
- 다만 전환 과정에서 아래 임시 구성 요소 사용 가능성을 허용한다.
  - 구 API 호환 라우트: `/api/users`에서 `/api/players`로 위임하는 얇은 포워딩 계층
  - 임시 타입 별칭: 대규모 참조 정리를 완료할 때까지 `User*`를 `Player*`에 연결하는 짧은 과도기 타입
- 이 임시 구성 요소들은 구현 완료 시 제거를 기본 원칙으로 한다.

### 수정 구성 요소

- 프런트 상태 계층
  - `src/app/stores/user.store.ts`를 `player.store.ts`로 전환한다.
  - `useUserStore`는 `usePlayerStore`로 변경한다.
  - 상태 키 `userList`, `userPageData`, `userInfo`, `myInfo`는 각각 `playerList`, `playerPageData`, `playerInfo`,
    `myPlayerInfo`로 전환한다.
- 프런트 데이터 패칭 계층
  - `src/app/composables/users/*`를 `src/app/composables/players/*`로 옮긴다.
  - `useGetUserList`, `useGetMyInfo`는 `useGetPlayerList`, `useGetMyInfo`로 변경한다.
  - 내부 호출 API를 `/users`, `/users/me`에서 `/players`, `/players/me`로 변경한다.
- 프런트 뷰 계층
  - `src/app/components/users/UserListTable.vue`는 `components/players/PlayerListTable.vue`로 전환한다.
  - `src/app/pages/players/index.vue`의 참조 대상도 새 컴포넌트명으로 맞춘다.
  - 헤더 및 공통 네비게이션은 `myInfo`, `userStore` 같은 내부 식별자를 새 스토어 기준으로 교체한다.
- 쿼리 키 계층
  - `queryKeys.users`를 `queryKeys.players`로 교체한다.
  - 키 문자열도 `users`에서 `players`로 전환해 캐시 네임스페이스를 일치시킨다.
- 서버 API 계층
  - `src/server/api/users/*`를 `src/server/api/players/*`로 이동한다.
  - `/users/me`, `/users/check`, `/users/[id]` 등 하위 경로도 동일 구조로 `players` 기준으로 재구성한다.
  - 응답 메시지 상수와 성공/실패 메시지도 플레이어 명칭 기준으로 정리 대상에 포함한다.
- 서버 타입 계층
  - `src/server/types/dto/user.dto.ts`를 `player.dto.ts`로 전환한다.
  - `UserQueryDto`, `UserCreateDto`, `UserUpdateDto`, `UserOutDto`는 `PlayerQueryDto`, `PlayerCreateDto`,
    `PlayerUpdateDto`, `PlayerOutDto`로 변경한다.
  - `UserStatus`, `UserRole`은 의미상 플레이어 모델에 종속되므로 `PlayerStatus`, `PlayerRole`로 함께 전환한다.
- 서버 자동임포트 계층
  - `manual-server-imports.d.ts`의 `TypeUser`, `TypeNewUser`, `User*` 전역 타입을 `TypePlayer`, `TypeNewPlayer`, `Player*`로
    바꾼다.
  - 프런트 자동임포트 및 전역 타입 사용처도 같은 규칙으로 맞춘다.
- 인증/권한 계층
  - `auth.ts`는 반환 타입과 내부 변수 명칭을 `Player` 중심으로 조정한다.
  - 권한 검사 인자의 `resourceUserId`는 `resourcePlayerId`로 바꾸고, `/players/me` 기준 자기 정보 흐름으로 정렬한다.

### DB / 스키마 변경

- Drizzle 스키마
  - `src/server/db/table/users.table.ts`를 `players.table.ts`로 전환한다.
  - `usersTable` 상수명과 `db.query.usersTable` 접근 경로를 `playersTable` 기준으로 교체한다.
- 실제 DB 테이블
  - 물리 테이블명을 `users`에서 `players`로 바꾸는 것을 목표로 한다.
  - 관련 인덱스 명 `idx_users_name`은 `idx_players_name`으로 변경한다.
- 관계/외래키 명명
  - 참조 컬럼은 우선 `user_id` 유지 여부를 검토할 수 있으나, 목표는 `player_id`까지 정렬하는 것이다.
  - 다만 외래키 컬럼명까지 전환하면 모든 연관 테이블과 DDL 수정 범위가 폭증하므로 태스크 단계에서 "테이블명/상수명 우선, FK 컬럼명 후속" 여부를 확정한다.
- DDL 및 명세 문서
  - `PRD/DDL.md`, `PRD/Api-Specification.md`의 `users` 테이블과 FK 설명을 `players` 기준으로 갱신한다.

### API / 인터페이스 변경

- 엔드포인트 변경
  - `POST /api/users` -> `POST /api/players`
  - `GET /api/users` -> `GET /api/players`
  - `GET /api/users/{id}` -> `GET /api/players/{id}`
  - `PATCH/PUT /api/users/{id}` -> `PATCH/PUT /api/players/{id}`
  - `GET /api/users/me` -> `GET /api/players/me`
  - `PUT /api/users/me` -> `PUT /api/players/me`
  - `DELETE /api/users/me` -> `DELETE /api/players/me`
  - `GET /api/users/check` -> `GET /api/players/check`
- 타입 계약 변경
  - 클라이언트와 서버 사이의 요청/응답 타입을 `Player*Dto` 기준으로 통일한다.
  - 다른 도메인 DTO에서 포함하는 `user: UserOutDto | null`은 `player: PlayerOutDto | null`로 변경한다.
- 내부 관계 필드 변경
  - 캠페인/세션/문서/로그 DTO와 relation 필드의 `user` 명칭도 `player`로 맞춘다.
  - 단, 관리자 권한 의미까지 포함된 필드는 명칭 변경 후에도 역할 의미가 유지되는지 검토한다.

## 로직 흐름

1. 프런트 상태와 뷰의 기준 식별자를 `player`로 전환한다.
2. 프런트 컴포저블과 쿼리 키를 `/api/players` 계약으로 옮긴다.
3. 서버 API 라우트를 `players` 경로로 복제 또는 이동하고, 기존 핸들러 내부 로직을 `Player*Dto`, `playersTable` 기준으로 치환한다.
4. 서버 자동임포트와 공용 DTO 참조를 정리해 `PlayerOutDto`가 다른 도메인의 기본 참조 타입이 되게 한다.
5. Drizzle 테이블/relations와 `db.query.*` 경로를 `playersTable` 기준으로 전환한다.
6. DB 물리 테이블명과 인덱스명 변경이 필요한 경우 마이그레이션 또는 DDL 반영 계획에 따라 적용한다.
7. PRD, API 명세, Task List, 워크플로우 문서에서 `users` 용어를 `players` 기준으로 정리한다.
8. 검증 단계에서 구 `/users` 경로 잔존 여부, 자동임포트 오류, DTO 참조 누락, relation 깨짐 여부를 점검한다.

## 대안 검토

### 옵션 A

- 설명: 프런트 내부 명칭과 UI 문구만 `players`로 바꾸고 서버 API와 DB는 `users`로 유지한다.
- 장점:
  - 구현량이 작다.
  - DB와 서버 계약을 건드리지 않아 위험이 낮다.
- 단점:
  - `/players` 페이지가 `/users` API를 호출하는 혼재 구조가 계속 남는다.
  - PRD/API 명세와 코드 구조의 정합성이 회복되지 않는다.
- 미채택 이유:
  - 마스터 요구사항이 API까지 포함한 전면 전환이므로 목표와 맞지 않는다.

### 옵션 B

- 설명: 프런트와 서버 API/DTO는 `players`로 바꾸고 DB 테이블은 `users`로 유지한다.
- 장점:
  - 외부 계약과 내부 프런트 구조는 정리된다.
  - 물리 DB 마이그레이션 리스크를 피할 수 있다.
- 단점:
  - Drizzle 스키마와 DB 실테이블 사이에 의미 차이가 남는다.
  - 장기적으로 유지보수 시 "API는 players, DB는 users"라는 또 다른 혼재가 생긴다.
- 미채택 이유:
  - 이번 작업의 목표가 "단어 자체를 전부 바꾸는 것"이므로 중간 상태를 최종 상태로 두기 어렵다.

### 최종 선택

- 선택 이유:
  - 프런트, API, DTO, DB, 문서를 모두 `players` 기준으로 정렬해야 용어 혼재가 사라진다.
  - 이후 유지보수자가 "플레이어" 도메인을 한 이름으로 추적할 수 있어 학습 비용이 크게 줄어든다.
- 기대 효과:
  - 라우트, 타입, 저장소, API, DB, 문서가 같은 명칭 체계를 사용한다.
  - 플레이어 목록/상세/내 정보/권한 로직이 모두 하나의 도메인명으로 정리된다.

## 제약 및 참고 사항

- 자동임포트 규칙 때문에 새 타입/스토어/컴포저블 전환 시 직접 import를 남발하지 않는다.
- 기존 `.workflow` 과거 문서의 기록은 아카이브이므로 현 작업에서 일괄 수정하지 않는다. 단, 현재 작업 문서와 PRD 기준 문서는 새 용어로 정렬한다.
- DB 물리 변경이 포함되므로 실제 마이그레이션 파일 또는 스키마 반영 방식은 구현 전에 추가 확인이 필요하다.
- 외래키 컬럼명(`user_id`)까지 이번 작업 범위에 포함할지 여부는 구현 난이도와 배포 리스크를 고려해 태스크 단계에서 명시적으로 쪼개야 한다.
