# 설계 (Design)

## 요약
- 변경 대상: `src/app/composables/useSetMeta.ts`, `src/app/types/common.types.ts`, `src/app/pages/players/detail/[id].vue`
- 설계 방향: 메타 입력을 반응형 값까지 허용하도록 공통 컴포저블을 확장하고, 상세 페이지는 null 안전한 `computed` 제목을 전달한다
- 주요 결정 사항:
  - `SiteMetadata`는 문자열 고정값 대신 `MaybeRefOrGetter<string | undefined>` 형태를 허용한다
  - `useSetMeta` 내부에서 `toValue()`와 `computed()`로 최종 메타 값을 계산한다
  - 플레이어 상세 페이지는 fetch 이전 null 상태를 가정한 fallback 제목을 사용한다

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- `useSetMeta(meta: SiteMetadata)` 시그니처는 유지하되 `SiteMetadata`의 필드 입력 범위를 넓힌다
- 기존 문자열 전달 호출부는 수정 없이 그대로 동작해야 한다

## 로직 흐름
1. 페이지는 `title`, `url` 등에 문자열 또는 `computed`를 전달한다
2. `useSetMeta`는 각 입력값을 `computed(() => toValue(...))`로 해석한다
3. 제목, 설명, URL, OG 이미지 등 파생 메타도 모두 `computed`로 계산한다
4. 초기 렌더 시 데이터가 없으면 fallback 제목이 적용된다
5. 이후 스토어 값이 채워지면 `computed`가 다시 평가되어 메타 값도 갱신된다

## 대안 검토
### 옵션 A
- 설명: 플레이어 상세 페이지 한 곳에서 optional chaining 또는 fallback 문자열만 사용한다
- 장점: 수정 범위가 가장 작다
- 단점: 동일 문제가 다른 페이지에서 반복될 수 있고 공통 정책이 생기지 않는다
- 미채택 이유: 현재 문제는 개별 페이지보다 `useSetMeta`가 고정 문자열만 받는 구조적 한계에 가깝다

### 옵션 B
- 설명: `execute()` 성공 콜백 안에서만 `useSetMeta`를 호출한다
- 장점: 실제 데이터 기준으로 메타를 설정할 수 있다
- 단점: 페이지별 fetch 콜백에 메타 책임이 흩어지고, 로딩 중 기본 메타가 비게 된다
- 미채택 이유: 공통 규칙이 약해지고 페이지 가독성이 떨어진다

### 최종 선택
- 선택 이유: 공통 메타 컴포저블을 반응형 입력 지원 구조로 개조하면 현재 오류를 해결하면서 재발 가능성도 낮출 수 있다
- 기대 효과: null 초기 상태를 안전하게 흡수하고, 데이터 수신 후 메타를 자연스럽게 갱신할 수 있다

## 제약 및 참고 사항
- Nuxt 오토임포트와 전역 타입 선언을 활용하므로 직접 import는 추가하지 않는다
- 기존 `useSetMeta` 호출부 호환성을 유지해야 하므로 파괴적 시그니처 변경은 피한다
