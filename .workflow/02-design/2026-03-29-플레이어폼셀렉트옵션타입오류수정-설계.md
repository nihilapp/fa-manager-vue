# 설계 (Design)

## 요약
- 변경 대상: `src/app/components/common/FormSelect.vue`
- 설계 방향: `FormSelect`가 읽기 전용 옵션 배열도 허용하도록 프롭 타입 계약을 현실 사용 방식에 맞게 확장
- 주요 결정 사항: 호출부 수정 대신 공용 컴포넌트 `options` 타입을 `readonly SelectOption[]`로 변경

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/app/components/common/FormSelect.vue`
  - `options` 프롭 타입을 mutable 배열에서 readonly 배열 허용 타입으로 변경한다.
  - 내부 `find`, `v-for` 기반 읽기 전용 소비 로직은 그대로 유지한다.

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- `FormSelect` 컴포넌트의 외부 인터페이스 변경:
  - 기존: `options?: SelectOption[]`
  - 변경: `options?: readonly SelectOption[]`
- 이 변경은 하위 호환이다. 일반 배열과 readonly 배열 모두 전달 가능하다.

## 로직 흐름
1. 호출부는 `as const`가 적용된 옵션 배열을 그대로 `FormSelect`에 전달한다.
2. `FormSelect`는 `options`를 읽기 전용 컬렉션으로 취급하고 값 조회(`find`) 및 렌더링(`v-for`)만 수행한다.
3. 선택 시에는 옵션 배열 자체를 수정하지 않고 선택된 값만 `handleChange` 또는 `modelValue`에 반영한다.
4. 타입 시스템은 배열 변경 가능성 요구를 제거하므로 TS4104 오류가 사라진다.

## 대안 검토
### 옵션 A
- 설명: `NewPlayerForm.vue`에서 `playerStatusSelectOptions`, `playerRoleSelectOptions`를 일반 mutable 배열로 바꾸거나 복사본을 만들어 전달
- 장점: 오류 지점이 사용자 호출부에서 즉시 사라진다.
- 단점: 공용 컴포넌트가 실제로는 읽기 전용 소비만 하는데 호출부가 불필요하게 제약을 맞춰야 한다.
- 미채택 이유: 다른 호출부에도 동일한 강제 변환이 반복될 수 있고, `as const` 기반 리터럴 타입 보존 이점이 줄어든다.

### 최종 선택
- 선택 이유: 오류의 근본 원인은 공용 컴포넌트의 과도한 mutable 타입 요구다. 이를 완화하는 편이 범용성과 타입 정확성이 높다.
- 기대 효과: 현재 오류 해결과 동시에 향후 readonly 옵션 상수 재사용 시 동일한 문제를 예방한다.

## 제약 및 참고 사항
- 내부 로직이 옵션 배열을 직접 변경하지 않는다는 전제 하에만 안전하다.
- context7 MCP는 현재 세션에서 확인되지 않아 로컬 코드와 문서만 근거로 설계를 확정한다.
