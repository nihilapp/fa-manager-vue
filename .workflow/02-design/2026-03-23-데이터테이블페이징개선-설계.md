## 로직 흐름
1. `/api/users`는 기존처럼 `ListDataType<UserOutDto>` 응답을 반환한다.
2. `user.store.ts`의 `getUserList` 성공 시 응답에서 `list`를 분리한다.
3. 분리한 `list`는 `userInfoList.value`에 저장한다.
4. 나머지 메타(`totalElements`, `filteredElements`, `currentPage` 등)는 `userPageData.value`에 저장한다.
5. `userInfoListCount.value`는 기존 코드 호환성을 위해 `userPageData.value.totalElements`와 동기화한다.
6. 유저 목록 화면 또는 후속 소비 코드는 `items=userInfoList`, `pagination=userPageData` 형태로 `DataTable`에 전달한다.
7. `DataTable`은 `pagination`이 없으면 페이지네이터를 렌더링하지 않는다.
8. `DataTable`은 `pagination`이 있으면 정규화된 메타를 바탕으로 `currentPage`, `pageSize`, `totalPages`, `hasPrev`, `hasNext`, `prevPage`, `nextPage`, `filteredElements`를 계산 또는 보정한다.
9. `DataTable`은 현재 row 개수(`items.length`)와 정규화 메타를 조합해 `startItem`, `endItem`, 페이지 버튼 목록을 안정적으로 표시한다.

## 대안 검토
### 옵션 A: 각 스토어에서 `ListDataType<T>` 전체를 그대로 보관
- 설명: `userListData` 같은 단일 ref에 응답 전체를 저장하고, 화면에서 `.list`와 메타를 직접 꺼내 쓴다.
- 장점:
  - 구현이 가장 단순하다.
  - API 응답 구조와 1:1 대응된다.
- 단점:
  - 목록 배열과 메타 사용처가 항상 같이 묶인다.
  - 테이블에 `list`를 전달하기 위해 소비 코드가 매번 `.list`를 해체해야 한다.
  - 마스터가 지시한 상태 분리 규칙과 맞지 않는다.
- 미채택 이유: 공용 컴포넌트와 스토어 역할을 분리하려는 이번 목표에 부합하지 않는다.

### 옵션 B: 스토어에서 `list`와 메타를 분리하고, 메타 타입을 `Omit` 별칭으로 공통화
- 설명: 배열과 페이지 메타를 별도 ref로 저장하고, `DataTable`은 분리된 계약만 소비한다.
- 장점:
  - 스토어와 공용 컴포넌트의 책임이 분명해진다.
  - 모든 리스트 도메인에 동일한 패턴을 복제하기 쉽다.
  - 페이지 메타만 필요한 UI가 생겨도 재사용하기 좋다.
- 단점:
  - 기존 상태 구조보다 ref가 하나 늘어난다.
  - 일부 기존 호출부는 `pagination` prop 전달 방식 변경이 필요하다.
- 선택 이유: 마스터가 요구한 구조와 가장 정확히 일치하고, 향후 확장성도 가장 좋다.

## 후속 확장 계획
- `campaign`, `character`, `session`, `log-history`, `currency-transaction` 등 `ListDataType<T>`를 소비하는 모든 스토어에 `xxxList + xxxPageData` 패턴을 동일 적용한다.
- 공통 규칙:
  - 목록 배열 ref 이름: 도메인명 + `List`
  - 페이지 메타 ref 이름: 도메인명 + `PageData`
  - 공통 타입: `ListPageData<T>`
- 실제 후속 구현 시 각 도메인 목록 페이지는 `DataTable`에 항상 `:items="도메인List"`와 `:pagination="도메인PageData"`를 함께 전달하도록 통일한다.
