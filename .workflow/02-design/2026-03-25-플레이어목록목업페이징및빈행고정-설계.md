# 설계 (Design)

## 요약
- 변경 대상: `src/app/pages/players/index.vue`, `src/app/components/table/DataTable.vue`
- 설계 방향: 플레이어 목록 페이지는 실제 API 대신 내부 목업 25건과 로컬 pagination 계산으로 동작시키고, `DataTable.vue`는 `pageSize` 기준으로 placeholder row를 채워 마지막 페이지에서도 높이를 고정
- 주요 결정 사항:
  - 목업 데이터는 페이지 파일 내부 상수/계산 값으로 관리
  - 페이지 크기는 고정값으로 두고, 현재 페이지에 따라 슬라이스된 데이터만 `DataTable`에 전달
  - `pagination` 객체는 `ListPageData` 형식과 호환되게 페이지 내부에서 직접 생성
  - `DataTable.vue`는 실제 데이터 행 뒤에 placeholder row를 추가 렌더링
  - 플레이어 페이지 바깥 `section` 배경은 제거

## 아키텍처 변화
### 신규 구성 요소
- `players/index.vue` 내부 목업 플레이어 목록 생성 상수
- `players/index.vue` 내부 현재 페이지 상태와 계산된 pagination 객체
- `DataTable.vue` 내부 placeholder row 계산용 computed

### 수정 구성 요소
- `src/app/pages/players/index.vue`
  - `useGetUserList()` 및 `execute()` 기반 실데이터 조회 제거
  - 25건 규모의 목업 플레이어 배열 정의
  - 현재 페이지 상태(`currentPage`)와 `pageSize` 기준 슬라이스 데이터 계산
  - `ListPageData<UserOutDto>`와 호환되는 pagination 객체를 계산해 `DataTable`에 전달
  - `page-change` 이벤트를 받아 로컬 현재 페이지를 갱신
  - 페이지 래퍼의 추가 배경 제거

- `src/app/components/table/DataTable.vue`
  - `normalizedPagination.pageSize`와 `items.length`를 이용해 빈 행 개수를 계산
  - 빈 행은 실제 행과 동일한 높이/셀 구조를 가지되 내용 없이 렌더링
  - 데이터가 일부라도 있으면 empty state 대신 실제 행 + placeholder row 조합을 렌더링

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- `DataTable.vue`의 props 및 emits 외부 계약은 유지
- `players/index.vue`에서 `@page-change`를 실제로 사용하도록 변경

## 로직 흐름
1. `players/index.vue`에서 목업 플레이어 데이터 25건과 고정 `pageSize`를 정의한다.
2. `currentPage` ref를 기준으로 시작 인덱스와 종료 인덱스를 계산해 현재 페이지 데이터를 잘라낸다.
3. 같은 정보로 `totalElements`, `filteredElements`, `totalPages`, `hasPrev`, `hasNext`, `prevPage`, `nextPage` 등을 포함한 pagination 객체를 만든다.
4. 페이지는 `DataTable`에 현재 페이지 데이터와 pagination 객체를 함께 전달한다.
5. `Paginator`가 `page-change`를 emit하면 `players/index.vue`가 `currentPage`를 갱신한다.
6. `DataTable.vue`는 현재 페이지 데이터 길이를 보고 `pageSize - items.length`만큼 placeholder row 개수를 계산한다.
7. 실제 데이터 행을 렌더링한 뒤 placeholder row를 같은 셀 구조로 추가 렌더링한다.
8. 현재 페이지 데이터가 0건이고 전체 데이터도 0건이면 기존 empty state를 표시한다.

## 대안 검토
### 옵션 A
- 설명: 목업 데이터를 별도 store나 composable로 분리
- 장점: 향후 재사용 가능
- 단점: 이번 요청은 플레이어 페이지 단일 테스트 목적이라 구조가 과해짐
- 미채택 이유: 현재는 빠른 시각 테스트와 페이징 검증이 우선이므로 페이지 내부 계산이 더 단순함

### 옵션 B
- 설명: `DataTable.vue`가 placeholder row를 모르고, 페이지에서 빈 row 데이터를 미리 만들어 전달
- 장점: 공통 테이블 책임을 늘리지 않을 수 있음
- 단점: 모든 호출부가 빈 행 생성을 알아야 해서 재사용성이 떨어짐
- 미채택 이유: `pageSize` 계약은 공통 테이블이 알고 있으므로 `DataTable.vue` 내부 처리 쪽이 더 자연스러움

### 최종 선택
- 선택 이유: 목업 pagination은 페이지에서 만들고, 빈 행 보강은 공통 테이블에서 담당하면 각자의 책임이 가장 명확하다
- 기대 효과: 실제 API 없이도 플레이어 목록에서 충분한 양의 데이터와 마지막 페이지 동작을 안정적으로 테스트할 수 있다

## 제약 및 참고 사항
- 이번 작업은 테스트 목적의 임시 목업 전환이므로 실제 API 연동은 후속 복귀 작업이 필요하다.
- placeholder row는 내용 없이 높이만 맞춰야 하므로 hover/액션이 동작하지 않게 처리해야 한다.
- 프로젝트 Tailwind 클래스 사용 범위가 제한될 수 있어 기존 테이블 클래스 조합을 최대한 재사용하는 편이 안전하다.
