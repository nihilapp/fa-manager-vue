# 설계 (Design)

## 요약
- 변경 대상: `src/server/utils/auth.ts`, `src/server/utils/access.ts`, `src/server/api/users/check.get.ts`
- 설계 방향: 공통 인증 컨텍스트는 `authHelper`에 유지하고, `checkCommunityAccess`가 같은 기준을 재사용하도록 재구성한다. 운영/개발 환경 분기와 접근 허용 조건은 세 파일에서 동일한 해석을 공유한다.
- 주요 결정 사항:
  - 1차 조건인 개발 환경 우회는 `authHelper`에서 가장 먼저 처리한다.
  - 운영 환경에서는 `x-discord-id` 헤더가 없는 요청을 즉시 차단한다.
  - 운영 환경에서 관리자/본인 판정은 `authHelper`가 제공하는 컨텍스트를 바탕으로 `checkCommunityAccess`가 수행한다.
  - `users/check.get.ts`는 별도 정책을 갖지 않고 `checkCommunityAccess` 결과만 노출한다.

## 아키텍처 변화
### 신규 구성 요소
- 신규 파일 추가는 없음
- 다만 `authHelper` 반환 객체의 의미를 더 명확히 고정한다.
  - `user`: 조회된 실제 사용자 또는 개발 환경 fallback 사용자
  - `isAdmin`: `ROLE_ADMIN`, `ROLE_SUPER_ADMIN` 여부
  - `isDevelopmentBypass`: 개발 환경 우회 적용 여부
  - `hasPermission(resourceUserId?)`: 개발 환경이면 항상 `true`, 운영 환경이면 관리자 또는 본인일 때만 `true`
  - `error`: 인증 실패 시 표준 에러 응답

### 수정 구성 요소
- `src/server/utils/auth.ts`
  - 개발 환경이면 후속 조건을 평가하지 않고 통과시키는 현재 구조를 유지하되, 반환 컨텍스트 의미를 명확히 한다.
  - 운영 환경에서는 `x-discord-id` 부재 시 즉시 실패한다.
  - 운영 환경에서는 헤더로 조회한 사용자를 기준으로 관리자 여부와 본인 판정을 위한 컨텍스트를 제공한다.
- `src/server/utils/access.ts`
  - 지금처럼 `error`만 보고 통과시키는 단순 래퍼에서, 요청하신 2차~4차 조건을 실제로 해석하는 접근 체크 유틸로 바꾼다.
  - 단, “본인 여부” 판단에 필요한 대상 사용자 ID를 외부에서 받을 수 없는 경우에는 함수 계약을 확장하거나, 현재 요청 범위에서는 `users/check` 용도에 맞춘 자기 자신 기준 체크만 수행한다.
- `src/server/api/users/check.get.ts`
  - 자체 로직 추가 없이 `checkCommunityAccess(event)`만 호출한다.
  - 결과 해석의 기준선은 `access.ts`에만 둔다.

## 로직 흐름
1. `authHelper(event)`가 호출되면 가장 먼저 `import.meta.env.DEV`를 확인한다.
2. 개발 환경이면 후속 조건을 평가하지 않고 `isDevelopmentBypass=true` 상태의 인증 컨텍스트를 반환한다.
3. 운영 환경이면 `getDiscordId(event)`로 `x-discord-id` 헤더를 읽는다.
4. 헤더가 없으면 `REQUIRE_DISCORD_ID` 에러를 담은 컨텍스트를 반환한다.
5. 헤더가 있으면 해당 디스코드 ID로 사용자를 조회한다.
6. 사용자가 없으면 `USER_NOT_FOUND` 에러를 담은 컨텍스트를 반환한다.
7. 사용자가 있으면 role 기반으로 `isAdmin`을 계산하고 정상 컨텍스트를 반환한다.
8. `checkCommunityAccess(event)`는 먼저 `authHelper(event)`를 호출한다.
9. `authHelper`가 에러를 반환하면 즉시 차단 응답을 반환한다.
10. 개발 환경 우회가 활성화되어 있으면 즉시 통과 응답을 반환한다.
11. 운영 환경에서는 아래 해석 규칙 중 하나를 선택해 적용한다.
12. `users/check.get.ts`는 `checkCommunityAccess(event)` 결과를 그대로 응답한다.

## 조건 해석 설계
### 옵션 A: 요청 문구를 문자 그대로 구현
- 규칙:
  - 1차: 개발 환경이면 통과
  - 2차: 운영 환경에서 헤더에 디스코드 ID가 있으면 통과
  - 3차: 디스코드 ID가 있고 관리자면 통과
  - 4차: 디스코드 ID가 있고 본인이면 통과
- 결과:
  - 2차가 참인 순간 3차, 4차는 평가할 필요가 없으므로 사실상 죽은 조건이 된다.
  - `users/check`는 헤더만 있으면 일반 사용자도 모두 통과하게 된다.
- 장점:
  - 마스터 요청 문장을 가장 직접적으로 따른다.
- 단점:
  - 3차, 4차 조건이 설계상 무의미해진다.
  - “본인인가?”를 판정해야 한다는 요구와 충돌한다.

### 옵션 B: 2차를 선행 조건, 3차/4차를 실제 허용 조건으로 해석
- 규칙:
  - 1차: 개발 환경이면 통과
  - 2차: 운영 환경에서는 헤더에 디스코드 ID가 있어야 다음 단계로 진행
  - 3차: 관리자면 통과
  - 4차: 관리자 아니면 본인일 때만 통과
  - 아니면 차단
- 결과:
  - 분석 단계에서 드러난 논리 충돌이 해소된다.
  - 운영 환경에서 헤더만 있는 일반 사용자는 자동 통과하지 않는다.
- 장점:
  - 관리자/본인 판정 요구를 실제 로직으로 살릴 수 있다.
  - `authHelper`의 기존 `hasPermission` 모델과 자연스럽게 맞물린다.
- 단점:
  - 마스터의 “있으면 패스” 표현과는 문자 그대로 일치하지 않는다.
  - `checkCommunityAccess`에서 “본인”의 기준 대상을 별도로 정의해야 한다.

### 최종 선택
- 설계 기준 선택: 옵션 B
- 선택 이유:
  - 3차, 4차 조건을 실질 조건으로 살릴 수 있는 유일한 해석이다.
  - 현재 `authHelper`가 이미 관리자/본인 판정 모델을 가지고 있어 최소 변경으로 구현 가능하다.
  - 옵션 A는 조건 목록을 유지하더라도 실제 보안 의미가 거의 없다.

## 세부 설계 포인트
### 1. 개발 환경 처리
- 현재 `authHelper`의 fallback 사용자 조회 방식은 유지한다.
- 이유:
  - 기존 코드 다수가 `user.id`를 직접 사용하므로, 개발 환경에서도 `user`가 존재하는 편이 회귀 위험이 낮다.
- 보조 원칙:
  - 개발 환경 통과는 접근 제어 우회일 뿐이며, 운영 환경 조건 검증을 대체하지 않는다.

### 2. 운영 환경 헤더 처리
- `x-discord-id` 헤더 유무는 `authHelper`가 단일 진입점에서만 판단한다.
- 다른 파일에서 헤더를 직접 다시 읽는 패턴은 추가하지 않는다.

### 3. 관리자/본인 판정
- 관리자 판정은 role 기반 고정: `ROLE_ADMIN`, `ROLE_SUPER_ADMIN`
- 본인 판정은 `resourceUserId === user.id` 규칙을 따른다.
- `checkCommunityAccess`에서 본인 대상을 직접 알 수 없는 경우:
  - 현재 요청 범위에서는 `users/check`를 “현재 요청자가 자기 자신으로 인증 가능한가”를 확인하는 엔드포인트로 간주한다.
  - 따라서 운영 환경에서 헤더로 식별된 사용자가 존재하면 자기 자신 기준 본인 판정은 참으로 본다.

### 4. 차단 흐름
- `checkCommunityAccess`가 실패 응답을 반환하면 서버 미들웨어는 `/block` 리다이렉트를 유지한다.
- `users/check.get.ts`는 같은 실패 응답을 그대로 노출해 프런트 또는 상위 호출자가 접근 불가 상태를 해석할 수 있게 한다.

## 대안 검토
### 옵션 A: `checkCommunityAccess`는 그대로 두고 `authHelper`만 수정
- 설명: 접근 유틸은 현재처럼 `error`만 보고, 인증 헬퍼만 조정한다.
- 장점:
  - 수정 범위가 작다.
- 단점:
  - 요청하신 3차, 4차 조건을 `access.ts`에서 실제로 표현하지 못한다.
- 미채택 이유: 이번 요청의 핵심이 `access.ts` 재구성까지 포함하므로 부족하다.

### 옵션 B: `checkCommunityAccess` 시그니처를 확장해 대상 사용자 ID를 받는다
- 설명: `checkCommunityAccess(event, resourceUserId)`로 바꾸고 본인 판정을 일반화한다.
- 장점:
  - “본인 여부”를 가장 정확하게 모델링할 수 있다.
- 단점:
  - 현재 수정 범위 밖의 호출부 변경이 연쇄적으로 필요하다.
- 미채택 이유: 이번 요청은 세 파일 중심 수정이며, 추가 승인 없는 범위 확장이 된다.

### 옵션 C: 현재 시그니처 유지 + `users/check` 용도에 맞춘 자기 자신 기준 판정
- 설명: `checkCommunityAccess(event)`는 현재 사용자 자신의 접근 유효성만 체크한다.
- 장점:
  - 세 파일 안에서 구현 가능하다.
  - 현재 호출 구조를 깨지 않는다.
- 단점:
  - 범용 “본인 여부” 체크 함수로는 부족하다.
- 선택 이유: 마스터 승인 범위를 넘지 않으면서 요청하신 조건을 가장 일관되게 반영할 수 있다.
