# 설계 (Design)

## 요약
- 변경 대상: `CharacterEditorForm.vue`
- 설계 방향: 공용 폼의 상태 타입, 기본값 매핑, 제출 payload, 템플릿 섹션을 최소 필드 집합으로 축소한다.
- 주요 결정 사항: 페이지 래퍼는 유지하고 공용 폼만 단순화한다.

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/app/components/characters/CharacterEditorForm.vue`
  - 폼 상태를 이름, 상태, 시작 레벨, 시작 자금으로 줄인다.
  - 기본 정보/클래스/시작 자금 3개 중심 섹션으로 재구성한다.

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- `submit` payload를 축소한다.
  - 유지: `name`, `status`, `startLevel`, `startCurrencyCp`, `startCurrencySp`, `startCurrencyEp`, `startCurrencyGp`, `startCurrencyPp`
  - 제거: 나머지 모든 필드

## 로직 흐름
1. 생성 모드는 축소된 빈 기본값으로 폼을 초기화한다.
2. 수정 모드는 `useGetCharacterInfo` 결과에서 이름, 상태, 시작 레벨, 현재/시작 자금 관련 값만 추려 기본값으로 반영한다.
3. 클래스 정보는 기존 스토어의 `characterClassList`를 그대로 읽어 요약 표시한다.
4. 폼 제출 시 축소된 payload만 이벤트로 방출한다.

## 대안 검토
### 옵션 A
- 설명: 전체 폼 구조는 유지하고 일부 섹션만 접어 둔다.
- 장점: 나중에 다시 확장하기 쉽다.
- 단점: 코드와 UI가 계속 비대하고 현재 요구사항과 어긋난다.
- 미채택 이유: 마스터 요청은 접기보다 제거에 가깝다.

### 최종 선택
- 선택 이유: 실제 유지할 필드만 남겨 UI와 후속 로직 연결 지점을 함께 단순화할 수 있다.
- 기대 효과: 생성/수정 페이지가 훨씬 짧아지고 핵심 정보만 빠르게 입력할 수 있다.

## 제약 및 참고 사항
- 클래스 정보는 별도 테이블이므로 편집 기능 없이 표시만 유지한다.
- 시작 자금은 현행 데이터 구조와 충돌을 줄이기 위해 화폐 단위별 입력을 유지한다.
