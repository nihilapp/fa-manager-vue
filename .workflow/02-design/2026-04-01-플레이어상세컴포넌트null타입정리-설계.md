# 설계 (Design)

## 요약
- 변경 대상: `src/app/components/players/PlayerDetail.vue`, `src/app/pages/players/detail/[id].vue`
- 설계 방향: 부모 페이지가 데이터 존재를 확인하고, 자식 `PlayerDetail`은 필수 props만 받아 렌더하는 구조로 전환
- 주요 결정 사항:
  - `PlayerDetail`은 `playerInfo`와 `playerSessionTotal`을 props로 받는다
  - 상태 뱃지 정보는 nullable Map 직접 조회 대신 `getPlayerStatusInfo`를 사용한다
  - 오류 화면 분기는 상세 페이지에서 담당한다

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/app/pages/players/detail/[id].vue`
  - `playerInfo`와 `playerSessionPageData`를 읽어 성공 화면과 오류 화면을 분기한다
  - 데이터가 있을 때만 `PlayerDetail`에 확정 타입 props를 전달한다
- `src/app/components/players/PlayerDetail.vue`
  - 스토어에서 nullable `playerInfo`를 직접 읽지 않는다
  - 필수 props를 기준으로 화면을 렌더한다
  - 상태 표시는 `getPlayerStatusInfo()` 결과를 사용한다

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- `PlayerDetail` 컴포넌트의 입력 계약이 props 기반으로 바뀐다

## 로직 흐름
1. 상세 페이지는 store의 `playerInfo`와 `playerSessionPageData`를 읽는다
2. `playerInfo`가 있으면 `PlayerDetail`에 `playerInfo`, `playerSessionTotal`을 전달한다
3. `playerInfo`가 없으면 페이지 레벨 오류 화면을 렌더한다
4. `PlayerDetail`은 nullable 없는 props만 사용해 상세 정보를 렌더한다
5. 상태 배지는 `getPlayerStatusInfo(playerInfo.status)` 결과를 그대로 사용한다

## 대안 검토
### 옵션 A
- 설명: `PlayerDetail.vue` 내부에 `computed(() => playerInfo.value!)` 같은 강제 단언 래퍼를 둔다
- 장점: 수정 범위가 작다
- 단점: 타입 경고는 줄어도 구조적으로 nullable 상태에 계속 의존한다
- 미채택 이유: 표현 컴포넌트가 스토어 nullable 상태를 직접 아는 구조를 유지하게 된다

### 옵션 B
- 설명: 템플릿에서 optional chaining을 반복 사용한다
- 장점: 빠르다
- 단점: 경고를 억누르는 수준이고, 컴포넌트 계약이 모호하다
- 미채택 이유: 타입 모델이 개선되지 않는다

### 최종 선택
- 선택 이유: 부모가 데이터 로딩 상태를 책임지고 자식은 확정 데이터만 받도록 분리하는 편이 타입과 구조 모두 깔끔하다
- 기대 효과: nullable 경고 제거, 컴포넌트 책임 분리, 향후 재사용 계약 명확화

## 제약 및 참고 사항
- 오토임포트 규칙에 따라 별도 import는 추가하지 않는다
