# 설계 (Design)

## 요약
- 변경 대상: `src/server/api/players/[id]/campaigns.get.ts`, `src/server/api/players/[id]/sessions.get.ts`
- 설계 방향: 플레이어 상세 전용 하위 목록 API를 추가하고, 기존 공용 목록 DTO/필터 체계를 최대한 재사용한다.
- 주요 결정 사항:
  - 캐릭터는 기존 `/api/characters?userId=` 재사용으로 유지한다.
  - 참여 캠페인 목록은 `campaignMembersTable` 기반 id 집합을 만들고, 그 위에 `CampaignQueryDto` 필터를 합성한다.
  - 참여 세션 목록은 `sessionPlayersTable` 기반 id 집합을 만들고, 그 위에 `SessionQueryDto` 필터를 합성한다.

## 아키텍처 변화
### 신규 구성 요소
- `src/server/api/players/[id]/campaigns.get.ts`
  - 특정 플레이어가 참여한 캠페인 목록을 페이징/필터링해 반환한다.
- `src/server/api/players/[id]/sessions.get.ts`
  - 특정 플레이어가 참여한 세션 목록을 페이징/필터링해 반환한다.

### 수정 구성 요소
- `.workflow/*`
  - 이번 작업의 산출물 기록

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- `GET /players/:id/campaigns`
  - Path: `id`
  - Query: `CampaignQueryDto` 기반 + `role?: CampaignRole`
  - Response: `BaseApiResponse<ListData<CampaignOutDto>>`
- `GET /players/:id/sessions`
  - Path: `id`
  - Query: `SessionQueryDto` 기반 + `role?: SessionRole`, `characterId?: number`
  - Response: `BaseApiResponse<ListData<SessionOutDto>>`

## 로직 흐름
1. 요청에서 플레이어 id를 읽고 권한을 `[id].get`와 동일하게 관리자 전용으로 검증한다.
2. 조인 테이블에서 해당 플레이어의 참여 대상 id 집합을 먼저 구한다.
3. 대상 id 집합이 비어 있으면 빈 `ListData`를 즉시 반환한다.
4. 대상 테이블(`campaignsTable`, `sessionsTable`)에 대해 기존 DTO 기반 필터, 정렬, 페이징을 적용한다.
5. 결과는 기존 목록 API와 동일한 `ListData` 포맷으로 반환한다.

## 대안 검토
### 옵션 A
- 설명: `players/[id].get` 응답 내부 관계 배열을 프런트에서 직접 잘라 쓰게 한다.
- 장점: 서버 엔드포인트 추가가 없다.
- 단점: 대용량 데이터에 취약하고 독립 필터링/정렬이 불가능하다.
- 미채택 이유: 마스터 요구의 `pageSize=10` 기반 독립 리스트 운용을 충족하지 못한다.

### 옵션 B
- 설명: 공용 `campaigns/index.get.ts`, `sessions/index.get.ts`에 `participantUserId` 같은 필터를 추가한다.
- 장점: 파일 수가 늘지 않는다.
- 단점: 공용 API 책임이 커지고 소유 목록/참여 목록 의미가 섞인다.
- 미채택 이유: 플레이어 상세 전용 요구를 분리된 경로로 표현하는 편이 명확하다.

### 최종 선택
- 선택 이유: 플레이어 하위 리소스 경로가 용도를 가장 잘 드러내고, 프런트도 직관적으로 사용할 수 있다.
- 기대 효과: 상세 화면의 각 리스트가 독립적인 page/filter/sort 흐름을 갖게 된다.

## 제약 및 참고 사항
- 캐릭터 목록 전용 하위 엔드포인트는 이번 범위에 포함하지 않는다.
- 타입체크는 현재 프로젝트 환경 이슈로 완전 검증이 막힐 가능성이 있다.
