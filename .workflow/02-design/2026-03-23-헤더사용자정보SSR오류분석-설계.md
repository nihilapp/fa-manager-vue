# 설계 (Design)

## 요약
- 변경 대상: `CommonHeader.vue`, `app.vue`, `user.store.ts`, 필요 시 `common.store.ts`
- 설계 방향: SSR 시점에도 헤더가 안전하게 렌더되도록 null-safe 표현을 적용하고, 사용자 정보 조회를 클라이언트 마운트 전후 불일치가 최소화되도록 초기화 흐름으로 재배치한다.
- 주요 결정 사항:
  - 헤더는 사용자 정보 부재를 정상 상태로 처리한다.
  - 사용자 정보 조회는 `onMounted` 단일 의존에서 벗어나 앱 초기 진입 시점으로 이동한다.
  - 관리자 메뉴 공개 여부는 사용자 정보 반영 후에도 일관되게 계산되도록 반응형 계약으로 정리한다.

## 아키텍처 변화
### 신규 구성 요소
- 신규 파일 추가는 필수 아님
- 필요 시 헤더 표시용 계산값(`displayUserName`, `isMyInfoReady`) 같은 computed를 도입해 템플릿의 null 접근을 제거한다.

### 수정 구성 요소
- `src/app/components/layout/common-layout/CommonHeader.vue`
  - `myInfo!.name` 직접 접근 제거
  - 사용자 정보가 없을 때도 렌더 가능한 표시 규칙 도입
  - 예시 계약:
    - `myInfo` 존재 시: 실제 이름 표시
    - `myInfo` 부재 시: 빈 문자열, 플레이스홀더, 혹은 해당 영역 조건부 숨김 중 하나를 사용
- `src/app/app.vue`
  - `getMyInfo()` 호출 위치를 `onMounted` 전용 흐름에서 분리
  - SSR/CSR 공통 진입점에서 1회 실행되도록 `callOnce` 또는 동등한 초기화 방식 적용 검토
  - `/block` 경로 예외는 유지하되, 경로 검사와 정보 조회 타이밍의 계약을 명확히 한다.
- `src/app/stores/user.store.ts`
  - `myInfo`가 `null`일 수 있다는 사실을 유지하되, 이를 소비하는 컴포넌트가 안전하게 다루도록 보조 상태 또는 호출 계약 정리
  - `getMyInfo()` 내부의 디버그 로그 제거 검토
  - 실패 시 `null` 유지와 클라이언트 리다이렉트 조건을 유지하되, SSR에서는 예외 없이 호출 가능해야 한다.
- `src/app/stores/common.store.ts`
  - `navItems` 초기값에 `isAdmin.value`를 고정하지 않고, `isAdmin` 변화가 메뉴 공개 여부에 반영되도록 계산 구조 재구성 검토

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 외부 API 계약 변경 없음
- 내부 인터페이스 변경:
  - 헤더는 `UserOutDto | null`을 직접 렌더하지 않고 표시용 계산값을 사용
  - 앱 초기화는 "사용자 정보가 아직 없을 수 있음"과 "조회 중/조회 실패"를 구분 가능한 흐름으로 구성

## 로직 흐름
1. 앱 진입 시 `/block` 여부를 먼저 판단한다.
2. `/block`이 아니면 사용자 정보 조회 초기화 함수를 앱 초기 단계에서 1회 실행한다.
3. SSR 중 `/users/me` 응답이 아직 없거나 실패하더라도 `myInfo`는 `null` 상태로 유지되고, 헤더는 이를 정상 상태로 렌더한다.
4. 사용자 정보 조회 성공 시 `myInfo`가 갱신되고, 헤더 표시 이름과 관리자 메뉴 공개 상태가 반응형으로 업데이트된다.
5. 사용자 정보 조회 실패 시:
  - SSR: 예외를 던지지 않고 null-safe 상태로 렌더
  - CSR: 기존 정책대로 `/block` 리다이렉트 수행

## 예외 흐름 및 계약
- `myInfo === null`은 오류가 아니라 허용된 초기 상태다.
- 헤더는 초기 상태에서 절대 예외를 발생시키지 않는다.
- 관리자 메뉴는 초기 렌더에서 숨김이어도 허용되며, 사용자 정보 반영 후 노출 여부가 다시 계산되어야 한다.
- `/block` 경로에서는 불필요한 사용자 정보 조회를 생략해도 된다.

## 대안 검토
### 옵션 A
- 설명: `CommonHeader.vue`만 수정하여 `myInfo?.name` 또는 `v-if="myInfo"`로 방어
- 장점:
  - 수정 범위가 가장 작다.
  - SSR 크래시를 즉시 차단할 수 있다.
- 단점:
  - 사용자 정보 초기화 시점 문제는 그대로 남는다.
  - 관리자 메뉴 공개 여부 같은 파생 상태 불일치가 지속될 수 있다.
- 미채택 이유:
  - 단기 핫픽스로는 유효하지만, 현재 분석에서 드러난 초기화 타이밍 문제를 해결하지 못한다.

### 옵션 B
- 설명: 사용자 정보 조회를 SSR 가능한 앱 초기화 지점으로 이동하고, 헤더는 null-safe 렌더링으로 보완
- 장점:
  - SSR 크래시와 초기 상태 불일치를 함께 완화할 수 있다.
  - 이후 공용 레이아웃과 권한 기반 UI에도 같은 규칙을 재사용할 수 있다.
- 단점:
  - 수정 범위가 `app.vue`, 스토어, 헤더로 넓어진다.
  - `useGet()`의 초기화 호환성을 함께 고려해야 한다.

### 최종 선택
- 선택 이유:
  - 구현은 옵션 B를 채택하되, 우선순위는 "헤더 null-safe 보장"을 가장 먼저 두는 2단계 방식을 사용한다.
  - 즉, 헤더 방어를 필수로 적용하고, 동시에 사용자 정보 조회를 앱 초기화 지점으로 옮겨 구조적 원인도 줄인다.
- 기대 효과:
  - SSR 예외 제거
  - 초기 렌더와 하이드레이션 간 사용자 정보 불일치 감소
  - 관리자 메뉴 표시 정책의 반응성 회복 기반 확보

## 제약 및 참고 사항
- `useGet()`가 SSR에서 완전히 동일하게 동작하는지는 구현 전 확인이 필요하다.
- Nuxt 오토임포트 규칙을 따라 추가 import 없이 해결 가능한 범위를 우선 사용한다.
- 설계 외에 인증 정책 자체를 바꾸지는 않는다. 이번 범위는 렌더 안전성과 초기화 시점 정리에 한정한다.
