# 설계 (Design)

## 요약
- 변경 대상: `src/app/composables/query/useDelete.ts`, `usePatch.ts`, `usePost.ts`, `usePut.ts`
- 설계 방향: Nuxt 오토임포트 대상 composable의 공용 export 표면을 함수 중심으로 축소하여 내부 제네릭 식별자 `TBody`가 전역 오토임포트 심볼로 승격되지 않게 한다.
- 주요 결정 사항:
  - 네 파일의 `UseXxxReturn`, `UseXxxOptions` 타입 선언은 파일 내부 전용으로 전환한다.
  - 함수 시그니처와 런타임 동작은 유지한다.
  - 타입을 별도 파일로 분리하는 확장 설계는 이번 작업 범위에서 제외한다.

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 런타임 API 변경 없음
- 모듈 외부에서 직접 import 가능한 타입 심볼은 축소된다.
- 현재 코드 검색 기준 직접 소비 흔적이 없어 회귀 위험은 낮다.

## 로직 흐름
1. Nuxt 오토임포트 스캐너는 `imports.dirs`에 포함된 composable 파일의 top-level export를 수집한다.
2. 각 CRUD composable에서 export 대상이 함수 `useXxx` 하나만 남도록 타입 export를 제거한다.
3. `.nuxt/imports.d.ts` 재생성 시 `TBody`와 `UseXxxOptions`, `UseXxxReturn` 전역 export 생성이 사라져야 한다.
4. 개발 서버 경고가 더 이상 출력되지 않아야 하며, composable의 내부 타입 안정성은 그대로 유지되어야 한다.

## 대안 검토
### 옵션 A
- 설명: 각 파일의 제네릭 파라미터 이름을 `TDeleteBody`, `TPatchBody`처럼 고유하게 변경한다.
- 장점: 경고 메시지 자체는 빠르게 제거할 가능성이 있다.
- 단점: 내부 제네릭 식별자가 계속 전역 심볼로 노출되는 구조는 남는다.
- 미채택 이유: 증상 완화에 그치고 공용 API 표면 오염 문제를 해결하지 못한다.

### 옵션 B
- 설명: 공용 타입을 별도 `query.types.ts`로 분리하고 composable 파일은 함수만 export한다.
- 장점: 타입 구조가 더 명시적이고 재사용에 유리하다.
- 단점: 파일 추가와 타입 import 경로 정리가 필요해 현재 문제 대비 수정 범위가 커진다.
- 미채택 이유: 이번 요청은 경고 제거가 우선이며, 사용처도 확인되지 않아 과도한 확장이다.

### 최종 선택
- 선택 이유: export 표면을 축소하는 방식이 원인을 직접 제거하면서도 수정 범위가 가장 작다.
- 기대 효과: `TBody` 전역 오토임포트 생성이 사라지고, 개발 서버 경고와 타입 공간 오염이 동시에 해소된다.

## 제약 및 참고 사항
- `src/app/types/manual-app-imports.d.ts`와 Nuxt 오토임포트 규칙상 직접 import 추가는 하지 않는다.
- `.nuxt` 생성물은 소스 파일이 아니므로 직접 수정하지 않고 재생성 결과로 확인한다.
