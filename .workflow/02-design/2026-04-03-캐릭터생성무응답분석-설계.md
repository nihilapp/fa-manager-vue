# 설계 (Design)

## 요약
- 변경 대상: 캐릭터 생성 DTO, 캐릭터 생성 API, 생성 폼 payload 계약
- 설계 방향: `race`를 사용자 입력 기준 선택값으로 완화하되, 현재 DB `not null` 제약은 유지하고 서버에서 빈 문자열로 정규화한다
- 주요 결정 사항: DB 마이그레이션 없이 API 계약과 타입 정의를 현재 폼 구조에 맞춘다

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/server/types/dto/character.dto.ts`
  - `CharacterCreateDto.race`를 필수에서 선택으로 전환한다
- `src/server/api/characters/index.post.ts`
  - 생성 시 필수값 검증에서 `race`를 제외한다
  - insert 시 `race`는 `body.race?.trim() ?? ''` 형태로 정규화해 저장한다
- `src/app/components/characters/CharacterCreateForm.vue`
  - 생성 payload에서 강제 `race: ''`를 직접 덮어쓰는 구조를 제거하거나, 선택값이라는 의도가 드러나는 형태로 정리한다
  - API 실패 시 사용자에게 실패 사실이 드러나도록 최소 에러 처리 경로를 연결한다

### DB / 스키마 변경
- `src/server/db/table/characters.table.ts`
  - 이번 변경에서는 수정하지 않는다
  - 현재 `race` 컬럼은 `notNull`이므로, 저장값은 `null` 대신 빈 문자열로 유지한다

### API / 인터페이스 변경
- 생성 API의 필수 계약은 `name` 중심으로 단순화한다
- 클라이언트와 서버 타입 모두 `race` 미전송을 허용하도록 맞춘다
- 응답 구조는 변경하지 않는다. 조회/출력 시 `race`는 기존처럼 `string`을 유지하되 비어 있을 수 있다

## 로직 흐름
1. 생성 폼은 이름, 상태, 시작 레벨, 캠페인, 시작 자금, 클래스 정보만 수집한다
2. submit 시 캐릭터 생성 payload는 `race`를 필수값으로 강제하지 않는다
3. 서버 생성 API는 `name` 존재 여부만 필수로 검사한다
4. 서버는 `race`가 없으면 빈 문자열로 정규화해 `characters` 테이블에 저장한다
5. 생성 성공 시 기존처럼 캐릭터 상세 이동을 수행한다
6. 생성 실패 시 폼은 조용히 종료되지 않고 사용자에게 실패 사실을 알린다

## 대안 검토
### 옵션 A
- 설명: DB 컬럼 `race`를 nullable로 변경하고 전체 타입을 `string | null`로 확장한다
- 장점: 의미상 "선택값"을 DB까지 정확히 반영할 수 있다
- 단점: 마이그레이션과 기존 조회/필터/출력 타입 영향을 모두 검토해야 한다
- 미채택 이유: 현재 요청은 무응답 복구와 필수값 완화가 핵심이며, nullable 전환은 범위가 커진다

### 옵션 B
- 설명: 프런트에 `race` 입력 필드를 다시 추가해 기존 서버 계약을 유지한다
- 장점: 서버와 DB를 거의 건드리지 않는다
- 단점: 마스터 요청인 "선택으로 변경"과 반대 방향이며, 현재 단순화된 UI 흐름도 되돌린다
- 미채택 이유: 요구사항과 불일치한다

### 최종 선택
- 선택 이유: DB 구조를 건드리지 않고도 사용자 관점의 선택값 요구사항을 만족하면서 현재 생성 실패 원인을 제거할 수 있다
- 기대 효과: 캐릭터 생성이 즉시 복구되고, 이후 race 입력이 다시 필요해져도 별도 UI 설계로 독립 대응이 가능하다

## 제약 및 참고 사항
- `CharacterOutDto.race`는 그대로 `string`이므로, 화면에서는 빈 문자열을 만났을 때 `-` 같은 대체 표기를 쓰는지 후속 검토가 필요할 수 있다
- 에러 피드백 방식은 기존 공통 UI 구성 요소 존재 여부를 확인한 뒤 최소 침습적으로 적용해야 한다
