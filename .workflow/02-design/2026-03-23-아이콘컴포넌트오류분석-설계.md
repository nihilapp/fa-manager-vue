# 설계 (Design)

## 요약
- 변경 대상: `src/app/components/GetIcon.vue`
- 설계 방향: `@iconify/vue`의 `Icon`을 템플릿 전역 등록에 의존하지 않고, `GetIcon` 내부에서 명시적으로 참조 가능한 렌더링 대상로 고정한다.
- 주요 결정 사항: Nuxt 전역 컴포넌트 등록 확장 대신 `GetIcon` 단일 수정으로 장애 지점을 차단한다.

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/app/components/GetIcon.vue`
  - 현재 템플릿의 `<Icon />` 직접 사용을 제거한다.
  - `<script setup>`에서 오토임포트된 `Icon` 심볼을 템플릿이 확실히 참조할 수 있는 로컬 식별자 또는 동적 컴포넌트 대상으로 연결한다.
  - 템플릿은 전역 컴포넌트 이름 해석이 아닌 로컬 참조 기반으로 아이콘을 렌더링한다.

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 없음
- `GetIcon`의 props 계약 `iconName: IconName` 유지
- `GetIcon`를 사용하는 상위 컴포넌트 인터페이스 변경 없음

## 로직 흐름
1. 레이아웃 또는 화면 컴포넌트가 기존과 동일하게 `GetIcon`에 `iconName`을 전달한다.
2. `GetIcon`의 `<script setup>`은 Nuxt 오토임포트 심볼 `Icon`을 로컬 렌더링 대상으로 고정한다.
3. 템플릿은 전역 `<Icon>` 태그를 찾지 않고, 로컬 식별자를 통해 실제 `@iconify/vue` 컴포넌트를 렌더링한다.
4. `CommonFooter`, `CommonHeader`, `DataTable` 등 모든 호출처는 수정 없이 정상 렌더링된다.
5. 개발서버 초기 진입 시 `Failed to resolve component: Icon` 경고와 500이 제거된다.

## 대안 검토
### 옵션 A
- 설명: `nuxt.config.ts`에 별도 컴포넌트 등록 또는 외부 모듈을 추가해 `Icon`을 전역 Vue 컴포넌트로 등록한다.
- 장점:
  - 템플릿에서 `<Icon />`를 직접 사용하는 패턴을 유지할 수 있다.
  - 향후 다른 컴포넌트에서도 동일 이름을 바로 사용할 수 있다.
- 단점:
  - Nuxt 설정 변경과 재생성 범위가 커진다.
  - 전역 등록 방식은 현재 프로젝트의 오토임포트 규칙과 경계를 흐릴 수 있다.
  - 지금 문제는 `GetIcon` 내부 한 지점에서 발생하므로 변경 대비 효과가 과하다.
- 미채택 이유:
  - 긴급 장애 대응으로는 수정 범위가 불필요하게 크고, 설정 변경 부작용 가능성이 있다.

### 옵션 B
- 설명: `GetIcon.vue`에서 `import { Icon } from '@iconify/vue'`를 직접 선언한다.
- 장점:
  - 가장 직관적이고 Vue SFC 관점에서 즉시 이해 가능하다.
  - 전역 등록 여부와 무관하게 안정적으로 동작한다.
- 단점:
  - 프로젝트 지침상 Nuxt 오토임포트/전역 선언으로 해결 가능한 대상은 직접 import 하지 말아야 한다.
- 미채택 이유:
  - 프로젝트 규칙 위반 소지가 있어 유지하지 않는다.

### 최종 선택
- 선택 이유:
  - `GetIcon` 내부에서 오토임포트된 `Icon` 심볼을 로컬 렌더링 대상으로 명시적으로 연결하면 직접 import 없이도 템플릿 해석 실패를 없앨 수 있다.
  - 수정 범위가 단일 파일이라 기존 사용자 변경과 충돌 가능성이 가장 낮다.
  - 모든 호출처를 그대로 유지하면서 장애를 해소할 수 있다.
- 기대 효과:
  - 접속 직후 500 제거
  - `GetIcon` 사용 지점 전체 안정화
  - 향후 유사한 외부 컴포넌트 사용 시 "템플릿 전역 등록"과 "오토임포트 심볼"의 경계가 분명해짐

## 제약 및 참고 사항
- `manual-app-imports.d.ts`와 `.nuxt/types/imports.d.ts`는 타입과 심볼 가시성만 제공하므로, 런타임 컴포넌트 등록 보장을 대신하지 못한다.
- 구현은 `GetIcon.vue` 단일 파일 수정과 최소 검증으로 제한한다.
- 설계 외에 Nuxt 전역 설정 변경이 필요해지면 다시 2단계로 복귀한다.
