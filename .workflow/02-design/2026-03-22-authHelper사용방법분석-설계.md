# 설계 (Design)

## 요약
- 변경 대상: `src/server/utils/auth.ts`의 사용 규약 해석 방식과 후속 정리 기준
- 설계 방향: `authHelper`를 "인증/인가 컨텍스트 생성기"로 정의하고, 반환 필드별 표준 사용 흐름을 계약 단위로 명확히 정리한다.
- 주요 결정 사항:
  - 호출 직후 `error`를 선처리하는 것을 표준 흐름으로 고정한다.
  - 인가 판단은 `isAdmin` 직접 비교보다 `hasPermission(resourceOwnerId)` 중심으로 설명한다.
  - 개발 환경 분기는 운영 인증과 별도의 우회 흐름으로 명시한다.
  - 추후 리팩터링 시 인증, 권한 판정, 개발 우회를 분리 가능한 계약으로 본다.
- 참조 문서:
  - `.workflow/01-analysis/2026-03-22-authHelper사용방법분석-분석.md`
  - `PRD/PRD.md`
  - `PRD/Coding-Rules.md`

## 아키텍처 변화

### 신규 구성 요소
- 없음
- 이번 단계는 코드 추가가 아니라 `authHelper` 사용 규약과 설명 구조를 설계하는 문서화 단계다.

### 수정 구성 요소
- 코드 수정 없음
- 다만 이후 문서화 또는 리팩터링이 필요할 경우 아래 논리 단위로 분리해 다룬다.
  - 인증 입력 계층: `getDiscordId(event)`로 `x-discord-id`를 읽는 단계
  - 사용자 해석 계층: Discord ID로 실제 사용자 레코드를 찾는 단계
  - 권한 컨텍스트 계층: `isAdmin`, `hasPermission`, `canAccessSelf`, `requireUser`를 노출하는 단계
  - 개발 우회 계층: `NODE_ENV === 'development'`일 때 대체 사용자를 선택하고 권한을 우회하는 단계

### DB / 스키마 변경
- 없음
- 현재 설계 범위는 사용 규약 분석이므로 DB 스키마 변경은 포함하지 않는다.

### API / 인터페이스 변경
- 현재 단계에서는 인터페이스 변경 없음
- 다만 설명 및 후속 구현 기준으로 `authHelper(event)`의 계약을 아래처럼 고정한다.
  - 입력: `H3Event`
  - 출력: 인증 결과(`user`, `error`)와 인가 보조 함수(`isAdmin`, `hasPermission`, `checkAdmin`, `canAccessSelf`, `requireUser`)를 포함한 컨텍스트
  - 전제: 호출자는 `error`를 선처리한 뒤 나머지 필드를 사용해야 한다.

## 로직 흐름
1. 요청 핸들러는 `await authHelper(event)`를 호출한다.
2. `authHelper`는 `getDiscordId(event)`로 `x-discord-id` 헤더를 읽는다.
3. 개발 환경이면:
   - 헤더 기반 사용자 조회를 우선 시도한다.
   - 실패 시 관리자 계정을 우선 선택하고, 그것도 없으면 첫 사용자로 대체한다.
   - `isDevelopmentBypass = true`인 컨텍스트를 반환한다.
4. 운영 환경이면:
   - 헤더가 없으면 `error = UNAUTHORIZED`인 컨텍스트를 반환한다.
   - 헤더는 있으나 사용자가 없으면 `error = USER_NOT_FOUND`인 컨텍스트를 반환한다.
   - 사용자가 있으면 정상 컨텍스트를 반환한다.
5. 호출자는 반환 직후 `if (error) return error;`를 먼저 수행한다.
6. 이후 사용 목적에 따라 분기한다.
   - 인증만 필요: `user` 사용
   - 관리자 전용: `isAdmin` 또는 `checkAdmin()` 사용
   - 자원 소유권 판정: DB에서 자원을 조회한 뒤 `hasPermission(resource.userId)` 사용
   - 복합 인가: 도메인 조건(예: 세션 참가 여부)과 `hasPermission(...)`, `isAdmin`을 함께 조합
7. 후속 DB 수정 시에는 `user.id`를 메타 정보 작성자 ID로 사용한다.

## 사용 규약 설계

### 표준 호출 패턴
```ts
const { user, isAdmin, hasPermission, error } = await authHelper(event);
if (error) return error;
```

### 규약 1. `error` 선처리
- 모든 운영 경로는 `error`를 우선 반환 처리해야 한다.
- 이 규약을 지키지 않으면 `user`가 `null`일 수 있고, 후속 로직에서 런타임 오류나 인증 누락이 생길 수 있다.
- `src/server/api/users/me.get.ts`는 현재 이 규약을 따르지 않는 예외 사례로 본다.

### 규약 2. `hasPermission(resourceOwnerId)` 중심 인가
- 본인 또는 관리자 허용이 필요한 엔드포인트는 `hasPermission`을 표준으로 사용한다.
- 직접 `user.id === resource.userId || isAdmin`를 매번 풀어 쓰지 않고, 컨텍스트 함수에 의미를 위임하는 구조가 현재 아키텍처와 맞다.

### 규약 3. `isAdmin`은 보조 조건으로 사용
- 관리자 전용 API에서 직접 사용하거나, 일반 소유권 외 추가 기능 허용에만 사용한다.
- 예: 다른 사용자 대신 로그를 작성할 수 있는지 결정

### 규약 4. 개발 환경 동작은 운영 인증과 별도로 본다
- 개발 환경에서는 인증 실패보다 테스트 편의성이 우선된다.
- 따라서 로컬 동작을 운영 인증 계약의 검증 근거로 삼지 않는다.

## 대안 검토

### 옵션 A. 현재처럼 단일 헬퍼로 유지
- 설명: 인증, 사용자 조회, 관리자 판정, 소유권 검사, 개발 우회를 하나의 함수에서 제공
- 장점:
  - 호출부가 짧고 일관적이다.
  - 대부분의 API가 동일한 구조분해 패턴으로 사용할 수 있다.
- 단점:
  - 인증과 인가, 개발 우회의 책임이 섞인다.
  - 사용자가 `error` 선처리를 누락하기 쉽다.
  - 반환 표면이 넓어 실제 필요보다 많은 기능을 노출한다.
- 이번 분석 기준: 현 구조 설명에는 적합하지만 장기적으로는 역할 경계가 흐리다.

### 옵션 B. 인증과 인가 헬퍼를 분리
- 설명: `resolveAuthUser(event)`와 `createAccessPolicy(user)` 같은 형태로 분리
- 장점:
  - 역할이 명확해진다.
  - 테스트와 운영 흐름을 나눠 설명하기 쉽다.
  - `error` 처리 규약을 타입 수준에서 더 강제하기 좋다.
- 단점:
  - 기존 API 전역 수정이 필요하다.
  - 현재 요청 범위는 분석이므로 변경 파급이 과하다.
- 미채택 이유:
  - 이번 작업은 사용방법 분석과 문서화가 목적이며, 구조 변경은 범위를 벗어난다.

### 옵션 C. 엔드포인트별로 권한 조건을 직접 작성
- 설명: `authHelper` 없이 각 핸들러에서 `discordId`, `user`, `isAdmin`을 개별 조합
- 장점:
  - 각 엔드포인트의 권한 정책이 로컬에서 모두 드러난다.
- 단점:
  - 중복이 커지고 일관성이 깨진다.
  - 실수 가능성이 증가한다.
- 미채택 이유:
  - 현재 코드베이스는 공통 헬퍼 중심 구조이며, 이 방향을 버리면 유지보수 비용이 커진다.

### 최종 선택
- 선택 이유:
  - 이번 단계에서는 옵션 A 구조를 유지하되, 사용 규약을 명시적으로 설계하는 것이 가장 현실적이다.
  - 분석 결과를 기준으로 호출 순서와 필드별 책임을 문서화하면, 별도 코드 변경 없이도 오용 위험을 줄일 수 있다.
- 기대 효과:
  - `authHelper`를 처음 보는 작업자도 어떤 필드를 언제 써야 하는지 바로 판단할 수 있다.
  - 추후 리팩터링이 필요할 경우 인증/인가/개발우회 분리 경계를 이 문서에서 바로 가져갈 수 있다.

## 제약 및 참고 사항
- 현재 세션에서는 context7 MCP를 사용할 수 없어 로컬 정적 분석을 기준으로 설계했다.
- `users/me.get.ts`는 미완성 또는 예외 구현으로 보이며, 규약 준수 사례로 사용하지 않는다.
- 다음 단계 태스크 확립에서는 문서화 작업과 후속 선택 작업(예: 사용 규약 문서 정리, 예외 사용처 보강 후보 식별)을 원자 단위로 분해할 수 있다.
