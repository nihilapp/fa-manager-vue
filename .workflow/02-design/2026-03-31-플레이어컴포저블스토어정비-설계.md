# 설계 (Design)

## 요약
- 변경 대상: player/session 스토어, players 컴포저블 일괄
- 설계 방향: 기본 축과 플레이어별 파생 축을 분리하고 모든 조회 응답은 스토어 액션을 통해 저장
- 주요 결정 사항: 목록/페이지/개별 정보 액션 패턴 통일, players 미구현 API 컴포저블 추가

## 아키텍처 변화
### 신규 구성 요소
- `useGetPlayerCampaignList.ts`: 플레이어별 캠페인 목록 조회 후 campaign 전용 또는 추후 확장 스토어 패턴에 맞출 기반 제공
- `useCheckPlayerAccess.ts`: `/players/check` 래퍼
- `useUpdateMyInfo.ts`: `/players/me` 수정 래퍼
- `useDeleteMyInfo.ts`: `/players/me` 삭제 래퍼
- `useDeletePlayer.ts`: `/players/:id` 삭제 래퍼

### 수정 구성 요소
- `player.store.ts`
  - 기본 축: `playerList`, `playerPageData`, `playerInfo`
  - 파생 축: `myInfo`
  - 액션 통일: `setPlayerList`, `setPlayerPageData`, `clearPlayerList`, `setPlayerInfo`, `clearPlayerInfo`, `setMyInfo`, `clearMyInfo`, `setMyPageData`는 두지 않고 `myInfo`는 개별 정보 축으로만 유지
- `session.store.ts`
  - 기본 축: `sessionList`, `sessionPageData`, `sessionInfo`
  - 파생 축: `playerSessionList`, `playerSessionPageData`, 필요 시 향후 동일 패턴 확장
  - 액션 통일: 기본 목록과 파생 목록 초기화 액션을 분리
- 기존 players 컴포저블
  - 성공 시 스토어 저장
  - 실패 시 대응 상태만 초기화
  - 뮤테이션 후 필요한 경우 관련 조회 재호출 또는 직접 상태 동기화

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 프런트엔드 컴포저블 인터페이스만 추가/정리
- 서버 API 계약은 유지

## 로직 흐름
1. 페이지나 컴포넌트가 도메인 컴포저블의 `execute`를 호출한다.
2. 컴포저블은 `useGet/usePost/usePut/useDelete`를 통해 API를 호출한다.
3. 성공 응답이면 응답 타입에 맞춰 스토어 액션을 호출한다.
4. 목록 응답이면 `list`와 `pageData`를 분리해 저장한다.
5. 개별 응답이면 해당 `info` 축에 저장한다.
6. 실패 응답이면 호출한 축에 대응하는 초기화 액션만 실행한다.
7. 뮤테이션 성공 후에는 직접 스토어를 보정하거나 관련 조회를 재실행해 일관성을 맞춘다.

## 대안 검토
### 옵션 A
- 설명: 모든 players 관련 상태를 `player.store.ts` 하나에 몰고 session/campaign 파생 목록도 player store에 저장
- 장점: players 페이지 구현 시 참조 경로가 단순함
- 단점: 도메인 경계가 무너지고 session/campaign 스토어 확장성과 재사용성이 떨어짐
- 미채택 이유: 마스터가 언급한 플레이어별 목록은 각 도메인 스토어의 파생 축으로 관리하는 편이 장기적으로 명확함

### 옵션 B
- 설명: 뮤테이션 성공 시 스토어를 직접 갱신하지 않고 무조건 전체 목록 재조회
- 장점: 구현이 단순함
- 단점: 네트워크 호출이 늘고 개별 정보/내 정보 동기화가 늦어짐
- 미채택 이유: 현재 구조에서는 직접 저장과 부분 재조회 혼합이 더 실용적임

### 최종 선택
- 선택 이유: 도메인 스토어의 기본 축과 플레이어별 파생 축을 유지하면서도 액션 패턴을 통일할 수 있음
- 기대 효과: 이후 campaigns/sessions/characters 컴포저블 작성 시 동일한 저장 규칙을 재사용 가능

## 제약 및 참고 사항
- Nuxt 오토임포트 대상은 직접 import 하지 않는다.
- 기존 페이지 참조명을 최대한 유지해 불필요한 컴포넌트 수정 범위를 줄인다.
- `GET /players/check`는 현재 사용처가 없어도 추후 인증/접근 제어용으로 유지한다.
