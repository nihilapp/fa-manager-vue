# 설계 (Design)

## 요약
- 변경 대상: `src/app/pages/characters/index.vue`, `src/app/components/characters/CharacterList.vue`, `src/app/components/characters/CharacterListTable.vue`
- 설계 방향: 플레이어 목록과 동일하게 페이지는 메타와 래퍼만 두고, 실제 목록 조회와 테이블 렌더링은 전용 컴포넌트로 분리한다.
- 주요 결정 사항:
  - 컨테이너 컴포넌트가 `useGetCharacterList`와 페이지네이션을 담당한다.
  - 테이블 컬럼은 요청한 6개 컬럼만 구성하고 액션 컬럼은 두지 않는다.
  - 비욘드 링크는 현재 DTO에 필드가 없어 fallback `-` 또는 비활성 버튼으로 처리한다.

## 아키텍처 변화
### 신규 구성 요소
- `src/app/components/characters/CharacterList.vue`
  - 캐릭터 목록 조회, 페이지 전환, 생성 버튼 클릭을 담당한다.
- `src/app/components/characters/CharacterListTable.vue`
  - 캐릭터 리스트를 `DataTable`로 렌더링하고 클래스/링크 포맷팅을 담당한다.

### 수정 구성 요소
- `src/app/pages/characters/index.vue`
  - 메타 설정 후 `CharacterList`를 렌더링한다.

## 로직 흐름
1. `characters/index.vue`가 `CharacterList`를 렌더링한다.
2. `CharacterList`는 마운트 시 `useGetCharacterList({ page, size })`를 호출한다.
3. 스토어의 `characterList`, `characterPageData`를 `CharacterListTable`에 전달한다.
4. `CharacterListTable`은 클래스 배열을 문자열로 합치고, 비욘드 링크는 응답에 값이 없으므로 `-`로 표기한다.
5. 상단 `캐릭터 생성` 버튼은 후속 캐릭터 생성 경로 준비 전까지 `/characters/add-character` 경로를 사용하도록 맞춘다.

## 대안 검토
### 옵션 A
- 설명: `characters/index.vue` 한 파일에서 목록 로딩과 테이블 컬럼 정의를 모두 처리한다.
- 장점: 파일 수가 적다.
- 단점: 플레이어 목록과 구조가 달라져 재사용 패턴이 약해진다.
- 미채택 이유: 마스터 요청이 플레이어 목록 페이지를 참고한 동일 패턴 구현이다.

### 옵션 B
- 설명: 플레이어 목록 테이블을 범용화해 캐릭터 목록도 함께 사용한다.
- 장점: 공통화 수준이 높다.
- 단점: 컬럼 구조가 달라 제네릭 확장이 더 커지고 현재 범위를 넘는다.
- 미채택 이유: 이번 작업은 캐릭터 목록 화면을 빠르게 안정화하는 편이 우선이다.

### 최종 선택
- 선택 이유: 페이지/컨테이너/테이블 3단 구조를 그대로 맞추면 일관성이 높고 이후 상세/필터 확장도 수월하다.
- 기대 효과: 플레이어 목록과 같은 UX 흐름으로 캐릭터 목록을 바로 사용할 수 있다.

## 제약 및 참고 사항
- `비욘드 링크` 실제 필드가 DTO에 추가되기 전까지는 placeholder 표기로 남는다.
- 캐릭터 생성 경로가 아직 비어 있으면 버튼 연결만 준비하고 후속 작업 대상으로 남길 수 있다.
