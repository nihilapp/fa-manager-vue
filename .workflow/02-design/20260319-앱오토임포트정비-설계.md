# 2단계 설계 문서

## 참조 문서
- `.workflow/01-analysis/20260319-앱오토임포트정비-분석.md`

## 변경 목표
- 앱 레이어의 메타 유틸과 API 쿼리 컴포저블을 실제 폴더 구조와 Nuxt 동작 방식에 맞게 재구성합니다.
- `src/server`와 같은 선언형 오토임포트 기준을 앱 레이어에도 적용합니다.

## 설계 방안
- `src/app/types/common.types.ts`에 앱 메타 타입 외에도 서버 응답 공유 타입 별칭을 추가합니다.
- `src/app/composables/query` 아래에 공통 요청 헬퍼를 두고 인증 헤더 생성, 성공/실패 콜백 처리, 오류 정규화를 통합합니다.
- `useGet`은 `query`, `enabled`, `watch`, `key`를 받는 구조로 바꾸고 `watch`와 수동 `execute()` 중복 실행을 제거합니다.
- `POST`, `PUT`, `PATCH`, `DELETE`는 공통 mutation 헬퍼를 사용해 동일한 상태 인터페이스를 반환합니다.
- `useSetMeta`는 제목, canonical URL, 이미지 URL, OG/Twitter 값을 일관된 해석 규칙으로 계산하고 `useSeoMeta`와 `useHead`를 함께 사용합니다.
- `nuxt.config.ts`의 앱 `imports.autoImport`를 활성화하고 앱 전용 util/composable/config import와 공용 preset을 등록합니다.
- `src/app/types/manual-app-imports.d.ts`는 Nuxt 생성 타입과 충돌하지 않도록 안내용 최소 파일로 축소합니다.

## 로직 및 계약
- `useGet` 성공 시에만 성공 콜백을 호출하고, 에러 응답 또는 네트워크 예외는 공통 에러 처리 후 `errorCallback`으로 전달합니다.
- mutation 계열은 실행마다 최신 쿠키 토큰을 읽어 헤더를 구성하고 `status`, `error`, `response`를 반응형으로 갱신합니다.
- `useSetMeta`는 빈 제목에서도 사이트 기본 제목이 깨지지 않아야 하며, 상대 경로 입력 시 canonical/이미지 URL을 안전하게 조합해야 합니다.
- 오토임포트는 `appConfig`, `useSetMeta`, `useGet`~`useDelete`, `handleApiError` 계열을 명시적으로 노출합니다.

## 대안 검토
- 대안 1: 기존 수동 전역 선언 파일을 유지하고 오토임포트를 계속 비활성화합니다.
  - 미채택 이유: 실제 런타임 동작과 타입 선언이 분리되어 유지보수 비용이 커집니다.
- 대안 2: `POST`, `PUT`, `PATCH`, `DELETE` 파일을 그대로 유지하고 중복만 부분 수정합니다.
  - 미채택 이유: 인증 헤더, 에러 처리, 상태 갱신 규약이 계속 분산되어 이후 수정 비용이 커집니다.
- 대안 3: 앱 전역 import를 모두 상대 경로 명시 import로 유지합니다.
  - 미채택 이유: 사용자 요청인 `src/server` 수준의 선언형 오토임포트 방향과 맞지 않습니다.

## 예외 흐름
- 네트워크 예외에서 서버 응답 본문이 없는 경우에는 공통 에러 처리 콜백을 생략하고 원본 예외를 유지합니다.
- 사이트 기본 URL이 비어 있으면 canonical/이미지 URL은 상대 경로 기준으로 유지됩니다.
- 타입체크 결과 다른 미정비 앱 코드가 추가로 드러날 수 있으며, 그 경우 현재 설계 범위 안에서 경로 정리까지 함께 처리합니다.
