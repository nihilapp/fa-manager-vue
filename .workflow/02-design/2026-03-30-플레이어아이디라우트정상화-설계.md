## 요약
- 변경 대상: `src/server/api/players/[id].get.ts`, `src/server/api/players/[id].put.ts`, `src/server/api/players/[id].delete.ts`
- 설계 방향: `me`는 자기 자신 전용으로 유지하고 `[id]`는 어드민 대상 플레이어 관리로 복구
- 주요 결정 사항:
  - 세 라우트 모두 `getRouterParam(event, 'id')`를 사용한다.
  - 세 라우트 모두 `isAdmin` 검사 후 동작한다.
  - `GET`은 조회, `PUT`은 수정, `DELETE`는 soft delete만 수행한다.

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `players/[id].get.ts`: 대상 플레이어 상세 조회
- `players/[id].put.ts`: 대상 플레이어 수정
- `players/[id].delete.ts`: 대상 플레이어 soft delete

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 외부 URL 유지
- 내부 대상 식별 기준을 `user!.id`에서 route param `id`로 교체

## 로직 흐름
1. `authHelper(event)`로 사용자 컨텍스트를 가져온다.
2. 어드민이 아니면 `FORBIDDEN`을 반환한다.
3. route param `id`를 숫자로 파싱한다.
4. 대상 플레이어를 `deleteYn = 'N'` 조건으로 조회한다.
5. `GET`은 조회 결과를 반환한다.
6. `PUT`은 이름 중복 검사 후 대상 플레이어를 수정한다.
7. `DELETE`는 soft delete 메타를 기록한다.

## 대안 검토
### 옵션 A
- 설명: `[id]` 라우트를 제거하고 `me`만 유지
- 장점: 단순함
- 단점: 어드민 대상 플레이어 관리 요구를 만족하지 못함
- 미채택 이유: 마스터 요구와 불일치

### 최종 선택
- 선택 이유: 기존 라우트 구조를 유지하면서 책임만 올바르게 복구할 수 있다
- 기대 효과: `me`와 `[id]` 책임이 명확해지고 어드민 관리 기능이 정상 동작한다

## 제약 및 참고 사항
- `[id]` 수정은 어드민 전용이므로 본인 수정 제약을 적용하지 않는다.
