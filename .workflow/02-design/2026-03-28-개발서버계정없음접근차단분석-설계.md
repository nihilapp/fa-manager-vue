- 설계 방향: 개발 우회 플래그가 활성화된 경우 소비자 레이어에서 `error`보다 먼저 처리하도록 우선순위 조정
- 주요 결정 사항: `authHelper()`의 반환 계약은 유지하고, 실제 차단 여부를 판단하는 소비자만 수정

### API / 인터페이스 변경

  - 변경: 개발 우회 상태에서는 성공 응답으로 처리하며 `data`는 `null`일 수 있음

  - 기존: `error` 우선
  - 변경: 개발 우회 우선

## 로직 흐름
1. 요청이 페이지 라우트로 들어오면 `checkCommunityAccess()`가 `authHelper()` 결과를 받는다.
2. `isDevelopmentBypass`가 `true`면 `error` 존재 여부와 무관하게 접근 허용 응답을 반환한다.
3. 개발 우회가 아니면 기존처럼 `error`, `user`, 관리자/본인 권한 순서로 판정한다.
4. 앱 부트스트랩의 `/players/me` 호출도 같은 우선순위를 적용한다.
5. 개발 우회 상태에서 `user`가 없으면 성공 응답의 `data`는 `null`이 되고, 클라이언트는 사용자 정보만 비운 채 페이지 접근은 유지한다.

## 대안 검토
### 옵션 A
- 설명: `authHelper()` 자체가 개발환경에서 플레이어 미존재 에러를 만들지 않도록 변경
- 장점:
  - 소비자 모두가 자동으로 동일 동작을 따름
- 단점:
  - 모든 API에서 `user === null` 상태가 확산되어 `user!.id` 가정이 많은 현재 코드와 충돌 가능성이 큼
  - 개발 우회 허용 범위를 의도보다 넓게 열 수 있음
- 미채택 이유:
  - 이번 요구는 접근 차단 해소가 핵심이며, 인증 계약 전체를 바꾸면 영향 범위가 과도하게 커진다.

### 최종 선택
- 선택 이유:
  - 실제 차단 지점인 페이지 접근 가드와 부트스트랩 API만 수정하면 문제를 직접 해소할 수 있다.
  - 다른 CUD API의 기존 보호 로직을 건드리지 않아 부작용을 최소화한다.
- 기대 효과:
  - DB가 비어 있는 개발환경에서도 `/block` 리다이렉트 없이 화면 진입 가능
  - 개발 우회와 일반 인증 오류의 해석 우선순위가 명시적으로 정리됨

## 제약 및 참고 사항
- 개발 우회 상태에서 `user`가 `null`일 수 있으므로, 이후 화면에서 사용자 실체를 전제하는 기능은 별도 보호가 필요하다.
- 이번 작업은 접근 차단 해소에 한정하며, 초기 관리자 생성 UX까지 확장하지 않는다.
