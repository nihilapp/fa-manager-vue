# 설계 (Design)

## 요약
- 변경 대상: `CharacterCreateForm.vue`, `CharacterEditorForm.vue`, `character.store.ts`
- 설계 방향: 플레이어 폼과 동일하게 `zod` 스키마를 `useForm.validationSchema`에 연결하되, 두 캐릭터 폼이 동일한 스키마를 재사용한다.
- 주요 결정 사항: 기본 캐릭터 필드만 `zod` 검증 대상으로 두고 클래스 배열 입력은 이번 범위에서 제외한다.

## 아키텍처 변화
### 수정 구성 요소
- `src/app/stores/character.store.ts`
  - 캐릭터 상태 enum 상수와 숫자 문자열 검증 유틸리티를 정의
  - `createCharacterFormValidationSchema()`를 추가해 두 폼에서 공용 사용
- `src/app/components/characters/CharacterCreateForm.vue`
  - `useForm`에 `validationSchema` 연결
- `src/app/components/characters/CharacterEditorForm.vue`
  - `useForm`에 동일 `validationSchema` 연결

### API / 인터페이스 변경
- 외부 API 변경 없음
- 캐릭터 폼은 제출 전에 `zod` 검증을 통과해야 `handleSubmit` 콜백이 실행된다.

## 로직 흐름
1. 스토어에서 상태 enum과 숫자 문자열 검증 함수를 조합한 `z.object` 스키마를 만든다.
2. 생성/수정 폼은 `createCharacterFormValidationSchema()`를 `useForm.validationSchema`에 전달한다.
3. 사용자가 잘못된 값을 입력하면 `FormInput`, `FormSelect`가 `Field` 에러 메시지를 출력한다.
4. 검증을 통과한 경우에만 기존 제출 로직이 실행된다.

## 대안 검토
### 옵션 A
- 설명: 생성 폼과 수정 폼 각각에 동일 스키마를 중복 선언한다.
- 장점: 파일 단위로 이해가 쉽다.
- 단점: 검증 규칙 변경 시 두 곳을 함께 수정해야 한다.
- 미채택 이유: 동일 입력 구조에 대한 중복 유지비용이 크다.

### 옵션 B
- 설명: 스토어에 공용 스키마 팩토리를 두고 두 폼이 재사용한다.
- 장점: 규칙 일관성이 유지되고 향후 필드 확장 시 변경 지점이 한 곳이다.
- 단점: 스토어가 폼 검증 책임 일부를 갖게 된다.
- 미채택 이유: 없음

### 최종 선택
- 선택 이유: 캐릭터 생성/수정 기본 입력 구조가 동일해 공용 스키마화가 가장 단순하다.
- 기대 효과: 플레이어 폼과 동일한 검증 진입점을 확보하면서도 중복을 줄일 수 있다.

## 제약 및 참고 사항
- 숫자 입력 필드는 현재 문자열로 보관하므로 `zod`에서 숫자형이 아니라 정수 문자열 검증으로 처리한다.
- `campaignId`는 현재 `선택 안 함`이 허용되므로 `number | null`을 유지한다.
