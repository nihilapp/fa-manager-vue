# 설계 (Design)

## 요약
- 변경 대상: `Box.vue` 프롭 계약과 기존 `padding`/`border-style`/`border-color` 호출부
- 설계 방향: `Box`는 레이아웃 패딩을 제거하고, `border`와 `color`만 공통 프롭으로 유지한다
- 주요 결정 사항: `color`는 `StatusColor`를 그대로 사용하며 border/background를 함께 제어한다

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/app/components/common/Box.vue`
  - `borderStyle`를 `border`로 축약한다
  - `borderColor`와 `padding`을 제거한다
  - `color?: StatusColor`를 추가한다
  - `StatusColor`별 border/background 조합을 내부 variant로 직접 정의한다
- `Box` 호출부
  - `padding` 사용은 모두 `class`의 `p-*` 또는 기존 레이아웃 클래스 조합으로 치환한다
  - `border-style`은 `border`로 변경한다
  - `border-color`는 `color`로 변경한다

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- `Box` 컴포넌트 인터페이스
  - 기존: `borderStyle?: 'solid' | 'dashed' | 'none'`, `borderColor?: 'slate' | 'stone'`, `padding?: 'none' | 'sm' | 'md' | 'lg'`
  - 변경: `border?: 'solid' | 'dashed' | 'none'`, `color?: StatusColor`

## 로직 흐름
1. `Box`는 기본으로 `rounded-2`만 제공하고 배경/테두리 색은 `color`에 따라 정한다
2. `border='solid' | 'dashed' | 'none'`에 따라 테두리 스타일만 결정한다
3. `color`가 없거나 기본값이면 neutral 계열(`gray`) border/background를 적용한다
4. 호출부는 패딩과 텍스트 색 같은 세부 디자인을 `class`에서 직접 지정한다
5. 에러/상태 박스처럼 배경과 테두리를 함께 쓰는 곳은 `color="red"` 같은 방식으로 표현한다

## 대안 검토
### 옵션 A
- 설명: `border`, `borderColor`, `backgroundColor`를 각각 독립 프롭으로 둔다
- 장점: 조합 자유도가 높다
- 단점: 공통 컴포넌트가 다시 세부 디자인 조합 책임을 많이 가져가고, 호출부에서 잘못된 조합이 늘어난다
- 미채택 이유: 마스터가 요청한 단순화 방향과 맞지 않는다

### 옵션 B
- 설명: `Box`의 모든 디자인 프롭을 제거하고 `class`만 남긴다
- 장점: 구현이 가장 단순하다
- 단점: 반복적으로 쓰이는 상태성 박스의 border/background 표현이 매번 중복된다
- 미채택 이유: 마스터가 색상과 테두리 정도는 쓰기 편하게 유지하길 원했다

### 최종 선택
- 선택 이유: 공통 컴포넌트의 책임을 줄이면서도 자주 반복되는 상태성 박스 표현은 간단히 유지할 수 있다
- 기대 효과: 호출부가 더 직관적이고, `StatusColor` 체계와 시각 표현이 맞춰진다

## 제약 및 참고 사항
- `StatusColor` 타입은 오토임포트/전역 선언으로 해결되는지 확인된 상태이므로 직접 import를 추가하지 않는다
- `Box`의 `color` 매핑은 `StatusBadge`보다 약간 넓은 영역에 쓰이므로 background는 동일 계열의 연한 톤으로 유지하는 것이 안전하다
