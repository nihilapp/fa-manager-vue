# 설계 (Design)

## 요약
- 변경 대상: `src/app/components/common/Button.vue`
- 설계 방향: 버튼 기능은 공통 계약으로 흡수하고, 시각 표현은 컴포넌트 내부의 최소 베이스 클래스와 호출부 커스텀 클래스를 조합하는 방식으로 유지
- 주요 결정 사항:
  - `mode`는 `fill`, `ghost`, `outline` 3가지만 지원
  - 버튼 스타일은 별도 전역 CSS 의존을 최소화하고 `Button.vue` 내부 계산 클래스로 우선 정의
  - 전역 유틸리티를 새로 만들 경우에는 Tailwind 유틸리티 클래스 형태로만 추가 가능하지만, 이번 설계에서는 컴포넌트 내부 정의를 우선 채택

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/app/components/common/Button.vue`
  - `type` 기본값을 `button`으로 고정해 폼 내부 오동작을 방지
  - `mode`, `disabled`, `loading` 조합으로 실제 인터랙션 가능 여부를 계산
  - 텍스트 버튼과 아이콘 버튼을 하나의 템플릿 구조로 통합
  - `buttonClass`, `iconClass`를 베이스 클래스 위에 병합해 호출부 커스터마이즈를 허용
  - `run` emit은 유지하되, 클릭 이벤트 원본도 함께 전달할 수 있도록 확장
  - `aria-label`은 아이콘 전용 버튼일 때 필수 의미를 가지도록 처리

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- `Button.vue` props 변경 계획
  - 유지: `iconName`, `label`, `ariaLabel`, `disabled`, `type`, `buttonClass`, `iconClass`
  - 정리: `color`는 이번 설계에서 제거 대상. 실제 색상 표현은 호출부 클래스 또는 내부 모드 클래스가 담당
  - 확장: `mode?: 'fill' | 'ghost' | 'outline'`
  - 확장: `loading?: boolean`
- `Button.vue` emit 변경 계획
  - 유지 및 확장: `run` emit은 유지하되 `MouseEvent`를 payload로 전달

## 로직 흐름
1. 호출부가 `iconName`, `label`, `mode`, `disabled`, `loading`, 커스텀 클래스를 전달한다.
2. 컴포넌트는 `type` 기본값을 `button`으로 적용하고, `disabled || loading`이면 실제 비활성 상태로 계산한다.
3. `mode` 값에 따라 내부 베이스 버튼 클래스를 계산한다.
4. 공통 베이스 클래스에는 최소한의 레이아웃, 정렬, 간격, 전환, 비활성 상태 표현만 포함한다.
5. `fill`은 기본 배경과 텍스트 대비가 있는 일반 버튼 의미로 처리한다.
6. `ghost`는 배경과 테두리를 최소화한 텍스트 중심 버튼 의미로 처리한다.
7. `outline`은 테두리와 텍스트 중심의 중립 버튼 의미로 처리한다.
8. 최종 버튼 클래스는 `base + mode + state + buttonClass` 순서로 병합한다.
9. 아이콘 클래스는 `iconClass`와 내부 아이콘 상태 클래스를 분리 병합한다.
10. 클릭 시 비활성 상태가 아니면 `run` 이벤트를 발생시키고 원본 `MouseEvent`를 함께 전달한다.
11. `label`이 있으면 아이콘과 텍스트를 함께 렌더링하고, 없으면 아이콘 전용 버튼으로 렌더링한다.
12. 아이콘 전용 버튼은 `ariaLabel`을 버튼 접근성 이름으로 사용한다.

## 대안 검토
### 옵션 A
- 설명: `src/app/assets/styles/utility/button.css`에 모드별 유틸리티를 추가하고 컴포넌트는 그 클래스만 조합
- 장점: 여러 컴포넌트에서 동일 유틸리티 재사용 가능
- 단점: 지금 단계에서는 공통 버튼 계약이 아직 좁아 전역 유틸리티를 먼저 굳히면 오히려 변경 비용이 커질 수 있음
- 미채택 이유: 마스터 의견대로 현시점에는 컴포넌트 내부 정의가 더 단순하고 안전함

### 옵션 B
- 설명: `color`, `size`, `mode`를 모두 props로 열어 사실상 디자인 시스템 버튼으로 확장
- 장점: 호출부 제어 범위가 넓음
- 단점: 이번 요청의 "기본적인 기능만"이라는 범위를 넘고 컴포넌트 책임이 과해짐
- 미채택 이유: 아직 디자인 시스템 기준이 확정되지 않아 조기 일반화 위험이 큼

### 최종 선택
- 선택 이유: `Button.vue` 내부에 최소 베이스 클래스를 두고, `mode`와 상태만 공통화하면 기능은 채우면서도 디자인 자유도를 유지할 수 있음
- 기대 효과: 호출부는 `buttonClass`와 `iconClass`만으로 스타일을 쉽게 덧씌우고, 공통 버튼은 접근성/비활성화/이벤트 계약을 안정적으로 제공함

## 제약 및 참고 사항
- Nuxt 오토임포트 규칙을 유지해야 하므로 전역 선언 대상은 직접 import하지 않는다.
- 프로젝트 기본 Tailwind 클래스는 제한될 수 있어, 내부 클래스 정의는 `src/app/assets/styles/tailwind.css`에서 확인된 범위 안에서만 사용한다.
- 설계 범위는 공통 버튼의 기능 보강까지만 한정하며, 로딩 스피너나 사이즈 시스템 같은 확장 디자인 기능은 포함하지 않는다.
