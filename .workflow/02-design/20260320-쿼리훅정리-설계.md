# 20260320-쿼리훅정리-설계

## 1. 참조 문서
- `.workflow/01-analysis/20260320-쿼리훅정리-분석.md`

## 2. 목표 구조
- `useGet`은 `{ api, enabled, fetcher, key, query, onSuccess, onError, staleTime, gcTime }` 객체 하나를 받습니다.
- `usePost`, `usePut`, `usePatch`, `useDelete`는 `{ api, enabled, fetcher, key, onSuccess, onError }` 객체 하나를 받습니다.
- 모든 훅은 `response`와 `execute`를 공통 반환값으로 제공합니다.
  - `useGet.execute`는 `refetch`
  - mutation 훅의 `execute`는 `mutateAsync`

## 3. 공통 처리 설계
- `useApiRequest.ts`에는 다음 두 함수만 남깁니다.
  - 성공 응답 후 공통 에러 여부와 콜백을 정리하는 `handleApiResponse`
  - Axios 에러에서 `BaseResponse`를 추출하여 공통 처리하는 `handleApiRequestError`
- `api-error-handler.ts`는 `checkAndHandleApiError`만 외부 API로 유지하고, 내부에서만 사용할 리다이렉트/로깅 helper를 둡니다.

## 4. 대안 검토
- 대안 1: `createApiMutation`을 유지한 채 옵션만 객체화
  - 미채택 사유: 마스터 요구사항인 “각 파일에서 직접 `useMutation` 호출”과 맞지 않습니다.
- 대안 2: 에러 처리까지 각 훅 파일로 완전히 분산
  - 미채택 사유: 인증 만료 리다이렉트와 공통 에러 콜백 호출이 다시 중복됩니다.

## 5. 예외 및 검증 포인트
- `enabled`가 `false`인 mutation은 실행을 막아야 합니다.
- `fetcher`를 넘기지 않는 경우 기본 `apiClient` 요청으로 동작해야 합니다.
- 네트워크 에러와 `BaseResponse.error` 기반 에러를 모두 기존처럼 처리해야 합니다.
