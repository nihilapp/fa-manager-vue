# 설계 (Design)

## 요약
- 변경 대상: `BaseApiResponse<T>` 인터페이스와 직전 워크플로 문서
- 설계 방향: 공용 응답 인터페이스는 `data: TData`만 보장하고, 성공 시 nullable은 제네릭으로 표현
- 주요 결정 사항: 에러 응답은 계속 `BaseApiResponse<null>` 별칭을 유지한다

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/server/types/response.types.ts`
  - `data` 필드를 `TData`로 수정한다.
- `.workflow/05-verification/2026-03-29-응답타입명정리-검증.md`
  - 직전 작업 검증 문구 중 `T | null` 전제를 암묵적으로 깔 수 있는 표현을 보정한다.
- `.workflow/06-reports/2026-03-29-응답타입명정리-보고.md`
  - 공용 응답 계약 설명을 최신 계약으로 맞춘다.

- 없음

### API / 인터페이스 변경
- JSON 구조 변화 없음
- TypeScript 인터페이스 계약만 더 엄밀하게 조정

## 로직 흐름
1. `BaseApiResponse<T>`의 `data`를 `T`로 변경한다.
2. 에러 응답은 기존대로 `BaseApiResponse<null>`를 사용한다.
3. `tsc --noEmit`로 이번 조정이 기존 사용처를 깨지 않는지 확인한다.
4. 직전 작업 문서도 최신 계약에 맞게 보정한다.

## 대안 검토
### 옵션 A
- 설명: `data: T | null` 유지
- 장점: 에러 응답을 인터페이스 하나로 쉽게 수용
- 단점: 성공 응답의 타입 정확도가 떨어진다
- 미채택 이유: 마스터 요구와 반대다

### 최종 선택
- 선택 이유: nullable 여부는 인터페이스가 아니라 제네릭 인자로 표현하는 것이 더 정확하다
- 기대 효과: 성공 응답 타입이 불필요하게 넓어지지 않는다

## 제약 및 참고 사항
- 런타임 변경이 아니라 타입 계약 수정이라 검증은 타입체크 중심으로 수행한다.
