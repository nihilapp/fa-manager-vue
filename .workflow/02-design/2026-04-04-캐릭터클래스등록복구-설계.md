# 설계 (Design)

## 요약
- 변경 대상: 캐릭터 클래스 CRUD API, 클래스 요청 프론트 컴포저블, 클래스 payload 필터
- 설계 방향: 서버 엔드포인트를 실제 CRUD 로직으로 복구하고 프론트는 서브클래스 없이도 전송
- 주요 결정 사항: API 응답은 갱신된 `CharacterOutDto`를 반환해 기존 상태 갱신 흐름을 유지

## 아키텍처 변화
### 신규 구성 요소
- 각 클래스 API 파일 내부의 `loadCharacterDetail` 헬퍼

### 수정 구성 요소
- `src/server/api/characters/[id]/classes/index.post.ts`
- `src/server/api/characters/[id]/classes/[className].put.ts`
- `src/server/api/characters/[id]/classes/[className].delete.ts`
- `src/app/stores/character.store.ts`
- `src/app/composables/characters/useCreateCharacterClass.ts`
- `src/app/composables/characters/useUpdateCharacterClass.ts`
- `src/app/composables/characters/useDeleteCharacterClass.ts`

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 클래스 CRUD 응답 타입을 `CharacterOutDto` 기준으로 정합화

## 로직 흐름
1. 프론트는 클래스명만 있으면 클래스 payload를 생성한다.
2. POST API는 캐릭터 권한과 기존 클래스 중복을 확인한 뒤 `character_classes`에 insert 한다.
3. PUT API는 대상 클래스 존재 여부를 확인한 뒤 `subClassName`, `level`을 갱신한다.
4. DELETE API는 대상 클래스 존재 여부를 확인한 뒤 row를 삭제한다.
5. 각 API는 갱신된 캐릭터 상세를 다시 계산해 반환한다.

## 대안 검토
### 옵션 A
- 설명: 프론트에서만 전송 로직을 고치고 서버는 나중에 수정
- 장점: 수정 범위가 작다.
- 단점: 서버가 실제로 저장을 안 하므로 장애가 해결되지 않는다.
- 미채택 이유: 주원인이 서버라서 불충분하다.

### 최종 선택
- 선택 이유: 실제 저장 책임은 서버에 있으므로 엔드포인트 복구가 우선이며, 프론트 보조 이슈도 동시에 제거해야 한다.
- 기대 효과: 생성/수정/삭제 전체 경로 정상화
