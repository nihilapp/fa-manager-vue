# 설계 (Design)

## 요약
- 변경 대상: `src/app/layouts/default.vue`, `src/app/components/layout/common-layout/CommonContent.vue`
- 설계 방향: 기본 레이아웃은 뷰포트 높이 안에서 헤더/메인/푸터를 세로 flex로 고정하고, `main`만 남은 높이를 점유하면서 독립 스크롤을 담당하도록 구조화
- 주요 결정 사항:
  - 전역 `#__nuxt`의 `overflow: hidden`은 유지
  - `default.vue` 최상위 래퍼에 `min-h-0`을 포함해 자식 축소 허용
  - `CommonContent.vue`의 `<main>`에 `flex-1 min-h-0 overflow-y-auto`를 부여해 실제 스크롤 컨테이너를 명시

## 로직 흐름
1. `body div#__nuxt`가 `100dvh` 높이와 `overflow: hidden`으로 전체 앱 영역을 제한한다.
2. `default.vue`의 최상위 컨테이너가 해당 높이를 그대로 이어받아 `header`, `main`, `footer`를 세로 flex로 배치한다.
3. `header`와 `footer`는 콘텐츠 높이만큼만 차지하고 축소되지 않는다.
4. `CommonContent.vue`의 `main`은 `flex-1`로 남은 높이를 모두 차지하고, `min-h-0`으로 flex 문맥 내 축소를 허용한다.
5. `main` 내부 콘텐츠가 높이를 초과하면 `overflow-y-auto`에 의해 `main` 내부에서만 스크롤이 발생한다.

## 대안 검토
### 옵션 A
- 설명: `default.vue`는 그대로 두고 각 페이지 루트에 개별적으로 `overflow-y-auto`와 높이 클래스를 부여
- 장점:
  - 특정 페이지만 선택적으로 제어 가능
- 단점:
  - 기본 레이아웃을 사용하는 모든 페이지에서 중복 수정이 필요함
  - 페이지별 구현 편차가 생겨 스크롤 정책이 쉽게 깨짐
- 미채택 이유:
  - 요구사항이 기본 레이아웃 단위의 공통 동작 변경이므로 페이지별 대응은 유지보수 비용이 크다

### 옵션 B
- 설명: 전역 `body` 또는 `#__nuxt`의 `overflow`를 다시 열어 문서 스크롤로 되돌림
- 장점:
  - 즉시 스크롤은 생길 수 있음
- 단점:
  - 헤더/푸터 고정 요구와 충돌함
  - 현재 프로젝트의 전역 레이아웃 제약(`alt-body.css`)을 깨뜨릴 수 있음
- 미채택 이유:
  - 문제 원인은 전역 스크롤 차단이 아니라 내부 스크롤 컨테이너 부재이므로, 전역 정책을 바꾸는 것은 과도하다

### 최종 선택
- 선택 이유:
  - 수정 범위가 작고 명확하며, 기본 레이아웃을 사용하는 모든 화면에 일관된 스크롤 정책을 적용할 수 있다
  - 전역 컨테이너 정책을 유지하면서 `main`의 책임만 분명히 하므로 부작용 가능성이 낮다
- 기대 효과:
  - 헤더와 푸터 위치가 유지되고 긴 콘텐츠는 `main` 안에서만 스크롤된다
  - 이후 페이지 단위 화면도 별도 스크롤 처리 없이 공통 레이아웃 규약을 그대로 활용할 수 있다

## 제약 및 참고 사항
- `default` 레이아웃을 사용하는 페이지 중 부모 문서 스크롤을 전제로 한 높이 계산이 있다면 렌더링 확인이 필요하다.
- Tailwind 기본 클래스 가용성 이슈를 고려해 이미 프로젝트에서 사용 중인 범위의 클래스 조합(`flex`, `flex-col`, `flex-1`, `h-full`, `min-h-0`, `overflow-y-auto`)만 사용한다.
