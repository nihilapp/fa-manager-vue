# 설계 (Design)

## 요약
- 변경 대상: 도메인 컴포저블 4종과 응답 추출 보조 로직
- 설계 방향: API 엔드포인트별 실제 응답 타입을 컴포저블 제네릭/콜백 타입에 직접 반영하고, 공용 유니온 응답 타입 제거 후 `players` 패턴처럼 파일당 하나 구조로 재배치
- 주요 결정 사항:
  - 서버 API는 변경하지 않음
  - 도메인별 공용 `XxxResponse = A | B | C` 유니온 제거
  - 훅 단위로 `ListData<T>` 또는 `T`를 명시
  - `sessions/logs`는 현재 서버 계약인 `ListData<SessionOutDto>`로 고정

### API / 인터페이스 변경
- 외부 API 경로 및 서버 응답 구조는 유지
- 프론트 컴포저블의 `callback` 시그니처가 더 구체적인 정확 타입으로 변경됨
- 내부 동기화 함수는 "정확 타입"을 받도록 분리

## 로직 흐름
1. 각 컴포저블 훅이 호출하는 API 경로를 기준으로 실제 서버 응답 래퍼를 확정한다.
2. 훅별 `useGet`/`usePost`/`usePut`/`useDelete` 제네릭을 정확 타입으로 교체한다.
3. 기존 묶음 파일을 삭제하고 파일당 하나의 컴포저블로 분해한다.
4. 기존 공용 유니온 기반 `sync*Response`를 없애거나 파일 내부의 정확 타입 처리로 대체한다.
5. `sessions/logs`는 `ListData<SessionOutDto>`에서 `session.logs`를 추출하는 현재 동작만 남기고, `SessionLogOutDto` 직접 응답 분기 제거를 검토한다.
6. 타입 체크 또는 최소한 정적 검색으로 유니온 잔존 여부와 호출부 호환성을 확인한다.

## 대안 검토
### 옵션 A
- 설명: 서버 API를 모두 `BaseApiResponse.data(T)` 또는 `BaseApiResponse.page(ListData<T>)` 한 형태로 통일한 뒤 컴포저블을 단순화
- 장점: 프론트 타입이 더 일관되고 도메인별 공용 옵션 재사용이 쉬움
- 단점: 서버 엔드포인트 다수 수정이 필요하고, 이번 요청 범위를 넘어서며 런타임 영향이 커짐
- 미채택 이유: 현재 요청은 "컴포저블과 API를 맞춘 타입 정비"가 중심이며, 서버 계약 변경은 별도 작업으로 분리하는 편이 안전함

### 옵션 B
- 설명: 기존 유니온 타입을 유지하되 주석만 보강
- 장점: 수정 범위가 작음
- 단점: 타입 정확도 문제가 해결되지 않고, 엔드포인트별 계약이 계속 숨겨짐
- 미채택 이유: 마스터 요청인 "유니온 없이 있는 그대로 타입 사용"을 충족하지 못함

### 최종 선택
- 선택 이유: 서버 계약은 유지하면서 프론트 타입만 실제 응답에 맞게 좁히면 리스크 대비 효과가 가장 큼
- 기대 효과: 도메인 컴포저블의 응답 타입이 명확해지고, 불필요한 타입 가드와 광범위 유니온 의존이 줄어듦

## 제약 및 참고 사항
- `server/api` 주석은 유지해야 하므로 서버 변경이 필요한 경우 별도 작업으로 분리
- Nuxt 오토임포트 규칙을 지키며 직접 import 추가 없이 수정
- `sessions/logs` 서버 응답 구조는 설계상 어색하지만 현재는 타입 정합성만 우선 반영
