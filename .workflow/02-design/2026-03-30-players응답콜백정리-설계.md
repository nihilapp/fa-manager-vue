# 설계 (Design)

## 요약
- 변경 대상: 공통 API 응답 처리 유틸리티와 플레이어 전용 커스텀 훅
- 설계 방향: 성공 콜백 계약을 `TData`에서 `BaseApiResponse<TData>`로 승격하고, 각 훅 내부에서 `response.data`를 분해해 기존 스토어 갱신을 유지
- 주요 결정 사항: 플레이어 전용 훅만 우회 수정하지 않고 공통 래퍼 계약을 일관되게 정비

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 변경 전: `onSuccess(data: TData)` / `callback(data: PlayerOutDto)` 또는 `callback(list, totalCnt)`
- 변경 후: `onSuccess(response: BaseApiResponse<TData>)` / `callback(response: BaseApiResponse<...>)`

## 로직 흐름
1. 공통 요청 래퍼가 서버 응답을 `BaseApiResponse<TData>`로 수신한다.
2. `handleApiResponse`가 오류 여부를 검사한다.
3. 성공이면 `onSuccess`에 `response` 전체를 전달한다.
4. 플레이어 전용 훅은 `response.data`로 스토어를 갱신한다.
5. 외부 `callback`에는 동일한 `response` 전체를 그대로 전달한다.
6. 호출부는 필요에 따라 `response.data`, `response.code`, `response.message`를 선택적으로 사용한다.

## 대안 검토
### 옵션 A
- 설명: 플레이어 전용 훅 내부에서만 원본 응답을 별도로 보관하고 외부 콜백에 재구성해서 전달
- 장점: 공통 쿼리 래퍼 변경 범위가 작다
- 단점: 공통 계약과 실제 전달 데이터가 분리되어 유지보수성이 나빠진다
- 미채택 이유: 성공 콜백의 기준이 도메인별로 달라져 이후 다른 도메인에도 같은 문제가 반복된다

### 최종 선택
- 선택 이유: 공통 계약을 먼저 바로잡아야 플레이어 훅과 향후 다른 도메인의 동작이 일관된다
- 기대 효과: 성공 응답 메타데이터 활용성이 높아지고, 커스텀 훅의 콜백 시그니처가 단일 패턴으로 정리된다

## 제약 및 참고 사항
- 기존 워크트리 변경사항은 유지하고 이번 작업 관련 파일만 건드린다.
- 호출부 변경은 현재 필요하지 않지만, 이후 콜백을 추가하는 코드에서는 `response.data` 접근이 필요하다.
