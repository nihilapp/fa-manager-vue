## 요약

- 변경 대상: `src/server/utils/auth.ts`, `src/server/utils/access.ts`, `src/server/middleware/access.ts`, `src/server/api` 내 CUD 라우트의 권한 호출 방식
- 설계 방향: 개발환경 우선 통과, 운영환경 디스코드 식별, 어드민 전권, 일반 사용자 수정/삭제 소유권 검사를 하나의 공통 판정 함수로 강제
- 주요 결정 사항:
  - 인증 정보 해석과 권한 판정을 분리한다.
  - `GET`은 공통 권한 차단 대상에서 제외하고, `POST`/`PUT`/`DELETE`만 공통 판정 대상에 포함한다.
  - 각 라우트가 `error`, `isAdmin`, `hasPermission`를 직접 조합하지 않고, 공통 권한 결과만 사용하도록 수렴시킨다.

## 아키텍처 변화

### 신규 구성 요소

- `resolveAuthContext(event)`
  - 책임: 요청의 실행 환경, 디스코드 식별자, 사용자 조회 결과를 표준 컨텍스트로 반환
  - 반환값 초안:
    - `mode`: `'development' | 'production'`
    - `discordId`: `string | null`
    - `user`: `PlayerOutDto | null`
    - `isAdmin`: `boolean`
    - `isDevelopment`: `boolean`
  - 특징: 더 이상 "개발 우회인데 error도 있음" 같은 모순 상태를 만들지 않는다.

- `authorizeWriteAccess(event, options?)`
  - 책임: 수정/삭제/생성 요청에 대한 단일 권한 판정
  - 입력 초안:
    - `ownerId?: number | null`
    - `adminOnly?: boolean`
    - `requireUser?: boolean`
  - 반환값 초안:
    - `allowed: boolean`
    - `code`
    - `message`
    - `user`
    - `isAdmin`
    - `isDevelopment`
  - 판정 순서:
    1. 개발환경이면 즉시 허용
    2. 운영환경에서 `discordId` 없으면 거부
    3. 운영환경에서 사용자 조회 실패면 거부
    4. 어드민이면 즉시 허용
    5. `adminOnly`면 거부
    6. 일반 사용자는 `ownerId === user.id`일 때만 허용

- `authorizeReadAccess(event, options?)`
  - 책임: 페이지 진입과 특수 조회 API에서 필요한 최소 인증 판정
  - 기본 정책: 마스터 요구사항에 따라 `GET`은 권한 차단 대상이 아니므로, 읽기 요청은 개발환경/운영환경 식별만 확인하고 소유권 검사는 하지 않는다.
  - 적용 범위:
    - 페이지 접근 미들웨어
    - `/players/me` 같은 부트스트랩 API
    - 운영환경에서 식별자 필수인 엔드포인트

### 수정 구성 요소

- `src/server/utils/auth.ts`
  - 현재 `authHelper()`의 `error`, `isDevelopmentBypass`, `hasPermission` 혼합 반환 계약을 제거한다.
  - 인증 컨텍스트 해석 전용 함수와 권한 판정 전용 함수로 역할을 분리한다.
  - `discord_id` 쿠키가 존재해도 개발환경이면 개발환경 우선으로 처리되도록 식별 우선순위를 재정의한다.

- `src/server/utils/access.ts`
  - 페이지 접근용 함수는 `authorizeReadAccess()`만 사용한다.
  - `/block` 리다이렉트 기준은 운영환경에서 식별자 부족 또는 사용자 미존재일 때만 유지한다.

- `src/server/middleware/access.ts`
  - 비 API 페이지 요청은 기존처럼 공통 접근 체크를 거치되, 판단 근거는 새 읽기 권한 함수 하나로 통일한다.

- `src/server/api/*`
  - `GET`: 기존 데이터 조회 로직을 유지하되, 읽기 인증이 필요한 경우 공통 읽기 권한 함수만 사용한다.
  - `POST`/`PUT`/`DELETE`: 각 라우트에서 직접 `if (error) return error;`, `if (!isAdmin)`, `if (!hasPermission(...))`를 조합하지 않고 `authorizeWriteAccess()`로 통일한다.
  - 리소스 소유권이 필요한 라우트는 DB 조회 뒤 `ownerId`만 공통 함수에 넘긴다.

### DB / 스키마 변경

- 없음

### API / 인터페이스 변경

- 서버 내부 인터페이스만 변경한다.
- 외부 API URL, DTO, 응답 포맷은 유지한다.
- 단, 개발환경에서 권한 거부 대신 성공 또는 후속 비즈니스 검증 결과가 반환될 수 있으므로 실행 결과는 달라진다.

## 로직 흐름

1. 모든 요청은 먼저 `resolveAuthContext(event)`로 환경과 사용자 컨텍스트를 해석한다.
2. 개발환경이면 다른 조건을 보지 않고 `isDevelopment = true`, `allowed = true`로 즉시 통과한다.
3. 운영환경이면 디스코드 아이디 존재 여부를 확인한다.
4. 디스코드 아이디가 없으면 읽기 페이지와 `/players/me`는 접근 차단, 쓰기 요청은 `UNAUTHORIZED`를 반환한다.
5. 디스코드 아이디가 있으면 사용자 조회를 수행한다.
6. 사용자가 어드민이면 쓰기 요청도 즉시 통과한다.
7. 사용자가 일반 사용자이면 `GET`은 통과시키고, `POST`/`PUT`/`DELETE`는 리소스의 `ownerId`와 `user.id`를 비교한다.
8. 일치하면 허용하고, 불일치하면 `FORBIDDEN`을 반환한다.
9. 각 라우트는 공통 판정 결과에서 `user`와 `allowed`만 사용하고, 자체적인 우선순위 해석 로직을 두지 않는다.

## 리소스별 적용 규칙

### 페이지 접근

- 기준 함수: `authorizeReadAccess()`
- 개발환경: 항상 통과
- 운영환경: 디스코드 식별이 없거나 사용자 조회 실패 시 `/block`

### `/players/me`

- 기준 함수: `authorizeReadAccess()`
- 개발환경: `user`가 없어도 성공 응답 허용
- 운영환경: 디스코드 식별과 사용자 조회가 모두 필요

### 관리자 전용 CUD

- 대상 예시: `log-histories` CUD
- 기준 함수: `authorizeWriteAccess({ adminOnly: true })`
- 개발환경: 통과
- 운영환경 어드민: 통과
- 운영환경 일반 사용자: 거부

### 소유권 기반 CUD

- 대상 예시: `sessions`, `currency-transactions`, `characters`, `players/me`
- 기준 함수: `authorizeWriteAccess({ ownerId })`
- 개발환경: 통과
- 운영환경 어드민: 통과
- 운영환경 일반 사용자: `ownerId === user.id`일 때만 통과

## 구현 규칙

- 개발환경 우선순위는 모든 권한 판단의 첫 줄에서 처리한다.
- `GET` 라우트에는 소유권 기반 권한 검사를 넣지 않는다.
- CUD 라우트는 반드시 "리소스 조회 -> ownerId 추출 -> 공통 권한 판정" 순서를 따른다.
- `user!.id` 직접 사용은 공통 권한 판정 후, `user` 존재가 보장된 분기에서만 허용한다.
- `authHelper()` 또는 후속 함수는 더 이상 `error`와 `isDevelopmentBypass`를 동시에 반환하지 않는다.

## 대안 검토

### 옵션 A: 기존 `authHelper()`만 수정하고 각 라우트는 유지

- 설명: `authHelper()`가 개발환경일 때 `error`를 없애거나 `isAdmin`을 강제로 `true`처럼 보이게 만들어 현재 라우트 코드를 최대한 건드리지 않는 방법
- 장점:
  - 초기 수정량이 적어 보인다
  - 빠르게 일부 라우트는 통과시킬 수 있다
- 단점:
  - `user!.id`를 직접 쓰는 라우트는 그대로 깨진다
  - `isAdmin` 직접 검사 라우트와 `hasPermission()` 라우트가 서로 다른 계약에 계속 의존한다
  - 다시 같은 문제가 생겨도 원인을 추적하기 어렵다
- 미채택 이유:
  - 마스터 요구사항인 "하나의 부분만 수정하면 동작 제어 가능"을 만족하지 못한다

### 옵션 B: Nitro 전역 미들웨어에서 HTTP 메서드만 보고 일괄 우회

- 설명: `server/middleware/access.ts`에서 `POST`/`PUT`/`DELETE`까지 강제로 허용하거나 거부하는 방식
- 장점:
  - 진입점은 하나다
- 단점:
  - `/api` 경로는 현재 미들웨어에서 우회되고 있고, 라우트별 리소스 소유권 정보가 없다
  - 어떤 요청이 누구 소유 리소스를 수정하는지 미들웨어만으로 판정할 수 없다
- 미채택 이유:
  - 소유권 판단이 필요한 요구사항 4번을 만족할 수 없다

### 최종 선택

- 선택 이유:
  - 인증 컨텍스트 해석과 쓰기 권한 판정을 분리하면 우선순위를 코드 한 곳에 명확히 고정할 수 있다
  - 각 라우트는 리소스 소유자 정보만 전달하고, 판정 기준 자체는 공통 함수 하나가 맡는다
  - 개발환경 우선 통과와 운영환경 소유권 판단을 동시에 만족할 수 있다
- 기대 효과:
  - 마스터가 원하는 정책 변경 시 공통 권한 함수만 수정하면 전체 동작을 제어할 수 있다
  - 라우트별 예외 처리 중복이 줄고 `/block` 및 CUD 권한 실패 원인이 예측 가능해진다

## 제약 및 참고 사항

- 운영환경에서 `GET`을 전부 무제한 허용하면 데이터 노출 범위가 넓어질 수 있으므로, 이는 현재 마스터 정책을 그대로 반영한 설계임을 명시한다.
- 개발환경에서 `user`가 없는 상태로 CUD를 허용하면 작성자 메타(`creatorId`, `updaterId`, `deleterId`) 처리 방식을 별도로 정해야 한다.
- 태스크 단계에서는 라우트를 `adminOnly`, `ownerBased`, `readOnly`, `selfWrite` 유형으로 분류해 순차 전환해야 한다.
