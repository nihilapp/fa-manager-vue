# 설계 (Design)

## 요약
- 변경 대상: `authHelper`, 페이지 접근 체크 유틸, 서버 미들웨어, `users/check`, 예외 CUD API
- 설계 방향: 인증/인가 규칙을 `authHelper` 중심의 단일 인증 컨텍스트로 통합하고, 페이지 접근 가드와 CUD API가 같은 기준을 재사용하도록 정리
- 주요 결정 사항:
  - 운영 환경에서는 `X-Discord-ID` 필수, 유저 존재 필수, 일반 유저는 본인 리소스만 허용, 관리자 계열은 우회 허용
  - 개발 환경에서는 인증 및 소유권 제한을 완화하는 “개발 우회 모드”를 `authHelper` 안에서 처리
  - 페이지 접근 판단도 `authHelper`와 동일한 규칙 집합을 재사용하되, 페이지용 경량 래퍼를 둔다
  - 예외 엔드포인트의 중복 헤더 체크를 제거하고 공통 헬퍼만 신뢰한다

## 아키텍처 변화
### 신규 구성 요소
- `AuthContext` 성격의 표준 반환 구조
  - 별도 파일 분리 여부는 구현자가 판단하되, 최소한 `authHelper` 반환값을 다음 의미로 고정한다
  - `user`: 현재 인증된 실제 유저 또는 개발 우회 모드에서의 대체 사용자/없음
  - `isAdmin`: 관리자 계열 여부
  - `isDevelopmentBypass`: 개발 환경 완화가 적용되었는지 여부
  - `hasPermission(resourceUserId?)`: 현재 요청자가 해당 리소스 소유권 제한을 통과하는지 판정
  - `requireUser()`: 실제 유저가 반드시 필요한 시점에 명시적으로 검증하는 보조 함수 후보

### 수정 구성 요소
- `src/server/utils/auth.ts`
  - 전체 정책의 중심
  - 운영 환경과 개발 환경을 분기
  - 운영 환경:
    - `X-Discord-ID` 없으면 `UNAUTHORIZED`
    - 해당 유저 없으면 `NOT_FOUND`
    - `ROLE_ADMIN`, `ROLE_SUPER_ADMIN`이면 관리자 권한 부여
    - 일반 유저는 `hasPermission(resourceUserId === user.id)` 조건만 통과
  - 개발 환경:
    - 헤더 없어도 통과
    - 소유권 제한도 우회
    - 단, 실제 쓰기 작업에서 `creatorId`/`updaterId`가 필요한 경우를 위해 사용자 식별 전략을 함께 제공해야 함
- `src/server/utils/access.ts`
  - 현재의 역전된 DEV 판정을 제거
  - `authHelper` 또는 그 하위 경량 유틸을 호출해 “페이지 진입 가능 여부”만 판단
  - 페이지는 사용자 객체 전체보다 `error 여부`가 중요하므로 래퍼 함수로 유지 가능
- `src/server/middleware/access.ts`
  - 공통 접근 유틸 결과가 에러일 때만 `/block` 리다이렉트
  - `/api`, Nuxt 자산, `/block`, 정적 자산 예외 유지
- `src/server/api/users/check.get.ts`
  - 페이지와 동일 규칙을 반환하는 진단용/프론트 확인용 엔드포인트로 유지
- 예외 API 파일
  - `src/server/api/sessions/[id].delete.ts`처럼 직접 `getDiscordId`를 다시 읽는 중복 코드를 제거
  - `authHelper` 결과만으로 판단하게 통일

### DB / 스키마 변경
- 없음
- 단, 개발 환경 완화 시 실제 DB 유저 없이도 쓰기 요청을 허용할지 결정되면, `creatorId`/`updaterId` 기본값 처리 규칙을 점검해야 한다

### API / 인터페이스 변경
- 내부 인터페이스 변경이 핵심이며 외부 API 시그니처는 최대한 유지한다
- `authHelper` 내부 계약
  - 운영:
    - `error`가 없으면 `user`는 반드시 존재
    - `hasPermission`은 본인 또는 관리자만 통과
  - 개발:
    - `error` 없이 통과 가능
    - `hasPermission`은 항상 `true`
    - 구현자가 실제 `user.id`가 필요한 엔드포인트에서 안전하게 사용할 수 있도록 보조 규칙 필요
- `checkCommunityAccess` 내부 계약
  - `authHelper`의 운영/개발 규칙을 재사용하되, 페이지 접근에 필요한 최소 정보만 응답

## 로직 흐름
1. 요청이 페이지 요청인지 API 요청인지에 관계없이, 인증/인가가 필요한 곳은 모두 `authHelper(event)`를 기준 진입점으로 사용한다.
2. `authHelper`는 먼저 `import.meta.env.DEV`를 확인한다.
3. 개발 환경이면 “개발 우회 모드”를 활성화한다.
4. 개발 우회 모드에서는 `X-Discord-ID` 유무와 소유권 제한을 강제하지 않는다.
5. 운영 환경이면 `getDiscordId(event)`로 `X-Discord-ID`를 읽는다.
6. 헤더가 없으면 `UNAUTHORIZED`, 헤더는 있으나 유저가 없으면 `NOT_FOUND`를 반환한다.
7. 유저가 있으면 role을 기준으로 `isAdmin`을 계산한다.
8. `hasPermission(resourceUserId)`는 운영 환경에서 `isAdmin === true`면 무조건 통과, 아니면 `resourceUserId === user.id`일 때만 통과한다.
9. 페이지 접근 유틸은 `authHelper` 결과에서 `error`만 확인하고 실패 시 `/block`으로 보낸다.
10. CUD API는 `authHelper` 결과에서 `user`, `hasPermission`, `isAdmin`을 사용해 기존 도메인 규칙을 계속 적용한다.

## 세부 설계 포인트
### 1. 개발 환경 우회 전략
- 권장 방향:
  - 개발 환경에서는 `hasPermission`을 항상 `true`로 둔다.
  - 헤더가 없어도 `error: null`을 반환한다.
  - 다만 `user!.id`를 바로 쓰는 기존 코드가 많으므로, 개발 우회 시 사용할 대체 사용자 전략이 필요하다.
- 대체 사용자 전략 옵션:
  - 옵션 A: `user`를 `null`로 두고, 각 API가 `user!.id`를 쓰는 지점을 개별 보완
  - 옵션 B: DB의 첫 관리자/첫 유저를 개발 기본 사용자로 자동 선택
  - 옵션 C: 개발 전용 더미 식별자(`0` 또는 `-1`)를 공통 메타 기본값으로 사용
- 권장 선택:
  - 분석 기준으로는 옵션 B가 가장 덜 침습적이다.
  - 이유: 기존 API 다수가 `creatorId`, `updaterId`, `user!.id`를 직접 사용하므로 `null user` 방식은 수정 범위가 커진다.
  - 구현자는 개발 환경에서 헤더가 없을 때 “첫 관리자 > 첫 일반 유저 > 없음” 순으로 fallback 사용자를 찾는 구조를 우선 검토한다.

### 2. 페이지 접근과 API 인증의 통합
- `checkCommunityAccess(event)`는 독립 규칙을 가지면 안 된다.
- 내부적으로는 `authHelper(event)` 또는 그 하위 함수(`resolveRequestActor` 같은 이름 후보)를 호출해야 한다.
- 페이지는 “유저가 누구냐”보다 “이 요청을 보여 줄 수 있느냐”가 중요하므로, 응답은 `BaseResponse.data/error` 형식만 유지하면 충분하다.

### 3. 예외 API 정리
- `sessions/[id].delete.ts`처럼 `authHelper` 전에 `getDiscordId`를 직접 검사하는 코드는 제거한다.
- 이후 원칙:
  - 인증/인가 관련 분기문은 `authHelper` 밖에서 직접 구현하지 않는다.
  - 도메인 규칙만 엔드포인트 내부에 남긴다.
- 도메인 규칙 예시:
  - 캠페인 수정: 캠페인 소유자 또는 관리자
  - 세션 수정: 세션이 속한 캠페인의 소유자 또는 관리자
  - 캠페인 멤버 가입/이탈: 본인 또는 관리자

### 4. 관리자 우회 규칙
- 운영에서는 `ROLE_ADMIN`, `ROLE_SUPER_ADMIN`만 우회 가능
- 코드상 특정 디스코드 ID를 하드코딩하지 않는다
- “어드민은 나 하나”는 데이터 운영 정책으로 본다

## 대안 검토
### 옵션 A: 각 엔드포인트에서 `import.meta.env.DEV` 분기 직접 추가
- 설명: 모든 CUD API가 `authHelper` 밖에서 개발 예외를 개별 처리
- 장점:
  - 빠르게 임시 대응 가능
- 단점:
  - 예외 로직이 수십 개 파일에 퍼짐
  - 운영/개발 기준이 다시 어긋날 가능성이 큼
  - 페이지 접근 가드와도 규칙 공유가 안 됨
- 미채택 이유: 유지보수성이 너무 낮고, 마스터가 원하는 “딱 하나의 규칙”에 맞지 않는다.

### 옵션 B: `authHelper`와 `checkCommunityAccess`를 별도 규칙으로 유지
- 설명: API 인증은 `authHelper`, 페이지 접근은 별도 유틸이 각각 규칙을 가짐
- 장점:
  - 페이지와 API 요구사항을 분리하기 쉽다
- 단점:
  - 같은 서비스에서 접근 가능/불가능 기준이 서로 달라질 수 있다
  - 버그 발생 시 어디를 기준으로 봐야 하는지 모호하다
- 미채택 이유: 현재도 이미 그 문제를 겪고 있고, 이번 작업의 목적 자체가 통합이다.

### 옵션 C: `authHelper` 중심 단일 인증 컨텍스트 + 페이지 래퍼
- 설명: 공통 규칙은 `authHelper`에 두고, 페이지 접근은 이 결과를 감싼 얇은 유틸로 처리
- 장점:
  - 운영/개발 규칙이 한 곳에 모인다
  - CUD API와 페이지 접근이 같은 기준을 쓴다
  - 예외 엔드포인트 정리가 쉬워진다
- 단점:
  - 개발 환경 fallback user 전략을 별도로 정해야 한다
  - 기존 `user!.id` 가정 코드가 많아 초기 정리가 필요하다

### 최종 선택
- 선택 이유: 이미 다수 API가 `authHelper`를 사용하고 있으므로, 최소 변경으로 전체 규칙을 통일할 수 있는 중심축이기 때문이다.
- 기대 효과:
  - 마스터가 원하는 운영/개발 규칙을 한 군데에서 관리 가능
  - 페이지 접근과 CUD API 권한 체크가 같은 정책을 따른다
  - 이후 GET 보호 범위 확장이나 관리자 정책 변경 시 수정 지점이 명확해진다

## 구현자가 바로 이어서 해야 할 작업 제안
1. `authHelper` 반환 타입과 개발 우회 정책을 먼저 고정한다.
2. `checkCommunityAccess`를 `authHelper` 기반으로 다시 작성한다.
3. `sessions/[id].delete.ts` 등 중복 헤더 체크 API를 정리한다.
4. `user!.id`를 사용하는 CUD API 중 개발 우회 시 깨질 수 있는 지점을 점검한다.
5. 마지막에 운영/개발 각각의 테스트 시나리오를 문서화한다.

## 제약 및 참고 사항
- 개발 우회가 너무 강하면 실제 운영 권한 버그를 놓칠 수 있으므로, 테스트 문서에는 운영 기준 시나리오를 반드시 별도로 남겨야 한다.
- 운영 환경에서 페이지 요청에 `X-Discord-ID`를 어떻게 공급하는지는 상위 프록시/게이트웨이 설계에 달려 있다.
- 추후 “조회(GET)도 모두 보호”로 확장하려면 이번 설계 이후 공개 GET API 분류 작업이 별도 필요하다.
