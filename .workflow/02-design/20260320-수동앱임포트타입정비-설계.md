# 20260320-수동앱임포트타입정비-설계

## 참조 문서
- `.workflow/01-analysis/20260320-수동앱임포트타입정비-분석.md`

## 1. 설계 목표
- `manual-app-imports.d.ts`에서 전역 상수와 전역 타입 별칭이 실제 모듈 타입을 직접 참조하도록 재구성합니다.
- 선언 구조를 모듈별 별칭으로 단순화하여 추후 유지보수 시 어떤 심볼이 어느 패키지에서 오는지 즉시 파악 가능하게 합니다.

## 2. 변경 구조
- `AppImports = typeof import('#imports')`를 제거합니다.
- 아래 모듈 타입 별칭을 도입합니다.
  - `VueImports = typeof import('vue')`
  - `PiniaImports = typeof import('pinia')`
  - `NuxtAppImports = typeof import('nuxt/app')`
  - `PiniaNuxtImports = typeof import('@pinia/nuxt/dist/runtime/composables')`
  - `AppComponents = typeof import('#components')`
- 전역 상수 선언은 각 모듈 별칭의 실제 멤버를 직접 가리키도록 변경합니다.
- 전역 타입 별칭도 동일한 모듈 별칭을 재사용하도록 정리합니다.

## 3. 계약 및 흐름
- `vue` 계열 반응성 API는 `VueImports['ref']` 등으로 선언하여 함수 시그니처를 그대로 보존합니다.
- `pinia` 계열 스토어 API는 `PiniaImports['defineStore']`, `PiniaImports['storeToRefs']`로 선언합니다.
- `usePinia`는 `PiniaNuxtImports['usePinia']`를 사용하여 Nuxt 통합 시그니처를 유지합니다.
- Nuxt 앱 API는 `NuxtAppImports['useRoute']`, `NuxtAppImports['navigateTo']` 등으로 선언합니다.
- 전역 컴포넌트는 `AppComponents[...]`를 사용하여 기존 계약을 유지합니다.

## 4. 대안 검토
- 대안 1: `#imports` 기반 구조 유지
  - 미채택 이유: 현재 문제가 이 간접 경로에서 발생하고 있어 요구사항을 해소하지 못합니다.
- 대안 2: `.nuxt/types/imports.d.ts` 내부 `pnpm` 경로를 직접 복사
  - 미채택 이유: 버전 문자열이 경로에 포함되어 유지보수성이 낮고 생성 파일 구현 세부사항에 과도하게 결합됩니다.

## 5. 예외 및 검증 계획
- 선언 충돌 여부는 `nuxi typecheck`로 확인합니다.
- 만약 `nuxt/app` 공개 엔트리포인트에 없는 심볼이 있으면 해당 심볼만 제한적으로 별도 모듈 선언을 사용합니다.
