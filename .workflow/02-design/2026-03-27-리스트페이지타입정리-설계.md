# 설계 (Design)

## 요약
- 변경 대상: 서버 응답 타입, 리스트 직렬화 유틸, 프런트 테이블 타입 참조부
- 설계 방향: 응답 구조를 서버 기준으로 일원화하고, 비페이징 상태도 nullable 없이 표현
- 주요 결정 사항: 비페이징 기본값은 `0/false`, `ListPageData`는 `response.types.ts`에 정의

## 아키텍처 변화
### 신규 구성 요소
- `ListPageData<T>`를 `src/server/types/response.types.ts`에 추가

### 수정 구성 요소
- `ListDataType<T>`의 페이지 관련 nullable 제거
- `ListData` 클래스의 기본 직렬화 값 변경
- `DataTable`과 `Paginator`의 `null` 방어 코드 제거

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 목록 응답의 페이지 필드가 `null` 대신 `0/false`를 반환

## 로직 흐름
1. 서버 응답 타입에서 `ListPageData`를 정의하고 페이지 관련 필드를 non-nullable로 바꾼다.
2. `ListData` 클래스가 비페이징 상태에서 `0/false`를 직렬화하도록 수정한다.
3. 프런트 전역 타입 선언과 테이블 소비부가 새 타입을 직접 소비하도록 바꾼다.

## 대안 검토
### 옵션 A
- 설명: 프런트에서만 `null`을 계속 정규화
- 장점: 서버 변경이 적음
- 단점: 응답 구조 자체가 불편한 상태로 유지됨
- 미채택 이유: 마스터 요청의 핵심이 응답 타입 구조 자체 정리임

### 최종 선택
- 선택 이유: 서버가 일관된 값을 내려주면 프런트가 단순해진다.
- 기대 효과: 테이블 컴포넌트와 후속 UI 연동 코드가 더 단순해짐

## 제약 및 참고 사항
- `prevPage`, `nextPage`는 "없음" 상태를 `0`으로 표현하되, 실제 사용은 `hasPrev`, `hasNext`와 함께 본다.
