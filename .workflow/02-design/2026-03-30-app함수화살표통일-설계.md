# 설계 (Design)

## 요약
- 변경 대상: `src/app` 내 함수 선언식 사용 파일과 `player.store.ts`
- 설계 방향: 함수 선언식을 동일한 시그니처의 화살표 함수 상수로 치환하고, store 상태 초기화 액션을 명시적으로 추가
- 주요 결정 사항: exported function도 모두 `export const`로 통일, 내부 helper/액션도 `const` 화살표 함수로 통일, 상세 조회 에러 시 `clearPlayerInfo` 호출

## 아키텍처 변화
### 신규 구성 요소
- `player.store.ts`의 `clearPlayerInfo`: `playerInfo.value = null`을 담당하는 명시적 상태 초기화 액션

### 수정 구성 요소
- `src/app/stores/player.store.ts`
  - 액션 선언식을 화살표 함수로 변경
  - `clearPlayerInfo` 추가
  - 반환 객체에 새 액션 노출
- `src/app/composables/players/*.ts`
  - exported composable 선언식을 `export const` 화살표 함수로 변경
  - `useGetPlayerInfo.ts` 에러 처리에서 `clearPlayerInfo` 사용
- `src/app/composables/query/*.ts`
  - 내부 `execute`, `clear`, `refetch`를 화살표 함수로 변경
  - exported helper/composable도 동일 규칙 적용
- `src/app/composables/useSetMeta.ts`, `src/app/utils/*.ts`
  - local helper와 exported helper를 모두 화살표 함수로 통일

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 외부 API 계약 변경 없음
- store public interface에 `clearPlayerInfo(): void`가 추가됨

## 로직 흐름
1. `src/app`에서 함수 선언식이 남아 있는 파일을 검색해 변경 대상을 확정한다.
2. exported function은 `export const 함수명 = (...) => {}` 형태로, 내부 함수는 `const 함수명 = (...) => {}` 또는 `const 함수명 = async (...) => {}` 형태로 변환한다.
3. `player.store.ts`에 `clearPlayerInfo`를 추가하고 반환 객체에 포함한다.
4. `useGetPlayerInfo.ts`의 오류 처리에서 오타 호출을 제거하고 `clearPlayerInfo`를 호출해 상세 상태를 명시적으로 초기화한다.
5. 검색 재검증으로 `src/app` 범위에 함수 선언식이 남지 않았는지 확인한다.

## 대안 검토
### 옵션 A
- 설명: 스토어 액션만 화살표 함수로 바꾸고 나머지 app 함수는 유지
- 장점: 변경 범위가 작고 빠름
- 단점: 마스터가 명시한 "app 에서는 모든 함수를 화살표 함수로" 기준을 충족하지 못함
- 미채택 이유: 요구사항을 부분적으로만 만족하므로 제외

### 옵션 B
- 설명: 선언식 함수만 부분 치환하고 오류 경로 오타는 별도 작업으로 남김
- 장점: 요청 범위를 최소 해석할 수 있음
- 단점: `useGetPlayerInfo.ts`가 여전히 잘못된 호출을 유지해 상태 정리가 불완전함
- 미채택 이유: 새 `playerInfo` 초기화 액션 추가 요청과 자연스럽게 결합되는 수정이며, 분리할 이유가 없음

### 최종 선택
- 선택 이유: 함수 스타일 통일과 상태 초기화 보강을 동시에 반영하면 코드 일관성과 실제 동작 안정성을 함께 확보할 수 있음
- 기대 효과: `src/app` 함수 스타일이 단일 규칙으로 정리되고, 플레이어 상세 상태 초기화 흐름이 명확해짐

## 제약 및 참고 사항
- Nuxt 오토임포트 규칙을 유지하기 위해 import는 추가하지 않음
- 선언식 치환 과정에서 런타임 동작을 바꾸지 않도록 함수 시그니처와 반환 구조는 유지함
