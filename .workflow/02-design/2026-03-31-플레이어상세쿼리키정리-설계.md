# 설계 (Design)

## 요약

- 변경 대상: `useGetPlayerInfo`, `useGetPlayerSessionList`
- 설계 방향: API 경로 의미와 동일한 쿼리 키를 사용하고, 미완성 훅은 기존 공통 `useGet` 계약에 맞춰 완성
- 주요 결정 사항: 상세 조회는 `players.detail`, 내정보는 `players.me`로 분리 유지

## 아키텍처 변화

### 신규 구성 요소

- 없음

### 수정 구성 요소

- `src/app/composables/players/useGetPlayerInfo.ts`
  - `queryKeys.players.detail({ id })` 사용
- `src/app/composables/players/useGetPlayerSessionList.ts`
  - `query`, `callback`, `errorCallback` 옵션을 풀어 실제 `useGet` 호출 완성
  - `queryKeys.players.sessions({ playerId, ...query })` 사용

### DB / 스키마 변경

- 없음

### API / 인터페이스 변경

- 없음

## 로직 흐름

1. 플레이어 상세 조회는 `/players/:id` 요청과 동일한 `players.detail` 키로 캐시된다.
2. 내 정보 조회는 기존대로 `players.me` 키를 사용한다.
3. 플레이어 세션 조회 훅은 `playerId`와 쿼리 필터를 결합한 키로 요청한다.
4. 성공/실패 콜백은 기존 공통 `useGet` 계약을 따른다.

## 대안 검토

### 옵션 A

- 설명: `useGetPlayerInfo`만 수정하고 `useGetPlayerSessionList`는 미완성 상태로 둠
- 장점: 변경 범위가 더 작다
- 단점: 타입 검증이 계속 실패한다
- 미채택 이유: 남은 부분까지 진행하라는 요청과 맞지 않는다

### 최종 선택

- 선택 이유: 관련 플레이어 조회 흐름을 함께 정상화해야 검증까지 일관되게 끝난다
- 기대 효과: 플레이어 조회 계층의 키 의미와 타입 상태가 동시에 안정화된다

## 제약 및 참고 사항

- 현재 작업은 플레이어 조회 계층 내부 정리만 수행한다.
