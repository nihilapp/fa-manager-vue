# 설계 (Design)

## 요약
- 변경 대상: 서버 접근 체크 유틸, Nitro 서버 미들웨어, `users/check` 엔드포인트, `block` 페이지
- 설계 방향: 모든 페이지 요청이 서버에 도달하는 시점에 `X-Discord-ID` 헤더와 개발 환경 여부를 검사하고, 실패 시 `block`으로 리다이렉트
- 주요 결정 사항:
  - 접근 제어는 클라이언트 라우트 미들웨어가 아니라 Nitro 서버 미들웨어에서 수행
  - 디스코드 ID는 프론트 상태나 UI에 노출하지 않는다
  - `users/check`와 서버 미들웨어는 동일한 공통 검증 유틸을 재사용한다
  - `block` 라우트와 정적 자산 경로는 서버 미들웨어 예외 대상이다

## 아키텍처 변화
### 신규 구성 요소
- `src/server/utils/access.ts`
  - 요청 헤더 `X-Discord-ID`, 개발 환경 여부, 유저 존재 여부를 함께 검사하는 공통 접근 체크 유틸
  - `users/check` 엔드포인트와 서버 미들웨어가 같은 로직을 공유한다
- `src/server/middleware/access.ts`
  - 페이지 요청 시점에 접근 가능 여부를 판정하는 Nitro 서버 미들웨어
  - `/api`, `/_nuxt`, `/block`, 정적 자산 경로는 우회시킨다
  - 실패 시 렌더링 전에 `/block`으로 리다이렉트한다
- `src/app/pages/block.vue`
  - 차단 안내 전용 페이지
  - "개발 버전 전용" 및 "커뮤니티 전용 접근" 메시지를 명시
  - 검색 유입 또는 직접 URL 입력 사용자가 보게 될 안전한 종착지 역할을 한다

### 수정 구성 요소
- `src/server/api/users/check.get.ts`
  - 직접 헤더를 읽는 단발 구현 대신 공통 접근 체크 유틸을 재사용한다
- `src/app/utils/api-client.ts`
  - 디스코드 ID를 클라이언트에서 헤더로 주입하지 않도록 유지한다
- `src/app/components/layout/common-layout/CommonHeader.vue`
  - 디스코드 ID 표시 책임을 두지 않고 기존 내비게이션 역할만 유지한다
- `src/app/layouts/main.vue`
  - 디스코드 ID를 화면에 렌더링하지 않는다

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- `GET /api/users/check`
  - 요청 헤더 `X-Discord-ID`와 개발 환경 여부, 유저 존재 여부를 공통 유틸로 판정
  - 결과는 기존처럼 `BaseResponse` 형식을 유지
- 서버 내부 계약
  - 페이지 렌더링 전 접근 제어는 `checkCommunityAccess(event)` 하나로 통일
  - 클라이언트는 디스코드 ID를 별도 상태로 관리하지 않는다

## 로직 흐름
1. 사용자가 브라우저로 페이지 URL에 직접 접근한다.
2. Nitro 서버 미들웨어 `src/server/middleware/access.ts`가 해당 요청의 pathname을 확인한다.
3. 요청 경로가 `/block`, `/api`, `/_nuxt`, 공개 정적 자산이면 접근 검사를 생략한다.
4. 그 외 페이지 요청이면 `checkCommunityAccess(event)`를 호출한다.
5. `checkCommunityAccess`는 `getDiscordId(event)`로 `X-Discord-ID` 헤더를 읽는다.
6. 개발 환경이 아니면 즉시 `TEAM_ONLY` 에러를 반환한다.
7. 헤더가 없으면 `UNAUTHORIZED`, 헤더는 있으나 매칭 유저가 없으면 `NOT_FOUND`를 반환한다.
8. 검사가 실패하면 서버 미들웨어는 렌더링 전에 `/block`으로 리다이렉트한다.
9. 검사가 성공하면 원래 페이지 렌더링을 허용한다.
10. `GET /api/users/check`도 같은 유틸을 사용하므로, 필요 시 후속 API나 디버깅 도구에서 동일 기준으로 접근 상태를 확인할 수 있다.

## 예외 흐름
- `X-Discord-ID` 헤더가 없는 직접 유입 요청은 모두 `/block`으로 이동한다.
- `block` 페이지는 서버 미들웨어 예외 대상이므로 무한 리다이렉트가 발생하지 않는다.
- API 요청은 별도 인증 규칙을 가질 수 있으므로 서버 미들웨어에서는 제외하고, 개별 API는 기존 `authHelper` 또는 `users/check` 기준으로 처리한다.
- 정적 자산 경로를 차단하면 앱 자체가 깨질 수 있으므로 반드시 우회 대상에 포함한다.

## 대안 검토
### 옵션 A: 각 레이아웃에서 `onMounted`로 `users/check` 호출
- 설명: `default.vue`, `main.vue` 진입 시 클라이언트에서 접근 검사 수행
- 장점:
  - 구현이 직관적임
  - 컴포넌트 내부에서 헤더 상태를 바로 다룰 수 있음
- 단점:
  - 렌더링 후 차단되어 화면이 잠깐 노출될 수 있음
  - 레이아웃이 늘어나면 중복 구현이 생김
  - SSR 초기 진입 제어가 약함
- 미채택 이유: "모든 페이지를 걸친 전역 차단" 요구에 비해 제어 시점이 늦고 중복이 생긴다.

### 옵션 B: 각 페이지에서 `useGet('/users/check')` 호출
- 설명: 페이지 단위로 진입 시 체크 API를 붙이는 방식
- 장점:
  - 페이지별 예외 규칙을 두기 쉬움
- 단점:
  - 페이지마다 반복 구현 필요
  - 누락 페이지가 생기기 쉬움
  - 차단 로직과 화면 로직이 섞임
- 미채택 이유: 커뮤니티 전용 웹앱의 입구 통제라는 목적에 비해 관리 포인트가 너무 많다.

### 옵션 C: 전역 라우트 미들웨어 + 공통 상태 저장소
- 설명: 클라이언트 라우트 전환 시점에 검사하고 결과를 store에 반영하는 방식
- 장점:
  - 진입 차단이 가장 빠름
  - Nuxt 앱 내부 구현만으로 완료 가능함
- 단점:
  - 브라우저가 신뢰 가능한 요청 헤더를 자동으로 유지해 주지 않음
  - 디스코드 ID를 프론트 상태/쿠키에 들고 다니면 변조 위험이 커짐
  - 최초 SSR 요청 단계에서 차단하기 어렵다

### 옵션 D: Nitro 서버 미들웨어 + 공통 접근 체크 유틸
- 설명: Nuxt 렌더링 전에 서버가 직접 들어온 요청 헤더를 검사하는 방식
- 장점:
  - 페이지 최초 진입 시점에 바로 차단 가능함
  - 프론트가 디스코드 ID를 상태나 쿠키로 보관하지 않아도 됨
  - 검색 유입, 직접 URL 입력, 새로고침 모두 같은 기준으로 막을 수 있음
  - `users/check`와 같은 검증 로직을 재사용하기 좋음
- 단점:
  - 페이지 요청에 실제로 `X-Discord-ID` 헤더가 실려 들어오는 배포/프록시 전제가 필요함
  - 클라이언트 단독 SPA 내비게이션은 최초 서버 검사를 통과한 세션을 전제한다

### 최종 선택
- 선택 이유: 마스터가 말한 "요청 헤더" 전제를 그대로 활용하면서, 디스코드 ID를 프론트에 노출하지 않고도 최초 페이지 진입을 가장 이른 시점에 막을 수 있기 때문이다.
- 기대 효과:
  - 검색 유입과 직접 URL 접근을 서버 수준에서 차단할 수 있음
  - 프론트 상태나 쿠키 변조에 기대지 않아도 됨
  - 페이지 추가 시 별도 접근 가드 코드를 넣지 않아도 됨

## 제약 및 참고 사항
- 이 구조는 페이지 요청에 `X-Discord-ID`가 실제로 포함되는 프록시/중계 환경을 전제로 한다.
- 완전한 인증 체계는 아니며, 신뢰 가능한 상위 게이트웨이 없이 브라우저 단독 환경만으로는 근본적인 위조 방지가 불가능하다.
- `users/check`는 여전히 `BaseResponse` 오류 응답 형태를 쓰므로, 후속 클라이언트 소비 코드가 생기면 그 계약을 그대로 따라야 한다.
