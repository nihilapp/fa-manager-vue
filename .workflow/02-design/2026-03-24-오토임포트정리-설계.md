# 설계 (Design)

## 요약

- 설계 방향: 자동/전역 타입 선언과 중복되는 `import type`만 제거하고 런타임 import는 유지

## 로직 흐름
1. `nuxt.config.ts`, `manual-app-imports.d.ts`, `manual-server-imports.d.ts`, `.nuxt/imports.d.ts`를 기준으로 auto/global 범위를 확정한다.
2. 해당 범위에 포함되는 심볼을 import하는 파일만 선정한다.
3. 값 import는 유지하고, 타입 import 중 전역으로 대체 가능한 것만 제거한다.
4. `src/server/utils/drizzle.ts`처럼 namespace import가 필요한 파일은 제외한다.
5. ESLint로 수정 파일을 점검해 타입/문법 오류가 없는지 확인한다.

## 대안 검토
### 옵션 A
- 설명: 프로젝트 전체 import를 일괄 제거 후 오류를 보면서 복구
- 장점: 정리 범위를 크게 가져갈 수 있음
- 단점: 현재 워크트리가 이미 많이 더러워져 있고 오탐 복구 비용이 큼
- 미채택 이유: 불필요하게 광범위하고 안전하지 않음

### 최종 선택
- 선택 이유: 설정과 생성 선언 파일에서 근거가 확인된 파일만 선별 수정하면 위험을 최소화할 수 있음
- 기대 효과: 오토임포트 규칙과 실제 코드가 일치하고, 타입 선언 파일의 중복 import가 제거됨

## 제약 및 참고 사항
- 기존 사용자 변경이 많은 워크트리이므로 관련 파일 외 수정 금지
- `@tanstack/vue-query`의 원시 타입처럼 전역 선언 근거가 없는 import는 유지
