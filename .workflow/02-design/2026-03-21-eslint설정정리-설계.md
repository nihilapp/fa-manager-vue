# 설계 (Design)

## 요약
- 변경 대상: `eslint.config.mjs`
- 설계 방향: TypeScript 공통 규칙을 재사용 가능한 묶음으로 분리하고, `.ts`와 `.vue`가 이를 공통 적용받도록 재구성
- 주요 결정 사항: Vue 전용 규칙과 TypeScript 공통 규칙을 분리하고, 파일 유형별 파서를 명확하게 나눈다

## 아키텍처 변화
### 신규 구성 요소
- `sharedTsRules` 또는 이에 준하는 TypeScript 공통 규칙 객체
  - 목적: `.ts`와 `.vue`가 동일한 TypeScript 규칙 세트를 공유하게 함
  - 포함 대상: `@typescript-eslint/no-unused-vars`, `@typescript-eslint/no-shadow`, `@typescript-eslint/no-use-before-define` 등 현재 중복 정의된 TS 규칙

### 수정 구성 요소
- `files: ['**/*.{js,ts}']` 블록
  - 역할을 JS/TS 공통 일반 규칙 중심으로 축소
  - TypeScript 전용 규칙은 직접 선언하지 않고 공통 TS 규칙 묶음으로 분리
- `files: ['**/*.ts']` 블록
  - `parser: tseslint.parser`를 유지
  - TypeScript 공통 규칙을 직접 적용받는 1차 대상 파일군으로 사용
- `files: ['**/*.vue']` 블록
  - 대상 범위를 `.vue`로 한정
  - `parser: vue-eslint-parser`와 `parserOptions.parser = tseslint.parser`를 유지
  - Vue 전용 규칙과 TypeScript 공통 규칙을 함께 적용
- `files: ['**/*.{js,mjs,ts,vue}', ...]` 형태의 광범위한 Vue 블록
  - `.vue` 중심 블록과 일반 JS/TS 블록으로 책임 분리
  - `.ts`가 Vue 파서 블록에 포함되지 않도록 범위를 축소

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 외부 API 변경 없음
- 내부 설정 인터페이스 수준에서만 변경

## 로직 흐름
1. 전역 추천 설정과 공통 언어 옵션을 먼저 로드한다.
2. 일반 JS/TS 공통 규칙 블록은 언어 공통 스타일과 기본 예외 처리만 담당한다.
3. TypeScript 공통 규칙을 별도 객체로 정의한다.
4. `.ts` 블록은 TypeScript 파서를 직접 사용하고 `sharedTsRules`를 적용한다.
5. `.vue` 블록은 `vue-eslint-parser`를 사용하되 내부 스크립트 파서는 TypeScript 파서로 연결한다.
6. `.vue` 블록에 `sharedTsRules`와 Vue 템플릿 전용 규칙을 함께 적용한다.
7. import-x, globals, accessibility 규칙은 필요한 파일군에만 유지하여 기존 동작을 보존한다.

## 대안 검토
### 옵션 A
- 설명: 기존 Vue 블록과 TS 블록을 하나의 거대한 블록으로 병합하고 모든 파일에 `vue-eslint-parser`를 사용
- 장점: 설정 블록 수가 줄어 겉보기에는 단순해 보임
- 단점: `.ts` 파일까지 Vue 파서 경로로 처리되어 설정 책임이 모호해지고, 추후 유지보수 시 파일 유형별 동작 차이를 추적하기 어려움
- 미채택 이유: Vue SFC 문제를 해결하기 위해 일반 TS 파일의 파서 책임까지 희생할 필요가 없음

### 옵션 B
- 설명: 현재 구조를 유지한 채 Vue 블록 안의 TS 규칙만 수동 복제
- 장점: 수정 범위가 작음
- 단점: 중복 규칙이 계속 남고, `.ts`와 `.vue` 간 규칙 드리프트가 다시 발생할 가능성이 큼
- 미채택 이유: 마스터 요청인 "같은 규칙" 보장을 구조적으로 충족하지 못함

### 최종 선택
- 선택 이유: TypeScript 공통 규칙을 재사용 단위로 분리하고 `.ts`와 `.vue`가 이를 함께 사용하면, 파일 유형별 파서는 유지하면서도 규칙 일관성을 확보할 수 있다.
- 기대 효과:
  - `.vue` 내부 TS와 `.ts` 파일 간 규칙 편차 제거
  - ESLint 설정 중복 감소
  - 이후 TypeScript 규칙 변경 시 수정 지점 단일화

## 제약 및 참고 사항
- ESLint 플랫 설정은 배열 순서에 민감하므로, 공통 규칙 블록과 파일별 블록의 선언 순서를 유지보수 가능하게 정리해야 한다.
- Vue 템플릿 규칙은 TypeScript 규칙과 별개 책임으로 유지해야 한다.
- 실제 구현 시 `eslint.config.mjs` 안의 globals 및 import resolver 설정이 어느 블록에 속해야 하는지도 함께 정리해야 한다.
