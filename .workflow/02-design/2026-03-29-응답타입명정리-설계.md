# 설계 (Design)

## 요약
- 변경 대상: 서버 응답 인터페이스/헬퍼, 앱 공용 요청 응답 타입, 앱/서버 수동 오토임포트 선언
- 설계 방향: 프론트와 서버가 동일한 `BaseApiResponse<T>` 인터페이스를 기준으로 응답을 해석하고, 앱 에러 타입은 `BaseApiResponse<null>` 별칭으로만 유지
- 주요 결정 사항: 서버 응답 인터페이스의 `data`는 `T | null`로 두어 성공/실패 응답을 단일 제네릭 타입으로 수용

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/server/types/response.types.ts`
  - `BaseResponseType`를 `BaseApiResponse`로 변경한다.
  - `data` 필드를 `TData | null`로 정의해 에러 응답도 동일 타입으로 수용한다.
- `src/server/utils/base-response.ts`
  - 응답 생성 클래스를 `BaseApiResponse`로 변경한다.
  - `data`, `page`, `error` 메서드 반환 타입을 새 인터페이스 기준으로 맞춘다.
- `src/server/types/manual-server-imports.d.ts`
  - `BaseResponseType`, `BaseResponse` 전역 별칭을 제거하고 `BaseApiResponse` 전역 타입으로 통일한다.
- `src/app/composables/query/useApiRequest.ts`
  - 앱 공용 응답 타입을 서버의 `BaseApiResponse` 타입 별칭으로 재노출한다.
  - `ApiErrorResponse`는 `BaseApiResponse<null>` 별칭으로만 유지한다.
- `src/app/types/manual-app-imports.d.ts`
  - 전역 `BaseApiResponse`, `ApiErrorResponse`가 동일 구조를 가리키도록 선언을 정리한다.
- `src/server/api/**`, `src/server/utils/**`
  - `BaseResponse` 호출과 타입 주석을 `BaseApiResponse` 기준으로 치환한다.

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 런타임 응답 JSON 구조는 유지한다.
- 타입 계약상 명칭만 `BaseApiResponse`로 통일하고, 앱/서버에서 같은 인터페이스를 사용하도록 맞춘다.

## 로직 흐름
1. 서버 응답 인터페이스를 `BaseApiResponse<T>`로 변경하고 응답 헬퍼 클래스도 같은 이름으로 맞춘다.
2. 서버 수동 오토임포트 선언과 실제 서버 호출부를 새 이름으로 전환한다.
3. 앱 공용 요청 훅에서 응답 타입을 서버 타입 별칭으로 재사용하게 바꾸고 `ApiErrorResponse = BaseApiResponse<null>`로 정리한다.
4. 앱 수동 오토임포트 선언을 새 이름 기준으로 갱신한다.
5. 전체 검색과 타입체크로 `src/app`, `src/server` 범위에 구 명칭 잔존 여부를 확인한다.

## 대안 검토
### 옵션 A
- 설명: 앱에서만 `BaseApiResponse` 별칭을 유지하고 서버는 `BaseResponse`/`BaseResponseType`를 그대로 둔다.
- 장점: 서버 수정량이 적다.
- 단점: 프론트와 백엔드가 같은 응답 타입을 쓴다는 요구를 충족하지 못한다.
- 미채택 이유: 마스터 요청이 명칭과 오토임포트까지 포함한 전면 통일이기 때문이다.

### 옵션 B
- 설명: 성공 응답과 에러 응답을 계속 유니온으로 유지한다.
- 장점: 성공 시 `data`의 null 가능성을 좁게 표현할 수 있다.
- 단점: `ApiErrorResponse = BaseApiResponse<null>` 요구와 맞지 않고 공용 타입이 다시 이중 구조가 된다.
- 미채택 이유: 동일 계약 단일화보다 타입 체계가 복잡해진다.

### 최종 선택
- 선택 이유: `BaseApiResponse<T>` 하나를 공통 기반으로 삼고 에러 응답만 `BaseApiResponse<null>` 별칭으로 두면 프론트/백 명칭과 구조를 모두 통일할 수 있다.
- 기대 효과: 검색성, 오토임포트 일관성, 공용 응답 처리의 가독성이 개선된다.

## 제약 및 참고 사항
- `src/app`과 `src/server` 외 문서 문자열은 이번 작업 범위에서 제외한다.
- 서버 응답 헬퍼 파일 경로는 유지하고 export 이름만 바꿔 Nuxt 자동 오토임포트 영향 범위를 최소화한다.
