# 설계 (Design)

## 요약
- 변경 대상: 캐릭터 생성/수정 페이지와 공용 폼 UI 컴포넌트
- 설계 방향: 페이지는 래퍼만 담당하고, 실제 화면 구조와 폼 상태는 `CharacterEditorForm`으로 집중시킨다.
- 주요 결정 사항: 생성/수정 모두 동일한 공용 폼을 사용하고, 수정 페이지는 조회 결과를 폼 기본값에 주입한다.

## 아키텍처 변화
### 신규 구성 요소
- `src/app/components/characters/CharacterEditorForm.vue`
  - 생성/수정 모드에 따라 제목, 버튼 라벨, 기본값, 보조 안내 문구를 바꾸는 공용 UI 컴포넌트다.

  - 라우트 파라미터를 읽고 `CharacterEditorForm`을 수정 모드로 렌더링하는 페이지 래퍼다.

## 로직 흐름
1. 생성 페이지는 `CharacterEditorForm`을 `create` 모드로 렌더링한다.
2. 수정 페이지는 라우트 파라미터에서 `id`를 읽고 `useGetCharacterInfo`를 실행한다.
3. 공용 폼은 내부 `useForm`으로 기본값을 관리하고, 전달받은 `characterId` 및 스토어의 `characterInfo` 변경을 감지해 폼 값을 갱신한다.
4. 화면은 기본 정보, 전투/감지 수치, 능력치, 보유 자산/장비, 메모성 특성 섹션으로 나눠 렌더링한다.
5. 클래스 정보는 수정 페이지에서만 별도 요약 박스로 노출하고, 생성 페이지에서는 안내 문구로 대체한다.
6. 저장 버튼은 현재 실제 API를 호출하지 않고 `submit` 이벤트만 발생시키며, 사용자에게 후속 로직 연결 필요성을 설명한다.

## 대안 검토
### 옵션 A
- 설명: 생성 페이지와 수정 페이지를 각각 독립적인 폼 컴포넌트로 구현
- 장점: 페이지별 특화 UI를 빠르게 붙이기 쉽다.
- 단점: 필드 수가 많아 중복이 커지고 후속 로직 연결 시 수정 범위가 커진다.
- 미채택 이유: 마스터가 로직을 나중에 붙일 예정이라 공용 UI 계약이 있는 편이 유지보수에 유리하다.

### 옵션 B
- 설명: 공용 폼 컴포넌트 하나에 생성/수정 모드를 주입
- 장점: 레이아웃, 필드 정의, 제출 이벤트를 한 곳에서 관리할 수 있다.
- 단점: 모드 분기와 기본값 주입 로직이 약간 복잡해진다.

### 최종 선택
- 선택 이유: 후속 저장 로직 연결과 UI 수정 비용을 최소화할 수 있기 때문이다.
- 기대 효과: 캐릭터 생성/수정 UX가 즉시 통일되고, 로직 추가 시 이벤트 핸들러만 연결하면 된다.

## 제약 및 참고 사항
- 페이지 파일은 렌더링 컴포넌트만 감싸야 하므로 실질적인 템플릿 구조는 공용 컴포넌트에 둔다.
- Tailwind 기본 클래스 제약이 있으므로 현재 프로젝트에서 이미 사용 중인 클래스 패턴만 활용한다.
- 오토임포트 규칙을 유지하기 위해 Nuxt 자동 주입 가능한 대상은 직접 import 하지 않는다.
