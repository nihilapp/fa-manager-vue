# 설계 (Design)

## 요약

- 변경 대상: `useGet`, 플레이어 래퍼 컴포저블
- 설계 방향: composable 본체는 동기 반환, 내부 액션 메서드만 비동기 유지
- 주요 결정 사항: `useGet`의 top-level `async`를 제거하고 래퍼는 단순 return만 유지

## 아키텍처 변화

### 수정 구성 요소

- `src/app/composables/query/useGet.ts`
  - 함수 선언을 동기 함수로 변경한다.
  - 반환 타입을 `UseGetReturn<T>`로 유지한다.
  - `refetch` 내부 `return await execute()`는 `return execute()`로 단순화한다.
- `src/app/composables/players/useGetMyInfo.ts`
  - 별도 `await` 없이 `useGet` 반환 객체를 그대로 반환한다.
- `src/app/composables/players/useGetPlayerList.ts`
  - 별도 `await` 없이 `useGet` 반환 객체를 그대로 반환한다.
- `src/app/composables/players/useCreateNewPlayer.ts`
  - `usePost` 반환 객체를 그대로 반환한다는 점을 유지하고 성공 콜백은 실제 데이터 `T` 기준으로 맞춘다.

### API / 인터페이스 변경

- `useGet<T>()` 반환 타입이 `Promise<UseGetReturn<T>>`에서 `UseGetReturn<T>`로 바뀐다.
- 내부 `execute`, `refresh`, `refetch`는 계속 `Promise<BaseApiResponse<T> | undefined>`를 반환한다.

## 로직 흐름

1. `useGet`의 top-level async를 제거한다.
2. 래퍼 컴포저블이 추가 비동기 없이 내부 반환 객체를 그대로 돌려주게 맞춘다.
3. `await useGet...` 호출부가 남아 있는지 검색 후 필요한 경우 정리한다.
4. 타입체크로 반환 타입 변경 영향 여부를 확인한다.

## 대안 검토

### 옵션 A

- 설명: 래퍼에서 `Promise`를 벗기기 위해 별도 프록시 객체를 만든다.
- 장점: `useGet`를 건드리지 않을 수 있다.
- 단점: 불필요하게 복잡하고 composable 계약이 이중화된다.
- 미채택 이유: 원인은 `useGet`의 불필요한 async 하나다.

### 최종 선택

- 선택 이유: 동기 반환이어야 하는 composable는 top-level async를 제거하는 것이 가장 단순하고 정확하다.
- 기대 효과: 래퍼 사용 시 `await`가 필요 없어지고 타입도 직관적이 된다.

## 제약 및 참고 사항

- `usePost`, `usePut`, `usePatch`, `useDelete`는 이미 동기 반환이므로 구조 변경이 필요 없다.
