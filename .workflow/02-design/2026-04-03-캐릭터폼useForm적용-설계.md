# 설계 (Design)

## 요약
- 변경 대상: `CharacterCreateForm.vue`, `CharacterEditorForm.vue`, `character.store.ts`
- 설계 방향: 폼 상태는 컴포넌트 내부 `useForm`이 관리하고, 스토어는 초기값/응답 매핑/payload 변환 유틸리티만 담당한다.
- 주요 결정 사항: 메인 입력 필드만 `useForm`으로 전환하고, 생성 화면의 클래스 배열 입력은 기존 스토어 구조를 유지한다.

## 아키텍처 변화
### 수정 구성 요소
- `src/app/stores/character.store.ts`
  - 스토어 내부 폼 ref 제거
  - 빈 폼 값 생성 함수와 상세 응답 -> 폼 값 변환 함수 제공
  - payload 생성 함수를 `CharacterFormValues` 인자 기반 순수 함수로 전환
- `src/app/components/characters/CharacterCreateForm.vue`
  - `useForm` 초기값 설정
  - `handleSubmit`, `resetForm`, `isSubmitting` 사용
  - 공용 입력 필드를 `name` 기반으로 연결
- `src/app/components/characters/CharacterEditorForm.vue`
  - `useForm` 초기값 설정
  - 상세 응답 수신 시 `resetForm`으로 값 동기화
  - 제출 시 `handleSubmit`을 통해 수정 요청 payload 생성

### API / 인터페이스 변경
- 외부 API 변경 없음
- `buildCharacterSubmitPayload(formValues)` 형태로 스토어 인터페이스 변경

## 로직 흐름
1. 생성/수정 컴포넌트는 `createEmptyCharacterFormValues()`를 `useForm.initialValues`로 사용한다.
2. 템플릿의 기본 입력 필드는 `name` 속성으로 `Field`와 연결한다.
3. 생성 제출 시 `handleSubmit`이 받은 `formValues`를 `buildCharacterSubmitPayload(formValues)`에 전달하고, 이후 클래스 배열 payload를 순차 생성한다.
4. 수정 화면은 캐릭터 상세 응답을 수신하면 `getCharacterFormValuesFromInfo()` 결과로 `resetForm`을 수행한다.
5. 폼 초기화는 생성 화면에서는 빈 값, 수정 화면에서는 현재 캐릭터 정보 기준 값으로 되돌린다.

## 대안 검토
### 옵션 A
- 설명: 스토어 폼 상태는 유지하고 형식상 `useForm`만 감싼다.
- 장점: 기존 코드 수정량이 적다.
- 단점: 상태 출처가 중복되어 `useForm` 적용 의미가 약하고 동기화 복잡성이 남는다.
- 미채택 이유: 요청 취지에 비해 효과가 약하다.

### 옵션 B
- 설명: 클래스 배열까지 포함해 전부 `useForm`과 `useFieldArray`로 전환한다.
- 장점: 폼 상태가 완전히 일원화된다.
- 단점: 현재 프로젝트에 `useFieldArray` 사용 패턴이 없고 범위가 커진다.
- 미채택 이유: 이번 요청은 두 컴포넌트의 기본 `useForm` 적용이 핵심이며, 클래스 배열까지 확장하면 리스크가 커진다.

### 최종 선택
- 선택 이유: 입력 필드 대부분을 `useForm`으로 전환하면서도 동적 배열 리스크는 피할 수 있다.
- 기대 효과: 공용 폼 컴포넌트의 `name` 계약을 활용해 추후 검증 스키마 추가도 쉬워진다.

## 제약 및 참고 사항
- `manual-app-imports.d.ts`에 선언된 `useForm`은 직접 import 하지 않는다.
- `context7` MCP 서버가 없어 외부 검색 근거는 확보하지 못했다.
