# 설계 (Design)

## 요약
- 변경 대상: `scripts/get-icon-names.ts`, `package.json`, 생성 대상 타입 파일
- 설계 방향: 서버 API 내부에 남아 있는 생성 로직을 루트 실행 스크립트로 재구성하고, 패키지 목록은 `package.json`에서 동적으로 수집한다.
- 주요 결정 사항: 생성 대상 정리는 출력 디렉터리에서 `.types.ts`와 `index.ts`만 삭제하고, 통합 타입 `IconName`에는 기존과 동일하게 `''`를 포함한다.

## 아키텍처 변화
### 신규 구성 요소
- `scripts/get-icon-names.ts`
  - 루트 `package.json`을 읽어 `@iconify-json/*` 패키지명을 수집한다.
  - 각 패키지의 `icons.json`을 읽어 세트별 유니온 타입 파일을 생성한다.
  - 최종적으로 `src/app/types/icons/index.ts`를 생성한다.

### 수정 구성 요소
- `package.json`
  - `icons:types` 실행 명령을 추가해 루트에서 단독 실행 가능하게 한다.

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 런타임 API 계약 변경은 없다.
- 파일 생성 인터페이스만 `pnpm run icons:types`로 추가된다.

## 로직 흐름
1. 스크립트가 자신의 경로를 기준으로 프로젝트 루트를 계산한다.
2. 루트의 `package.json`을 읽어 `dependencies`, `devDependencies`를 병합한다.
3. 병합된 의존성에서 `@iconify-json/` 접두사를 가진 패키지만 추출한다.
4. 출력 디렉터리 `src/app/types/icons`를 생성하고 기존 생성물 `.types.ts`, `index.ts`를 정리한다.
5. 각 패키지의 `icons.json`에서 `prefix`와 `icons` 키를 읽고 정렬된 아이콘 이름 목록을 만든다.
6. `prefix.types.ts`에 `export type XxxIconName = | 'prefix:name' ...` 형식의 유니온 타입을 기록한다.
7. 모든 세트 타입을 import 한 뒤 `IconName` 유니온과 `''`를 포함하는 `index.ts`를 기록한다.
8. 생성 세트 수와 아이콘 총수를 콘솔에 출력하고 종료한다.

## 대안 검토
### 옵션 A
- 설명: 기존 `src/server/api/icons/index.get.ts`를 그대로 두고 API 호출로만 생성한다.
- 장점: 이미 동작하는 로직을 그대로 쓴다.
- 단점: 서버 라우트 실행이 전제되어 단독 실행 스크립트 요구사항을 충족하지 못한다.
- 미채택 이유: 마스터 요청이 루트 스크립트 복구와 단독 실행 경로 제공에 명확히 맞춰져 있다.

### 옵션 B
- 설명: 패키지 목록을 스크립트 내부 상수 배열로 다시 하드코딩한다.
- 장점: 구현이 단순하다.
- 단점: `package.json`과 목록이 분리되어 유지보수 시 누락 가능성이 생긴다.
- 미채택 이유: 마스터 요구사항이 `package.json`을 읽어 처리하는 방식이다.

### 최종 선택
- 선택 이유: `package.json` 기반 동적 수집과 로컬 파일 생성 조합이 요구사항과 유지보수성을 동시에 만족한다.
- 기대 효과: Iconify 패키지 변경 시 스크립트 수정 없이 재생성 가능하고, 실수로 삭제된 타입 파일도 한 명령으로 복구할 수 있다.
