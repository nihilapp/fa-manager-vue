# 설계 (Design)

## 요약
- 변경 대상: `RESPONSE_MESSAGE` 상수와 직접 사용 API
- 설계 방향: 권한 주체와 리소스 주체를 기준으로 메시지 문맥을 분리
- 주요 결정 사항: 중복 키 제거, 캠페인 전용 권한 메시지 추가, 세션 캠페인 불일치 메시지 추가

## 아키텍처 변화
### 신규 구성 요소
- `CAMPAIGN_FORBIDDEN`: 캠페인 소유자 또는 관리자 권한이 필요한 문맥 전용 메시지
- `SESSION_CHARACTER_NOT_IN_CAMPAIGN`: 세션 플레이어 등록 시 캐릭터 캠페인 불일치 전용 메시지

### 수정 구성 요소
- `PLAYER_FORBIDDEN`: 단일 정의만 유지하고 "플레이어 대상 권한 부족" 문맥으로 한정
- 플레이어 수정/삭제 API: 범용 메시지 대신 `UPDATE_PLAYER_FORBIDDEN`, `DELETE_PLAYER_FORBIDDEN` 사용
- 캠페인 수정/삭제/상태변경 및 캠페인-캐릭터 연결 API: `CAMPAIGN_FORBIDDEN` 사용

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 응답 본문의 `message` 문자열만 정리하고 응답 코드/데이터 구조는 유지

## 로직 흐름
1. `response-message.ts`에서 중복 선언과 미사용에 가까운 중복 의미 상수를 식별한다.
2. 동일 상수명 충돌은 제거하고, 실제 권한 주체가 다른 문맥은 새 상수로 분리한다.
3. 각 API 핸들러가 수행하는 권한/상태 검증 의미에 맞춰 메시지 참조를 교체한다.
4. 검색으로 남아 있는 구형 키와 신규 키 사용처를 검증한다.

## 대안 검토
### 옵션 A
- 설명: `PLAYER_FORBIDDEN` 하나로 모든 권한 실패를 유지
- 장점: 수정 범위가 작음
- 단점: 캠페인/세션 불일치 문맥이 계속 왜곡됨
- 미채택 이유: 마스터 요청의 핵심이 "어느 맥락이 좋은지" 판단하는 것이므로 문맥 분리가 필요함

### 최종 선택
- 선택 이유: API가 실제로 어떤 검증에 실패했는지 응답 메시지만 보고도 이해 가능해야 함
- 기대 효과: 중복 키로 인한 덮어쓰기 위험 제거, 유지보수 시 메시지 선택 기준 명확화

## 제약 및 참고 사항
- 기존 워크트리 변경이 많아 최소 범위 패치만 적용한다.
- 타입체크보다는 우선 로컬 검색과 diff 검증으로 변경 정확성을 확인한다.
