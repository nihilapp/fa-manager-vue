# 설계 (Design)

## 요약
- 변경 대상: `src/app/stores`, `src/app/composables/{campaigns,characters,sessions}`, `src/app/composables/query`, `src/app/utils/query-keys.ts`
- 설계 방향: 4개 도메인 스토어를 기준으로 상태 소유권을 고정하고, API 경로별 컴포저블이 공통 정규화 유틸을 통해 응답을 스토어 액션에 반영한다.
- 주요 결정 사항:
  - 상태 저장소는 `campaign`, `character`, `session`, `currency` 4개만 사용한다.
  - 캠페인 관련 하위 리소스는 전부 `campaign.store.ts` 내부 축에서 관리한다.
  - 세션 관련 하위 리소스는 전부 `session.store.ts` 내부 축에서 관리한다.
  - 서버 응답이 단건/목록 형태로 혼재된 경우를 흡수하기 위해 프런트 정규화 유틸을 추가한다.

## 로직 흐름
1. 페이지나 컴포넌트가 도메인 컴포저블의 `execute` 또는 `refetch`를 호출한다.
2. 컴포저블은 `useGet`, `usePost`, `usePut`, `useDelete`를 통해 API를 호출한다.
3. 응답이 오면 `useDomainResource` 유틸이 `response.data`를 단건 또는 목록으로 정규화한다.
4. 조회형 컴포저블은 정규화된 데이터를 해당 도메인 스토어의 목록/상세/하위 리소스 축에 저장한다.
5. 실패 시 컴포저블은 대응하는 `clear*` 액션을 호출해 stale 상태를 제거한다.
6. 변경형 컴포저블은 성공 후 가능한 경우 응답 데이터로 현재 상세 축을 갱신하고, 관련 목록/상세 조회 컴포저블을 `refetch`해 동기화한다.
7. 하위 리소스 변경은 별도 스토어를 만들지 않고 해당 도메인 스토어 내부 축을 재조회해 정합성을 맞춘다.

## 대안 검토
### 옵션 A
- 설명: 하위 리소스별로 `campaign-member.store`, `session-log.store`, `character-class.store` 같은 별도 스토어를 만든다.
- 장점:
  - 축이 단순해 보인다.
  - 각 하위 리소스의 조회/변경 책임이 분리된다.
- 단점:
  - 마스터가 지정한 4개 스토어 기준을 위반한다.
  - 도메인 간 소유권이 흐려지고 참조 지점이 늘어난다.
- 미채택 이유:
  - 현재 작업의 목표는 API 대응보다 먼저 도메인 경계와 상태 소유권을 고정하는 데 있다.

### 옵션 B
- 설명: 서버 API의 잘못된 응답 형태를 먼저 모두 수정한 뒤 프런트 컴포저블을 작성한다.
- 장점:
  - 프런트 코드가 단순해진다.
  - 타입 계약이 깔끔해진다.
- 단점:
  - 이번 작업 범위를 서버 수정까지 불필요하게 확장한다.
  - 기존 API 동작을 전수 검증해야 해 작업량이 급증한다.
- 미채택 이유:
  - 현재 요청은 프런트 API 대응 컴포저블과 스토어 정비가 핵심이다.

### 최종 선택
- 선택 이유:
  - 4개 스토어 기준을 지키면서도 세 도메인 API를 빠르게 감쌀 수 있다.
  - 서버 응답 불안정성을 프런트 정규화 유틸이 흡수해 구현 리스크를 줄인다.
- 기대 효과:
  - 이후 페이지 연결 시 API 경로 단위로 즉시 재사용 가능한 컴포저블 표면이 생긴다.
  - 캠페인/캐릭터/세션/화폐 현황 상태 관리 기준이 흔들리지 않는다.

## 제약 및 참고 사항
- `src/app/types/manual-app-imports.d.ts`, Nuxt 오토임포트, 전역 선언으로 해결되는 타입은 직접 import 하지 않는다.
- 서버의 일부 상세/CUD 라우트는 실제 반환 형태가 일관되지 않을 수 있으므로 응답 정규화 전제를 둔다.
- `currency` 스토어는 이번 작업에서 직접 API 대응 범위가 아니더라도 기준선 고정을 위해 함께 생성한다.
