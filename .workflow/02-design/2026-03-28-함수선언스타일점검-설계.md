## 요약
- 변경 대상: `src/app`, `src/server` 내 명명 함수 선언 40건
- 설계 방향: Vue SFC 내부 명명 함수는 화살표 함수로, 비 Vue 파일의 명명 화살표 함수는 함수 선언식으로 치환한다.
- 주요 결정 사항:
  - 인라인 콜백은 이번 작업 대상에서 제외한다.
  - 파일군을 `Vue 컴포넌트`, `app 일반 TS`, `server 일반 TS`, `server API 내부 지역 함수`로 나눠 순차 적용한다.
  - 기능 변경 없이 선언 스타일만 바꾸고, 이름과 파라미터/반환 타입 계약은 유지한다.

## 아키텍처 변화
### 신규 구성 요소
- 없음

### 수정 구성 요소
- `src/app/components/**/*.vue`
  - `function foo()` 또는 `async function foo()`를 `const foo = () => {}` 또는 `const foo = async () => {}`로 변환한다.
  - 이벤트 핸들러, 렌더링 보조 함수, 페이지네이션 보조 함수 모두 동일 규칙을 적용한다.
- `src/app/**/*.ts`
  - `const foo = () => {}` 또는 `export const foo = () => {}`를 `function foo() {}` 또는 `export function foo() {}`로 변환한다.
  - 제네릭 함수는 `function foo<T>() {}` 형태로 변환하고, 기존 반환 타입 표기는 유지한다.
- `src/server/**/*.ts`
  - 인증 유틸, API 내부 지역 함수, 기타 서버 유틸의 명명 화살표 함수를 함수 선언식으로 변환한다.
  - `defineEventHandler(async () => {})` 같은 프레임워크 진입점 콜백은 정책 대상이 아니라 유지한다.

### DB / 스키마 변경
- 없음

### API / 인터페이스 변경
- 외부 API 계약 변경 없음
- 함수명, 인자, 반환 타입, 호출 방식은 유지한다.
- 오토임포트 대상이나 전역 선언 대상은 새 import를 추가하지 않는다.

## 로직 흐름
1. Vue SFC 파일에서 명명 `function` 선언을 식별한다.
2. 동일 스코프와 동일 이름을 유지한 채 `const 이름 = (...) => {}` 형태로 치환한다.
3. 비 Vue TS 파일에서 명명 `const 이름 = (...) => {}` 선언을 식별한다.
4. `export` 여부와 제네릭/반환 타입을 보존한 채 `function 이름(...) {}` 형태로 치환한다.
5. `auth.ts`, `player.store.ts`, `useGet.ts`처럼 함수 간 참조가 많은 파일은 선언 순서 변경 없이 스타일만 전환한다.
6. 스와핑 후 전체 위반 패턴을 재검색해 누락 여부를 확인한다.
7. 필요 시 타입체크 또는 린트 범위 검증으로 문법/타입 회귀를 확인한다.

## 파일군별 적용 원칙
### 1. Vue 컴포넌트
- 대상 파일:
  - `src/app/components/table/DataTable.vue`
  - `src/app/components/table/Paginator.vue`
  - `src/app/components/players/PlayerListTable.vue`
- 적용 방식:
  - 선언식 함수를 같은 이름의 `const` 화살표 함수로 치환한다.
  - `async function`은 `const fn = async (...) => {}`로 바꾼다.
  - 템플릿에서 참조되는 이름은 유지한다.

### 2. app 일반 TS
- 대상 파일:
  - `src/app/composables/useSetMeta.ts`
  - `src/app/composables/query/useGet.ts`
  - `src/app/composables/query/usePost.ts`
  - `src/app/composables/query/usePut.ts`
  - `src/app/composables/query/usePatch.ts`
  - `src/app/composables/query/useDelete.ts`
  - `src/app/stores/player.store.ts`
- 적용 방식:
  - exported arrow function은 exported function declaration으로 치환한다.
  - 지역 화살표 함수는 지역 함수 선언식으로 바꾼다.
  - 함수가 클로저 상태를 읽는 경우에도 현재 스코프 안에서 동일하게 동작하도록 이름과 본문만 유지한다.

### 3. server 일반 TS 및 API 내부 함수
- 대상 파일:
  - `src/server/utils/auth.ts`
  - `src/server/api/characters/index.get.ts`
  - `src/server/api/characters/mine.get.ts`
  - `src/server/api/characters/[id].get.ts`
- 적용 방식:
  - `auth.ts`의 `hasPermission`, `checkAdmin`, `canAccessSelf`, `requireUser` 등은 `createAuthContext` 내부 지역 함수 선언식으로 전환한다.
  - API 핸들러 내부 합산 함수는 핸들러 내부 지역 함수 선언식으로 바꾼다.
  - 바깥 스코프 값 캡처 방식은 유지한다.

## 대안 검토
### 옵션 A
- 설명: 현재 위반 파일만 수동 스와핑하고, 재발 방지는 별도 규칙 없이 운영한다.
- 장점: 당장 수정 범위가 가장 작다.
- 단점: 이후 동일 패턴이 다시 유입될 가능성이 높다.
- 미채택 이유: 마스터 요청이 "지금부터 이렇게 되어있지 않으면 스와핑"이라는 운영 규칙 성격이라, 최소한 문서상 기준과 탐지 범위를 명확히 남겨야 한다.

### 옵션 B
- 설명: ESLint 규칙 또는 codemod를 먼저 추가한 뒤 전체 코드를 일괄 변환한다.
- 장점: 재발 방지와 대량 수정 효율이 좋다.
- 단점: 이번 요청은 먼저 점검 보고서와 순차 워크플로우 승인이 필요하고, 규칙 설계까지 포함하면 범위가 커진다.
- 미채택 이유: 현재 단계 목적은 기존 위반분 스와핑 설계 확정이며, 도구화는 후속 작업으로 분리하는 편이 안전하다.

### 최종 선택
- 선택 이유:
  - 현재 위반 파일 수가 제한적이어서 순차 수동 스와핑이 가능하다.
  - 함수 선언 스타일만 바꾸는 보수적 변경으로 리스크를 관리할 수 있다.
  - `.workflow` 절차상 설계와 태스크를 명확히 남기기에 적합하다.
- 기대 효과:
  - 마스터가 지정한 함수 선언 규칙을 현재 코드베이스에 반영할 수 있다.
  - 이후 같은 기준으로 재점검 가능한 탐지 패턴이 문서화된다.

## 제약 및 참고 사항
- 현재 세션에서는 `context7 MCP`를 사용할 수 없어 로컬 검색 결과를 설계 근거로 사용했다.
- 인라인 콜백과 프레임워크 진입점 콜백은 이번 정책 대상에서 제외한다.
- 코드 수정 단계에서는 새 import를 추가하지 않고, 오토임포트 규칙을 유지한다.
