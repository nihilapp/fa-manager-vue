# 3단계: 태스크 확립 (Tasking & Checklist)

## 목적

설계 문서를 실제 수행 가능한 작업 단위로 분해하는 단계이다. 이 단계의 핵심은 작업을 최대한 원자적으로 나누고, 각 작업의 완료 조건을 명확히 하여 구현 중 혼선을 줄이는 데 있다. 구현은 반드시 이 체크리스트를 기준으로 순차 진행되어야 한다.

## 파일명 규칙

`YYYY-MM-DD-작업명-태스크확립.md`

## 단계 지침

2단계를 실천하기 위한 최소 단위의 할 일을 나열합니다. 하나가 끝나면 다시 이 단계로 돌아와 다음 항목을 확인하십시오.

## 입력

* Design 문서
* Analysis 문서

## 반드시 수행할 작업

1. 설계 내용을 원자 단위 작업으로 분해한다.
2. 각 작업이 무엇을 완료해야 끝난 것으로 볼 수 있는지 정의한다.
3. 작업 순서와 의존성을 고려해 체크리스트를 구성한다.
4. 구현 중에는 항상 하나의 활성 작업만 선택한다.
5. 작업 완료 시 다음 작업으로 넘어간 사실을 명시한다.

## 작성 내용

### 체크리스트 항목

각 태스크는 가능한 한 단일 책임을 가져야 한다. 하나의 태스크 안에 엔티티 수정, 서비스 수정, 화면 수정, 테스트 작성이 모두 들어가면 안 된다. 태스크는 작고 독립적이어야 하며, 검증 가능한 완료 기준이 있어야 한다.

예시:

* [ ] Task 1: User 엔티티에 role 필드 추가
* [ ] Task 2: AuthService 내 권한 체크 로직 구현
* [ ] Task 3: 프런트엔드 사이드바 메뉴 가시성 처리

## 필수 규칙

각 태스크 완료 후 반드시 다음 문구를 남긴다.
`Task X 완료, 다음 작업으로 이동`

이 규칙은 구현의 흐름을 추적 가능하게 만들고, 세션 중단 후에도 재개 지점을 빠르게 찾기 참조하기 위한 것이다.

## 권장 문서 템플릿

```md
# 체크리스트 (Checklist)

## 요약
- 설계 기반:
- 총 태스크 수:
- 현재 활성 태스크:

## 작업 목록
- [ ] Task 1:
  - 목표:
  - 완료 조건:
  - 의존성:

- [ ] Task 2:
  - 목표:
  - 완료 조건:
  - 의존성:

- [ ] Task 3:
  - 목표:
  - 완료 조건:
  - 의존성:

## 진행 로그
- Task 1 완료, 다음 작업으로 이동
```

## 완료 조건

* 설계 전체가 빠짐없이 작업 단위로 분해되어 있어야 한다.
* 각 태스크의 목적과 완료 조건이 분명해야 한다.
* 작업 순서가 논리적이어야 한다.
* 구현자가 한 번에 하나의 작업만 수행할 수 있을 정도로 명확해야 한다.
