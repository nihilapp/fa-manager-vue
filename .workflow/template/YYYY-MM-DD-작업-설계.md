# 2단계: 설계 (Design)

## 목적

1단계 분석 결과를 바탕으로 실제 구현을 위한 기술적 청사진을 수립하는 단계이다. 이 단계에서는 무엇을 바꾸는지가 아니라, 어떤 구조와 흐름으로 바꿀 것인지 명확히 정의해야 한다. 이후 구현은 이 설계 문서를 기준으로 진행되어야 하며, 설계 밖의 변경은 허용되지 않는다.

## 파일명 규칙

`YYYY-MM-DD-작업명-설계.md`

## 단계 지침

1단계의 분석을 바탕으로 '어떻게' 바꿀 것인지 청사진을 그립니다. 구체적인 인터페이스나 로직의 흐름을 정의하십시오.

## 입력

* Analysis 문서
* PRD 문서
* 기존 아키텍처 및 코드 구조

## 반드시 수행할 작업

1. 필요한 구조 변경 사항을 식별한다.
2. 클래스, 함수, API, DB, 상태 구조의 변경 계획을 정의한다.
3. 데이터와 로직의 흐름을 단계별로 설명한다.
4. 고려했던 대안과 최종 선택 이유를 기록한다.
5. 구현자가 해석 없이 작업할 수 있을 정도로 구체화한다.

## 작성 내용

### 1. 아키텍처 변화

새로 추가되거나 변경될 클래스, 함수, DB 스키마, 인터페이스, API 계약을 설명한다. 무엇이 왜 필요한지까지 함께 적는다.

### 2. 로직 흐름도

데이터가 어떤 입력을 받아 어떤 처리 단계를 거쳐 어떤 결과를 반환하는지를 서술형 혹은 단계형으로 정리한다. 조건 분기, 예외 처리, 검증 순서가 있다면 반드시 포함한다.

### 3. 대안 고려

최선의 방법 외에 검토한 다른 방식들을 적고, 왜 채택하지 않았는지 설명한다. 이는 향후 유지보수자가 의사결정 맥락을 이해하는 데 중요하다.

## 완료 조건

* 구현 대상 구조가 충분히 구체화되어 있어야 한다.
* 데이터와 로직 흐름이 누락 없이 설명되어 있어야 한다.
* 대안 검토와 선택 이유가 남아 있어야 한다.
* Tasking 단계에서 원자 단위 작업으로 분해할 수 있을 정도로 명료해야 한다.
