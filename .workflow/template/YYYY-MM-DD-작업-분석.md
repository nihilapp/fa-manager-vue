# 1단계: 분석 (Analysis)

## 목적

현재 코드베이스와 요구사항 사이의 간극을 파악하고, 해당 요구사항이 기존 구조 안에서 어떤 방식으로 수용될 수 있는지 진단하는 단계이다. 이 단계의 핵심은 단순히 코드를 읽는 것이 아니라, 구현 시 영향을 받는 지점을 식별하고, 충돌 가능성 및 난이도를 조기에 가시화하는 데 있다.

## 파일명 규칙

`YYYY-MM-DD-작업명-분석.md`

## 단계 지침

현재 코드베이스와 요구사항 사이의 간극을 파악합니다. 영향 범위를 최소화하고 충돌 가능성을 진단하십시오.

## 입력

* PRD 문서
* 사용자 요청 사항
* 관련 소스 코드
* 기존 설계 문서 또는 이전 보고서

## 반드시 수행할 작업

1. 요구사항을 기능 단위로 재정리한다.
2. 관련된 파일, 모듈, 클래스, 함수, API, 데이터 구조를 식별한다.
3. 현재 동작 방식과 요구사항 간의 차이를 분석한다.
4. 수정 범위와 연쇄 영향 가능성을 평가한다.
5. 구현 가능 여부와 예상 기술 난제를 명시한다.

## 작성 내용

### 1. 요구사항 요약

사용자가 원하는 기능, 수정사항, 기대 결과를 요약한다. 단순 복붙이 아니라 구현 관점에서 해석된 요구사항이어야 한다.

### 2. 현황 파악

관련 소스 코드 파일 목록과 현재 로직을 정리한다. 파일명만 나열하지 말고, 각 파일이 어떤 역할을 수행하는지와 현재 흐름에서 어느 지점을 담당하는지를 설명한다.

### 3. 영향도 평가

이 작업으로 인해 수정되거나 영향을 받는 기존 모듈을 분석한다. 직접 수정 대상뿐 아니라 간접 영향 가능성이 있는 검증, 인증, API 응답 구조, 프런트 렌더링, DB 연관 로직도 포함할 수 있다.

### 4. 실행 판단

구현 가능 여부를 판단하고, 예상되는 기술적 난제나 구조적 제약을 적는다. 모호하거나 확신할 수 없는 부분은 불확실하다고 명시해야 한다.

## 완료 조건

* 요구사항이 구현 관점으로 재정리되어 있어야 한다.
* 관련 파일과 현재 로직이 충분히 파악되어 있어야 한다.
* 영향 범위와 기술적 난제가 문서화되어 있어야 한다.
* 다음 단계인 설계가 가능할 수준의 분석 깊이를 갖추어야 한다.
