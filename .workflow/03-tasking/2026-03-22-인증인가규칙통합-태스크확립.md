- [x] Task 1: `authHelper` 인증 컨텍스트와 개발 우회 정책 고정
  - 목표: `src/server/utils/auth.ts`를 운영/개발 공통 진입점으로 재구성하고 `AuthContext` 성격의 반환 계약을 확정한다.
  - 완료 조건: 운영 환경의 `X-Discord-ID` 필수 규칙, 관리자 우회 규칙, 개발 환경의 `isDevelopmentBypass` 및 `hasPermission` 정책, fallback user 전략이 `authHelper` 내부 계약으로 코드에 반영되어 있어야 한다.
  - 의존성: `2026-03-22-인증인가규칙통합-설계.md`

- [x] Task 2: 페이지 접근 규칙을 `authHelper` 기반으로 통합
  - 목표: `src/server/utils/access.ts`, `src/server/middleware/access.ts`, `src/server/api/users/check.get.ts`가 독립 규칙 없이 `authHelper` 결과를 재사용하도록 정리한다.
  - 완료 조건: 페이지 접근 유틸이 자체 인증 규칙을 가지지 않고 `authHelper` 또는 그 하위 공통 함수만 사용하며, 서버 미들웨어와 `users/check`가 동일한 접근 판단 결과를 소비해야 한다.
  - 의존성: Task 1 완료

- [x] Task 3: 예외 엔드포인트의 중복 인증 분기 제거
  - 목표: `src/server/api/sessions/[id].delete.ts` 등 `authHelper` 외부에서 직접 헤더를 재검사하는 API를 공통 인증 컨텍스트 기준으로 통일한다.
  - 완료 조건: 대상 API 파일에서 `getDiscordId` 직접 호출 또는 중복 인증 분기가 제거되고, 인증/인가 판단이 `authHelper` 결과만으로 수행되어야 한다.
  - 의존성: Task 1 완료

- [x] Task 4: CUD API의 개발 우회 호환성 점검 및 보강
  - 목표: `user!.id`, `creatorId`, `updaterId`를 사용하는 변경성 API가 개발 환경 우회 모드에서도 안전하게 동작하도록 정리한다.
  - 완료 조건: 설계에서 식별한 주요 CUD API가 개발 우회 모드에서 null 참조 없이 동작하도록 보완되며, 필요한 경우 `requireUser()` 같은 보조 검증 규칙이 함께 반영되어 있어야 한다.
  - 의존성: Task 1 완료, Task 3 완료

- [x] Task 5: 운영/개발 검증 시나리오 정리 및 검증 준비
  - 목표: 통합된 인증/인가 규칙이 설계와 일치하는지 확인할 수 있도록 최소 검증 시나리오와 문서 근거를 정리한다.
  - 완료 조건: 운영 환경의 헤더 필수, 유저 미존재, 일반 유저 소유권 제한, 관리자 우회, 개발 환경 우회 동작에 대한 검증 기준이 정리되고 5단계 검증 문서 작성에 바로 사용할 수 있어야 한다.
  - 의존성: Task 2 완료, Task 3 완료, Task 4 완료
